### 3.5 关键技术决策

#### 决策1: 网关架构选型

**方案A: 单体网关（推荐）**
- 技术栈: FastAPI + WebSocket + Redis
- 优点: 简单、快速开发、易维护
- 缺点: 单点故障

**方案B: 分布式网关**
- 技术栈: Kong + Redis Cluster
- 优点: 高可用、可扩展
- 缺点: 复杂、开发周期长

**决策**: 专业版v1使用方案A（单体网关），定制版再考虑方案B

#### 决策2: AI调用策略

**方案A: 网关统一代理（推荐）**
- 网关持有LLM API Key
- 用户无需配置
- 网关记录用量并计费

**方案B: 用户自带Key + 网关记录**
- 用户配置LLM API Key
- 网关仅记录用量
- 网关不计费

**决策**: 专业版使用方案A（统一代理），这是专业版的核心价值

#### 决策3: 数据存储策略

**专业版继续使用SQLite**：
- 用户数据存储在本地
- 网关不存储用户业务数据
- 网关仅存储：用户账户、连接状态、AI用量

**优点**：
- 隐私保护（数据不离开用户设备）
- 简化架构（无需数据库迁移）
- 快速开发（复用基础版代码）

### 5.3 AI成本估算

假设每个专业版用户：
- 每天10次AI调用
- 每次调用平均1000 tokens
- DeepSeek价格：¥0.001/1K tokens

每用户每月AI成本：
```
10次/天 × 30天 × 1000 tokens × ¥0.001/1K = ¥0.3/月
```

定价策略：
- 专业版：¥29/月 Early Bird / ¥49/月正常价
- AI成本：¥0.3/月/用户
- 基础设施成本：¥510/月（固定）
- 盈亏平衡点：510 ÷ (29 - 0.3) ≈ **18个付费用户**（Early Bird价）

## ⚠️ 六、风险评估与缓解

### 6.1 技术风险

| 风险 | 级别 | 概率 | 影响 | 缓解措施 |
|------|------|------|------|---------|
| WebSocket稳定性问题 | 高 | 60% | 用户体验差 | 实现心跳机制 + 自动重连 + 降级到HTTP轮询 |
| 微信小程序审核不通过 | 中 | 30% | 延期发布 | 提前准备资质材料 + 多次小版本迭代 |
| AI成本超预算 | 中 | 40% | 亏损运营 | 实施用量限制 + Token计数 + 动态调价 |
| 网关单点故障 | 中 | 20% | 服务中断 | 监控告警 + 快速重启 + 定制版升级为高可用 |

### 6.2 业务风险

| 风险 | 级别 | 缓解措施 |
|------|------|---------|
| 用户从基础版升级意愿低 | 高 | 专业版提供差异化价值（AI代理 + 小程序体验） |
| 竞品冲击 | 中 | 强化"关系经营"定位 + 快速迭代 |
| 付费转化率低 | 中 | 免费试用期 + 转化漏斗优化 |

### 6.3 合规风险

| 风险 | 缓解措施 |
|------|---------|
| 数据隐私合规 | 隐私声明 + 用户授权 + 数据不上传云端 |
| 微信小程序政策 | 遵守小程序规范 + 不涉及敏感领域 |
| 支付资质 | 接入微信支付 + 申请相关资质 |

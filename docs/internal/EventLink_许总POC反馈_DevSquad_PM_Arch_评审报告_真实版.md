| 维度 | 李总"暂停匹配引擎"(F-05) | 许总"主题互通"(F-04) | 关系 |

- **F-05(暂停)**: "系统主动说:你们俩可以谈XX合作"


### 1. 日视图技术实现方案和改动量

**技术现状分析：**
- ✅ 数据层已支持：`events.timestamp` 字段存在，可按日期分组
- ✅ 查询能力：可直接用 `DATE(timestamp)` 分组查询
- ❌ 视图层缺失：当前无日历视图UI组件

**实现方案：**

```
Frontend:
├── DayView组件
│   ├── 时间轴布局（Timeline）
│   ├── 事件卡片（按timestamp排序）
│   └── 主题标签聚合展示
└── API调用: GET /events?date=YYYY-MM-DD&group_by=theme

Backend:
├── EventController.getEventsByDay()
│   ├── 按date过滤
│   ├── 按timestamp排序
│   └── 返回带theme的events数组
└── 无需新表，仅新增查询接口
```

**改动量评估：**
- **极小改动**（2-3天）
  - 后端：1个新API端点（50行代码）
  - 前端：1个日视图组件（150行代码）
  - 测试：单元测试 + UI测试
- **无架构冲击**：复用现有数据模型

### 2. "主题互通"当前技术实现程度和差距

**当前能力矩阵：**

| 能力 | 技术实现 | 覆盖度 |
|------|---------|--------|
| 主题提取 | F-04关联发现引擎 | ✅ 100% |
| 关联类型 | 8种关联类型（associations表） | ✅ 100% |
| 图谱存储 | entities + associations | ✅ 100% |
| 图谱查询 | 递归关联查询（SQL WITH RECURSIVE） | ✅ 80% |
| 可视化 | 缺图谱前端渲染 | ❌ 0% |
| 无限扩展 | 数据模型支持，缺UI交互 | ⚠️ 50% |

**差距分析：**
1. **核心能力已具备**：associations表的`relation_type`支持无限扩展关联
2. **缺失表达层**：需"知识图谱可视化"组件（如D3.js/Cytoscape.js）
3. **查询性能未优化**：深度递归查询需索引优化

**技术实现方案：**

```
Graph Visualization Layer:
├── GraphView组件
│   ├── 节点：entities
│   ├── 边：associations（relation_type标注）
│   ├── 交互：点击节点展开相邻节点
│   └── 布局算法：力导向布局（Force-directed）
└── API: GET /graph?entity_id={id}&depth={1-3}

Performance优化:
├── associations表索引：
│   CREATE INDEX idx_assoc_source ON associations(source_entity, relation_type);
│   CREATE INDEX idx_assoc_target ON associations(target_entity, relation_type);
└── 缓存层：Redis缓存高频图谱查询（TTL 1h）
```

**改动量：**中等（5-7天）

### 3. 语音交互(TTS+ASR)技术准备度

**当前技术栈评估：**

| 组件 | 方案 | 准备度 | 风险点 |
|------|------|--------|--------|
| ASR | 微信同声传译插件 | ✅ 现成 | 依赖微信生态 |
| TTS | TTSScriptComposer | ✅ 已设计 | 未实现 |
| 语音UI | 缺失 | ❌ 无 | 需全新设计 |
| 对话管理 | 缺失 | ❌ 无 | 需状态机 |

**技术准备度评分：60%**

**风险点：**
1. **微信插件依赖**：非自主可控，API变更风险
2. **对话状态管理**：多轮对话上下文保持需状态机（当前无）
3. **语音唤醒逻辑**：缺"语音模式"切换机制

**推荐方案：**

```
Phase 1（保守方案）：
├── 输入：微信同声传译插件（ASR）
├── 处理：当前文本流（无需改动）
├── 输出：TTS纯文本播报（系统级TTS，无定制）
└── 改动量：极小（1-2天）

Phase 2（完整方案）：
├── ASR: 云服务（阿里云/腾讯云）
├── TTS: 定制声音（Azure/Elevenlabs）
├── 对话管理：状态机（idle/listening/processing/speaking）
└── 改动量：大（15-20天）
```

**Architect建议：维持Phase 1规划正确**  
理由：语音是体验增强，非核心功能。PoC已验证价值，Phase 1快速上线文本+简单TTS即可。

### 4. CarryMem"终身"能力技术现状

**CarryMem技术架构审视：**

```
Memory类型覆盖：
├── 短期记忆：事件流（events表）
├── 中期记忆：关系简报（relationship_briefs表）
├── 长期记忆：实体画像（entities.properties JSONB）
├── 偏好记忆：用户偏好（CarryMem protocol）
├── 规则记忆：行为模式（CarryMem protocol）
└── 身份记忆：角色定位（CarryMem protocol）
```

**"终身"能力评估：**

| 能力维度 | 技术支撑 | 当前状态 | 差距 |
|---------|---------|---------|------|
| 时间跨度 | 无数据保留期限设计 | ⚠️ 未定义 | 需数据归档策略 |
| 知识累积 | entities/associations持续增长 | ✅ 支持 | 无 |
| 记忆召回 | 全文检索（PostgreSQL FTS） | ✅ 基础支持 | 需语义检索 |
| 遗忘机制 | 无 | ❌ 缺失 | 需重要性衰减算法 |
| 跨平台同步 | 依赖CarryMem协议 | ✅ 协议支持 | 需实现验证 |

**关键技术缺口：**

1. **数据生命周期管理**：
```sql
-- 需要新增字段
ALTER TABLE events ADD COLUMN importance_score FLOAT DEFAULT 1.0;
ALTER TABLE events ADD COLUMN decay_timestamp TIMESTAMP;

-- 定期衰减任务（Cron job）
UPDATE events 
SET importance_score = importance_score * 0.9 
WHERE decay_timestamp < NOW() - INTERVAL '30 days';
```

2. **语义检索能力**：
   - 当前：PostgreSQL全文检索（关键词匹配）
   - 目标：向量检索（语义相似度）
   - 需集成：pgvector扩展 + Embedding模型

**Architect结论：**  
✅ 基础架构支持"终身"  
⚠️ 需补充数据治理策略（Phase 2）

## 1. 产品概述

### 1.1 产品愿景

PromiseLink 是 IAMHERE 数字名片与 CarryMem AI 关联发现引擎的融合产品，自动发现商务社交中隐藏的人脉关联和商业线索。产品定位不是"帮助记忆"，而是"帮助发现被忽略的机会"，并在发现后生成可执行的行动建议——从"知道"到"做到"。长期愿景是成为用户的"随身工作学习助理"和"智慧大脑"，帮助用户串联人脉、发现商机、沉淀知识。

### 1.2 目标用户

| 用户画像 | 核心需求 | 使用频率 | 价值感知 |
|---------|---------|---------|---------|
| 商务BD/销售 | 拜访客户前了解关联背景 | 高频（每日） | 避免信息盲区，提高拜访效率 |
| 投资人/FA | 发现项目间隐藏关联 | 中频（每周3-5次） | 发现投资组合交叉机会 |
| 创业者 | 了解合作伙伴/竞对关系 | 中频（每周2-3次） | 规避商业风险，发现合作可能 |

### 1.3 核心价值主张

| 维度 | 传统方案 | PromiseLink |
|------|---------|-----------|
| 信息获取 | 被动查询CRM/通讯录 | 主动推送隐藏关联 |
| 关联发现 | 依赖人工记忆和经验 | AI自动发现跨事件关联 |
| 风险预警 | 事后才知道竞对关系 | 实时竞对关系预警 |
| 数据输入 | 手动录入CRM | 名片/录音/会议自动采集 |
| 行动建议 | 发现后自行判断下一步 | 根据事件类型自动生成行动建议和任务分解 |

- 应用层负责前端展示和用户操作，无需服务器部署、无需AI模型维护、无需后端团队
- 引擎层负责后端服务和核心算法，提供标准HTTP API（含SDK和示例代码）
- 文档采用第三人称客观语言，不使用"你做/我做"等指派口吻

## 2. 用户故事

### 2.1 机会发现场景（⚪ Opportunity）

**US-01: 商机匹配发现**

> 作为商务BD，当扫描一张数字名片时，系统自动发现该名片持有人与系统中另一联系人存在商机匹配（技术供需/业务互补/资源对接，匹配度≥0.80），并在5秒内推送⚪商机Todo+🟢破冰行动，以便及时建立连接或牵线搭桥。

**验收标准**: 商机匹配度≥0.80时，系统在事件上报后10秒内生成⚪机会Todo，包含双方姓名、匹配维度、匹配分数和1-3条🟢破冰行动（如共同话题、引荐话术、切入角度）。

**US-02: 跨事件合作线索发现**

> 作为投资人，当两次不同会议中出现同一技术领域的关键人物时，系统自动发现两人存在技术重叠关联，推送潜在合作机会Todo，以便评估联合投资可能。

**验收标准**: 当两个person实体在同一technology实体下共现且tech_overlap关联强度>0.5时，系统生成⚪机会Todo。

**US-03: 供应链关联发现**

> 作为创业者，当系统发现新接触的供应商与现有客户存在供应链上下游关系时，推送供应链关联Todo，以便优化采购或销售策略。

**验收标准**: 当两个organization实体间建立supply_chain关联且strength>0.4时，系统生成⚪机会Todo。

### 2.2 风险预警场景（🔴 Risk）— Phase 2+ 实现

> **优先级说明**：Phase 1数据源有限（无爬取、无付费API），竞对关系数据极度稀疏，硬做风险预警易产生大量误报。Phase 1仅建立competitor/risk_link关联类型的数据结构，不主动推送🔴风险Todo。Phase 2引入竞对数据管理(F-08)后启用。

**US-04: 竞对关系预警**（Phase 2）

> 作为商务人士，当扫描一张名片时，系统发现该名片所属公司与系统中已有联系人所在公司存在直接竞争关系，立即推送🔴风险Todo，以便在沟通中注意信息边界。

**验收标准**: 当两个organization实体间建立competitor关联且strength>0.5时，系统在5秒内生成🔴高风险Todo，包含竞对依据和沟通建议。

**US-05: 风险关联链预警**（Phase 2）

> 作为商务人士，当系统发现新接触的人与已知风险实体存在风险关联链（如竞对公司的前员工），推送🔴风险Todo，以便评估合作风险。

**验收标准**: 当person实体与已有risk_link关联的实体建立新关联时，系统生成🔴风险Todo。

### 2.3 背景补充场景（🔵 Context）

**US-06: 人脉关系背景补充**

> 作为商务人士，当查看今日待办时，系统推送🔵背景Todo，告知即将会面的联系人与自己存在人脉关联（校友/前同事/同城/同行业圈/投资偏好匹配等），以便在沟通中建立亲近感或针对性准备。

**验收标准**: 当两个person实体间建立alumni/ex_colleague/同城等关联，或投资偏好与项目领域匹配度>0.6时，系统生成🔵背景Todo，包含关系类型、关系依据和破冰话题建议。

### 2.4 实体归一场景

**US-08: 实体归一人工确认**

> 作为系统管理员，当实体归一引擎产生置信度在[0.70, 0.85)区间的归一建议时，系统在管理界面展示待确认队列，包含完整属性对比、AI归一推理、置信度分数和历史相似案例，管理员可执行"确认归一"/"拒绝归一"/"稍后处理"三种操作，确认结果反馈至模型用于持续改进。

**验收标准**: 待确认队列展示所有[0.70, 0.85)区间的归一建议，每条建议包含属性对比表、AI推理说明、置信度数值；操作后系统在200ms内响应。

**US-09: 实体归一撤回**

> 作为系统管理员，当发现已自动合并的实体（置信度≥0.85）存在误合并时，可在管理界面一键撤回，系统自动拆分实体并恢复原始状态，同时记录操作日志。

**验收标准**: 撤回操作后实体恢复为合并前的独立状态，关联关系正确还原，操作日志完整记录撤回时间、操作人和原始合并信息。

### 2.5 数据录入场景

**US-10: 名片扫描事件上报**

> 作为IAMHERE小程序用户，当扫描或查看一张数字名片时，小程序自动将名片数据上报至PromiseLink API，系统在200ms内返回确认，后台异步完成实体抽取和关联发现。

**验收标准**: API在200ms内返回event_id和processing状态；后台处理完成后，相关Todo在10秒内可查询。

### 2.6 行动建议场景

**US-11: 会议行动项自动分解**

> 作为商务BD，当上传一场客户会议纪要后，系统不仅推送关联Todo，还自动从会议内容中提取行动项（任务、负责人、截止时间），生成结构化的待办清单，以便会后立即执行跟进。

**验收标准**: meeting事件处理后，输出包含actions列表，每条action包含task/who/when/priority四要素；行动项提取准确率≥75%（基于20场会议人工评估）。

## 3. 功能需求清单

### 3.1 P0 功能（Phase 1 必须）

#### F-01: 事件上报接口

| 属性 | 说明 |
|------|------|
| 功能描述 | 接收多来源事件数据（名片JSON、音频转写文本、会议纪要），标准化为统一Event Schema，并根据event_type路由至对应处理管线 |
| API | `POST /api/v1/events` |
| 输入 | event_type, source, title, timestamp, raw_text, metadata |
| 输出 | event_id, status(processing), pipeline(路由管线名), estimated_completion |
| 事件语义路由 | card_save→轻量管线(秒级响应); meeting→深度管线(分钟级); call→要点管线(分钟级); manual→补全管线(异步) |
| 验收标准 | ① API响应时间P95<200ms ② 名片JSON解析准确率>95%（基于20张样本名片） ③ 支持card_save/meeting/call/manual四种event_type ④ 重复事件（相同source+timestamp+title）自动去重，去重率100% ⑤ 路由准确率100%（event_type与管线一一对应） ⑥ 返回结果包含pipeline字段标识路由目标 |

#### F-02: 实体抽取服务（管线化）

| 属性 | 说明 |
|------|------|
| 功能描述 | 根据事件语义路由结果，走不同深度的抽取管线 |
| 主路径 | LLM抽取（Moka AI Claude Sonnet / OpenAI） |
| 降级路径 | spaCy + 自定义词典规则引擎 |
| 管线定义 | **轻量管线**(card_save): 仅抽取person/organization/technology三类实体，延迟P95<1s; **深度管线**(meeting): 抽取全部5类实体+话题标签(见§5.5)+决策项+行动项，延迟P95<5s; **要点管线**(call): 抽取person/organization+承诺项+待确认项，延迟P95<3s; **补全管线**(manual): 仅抽取缺失实体+属性丰富，延迟P95<2s |
| 验收标准 | ① LLM抽取准确率≥90%（基于100条人工标注样本） ② 降级路径抽取准确率≥75% ③ 轻量管线延迟P95<1s ④ 深度管线延迟P95<5s ⑤ 抽取结果包含实体类型、名称、属性字典 ⑥ 深度管线额外输出话题标签、决策项、行动项 |

#### F-03: 实体归一引擎

| 属性 | 说明 |
|------|------|
| 功能描述 | 5步级联归一算法，判断新实体是否与已有实体为同一对象 |
| 算法步骤 | Step 1: 精确匹配（confidence=1.0）→ Step 2: 别名扩展查找 → Step 3: 上下文相似度计算（公司+0.3, 技能+0.2, 共现+0.3）→ Step 4: 置信度分级处理 → Step 5: Human-in-the-Loop确认 |
| 置信度分级 | ≥0.85: 自动合并（记录日志，支持撤回）；[0.70, 0.85): 进入待确认队列；<0.70: 创建新实体 |
| 撤回机制 | 所有自动合并支持一键撤回，撤回后实体恢复独立状态，关联关系还原，操作日志完整记录 |
| 人工确认 | 展示完整属性对比 + AI归一推理 + 置信度分数 + 历史相似案例；操作项：确认归一/拒绝归一/稍后处理；确认结果反馈至模型持续改进 |
| 验收标准 | ① 精确匹配准确率100% ② 别名扩展召回率≥80%（基于50组别名对测试） ③ 上下文相似度计算延迟<500ms ④ 自动合并误合并率<5%（人工抽检100条） ⑤ 撤回操作成功率100%，数据一致性无损 ⑥ 待确认队列展示完整归一依据，操作响应<200ms |

#### F-04: 关联发现引擎

| 属性 | 说明 |
|------|------|
| 功能描述 | 基于共现分析和类型推断发现实体间8种关联关系 |
| 关联类型 | alumni, ex_colleague, same_city, competitor, tech_overlap, deal_link, risk_link, supply_chain |
| 算法步骤 | A. 共现频率分析 → B. 类型推断（同公司→ex_colleague, 同城市→same_city, 技能重叠→tech_overlap, 竞对DB匹配→competitor）→ C. 时间衰减+过滤（strength × exp(-λ × Δt), λ=0.01, 保留>0.3） |
| 验收标准 | ① Precision@5>70% ② Recall@10>60% ③ F1>0.65 ④ 单次关联发现延迟<2s ⑤ 8种关联类型全部可生成 ⑥ 关联强度计算误差<0.05（与人工标注对比） |

#### F-05: 商机匹配度计算

| 属性 | 说明 |
|------|------|
| 功能描述 | 五维打分法计算供需双方商机匹配度（含话题语义相似度维度），匹配维度不限于技术，也包括业务互补、资源对接等 |
| 维度 | 需求-能力关键词重叠(Jaccard相似度): +0.30; 行业领域一致性（见§5.4领域分类体系）: +0.25; 话题语义相似度（见§5.5话题标签体系）: +0.20; LLM语义理解: +0.15; 历史合作信号: +0.05 |
| 阈值 | ≥0.80: 强匹配（推送⚪机会Todo）; [0.60, 0.80): 潜在机会（记录不推送）; <0.60: 不推送 |
| 验收标准 | ① 匹配度计算延迟<1s ② 强匹配推送准确率≥80%（人工评估50条） ③ Jaccard相似度计算与手工计算偏差<0.02 ④ 总分示例：0.30+0.25+0.20+0.15+0.05=0.95(95%) ⑤ 行业领域一致性评分规则：L1相同=0.25，L1相邻=0.15，L1不相关=0 ⑥ 话题语义相似度计算：余弦相似度≥0.7计满分0.20，[0.4,0.7)按比例折算，<0.4计0 ⑦ 匹配维度覆盖技术供需/业务互补/资源对接三类场景 |

#### F-06: Todo生成与追踪引擎（原Action路由引擎/提醒生成引擎）

| 属性 | 说明 |
|------|------|
| 功能描述 | 基于关联发现结果和事件类型，生成差异化Todo项（信息型+行动型），支持状态追踪和完成度统计 |
| Todo类型 | ⚪ Opportunity（商机发现，match_score≥0.80=强匹配）; 🔴 Risk（竞对关系预警，Phase 2+启用）; 🔵 Context（人脉关系背景补充）; 🟢 Action（行动项，如破冰/跟进/引荐） |
| Todo性质 | **信息型**(⚪🔴🔵): 需要用户知悉，无明确执行动作; **行动型**(🟢): 需要用户执行，有明确task/who/when |
| 状态流转 | pending → in_progress → done / dismissed; 信息型可直接 done/dismissed; 行动型需经 in_progress |
| Action路由 | **card_save**: ⚪🔵信息型Todo+🟢破冰行动; **meeting**: ⚪🔵信息型Todo+🟢行动项分解(按会议类型A/B/C/D)+待确认清单; **call**: ⚪🔵信息型Todo+🟢跟进行动+承诺追踪; **manual**: ⚪🔵增量Todo+合并建议 |
| API | `GET /api/v1/todos?status=pending&limit=20`; `PATCH /api/v1/todos/{id}` (更新状态); `GET /api/v1/todos/stats` (完成度统计); `POST /api/v1/todos/{id}/feedback` |
| 输出Schema | card_save: `{todo(信息型), actions[🟢]}`; meeting: `{todos[](信息型), actions[🟢{task,who,when,priority}], decisions[], pending_decisions[]}`; call: `{todos[](信息型), commitments[], followups[🟢]}`; manual: `{todos_delta[](信息型), merge_suggestions[]}` |

### 5.3 核心数据模型

#### Event（事件）

| 字段 | 类型 | 必填 | 说明 |

| title | string | 是 | 人类可读标题，≤200字符 |
| timestamp | ISO 8601 | 是 | 事件发生时间 |
| raw_text | text | 是 | 原始内容（JSON字符串/转写文本） |
| entities | Entity[] | 否 | 抽取出的实体列表（引擎填充） |
| metadata | JSONB | 否 | 扩展字段（来源特定） |

| 字段 | 类型 | 必填 | 说明 |

| source_events | UUID[] | 是 | 来源事件ID列表 |
| created_at | ISO 8601 | 是 | 创建时间 |
| updated_at | ISO 8601 | 是 | 更新时间 |

#### Association（关联）

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|

### 5.4 领域分类体系设计（Domain Taxonomy）

> **设计原则（来自许总反馈）**：大厂有成熟的行业分类体系，PromiseLink不需要照搬，但需要了解后选择自己的方向。本节定义PromiseLink自有的轻量级领域分类策略。

#### 5.4.1 分类策略：方案D — 一级预定义 + 二级动态扩展

| 层级 | 定义方式 | 数量 | 用途 | 维护方式 |
|------|---------|------|------|---------|
| **L1 一级领域** | 硬编码预定义 | 18±3个 | F-05技术匹配度"行业一致性"维度、F-08竞对判断的行业归属 | 配置文件，版本管理 |
| **L2 二级标签** | LLM动态生成 + 用户反馈收敛 | 不设上限 | 实体属性丰富化、精准匹配 | Provisional标记，人工确认后固化 |

**选择理由**：
- L1硬编码确保核心功能的可验证性和可复现性
- L2动态扩展避免过早锁定细分方向，允许从用户数据中学习
- 完美平衡"知道大厂分类→选自己方向→不完全照搬"的战略定位

### 5.5 话题标签体系设计（Topic Tag Taxonomy）

> **设计背景**：许总分享的分类截图更偏向文本话题维度（非纯行业归类）。PromiseLink需要在行业分类（§5.4）之外，建立话题标签体系，用于提升匹配精度、丰富实体属性、以及会议类型分类。话题标签通过不同管线融入处理流程——轻量管线(card_save)不提取话题，深度管线(meeting)和要点管线(call)提取话题标签。

#### 5.5.1 话题标签与行业分类的分工

| 维度 | 行业分类(§5.4 L1/L2) | 话题标签(§5.5) |
|------|---------------------|---------------|
| **分类对象** | 组织实体（公司/机构） | 文本内容（需求描述/技能表述/会议议题） |
| **解决什么问题** | 这家公司跟谁竞争？属于哪个赛道？ | 这个需求在说什么？能跟谁的能力配？这场会议属于什么类型？ |
| **数据来源** | 公司属性、公开信息 | 事件raw_text中的LLM抽取 |
| **使用管线** | 所有管线 | 仅meeting管线和call管线 |
| **参与匹配度** | F-05行业一致性维度(+0.25) | F-05话题语义相似度维度(+0.20) |
| **会议类型分类** | 不参与 | meeting管线通过话题标签判定会议类型(A/B/C/D) |

**核心洞察**：同一行业的两家公司可能关注完全不同的话题；不同行业的公司可能关注同一话题。行业分类解决组织层面问题，话题标签解决内容层面问题。话题标签还承担会议类型分类的职责——将许总分享的各类文本分类归纳到会议的四大类型及子类。

#### 5.5.2 话题标签生成规则

```
话题标签来源（仅meeting和call管线触发）：
├── LLM从raw_text中提取（标记为provisional）
│   ├── meeting管线：提取需求话题、技能话题、行业热词
│   └── call管线：提取关键话题、痛点话题
├── 用户手动标注（优先级最高，直接confirmed）
└── 反馈闭环收敛（用户确认后provisional→confirmed）

话题标签与实体的关联：
├── person实体 → skills话题标签（如"边缘计算""React"）
├── organization实体 → 业务话题标签（如"工厂智能化改造""私域运营"）
├── technology实体 → 技术话题标签（如"大模型推理""IoT网关"）
└── project实体 → 需求话题标签（如"产线数字化""供应链优化"）
```

#### 5.5.3 话题标签在管线中的使用

| 管线 | 话题标签使用方式 | 输出 |
|------|----------------|------|
| **card_save(轻量)** | 不提取话题标签，仅使用已有confirmed标签参与匹配 | 匹配结果沿用已有话题维度 |
| **meeting(深度)** | 从会议纪要中提取话题标签→丰富实体属性→参与F-05话题语义相似度计算 | 新增话题标签+匹配结果+行动项 |
| **call(要点)** | 从通话记录中提取关键话题→丰富实体属性 | 新增话题标签+跟进计划 |
| **manual(补全)** | 不提取话题标签，仅确认/拒绝待定标签 | 标签状态更新 |

#### 5.5.4 话题标签与F-05技术匹配度的关系

F-05技术匹配度四维打分法更新：

| 维度 | 权重 | 说明 | 变化 |
|------|------|------|------|
| 技能-需求关键词重叠(Jaccard) | +0.30 | 原有，权重从0.35微调 | ↓0.05 |
| 行业领域一致性(§5.4) | +0.25 | 原有，不变 | — |
| **话题语义相似度(§5.5)** | **+0.20** | **新增：需求话题与技能话题在话题空间的语义距离** | **新增** |
| LLM语义理解 | +0.15 | 原有，权重从0.20微调 | ↓0.05 |
| 历史合作信号 | +0.05 | 原有，不变 | — |
| **合计** | **0.95** | 预留0.05给未来维度扩展 | — |

**话题语义相似度计算**：
- 输入：实体A的话题标签集合 + 实体B的话题标签集合
- 计算：LLM embedding向量的余弦相似度，取均值
- 阈值：相似度≥0.7计满分0.20，[0.4, 0.7)按比例折算，<0.4计0

#### 5.5.5 话题标签生命周期

```
LLM提取 → provisional标签
              │
              ├─ 用户确认 → confirmed标签 → 参与F-05匹配度计算
              ├─ 用户拒绝 → discarded → 不再显示
              └─ 30天无操作 → auto_expired → 清理
```

#### 5.5.6 话题标签与领域分类的关联

| 关联方式 | 说明 | 示例 |
|---------|------|------|
| 话题→L1归属 | 每个话题标签可映射到1-2个L1领域 | "工厂智能化改造"→D02智能制造 |
| 交叉验证 | 话题标签推断的L1与实体已有L1不一致时，标记待确认 | 公司L1=D06零售电商，但话题="供应链优化"→可能L1应为D08 |
| L2标签来源 | 高频confirmed话题标签可升级为L2领域标签 | "边缘计算"出现≥20次→升级为D02的L2标签 |

#### 5.5.7 话题标签与会议类型分类的映射

meeting管线在提取话题标签后，通过话题组合判定会议类型（对应F-14的4类输出Schema）：

| 会议类型 | 话题标签特征 | 典型话题组合 | 对应F-14输出 |
|---------|------------|------------|-------------|
| **A.内部协同** | 权责、分账、团队、合伙人、招商 | ["团队分工","分账比例","合伙人机制"] | topics+consensus+actions+faq |
| **B.对外商务** | 客户、方案、报价、竞标、谈判 | ["客户需求","方案汇报","商务条款"] | clientBg+painPoints+solution+commercial+actions |
| **C.项目复盘** | 复盘、回顾、总结、改进、GRAI | ["项目回顾","经验总结","流程优化"] | goal+keep+improve+prevent+insights+actions |
| **D.知识沉淀** | SOP、话术、培训、标准、规范 | ["流程标准化","话术整理","培训大纲"] | knowledgeTags+assets+sop+talkScript |

**分类逻辑**：
1. LLM从会议raw_text中提取话题标签
2. 话题标签与4类会议的关键词特征匹配
3. 取匹配度最高的类型作为会议类型
4. 匹配度均<0.4时，默认为A类（内部协同）
5. 用户可在前端修正会议类型，修正结果反馈至模型

**与许总文本分类的关系**：许总分享的各类文本分类（如行业标签、话题标签、内容类型等），可通过本节映射机制归纳到4大会议类型及其子类。例如：许总分类中的"商务谈判"→B类子类，"团队管理"→A类子类，"案例复盘"→C类子类，"流程规范"→D类子类。

## 8. 术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 事件 | Event | 系统处理的基本单元，如名片扫描、会议记录等 |
| 实体 | Entity | 从事件中抽取的命名对象，含5种类型 |
| 实体归一 | Entity Resolution | 判断不同表述是否指向同一真实对象的过程 |
| 关联 | Association | 两个实体间的关系，含7种类型 |
| 关联强度 | Association Strength | 关联的置信程度，取值0.0~1.0 |
| 提醒 | Alert | 系统主动推送给用户的有价值信息，含3种类型（已升级为Todo） |
| Todo | Todo | 统一的待办项，包含信息型（⚪🔴🔵）和行动型（🟢），支持状态追踪和完成度统计 |
| 商机匹配度 | Opportunity Match Score | 供需双方商机匹配程度（含技术供需/业务互补/资源对接），五维打分法计算 |
| 置信度 | Confidence | 实体归一算法对归一结果的确定性评估 |
| 共现 | Co-occurrence | 两个实体在同一事件中同时出现 |
| 时间衰减 | Time Decay | 关联强度随时间推移而降低的机制，λ=0.01 |
| 别名 | Alias | 同一实体的不同称呼，如"李总"和"李明" |
| Human-in-the-Loop | HITL | 人工参与确认的关键决策环节 |
| Jaccard相似度 | Jaccard Similarity | 两个集合交集与并集的比值，用于关键词重叠计算 |
| 人脉关系 | Personal Network | 人与人之间的社会关系连接，包括校友/前同事/同城/同行业圈/投资偏好等 |
| 会议类型 | Meeting Type | meeting管线对会议内容的分类判定，含4类：A内部协同/B对外商务/C项目复盘/D知识沉淀 |
| IAMHERE | — | 许总团队的数字名片小程序产品 |
| CarryMem | — | AI关联发现引擎，PromiseLink的核心技术提供方 |

| 领域分类体系 | Domain Taxonomy | PromiseLink自建的轻量级行业/领域分类系统，采用L1预定义+L2动态扩展策略（方案D），不照搬大厂分类 |
| L1一级领域 | L1 Domain | 硬编码的18±3个顶层领域类别（草案见§5.4.2），用于F-05行业一致性评分和F-08竞对判断 |
| L2二级标签 | L2 Tag | LLM动态生成的细分标签，标记为provisional后经用户确认升级为confirmed |
| Provisional标签 | Provisional Tag | 自动生成但未经人工确认的L2标签，不参与匹配度计算 |

| 事件语义路由 | Event Semantic Routing | 根据event_type将事件分发到不同处理管线的机制，card_save→轻量/meeting→深度/call→要点/manual→补全 |
| Action路由引擎 | Action Router | 根据事件类型和关联发现结果，生成差异化Todo项（信息型+行动型）的引擎（F-06） |
| 行动项 | Action Item | 从会议/通话内容中提取的结构化待办，包含task/who/when/priority四要素 |

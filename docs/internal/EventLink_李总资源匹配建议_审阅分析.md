## 零、定位校准：私密资源经营助手 vs 资源撮合平台

**李总明确澄清：EventLink不是资源撮合平台，而是自用的私密资源经营助手。**

这个区分影响整篇分析的多个判断，需要首先校准：

| 维度 | 资源撮合平台 | 私密资源经营助手 |
|------|-------------|----------------|
| 本质 | 市场，供需双方见面 | 私人笔记，帮你自己理清 |
| 数据归属 | 资源需公开/授权共享 | 全在你自己手里 |
| 冷启动 | 需要双边密度，否则匹配跑不起来 | 你本来就认识这些人，只需记下来 |
| 匹配逻辑 | A的供给匹配B的需求 | **你的需求匹配你人脉的供给** |
| 隐私压力 | 极高（谁愿意把资源亮出来） | 零（只有你自己看） |
| Resource归属 | 资源可以脱离具体人存在（平台上的"工厂资源"） | 资源永远附着在具体人身上（"张总能搞定工厂"） |
| 对标产品 | 58同城/阿里1688/FA撮合 | Notion/笔记+AI记忆 |

### 校准后的关键推论

**1. 冷启动难度大幅下降**

我之前担心"用户不愿公开资源"——那是撮合平台的痛点。私密助手不需要公开任何东西。你本来就知道"张总能帮忙搞定工厂"，只是没系统记下来。EventLink帮你的不是"发现别人的资源"，而是"别把你已经知道的资源忘了"。

**2. Resource在私密场景下是否需要独立成表？**

如果资源永远附着在具体人身上（"张总→能搞定工厂→我上个月确认过→可信度高"），那么PoC阶段Person.properties的resource JSONB字段就足够。独立成表的触发条件是：用户需要按资源类型跨人检索（"谁能帮忙搞定工厂？"），而这个需求要在Phase1验证。

**3. 对许总更合适了**

许总不需要撮合别人，他需要的是："我知道老王能搞定装修，但我上次联系他是三个月前，而且我手头刚好有个客户要装修"——这种"自己的需求匹配自己人脉"的场景，私密助手秒杀撮合平台。

**4. 和CarryMem的哲学完全一致**

CarryMem的核心是"AI的个人记忆层，只为你自己服务"。EventLink作为"私密资源经营助手"就是CarryMem在商务关系领域的垂直应用。不是社交平台，是记忆外脑。

**5. 但"记录意愿"仍是挑战**

冷启动从"不愿公开"变成了"愿不愿花精力记录"。你确实知道张总能搞定工厂，但你会不会打开App记下来？这比"不愿公开"好解决——AI从会议记录里提取后标"待确认"，用户只需确认不需手写。

## 一、李总说得对的部分（完全同意）

### 1. "关系是入口，资源才产生商业结果"

这是整篇建议的锚点。PRD v3.3的核心闭环"事件→实体→关联→Todo→提醒"确实停在"提醒你该联系谁"这一层，没有回答"联系他干嘛"。李总加的Resource→Demand→Match闭环，是商业价值从0到1的跃迁。

具体对比：

| 维度 | 现有PRD（关系视角） | 李总建议（资源视角） |
|------|-------------------|---------------------|
| 知道什么 | 李总，供应链负责人 | 李总，可协助对接华南代工厂 |
| 能做什么 | 提醒"该联系李总了" | 建议"张总需求可能匹配李总资源" |
| 用户价值 | 效率工具（别忘了） | 商业工具（帮你赚钱） |
| 付费意愿 | 低（备忘录替代品多） | 高（直接关联收入） |

### 2. 新增Resource/Demand/MatchOpportunity三个实体

现有PRD把资源塞在`Person.properties`里，确实不够。资源需要被检索、匹配、验证、和维护，独立成表是迟早的事。

李总给的Resource字段设计很实际：

| 关键字段 | 为什么重要 |
|---------|-----------|
| `availability_status` | 区分"确认可调用"和"听说可能可以"，避免错误推荐 |
| `call_condition` | "需先有产品介绍及试产规模"——没有这字段，AI会推荐不该推的资源 |
| `last_verified_at` + `maintenance_cycle` | 资源有时效性，过期资源不如没有 |
| `sensitivity_level` | 有些资源不能公开推荐 |
| `historical_result` | 引荐2次成功1次——这是资源可靠性的核心数据 |

这些字段一看就是真做过商务撮合的人写的。PRD v3.3的Person画像虽然也有"核心资源"字段，但没有这些运营属性。

### 3. "名片降为辅助录入"这个判断

PRD v3.3把名片扫码作为主入口之一，但名片的信息密度（姓名/公司/职位/电话）对于资源匹配来说太薄了。李总指出：

> 名片是信息采集方式之一，而不是产品价值本身。

这个判断正确。合作档案Excel导入+语音一句话记录，信息密度远高于名片。MVP的首次价值演示路径应该从"扫个名片看AI能发现什么"改成"导入你的合作档案看AI能提炼什么资源"。

但要注意：名片不删除，只是降优先级。名片仍然是用户最容易理解的信息采集方式，特别是在线下场景。

### 4. 资源维护≠联系提醒

"资源透支提醒"（近30天已向王主任提出3次资源请求，建议先反馈已有进展）这个设计，没有真实商务经验写不出来。

现有PRD只有"两个月未联系"这种时间触发提醒，缺少"索取频率"这种关系健康度指标。李总的四类资源维护提醒：

| 维护类型 | 示例 | 现有PRD有吗 |
|---------|------|-----------|
| 有效性复核 | "社区资源已6个月未确认，是否仍可对接？" | ❌ 没有 |
| 关系维护 | "李总拥有近期可能用到的供应链资源，已超过建议联系周期" | ✅ 有，但不关联资源 |
| 调用结果补录 | "上次经李总对接的工厂结果如何？" | ❌ 没有 |
| 资源透支提醒 | "近30天已向王主任提出3次资源请求" | ❌ 没有 |

其中"调用结果补录"和"资源透支提醒"是真正的差异化设计，值得采纳。

### 5. 匹配等级设计

李总的四档匹配等级比PRD的二档（≥0.80推送/<0.80不推）更实际：

| 等级 | 条件 | 系统动作 |
|------|------|---------|
| 线索 | 资源类别可能匹配，但需求或可调用性未确认 | 保存线索，提示用户确认 |
| 可评估机会 | 需求明确，资源可能可用 | 展示匹配依据，建议确认 |
| 可推进机会 | 需求、资源、关系与调用条件均明确 | 生成引荐/联系行动 |
| 暂不建议推进 | 存在权限、敏感、时机或关系风险 | 不推行动，仅记录原因 |

特别好的是"暂不建议推进"这个档位——PRD只会告诉你"匹配度0.9"，不会告诉你"虽然很匹配但你不该推"。

### 6. 验收指标从算法准确率转向业务闭环

李总§19提出的8个MVP核心业务指标，比PRD的LLM抽取≥90%、F1>0.65更接近真实价值：

| 指标 | 目标 | 为什么比F1更重要 |
|------|------|-----------------|
| 用户认可匹配建议比例 | ≥60% | AI说像不像不重要，用户认不认才重要 |
| 确认匹配后生成行动比例 | ≥60% | 推荐不行动=没推荐 |
| 行动完成或已对接比例 | ≥40% | 行动不闭环=没价值 |
| 资源反馈填写率 | ≥50% | 没反馈=无法学习 |

## 二、需要谨慎的地方（可能撑爆MVP）

### 1. 这是产品重构，不是功能叠加

李总建议把核心主线从：
```
名片/会议 → 联系人 → 关联 → Todo
```
改成：
```
人物/互动 → 资源供给+真实需求 → 可信匹配 → 引荐行动 → 合作结果 → 资源维护
```

这个改动不是加几个字段能解决的。完整落地需要：

| 改动项 | 工作量 | 依赖 |
|--------|--------|------|
| 新增Resource表+CRUD | 2天 | 数据模型设计 |
| 新增Demand表+CRUD | 1.5天 | 数据模型设计 |
| 新增MatchOpportunity表+CRUD | 1.5天 | Resource+Demand就绪 |
| 重写匹配算法（五维→八维） | 3天 | Resource/Demand数据可用 |
| 新增需求录入页面 | 2天 | 前端 |
| 新增资源池页面 | 2天 | 前端 |
| 新增匹配机会中心页面 | 2天 | 前端 |
| 引荐话术生成 | 1天 | LLM prompt |
| 结果反馈流程 | 1天 | 前端+后端 |
| 资源复核提醒 | 1天 | 定时任务 |
| **合计** | **约17天** | |

PoC只有3周（15个工作日），这个工作量直接翻倍。

### 2. "可调用资源"的冷启动问题（定位校准后重新评估）

李总的MVP假设用户愿意主动录入"谁能提供什么资源"。在"私密资源经营助手"定位下，冷启动障碍已大幅缓解，但仍有挑战：

| 冷启动障碍 | 撮合平台场景 | 私密助手场景 | 严重程度 |
|-----------|-------------|-------------|---------|
| 不愿公开 | ❌ 核心障碍 | ✅ 不存在（只有自己看） | 已消除 |
| 不知道谁有资源 | ❌ 严重 | ⚠️ 中等（多数人知道部分，但不完整） | 下降 |
| 不愿意花时间录入 | ⚠️ 中等 | ⚠️ 中等（仍然需要记录动作） | 未变 |
| 资源信息变化快 | ⚠️ 中等 | ⚠️ 中等（时效性问题仍在） | 未变 |

**核心结论**：冷启动从"隐私焦虑"降级为"记录意愿"。PRD v3.3的"AI从会议记录自动提取→标待确认→用户确认"路径，正好解决"不愿意花时间录入"的问题。用户只需做确认动作（点✓），不需要手写。

**关键问题**：首批5个PoC用户，每人每周产生多少会议/名片记录？如果平均≥3条，AI提取+确认的冷启动路径就能跑通。

### 3. 对许总来说可能过度设计（定位校准后重新评估）

许总的核心场景：
- 见了很多人，记不住
- 该联系谁，想不起来
- 见了面不知道该聊什么

在"私密资源经营助手"定位下，许总的场景和李总方案更匹配了：

| 维度 | 许总需求 | 私密资源助手匹配度 | 之前（撮合平台）匹配度 |
|------|---------|------------------|---------------------|
| 资源类型 | 办公设备租赁+中网书房，类型单一 | ✅ 高（不需要8种分类，简单够用） | 中 |
| 客户需求 | 企业管理层，需求标准 | ✅ 高（"我的客户需要XX，我认识谁能提供"） | 中 |
| 核心痛点 | "见了人别忘、该联系谁、聊什么" | ✅ 高（人脉提醒+资源认知合一） | 低 |
| 使用频率 | 每周见3-5个新客户 | ✅ 高频低密度（每次见面都能沉淀） | 中 |

许总场景的典型路径："我知道老王能搞定装修，但三个月没联系了，正好手头有个客户要装修"——这就是私密资源助手的杀手场景：**把遗忘变成可调用的资源**。

**结论修正**：许总是私密资源经营助手的优质首批用户。他不需要撮合别人，他需要的是"别把我已经知道的资源忘了"。

### 4. 匹配算法八维 vs 五维

李总提出的八维匹配评分：

| 维度 | 权重 |
|------|------|
| 资源类型与需求类型匹配 | 25% |
| 具体能力与需求内容匹配 | 20% |
| 地域/行业/时间条件匹配 | 15% |
| 资源可调用程度 | 15% |
| 关系信任程度 | 10% |
| 历史资源调用结果 | 10% |
| 当前互动时机 | 5% |
| 风险与禁忌扣分 | -5%至-20% |

对比PRD五维：

| 维度 | 权重 |
|------|------|
| 关键词重叠 | 30% |
| 行业一致性 | 25% |
| 话题语义相似度 | 20% |
| LLM语义判断 | 15% |
| 历史合作 | 5% |

问题在于：八维中的"资源可调用程度"和"关系信任程度"需要Resource表和调用历史数据。没有这些数据，这两维就是空转。

**建议**：PoC用五维，把"行业一致性"拆成"行业+资源类型"，把"历史合作"扩展为"历史+调用结果"。Phase2数据丰富了再上八维。

## 三、与现有PRD/技术设计的兼容性分析

好消息：**不是颠覆，是显式化。**

| 李总的新概念 | 现有PRD中的对应 | 兼容性 | 升级路径 |
|---|---|---|---|
| Resource（资源供给卡） | Person画像中"核心资源"+"平台背书资源"字段 | 高 | JSONB字段→独立表 |
| Demand（需求） | Todo中Opportunity类型 + Event中业务意图 | 中 | Todo扩展→独立表 |
| MatchOpportunity（匹配机会） | 商机匹配度算法结果 | 中 | 内存中计算→持久化记录 |
| 资源匹配八维评分 | 商机匹配度五维评分 | 中 | 五维扩展+新增维度 |
| 资源维护提醒 | Todo类型扩展 | 高 | 加🟡🟣两种Todo |
| 资源供给卡页面 | P6实体详情页的Tab | 高 | 新增Tab→独立页面 |
| 需求池页面 | 无 | 低 | Phase1新增 |
| 匹配机会中心 | 无 | 低 | Phase2新增 |
| 引荐话术生成 | Todo中suggestion字段 | 高 | 扩展suggestion模板 |
| 结果反馈流程 | 无 | 低 | Phase1新增 |

### 兼容性最高的改动（可以PoC做）

1. **Person.properties增加resource/demand JSONB字段**——不新建表，先验证
2. **Todo类型扩展**——加🟡待确认 + 🟣资源维护，改动极小
3. **导入Excel后抽取资源候选**——LLM提取，标为待确认
4. **匹配算法加"资源可调用度"维度**——从Person.properties读

### 兼容性中等的改动（Phase1做）

5. **需求录入独立页面**
6. **匹配等级从二档变四档**
7. **调用结果补录+资源透支提醒**

### 兼容性低的改动（Phase2做）

8. **Resource/Demand/MatchOpportunity独立建表**
9. **匹配机会中心页面**
10. **团队协作+资源授权共享**

## 四、Slogan建议

### 李总的方案

- 主Slogan：让重要的人，不止停留在微信里（保留）
- 副Slogan：连接可调用的资源，促成可落地的合作

### 我之前的方案

- 主Slogan：见面只是开始
- 副Slogan：下次联系的理由，我们给你找好了

### 综合建议

**保留"让重要的人，不止停留在微信里"作为主slogan**。理由：

1. 李总说得对——这个slogan"表达直白、易传播、能击中痛点、不会过度承诺"
2. 在Resource视角下，这个slogan反而更对了——"不止停留在微信里"暗示了"让关系产生更多价值"
3. "见面只是开始"虽然更有力度，但传播成本更高（需要上下文理解）

**副slogan需要缩短**。"连接可调用的资源，促成可落地的合作"14个字，不像slogan像产品描述。

| 方案 | 文案 | 字数 | 评价 |
|------|------|------|------|
| 李总原版 | 连接可调用的资源，促成可落地的合作 | 14 | 太长，像产品描述 |
| 缩短版A | 连接资源，匹配需求，推进合作 | 9 | 短了但节奏好 |
| 缩短版B | 让关系成为资源，让资源促成合作 | 10 | 适合路演PPT |
| 缩短版C | 见面只是开始 | 5 | 最短最有力度 |

**推荐搭配**：

| 场景 | Slogan |

## ⚠️ 待确认 / 假设 / Non-goals

### 待确认
- 许总是否认同杀手场景"别把我已经知道的资源忘了"？这是私密资源经营助手的核心价值主张
- 李总是否接受"私密资源经营助手"定位（而非"资源撮合平台"）？这个定位校准是否改变他对功能优先级的判断？
- 副slogan用"连接资源，匹配需求，推进合作"还是李总原版"连接可调用的资源，促成可落地的合作"？
- PoC首批5个用户，每人每周产生多少会议/名片记录？（决定AI提取+确认的冷启动路径能否跑通）

### 假设
- EventLink定位为"私密资源经营助手"，资源永远附着在具体人身上，不需要公开/授权/双边密度
- PoC不改核心数据模型（不加新表），只扩展JSONB字段和Todo类型
- 首批用户更可能从会议记录中确认资源线索，而非主动手写录入
- 名片仍然保留为信息采集方式之一，只是不作为MVP首次价值演示路径

### Non-goals
- 本分析不决定是否采纳李总方案（只提供分析供决策）
- 不涉及团队协作、企业版等Phase2+功能的设计
- 不涉及与外部系统（CRM/日历）的集成方案

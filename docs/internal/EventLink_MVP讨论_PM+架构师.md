# EventLink MVP范围精简讨论

## PM与架构师对话

**PM**: 我们先对齐一下，当前28个功能确实太多了。3人团队3周PoC，按每人每周能完成1-1.5个功能点算，最多也就13-15个功能。我们得砍掉一半以上。

**架构师**: 同意。而且PoC阶段更重要的是验证核心假设，不是堆功能。我先说说我理解的核心价值：**验证AI能否从碎片化沟通中自动提取商机线索，并生成可执行的行动建议**。

**PM**: 对，这就是我们的北极星。用户痛点是：微信、邮件、会议里藏着大量商机，但人工整理太累，CRM录入门槛高。我们要证明AI能自动做这件事。

**架构师**: 那我们倒推MVP-Core。要验证这个假设，最小闭环是什么？

**PM**: 
1. 用户能上报事件（邮件/会议记录）
2. 系统能识别出人、公司、商机
3. 系统能发现"这个人和那个商机有关"
4. 系统能生成"你该联系谁做什么"的Todo
5. 用户能看到这些Todo并反馈

**架构师**: 这就是5个核心功能。但还缺两个基础能力：
- 数据管理：用户得能看到识别出的实体和关联
- 安全基础：至少得有认证，不能裸奔

加上这两个，MVP-Core就是7个功能。

---

## 1. MVP核心价值定义

**共识结论**：
> **验证AI能否从非结构化沟通记录中自动提取商机实体、发现关联关系，并生成可执行的行动建议，形成从数据到行动的完整闭环。**

## 2. MVP功能三层分级

### **MVP-Core（PoC必须，7个功能）**

| 编号 | 功能 | 理由 |
|------|------|------|
| F-01 | 事件上报接口 | 数据入口，没有就没戏 |
| F-02 | 实体抽取服务 | 核心AI能力验证 |
| F-03 | 实体归一引擎 | 同一个人/公司的多次出现要能合并 |
| F-04 | 关联发现引擎 | 核心价值：发现隐藏的商机线索 |
| F-06 | Todo生成引擎 | 从洞察到行动的关键 |
| F-07 | Todo反馈闭环 | 验证用户会不会用 |
| F-22 | 认证授权 | 最小安全保障 |

**PM**: 等等，F-05商机匹配度计算不在Core里？

**架构师**: 我建议先砍掉。PoC阶段可以用简单规则（比如"最近联系+职位匹配"）生成Todo，不需要复杂的打分算法。匹配度是优化问题，不是验证问题。

**PM**: 有道理。那数据管理中心（F-20）呢？用户得能看到识别出的实体吧？

**架构师**: F-20太重了，"数据管理中心"听起来像个完整的CRUD系统。PoC阶段我们只需要一个简单的列表页：显示识别出的人、公司、商机，能点进去看详情就够了。这个可以算在F-02/F-03的交付物里，不单独算功能点。

| F-05 | 商机匹配度计算 | 优化Todo质量，提升用户信任 |
| F-10 | 实体图谱可视化 | 让关联关系更直观 |

**PM**: F-09人脉关系提取不在Plus里？

**架构师**: 人脉关系是Phase 2的重点。PoC阶段我们只关注"商机关联"，不做社交网络分析。F-09可以放到V1.0。

## 3. 日程管理设计

**PM**: 用户明确要求"本地先有日程管理，再考虑外部同步"。但F-33现在只写了"日程表双向对接"，没有本地日程能力。

**架构师**: 这是个好问题。我们需要拆分成两个功能：

**新增功能：F-33A 本地日程管理（MVP-Plus）**
- 用户可以手动创建/编辑/删除日程
- 日程可以关联到商机、联系人
- Todo可以转化为日程事件
- 提供日历视图（日/周/月）

**原F-33改为：F-33B 外部日历同步（V1.0）**
- 对接Google Calendar / Outlook / 飞书日历
- 双向同步：外部日程导入 + 本地日程推送
- 冲突检测与合并策略

**PM**: 为什么F-33A在MVP-Plus而不是Core？

**架构师**: 因为PoC阶段我们验证的是"AI能否从已发生的事件中提取价值"，不是"日程管理工具"。用户可以先用Todo功能，等PoC验证通过后再补日程能力。

**PM**: 同意。而且F-33A和F-06 Todo生成有协同效应：Todo可以直接拖到日历上变成日程。

## 4. 关联暴增性能方案

**架构师**: 这是个架构层面的问题。假设用户有：
- 500个联系人
- 200个公司
- 100个商机
- 5000条沟通记录

关联关系可能达到：500×200×100 = 1000万种可能的三元组。即使只保留有效关联，也可能有几千到几万条。

**PM**: 用户场景是什么？他们会怎么用这些关联？

**架构师**: 三个高频场景：
1. **查看某个商机的所有相关人和公司**（商机详情页）
2. **查看某个人参与的所有商机**（人物详情页）
3. **搜索"哪些商机和华为有关"**（全局搜索）

**架构师**: 我建议的架构方案：

### **4.1 数据模型优化**
```
关联表设计（association）:
- id
- source_type (person/company/opportunity)
- source_id
- target_type
- target_id
- relation_type (参与/负责/竞争/合作)
- confidence (0-1置信度)
- evidence_ids (JSON数组，指向原始事件)
- created_at
- updated_at

索引策略：
- (source_type, source_id, confidence DESC) -- 查"某实体的所有关联"
- (target_type, target_id, confidence DESC) -- 反向查询
- (source_type, target_type, confidence DESC) -- 类型过滤
```

### **4.2 分层查询策略**
- **L1 热数据**：最近30天 + 置信度>0.7 的关联，存Redis，毫秒级响应
- **L2 温数据**：最近1年的关联，PostgreSQL索引查询，<100ms
- **L3 冷数据**：历史归档，按需加载

### **4.3 搜索方案**
**PM**: 用户要搜"华为相关的商机"，怎么做？

**架构师**: 两阶段方案：
1. **Phase 1（MVP-Plus）**: PostgreSQL全文搜索 + 关联表JOIN
   - 适用于<10万条记录
   - 实现F-16模糊查找
   
2. **Phase 2（V1.0+）**: 引入Elasticsearch
   - 实体和关联都索引到ES
   - 支持复杂查询："华为 AND (销售机会 OR 合作意向) AND 负责人:张三"

**PM**: PoC阶段用PostgreSQL够吗？

**架构师**: 够。3周PoC不会有超过1000条记录。我们在架构上预留ES接口，但实现延后到V1.0。

### **4.4 性能保障措施**
- **分页加载**：关联列表默认只显示Top 20（按置信度排序）
- **懒加载**：详情页的关联图谱按需展开
- **后台任务**：关联发现引擎异步计算，不阻塞用户操作
- **缓存策略**：高频查询的关联结果缓存5分钟

**PM**: 如果用户真的有几千条关联，前端怎么展示？

**架构师**: 
1. **过滤器**：按关联类型、时间范围、置信度筛选
2. **聚合视图**：不展示所有关联，而是显示"与10家公司、25个人、8个商机相关"
3. **智能排序**：优先显示"最近活跃"和"高置信度"的关联

## 最终共识结论

### **MVP三层分级**

| 层级 | 功能数 | 交付时间 | 核心目标 |
|------|--------|----------|----------|
| **MVP-Core** | 7个 | Week 1-3 (PoC) | 验证核心假设 |
| **MVP-Plus** | 8个 | Week 4-6 (首版发布) | 可用性与体验 |
| **V1.0** | 13个 | Week 7-12 (正式版) | 完整性与合规 |

### **MVP-Core清单（PoC必须）**
1. F-01 事件上报接口
2. F-02 实体抽取服务
3. F-03 实体归一引擎
4. F-04 关联发现引擎
5. F-06 Todo生成引擎
6. F-07 Todo反馈闭环
7. F-20 认证授权

### **关键架构决策**
1. **日程管理**：拆分为本地日程（MVP-Plus）+ 外部同步（V1.0）
2. **关联性能**：PostgreSQL + 分层缓存（PoC），预留ES接口（V1.0）
3. **搜索方案**：PG全文搜索（MVP-Plus），ES（V1.0+）
4. **功能编号**：按模块重整为F-01~F-09（引擎）、F-10~F-19（数据）、F-20~F-29（安全）、F-30~F-39（集成）

### **砍掉的功能（从Phase 1移除）**
- F-05 商机匹配度（降级到MVP-Plus）
- F-08 竞对关系（Phase 2）
- F-09 人脉关系（Phase 2）
- F-14 会议分类（V1.0优化）
- F-15 行动建议（V1.0优化）
- F-17/F-21/F-24/F-25/F-31 安全合规（V1.0）
- F-30/F-32/F-33/F-34 体验优化（V1.0）

**PM**: 这样Phase 1从28个功能降到15个（7 Core + 8 Plus），3人团队6周可以完成。PoC只做7个核心功能，3周可行。

**架构师**: 同意。而且这7个功能形成完整闭环，能真正验证价值。后续迭代有清晰的优先级。

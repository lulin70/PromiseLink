### 核心问题:用户最担心什么?

### 5.1 GDPR合规缺口

| GDPR要求 | 用户权利 | 当前PRD | 缺口 |

**用户场景**:投资人上传"尽职调查报告",包含未公开财务数据
- **SOC2要求**:访问控制+审计日志+加密 → ❌ 全部缺失
- **PCI-DSS**(如涉及支付):不适用,但需明确排除支付卡信息

# DevOps工程师 - EventLink PRD v1.6 客户视角评审

## 1. 客户视角发现的问题

### P0 - 阻断性问题(用户无法正常使用)

**P0-1: 缺少服务可用性保障定义**
- **用户痛点**: 用户在商务活动现场扫描名片,如果服务挂了,错过关键商机
- **当前PRD缺失**: 没有定义SLA目标(可用性、响应时间、恢复时间)
- **用户影响**: 在重要商务场合无法使用 = 产品失去信任

**P0-2: 高峰期容量规划缺失**
- **用户场景**: 大型行业峰会,100+用户同时扫描名片
- **当前PRD缺失**: 没有定义并发用户数、峰值QPS、扩容策略
- **用户影响**: 活动现场集体卡顿 = 口碑崩塌

**P0-3: 数据丢失风险未定义**
- **用户最不能接受**: 扫描的名片数据丢失、会议纪要丢失、Todo列表丢失
- **当前PRD缺失**: 没有备份策略、数据持久化保障、灾难恢复计划
- **用户影响**: 丢失关键商务联系人 = 产品不可用

**P0-4: 弱网/离线场景未覆盖**
- **用户场景**: 地铁通勤、电梯、地下会议室、国际漫游
- **当前PRD缺失**: 没有定义离线模式、数据同步策略、弱网降级方案
- **用户影响**: 只能在WiFi环境使用 = 使用场景受限50%+

### P1 - 严重影响体验

**P1-1: 响应时间未量化**
- **用户期望**: 扫描名片3秒内看到基础信息,5秒内看到关联,10秒内看到Todo
- **当前PRD缺失**: 只说"轻量秒级"、"深度分钟级",没有具体SLO
- **用户影响**: 等待时间不可预期,焦虑感

**P1-2: 部署更新对用户的影响未定义**
- **用户场景**: 用户正在查看关联图谱时,后端发布新版本
- **当前PRD缺失**: 没有定义零停机部署、版本兼容性、回滚策略
- **用户影响**: 操作中断、数据未保存

**P1-3: 监控告警从技术视角而非用户视角**
- **当前问题**: 监控CPU/内存,但不监控"用户扫描名片成功率"、"Todo生成延迟"
- **用户影响**: 技术指标正常但用户体验已经退化

**P1-4: 数据增长后的性能退化未验证**
- **用户场景**: 用户从0到1000张名片,查询速度从1秒变5秒
- **当前PRD缺失**: 没有定义性能基准、容量测试、数据归档策略
- **用户影响**: 老用户体验越来越差

**P1-5: 多环境数据隔离未定义**
- **用户风险**: 测试环境的错误配置影响生产环境
- **当前PRD缺失**: 没有定义环境隔离策略、配置管理、权限控制
- **用户影响**: 生产事故风险

### P2 - 优化项

## 4. 是否同意通过P1 Gate: ❌ **不同意**

### 理由:

#### 阻断原因(必须解决才能通过):

1. **用户在关键场景无法使用**
   - 高峰期容量未规划 → 商务活动现场集体卡顿
   - 弱网/离线未支持 → 50%+使用场景受限
   - 数据丢失风险未控制 → 用户不敢依赖产品

2. **响应时间不可预期**
   - "轻量秒级"、"深度分钟级"太模糊
   - 用户不知道要等多久,焦虑感强
   - 需要量化SLO: P95延迟<3秒(card_save), <10秒(meeting)

3. **缺少用户视角的可观测性**
   - 当前监控是技术指标(CPU/内存),不是用户体验指标
   - 用户反馈问题时,无法快速定位根因
   - 需要业务指标监控 + 分布式追踪

#### 通过P1 Gate的前置条件:

**必须补充的内容:**

1. **服务可用性保障(P0)**
   - 定义SLA目标: 可用性99.9%, RTO<5分钟, RPO<5分钟
   - 设计高可用架构: 多副本、健康检查、自动故障转移
   - 备份恢复策略: 自动备份、验证、一键恢复

2. **性能与容量规划(P0)**
   - 量化SLO: card_save P95<3秒, meeting P95<10秒
   - 容量模型: 支持1000并发用户, 10万张名片
   - 扩容策略: 自动扩缩容,响应时间<30秒

3. **弱网/离线支持(P0)**
   - 离线模式: 缓存最近100条数据,支持查看和标记
   - 弱网降级: <100KB/s时禁用图谱可视化,保留核心功能
   - 数据同步: 网络恢复后自动同步,冲突解决策略

4. **用户视角监控(P1)**
   - 业务指标: 扫描成功率、关联发现延迟、Todo生成成功率
   - 用户旅程追踪: 从扫描到行动完成的端到端延迟
   - 实时告警: 用户影响阈值(如成功率<95%立即告警)

5. **部署策略(P1)**
   - 零停机部署: 滚动更新,健康检查
   - 版本兼容性: API版本管理,前后端兼容性测试
   - 回滚策略: 自动回滚条件,回滚时间<5分钟

**建议的Gate通过流程:**

1. **补充PRD章节**: "非功能需求 - 性能、可用性、容量"
2. **输出技术方案**: 高可用架构设计、监控告警方案、部署流程
3. **验证关键路径**: 
   - 压力测试: 100并发用户,验证响应时间
   - 故障演练: 模拟服务宕机,验证恢复时间
   - 弱网测试: 限速到100KB/s,验证核心功能可用
4. **评审通过标准**: 所有P0问题有明确解决方案,P1问题有排期计划

## 一、客户视角发现的问题

### P0 级问题（阻塞用户核心体验）

**P0-1: 缺少前端架构定义，用户界面结构完全未定义**
- **用户痛点**：用户打开App后不知道会看到什么，导航逻辑不清晰
- **架构影响**：前后端接口无法设计，开发无法启动
- **缺失内容**：
  - 页面结构定义（首页、扫描页、详情页、Todo列表页、数据管理页）
  - 页面间导航关系
  - 状态管理策略（本地状态 vs 服务端状态）

**P0-2: 实时性架构未定义，用户等待体验不可控**
- **用户场景**：扫描名片后，用户期望：
  - 0-1秒：看到基础信息（姓名、公司、职位）
  - 1-3秒：看到关联发现结果
  - 3-5秒：看到商机匹配和Todo建议
- **当前问题**：PRD只定义了"轻量秒级"和"深度分钟级"，但没有定义：
  - 渐进式加载策略
  - 前端轮询 vs WebSocket vs SSE
  - 超时处理和降级方案
- **架构风险**：用户可能看到空白页面或无限加载

**P0-3: 离线和弱网场景完全缺失**
- **用户场景**：
  - 地铁上扫描名片（弱网/无网）
  - 飞机上查看历史数据
  - 会议室信号差时录入会议纪要
- **当前问题**：
  - 没有离线存储策略（IndexedDB/SQLite）
  - 没有数据同步机制（冲突解决、增量同步）
  - 没有离线队列设计（操作缓存、自动重试）
- **用户影响**：弱网环境下产品不可用，商务场景致命

**P0-4: 数据规模增长的性能退化路径不明确**
- **用户场景**：
  - 新用户：10张名片，体验流畅
  - 3个月后：500张名片，搜索变慢
  - 1年后：2000张名片，关联发现超时
- **当前问题**：
  - 没有分页加载策略
  - 没有索引设计（全文搜索、关联查询）
  - 没有缓存策略（热数据 vs 冷数据）
- **架构风险**：用户数据越多，体验越差，导致流失

### P1 级问题（影响用户满意度）

**P1-1: API设计从数据模型正推，而非用户操作倒推**
- **问题**：当前PRD定义了20个功能，但没有定义用户操作到API的映射

  - 一个用户操作可能需要调用多个API（网络往返增加）

## 二、架构分析

### 2.1 用户操作到系统响应的时间线分析

**场景1：扫描名片**

```
用户操作：点击"扫描名片"按钮
├─ T+0ms: 打开相机
├─ T+500ms: 拍照完成
├─ T+600ms: 上传图片（假设100KB，4G网络）
├─ T+1000ms: OCR识别完成 → 用户看到基础信息（姓名、公司、职位）✅
├─ T+1500ms: 实体抽取完成
├─ T+2000ms: 实体归一完成
├─ T+3000ms: 关联发现完成 → 用户看到关联结果✅
├─ T+5000ms: 商机匹配完成 → 用户看到Todo建议✅
└─ T+10000ms: 深度分析完成（如果需要）
```

**架构要求**：
- 渐进式加载：1秒、3秒、5秒三个里程碑
- 前端需要状态机：loading → partial → complete
- 后端需要异步任务队列：快速返回 + 后台处理

**场景2：查看Todo列表**

```
用户操作：打开Todo列表页
├─ T+0ms: 发起请求 GET /todos?status=pending
├─ T+200ms: 返回Todo列表（分页，20条/页）→ 用户看到列表✅
├─ 用户滚动到底部
├─ T+400ms: 加载下一页
└─ T+600ms: 返回下一页数据
```

**架构要求**：
- 分页加载（无限滚动 vs 分页按钮）
- 预加载策略（提前加载下一页）
- 缓存策略（已加载的页面缓存5分钟）

### 2.2 数据规模增长的性能分析

| 数据规模 | 名片数 | 关联数 | Todo数 | 查询延迟 | 架构要求 |
|---------|--------|--------|--------|----------|----------|
| 新用户 | 10 | 50 | 20 | <100ms | 单表查询 |
| 3个月 | 500 | 2500 | 200 | <200ms | 索引优化 |
| 1年 | 2000 | 10000 | 800 | <500ms | 分页+缓存 |
| 重度用户 | 5000+ | 25000+ | 2000+ | <1000ms | 分库分表 |

**架构决策**：
- **索引设计**：
  - 名片表：`idx_user_name`, `idx_user_company`, `idx_user_created_at`
  - 关联表：`idx_user_entity_type`, `idx_user_relation_type`
  - Todo表：`idx_user_status_priority`, `idx_user_due_date`
- **分页策略**：
  - 列表页：20条/页
  - 搜索结果：50条/页
  - 关联发现：Top 10（按匹配度排序）
- **缓存策略**：
  - 热数据（最近7天）：Redis缓存，TTL=1小时
  - 冷数据（7天前）：按需加载，不缓存

### 2.3 离线和弱网架构设计

**离线存储策略**：
```
前端存储层次：
├─ Memory Cache（当前会话）
├─ IndexedDB（持久化，最多1000条记录）
│  ├─ cards（名片基础信息）
│  ├─ relations（关联关系）
│  └─ todos（待办事项）
└─ 操作队列（待同步操作）
   ├─ pending_uploads（待上传的名片）
   ├─ pending_updates（待更新的Todo状态）
   └─ pending_deletes（待删除的记录）
```

**同步策略**：
```
网络恢复后：
1. 上传操作队列（按时间戳顺序）
2. 拉取服务端增量更新（since=last_sync_time）
3. 冲突解决：
   - 同一记录的修改：Last-Write-Wins（服务端时间戳为准）
   - 删除冲突：删除优先
4. 更新本地IndexedDB
```

**弱网优化**：
- 请求超时：5秒（正常网络）→ 15秒（弱网检测后）
- 自动重试：指数退避（1s, 2s, 4s, 8s）
- 降级策略：关联发现失败 → 只显示基础信息

## 四、架构决策记录（ADR）

### ADR-001: 渐进式加载 vs 一次性加载

**决策**：采用渐进式加载

**理由**：
- 用户期望快速看到结果（1秒内看到基础信息）
- AI处理需要时间（关联发现3-5秒）
- 网络不稳定时，一次性加载容易超时

**实现**：
- 前端：状态机管理（loading → partial → complete）
- 后端：异步任务队列（快速返回 + 后台处理）
- API：轮询 or WebSocket（推荐轮询，简单可靠）

### ADR-002: 客户端缓存 vs 服务端缓存

**决策**：两者结合

**理由**：
- 客户端缓存：减少网络请求，支持离线
- 服务端缓存：减少数据库压力，支持多端同步

**实现**：
- 客户端：IndexedDB（持久化）+ Memory Cache（临时）
- 服务端：Redis（热数据）+ CDN（静态资源）

### ADR-003: 关系型数据库 vs NoSQL

**决策**：关系型数据库（PostgreSQL）

**理由**：
- 数据结构清晰（用户、名片、关联、Todo）
- 需要事务支持（批量操作、数据一致性）
- 需要复杂查询（关联发现、搜索过滤）
- JSONB字段支持灵活扩展

**未来扩展**：
- 搜索：引入Elasticsearch（全文搜索）
- 图谱：引入Neo4j（关系可视化）

### ADR-004: 单体架构 vs 微服务

**决策**：单体架构（Modular Monolith）

**理由**：
- 团队规模小，微服务增加复杂度
- 功能耦合度高（名片→关联→Todo）
- 部署简单，运维成本低

**模块化设计**：
```
Monolith
├─ API Gateway（路由、认证）
├─ Card Service（名片管理）
├─ Relation Service（关联发现）
├─ Todo Service（待办管理）
├─ AI Service（OCR、NLP）
└─ Shared（数据库、缓存、队列）
```

**未来扩展**：
- 如果AI服务成为瓶颈，可以独立部署
- 如果用户数>100万，可以拆分为微服务

### [ ] 问题1:本地部署的依赖管理和环境一致性缺失
**影响**:
- 用户在不同操作系统(Windows/macOS/Linux)上遇到不同的安装问题
- Python版本差异导致依赖冲突(特别是SQLite版本、LLM库依赖)
- "零依赖"承诺与实际LLM集成需求矛盾,用户困惑
- 支持成本高,大量时间花在环境问题排查上

**建议**:
- 提供多种打包方式:
  - Docker镜像(推荐):一键运行,环境隔离
  - PyInstaller打包的可执行文件(Windows/macOS)
  - pip安装包 + 详细的环境要求文档(Python 3.9+)
- 创建 `promiselink doctor` 命令检查环境健康度
- 提供 docker-compose.yml 用于本地快速启动
- 在文档中明确区分"核心零依赖"(Phase 1)和"AI增强需要配置"(Phase 2+)

**优先级**: **P0** - MVP阶段必须解决,否则用户无法顺利onboarding

### [ ] 问题2:数据迁移和版本升级策略缺失
**影响**:
- 用户从Phase 1升级到Phase 2时,SQLite schema变更导致数据丢失
- 实体归一算法改进后,历史关联需要重新计算但无机制触发
- 用户不敢升级版本,停留在旧版本积累技术债
- 云端同步版本与本地版本schema不一致导致数据损坏

**建议**:
- 实现数据库版本管理:
  - 使用Alembic或类似工具管理SQLite schema迁移
  - 在数据库中存储schema_version表
  - 启动时自动检测并执行必要的迁移脚本
- 提供 `promiselink migrate` 命令手动触发迁移
- 实现数据备份机制:
  - 升级前自动备份SQLite文件(带时间戳)
  - 提供 `promiselink backup/restore` 命令
- 在云端版本中实现schema版本协商机制
- 文档化每个版本的breaking changes和迁移路径

**优先级**: **P0** - 必须在Phase 2之前实现,否则Phase 1用户无法平滑升级

## 2. 跨领域关注点

### 安全领域观察:
**问题**:实体归一过程中的数据泄露风险
- 如果使用云端LLM进行实体归一,用户的敏感人脉数据会发送到第三方API
- 即使是本地LLM,模型本身可能存在数据泄露风险(模型训练数据污染)
- 建议:
  - 提供"完全本地模式":使用规则引擎而非LLM进行实体归一
  - 云端LLM调用时脱敏处理(替换真实姓名为占位符)
  - 文档中明确说明数据流向和隐私保护措施
  - 通过SOC 2/ISO 27001认证增强信任

### 产品领域观察:
**问题**:提醒疲劳和用户留存
- 如果提醒过于频繁或不相关,用户会关闭通知甚至卸载产品
- 缺少用户反馈机制(这个提醒有用吗?)导致算法无法改进
- 建议:
  - 实现提醒频率控制:每天最多N条,可配置
  - 提醒优先级排序:只推送高价值提醒
  - 用户反馈机制:👍/👎按钮,记录到数据库
  - A/B测试不同提醒策略,优化点击率和留存率
  - 提供"勿扰模式":用户可暂停提醒

```python
2. L2: 规则匹配(邮箱域名+姓名/手机号+姓名) - 本地,低成本
3. L3: 向量相似度(embedding缓存) - 本地/云端,中成本
4. L4: LLM判断(仅处理L3不确定的case) - 云端,高成本
5. 人工确认机制:置信度<0.8时标记待确认,不自动合并
```

**实现要点:**
- 为每个实体维护`canonical_id`和`aliases`列表
- 记录归一决策的`confidence_score`和`method`
- 提供`undo_merge`命令用于错误修正
- 缓存LLM判断结果避免重复调用

**优先级:P0** (核心功能,直接影响产品价值)

#### [ ] 问题3:LLM调用的错误处理和降级策略缺失

**影响:**
- API超时/限流/余额不足会导致整个处理管道卡死
- 不同LLM提供商的输出格式不一致(OpenAI vs Claude vs 本地模型)
- 网络抖动导致部分事件处理失败但无重试机制

**建议:**
```python
# 统一LLM抽象层
class LLMProvider(Protocol):
    def extract_entities(self, text: str) -> List[Entity]: ...
    def resolve_entity(self, entity: Entity, candidates: List[Entity]) -> Optional[str]: ...

# 降级策略
class RobustLLMService:
    def __init__(self, providers: List[LLMProvider], fallback: RuleBased):
        self.providers = providers  # [OpenAI, Claude, LocalLLM]
        self.fallback = fallback
    
    async def extract_with_retry(self, text: str, max_retries=3) -> List[Entity]:
        for provider in self.providers:
            try:
                return await provider.extract_entities(text)
            except (Timeout, RateLimitError) as e:
                logger.warning(f"{provider} failed: {e}")
                continue
        # 最终降级到规则引擎
        return self.fallback.extract_entities(text)
```

**实现要点:**
- 使用`tenacity`库实现指数退避重试
- 为每个LLM调用设置超时(5s)和熔断机制
- 记录降级事件到日志,用于后续人工review
- 提供`--no-llm`模式纯规则运行

**优先级:P0** (生产可用性的基础)

#### [ ] 问题4:与CarryMem的集成边界模糊,存在循环依赖风险

```python
class PromiseLinkDB:
```

     - 验证假设:用户是否愿意投入时间维护人脉图谱

2. **明确MVP成功指标**:
   - 用户完成首次事件输入的时间 < 3分钟
   - 7天内至少输入5个事件的用户占比 > 30%
   - 用户认为提醒"有价值"的比例 > 60%

**优先级:P0**(MVP定义不清会导致开发方向错误)

---

#### **问题3:提醒策略缺乏用户控制机制,存在严重打扰风险**

**影响:**
- 3种提醒类型(机会/风险/上下文)全部推送会造成信息过载
- 没有提醒频率控制,用户可能一天收到几十条提醒
- 缺乏提醒相关性评分机制,无法区分重要/次要提醒
- 用户无法自定义关注的关联类型,导致噪音过多

**建议:**
1. **设计提醒优先级算法**:
   ```
   提醒分数 = 关联强度 × 时效性 × 用户偏好权重
   - 关联强度:直接关联(1.0) > 二度关联(0.5) > 三度关联(0.2)
   - 时效性:即将发生的事件(1.0) > 近期事件(0.7) > 历史事件(0.3)
   - 用户偏好:用户可设置关注的关联类型权重
   ```

2. **提供用户控制选项**:
   - 提醒频率设置(实时/每日摘要/每周摘要)
   - 关联类型开关(用户可关闭不感兴趣的关联类型)
   - 提醒阈值设置(只推送分数>X的提醒)
   - "稍后提醒"和"不再提醒此类"功能

3. **设计提醒展示规范**:
   - 标题:一句话说清楚关联点(不超过50字)
   - 正文:关联路径可视化 + 行动建议
   - 操作:查看详情/标记已读/反馈无用

4. **建立提醒质量反馈循环**:
   - 每条提醒提供"有用/无用"反馈按钮
   - 根据反馈调整算法权重
   - 定期分析提醒点击率和反馈率

**优先级:P1**(影响核心用户体验,但可以在MVP后快速迭代)

---

#### **问题4:商业模式缺乏清晰的价值阶梯,免费版与付费版边界模糊**

**影响:**
- "基础版免费+云端订阅"的分界线不清晰
- 用户为什么要付费?团队共享的需求有多强?
- 录音转写按量计费与订阅制混合,定价复杂度高
- 缺乏从免费到付费的转化路径设计

**建议:**
1. **重新设计价值阶梯**:
   ```
   免费版(个人本地):
   - 手动输入事件(无限)
   - 基础关联发现(同事/同校)
   - 本地存储(无云端同步)
   - CLI工具
   
   专业版($9.99/月):
   - LLM自动抽取(每月500次)
   - 高级关联(竞对/商机/供应链)
   - 云端同步+多设备
   - Web仪表盘
   - 日历/邮件集成
   
   团队版($29.99/用户/月):
   - 团队图谱共享
   - 权限管理
   - CRM集成
   - API访问
   - 优先支持
   ```

2. **设计转化漏斗**:
   - 免费用户达到50个事件后,提示"升级解锁高级关联"
   - 提供7天专业版试用
   - 在CLI中嵌入升级提示(非侵入式)

3. **验证付费意愿**:
   - 在MVP阶段向早期用户询问:"如果这个功能收费,你愿意付多少?"
   - 测试不同定价点的转化率($4.99/$9.99/$19.99)

**优先级:P2**(商业模式可以在产品验证后再优化,但需要提前规划)

**建议:**

### 4. 整体评估

#### **可行性评分:7/10**

**评分理由:**
- ✅ **技术可行性高**(8/10):SQLite+LLM的技术栈成熟,实现难度不大
- ⚠️ **市场需求待验证**(6/10):核心假设(用户愿意维护私域人脉图谱)未经验证
- ⚠️ **商业模式不确定**(6/10):付费意愿和定价策略需要市场测试
- ✅ **团队能力匹配**(8/10):基于CarryMem的技术积累,团队有能力实现

#### **最大风险点**

**用户不愿意投入时间维护数据**
- **风险描述**:即使有LLM辅助,用户仍需要定期输入事件。如果输入成本>感知价值,产品会失败
- **验证方法**:
  1. MVP阶段追踪"7日留存率"和"人均事件输入数"
  2. 如果7日留存<20%或人均输入<3个事件,说明输入成本过高
  3. 需要降低输入成本(更多自动化数据源)或提高感知价值(更精准的提醒)

#### **核心建议**

1. **先验证需求,再完善功能**
   - 用最小MVP(手动输入+LLM抽取+3种关联)快速验证核心假设
   - 不要在Phase 1就追求完美的技术架构
   - 目标:2周内发布MVP,获得50个早期用户反馈

2. **聚焦单一用户群体**
   - 建议首选:早期创业者/投资人(他们对人脉敏感度最高)
   - 在这个群体中做到极致体验,再扩展到其他群体
   - 避免"什么人都想服务"的陷阱

3. **建立数据飞轮**
   - 用户输入越多 → 关联发现越准 → 提醒越有价值 → 用户越愿意输入
   - 关键:前10个事件的体验必须足够好,让用户看到价值
   - 考虑"导入LinkedIn/日历"作为冷启动加速器

4. **设计退出机制**
   - 如果MVP验证失败(留存率<20%),需要有明确的Pivot方向:
     - Pivot A:转向纯被动查询工具(类似个人CRM)
     - Pivot B:转向团队协作工具(放弃个人市场)
     - Pivot C:转向特定垂直领域(如投资人专用工具)

# PromiseLink 产品设计安全评审报告

## 1. 角色专业视角评审(安全专家)

### 🔴 P0 - 关键安全问题

- [ ] **问题1:本地SQLite数据库无加密保护**
  - **影响**:商务人脉图谱包含高度敏感信息(竞对关系、商机、人脉网络),本地SQLite文件若无加密,任何获得文件系统访问权限的恶意软件/内部人员都可直接读取完整人脉图谱。这是数据泄露的最大风险点。
  - **建议**:
    - 使用SQLCipher替代标准SQLite,提供透明的AES-256加密
    - 密钥派生方案:用户主密码 → PBKDF2(100k+ iterations) → 数据库加密密钥
    - 密钥存储:macOS使用Keychain,Linux使用libsecret,Windows使用DPAPI
    - 提供密钥轮换机制(重新加密数据库)
  - **优先级**:P0 - MVP必须解决,否则产品不可用于商务场景

- [ ] **问题2:团队共享场景下的权限模型缺失**
  - **影响**:设计文档提到"团队共享"但未定义权限模型。可能导致:
    - 团队成员A看到成员B的私密人脉关系
    - 离职员工仍可访问团队图谱
    - 无法实现"只共享特定项目相关的实体"
    - IDOR漏洞:通过修改entity_id访问其他成员的私有实体
  - **建议**:
    - 实体级权限模型:每个Entity/Association/Event必须有owner_id和visibility字段
    - 访问控制矩阵:
      ```
      private: 仅owner可见
      team: 团队成员可见
      project: 特定项目成员可见
      public: 组织内可见
      ```
    - API层强制校验:`SELECT * FROM entities WHERE id=? AND (owner_id=? OR visibility IN (...))`
    - 审计日志:记录所有跨用户访问行为
  - **优先级**:P0 - Phase 4商务版的前置条件

- [ ] **问题3:LLM API Key管理的安全风险**
  - **影响**:
    - 用户自配API Key若明文存储在配置文件,易被窃取
    - API Key泄露后攻击者可消耗用户配额或访问LLM服务
    - 云端版本若集中管理用户API Key,成为高价值攻击目标
  - **建议**:
    - 本地版:API Key必须加密存储(使用系统密钥环)
    - 云端版:采用代理模式,用户API Key加密存储在服务端,使用时解密调用
    - 实施API Key轮换策略和使用量监控
    - 提供"只读API Key"选项(如OpenAI的restricted keys)
    - 错误处理不泄露API Key信息(日志脱敏)
  - **优先级**:P0 - Phase 2引入LLM时必须解决

### 🟠 P1 - 重要安全问题

- [ ] **问题4:实体归一过程的隐私泄露风险**
  - **影响**:实体归一需要跨事件比对人名/公司名,可能意外关联用户不希望关联的身份:
    - 例:用户在不同场合使用不同身份(本名/笔名/网名)
    - 例:同名不同人被错误合并,泄露无关人员信息
    - LLM调用时将多个事件上下文发送给第三方服务
  - **建议**:
    - 实体归一前需用户确认(特别是人物实体)
    - 提供"隔离身份"功能:用户可标记某些实体不参与归一
    - LLM调用时数据最小化:只发送必要字段,不发送完整事件内容
    - 本地LLM选项:支持Ollama等本地模型避免数据出境
    - 归一日志可审计和回滚
  - **优先级**:P1 - Phase 2必须解决

- [ ] **问题5:关联推理的数据投毒攻击**
  - **影响**:恶意用户可能通过精心构造的输入污染关联图谱:
    - 例:虚假的"竞对关系"误导商业决策
    - 例:伪造的"同校关系"进行社会工程学攻击
    - 团队共享场景下,单个成员的恶意输入影响整个团队
  - **建议**:
    - 关联置信度评分:标记自动推理vs人工确认
    - 关联来源追溯:记录每条关联的生成路径
    - 异常检测:识别批量创建的可疑关联
    - 用户反馈机制:允许标记错误关联
    - 团队版:管理员审核高敏感关联(竞对/风险)
  - **优先级**:P1 - Phase 3引入外部数据源前必须解决

- [ ] **问题6:多跳查询的拒绝服务(DoS)风险**
  - **影响**:恶意用户可构造深度查询消耗服务器资源:
    - 例:查询"A的朋友的朋友的朋友..."(10跳以上)
    - 邻接表方案在大图上的递归查询可能导致数据库锁死
    - 云端版本可能被滥用进行资源耗尽攻击
  - **建议**:
    - 查询深度限制:默认最大3跳,可配置但不超过5跳
    - 查询超时机制:单次查询不超过5秒
    - 结果集大小限制:最多返回1000个节点
    - 速率限制:单用户每分钟最多10次复杂查询
    - 查询成本预估:拒绝明显过载的查询
  - **优先级**:P1 - 云端版本上线前必须解决

### 🟡 P2 - 中等安全问题

- [ ] **问题7:导出功能的数据泄露风险**
  - **影响**:Graphviz DOT导出、CSV导出等功能可能导致:
    - 完整人脉图谱被导出到不安全位置
    - 导出文件未加密,易被第三方获取
    - 云端版本的导出文件存储在公开可访问的位置
  - **建议**:
    - 导出文件自动加密(ZIP加密或GPG)
    - 导出操作需二次确认和审计日志
    - 云端版本:导出文件临时存储,24小时后自动删除
    - 导出范围控制:允许部分导出而非全量
    - 水印/指纹:在导出数据中嵌入用户标识,追溯泄露源
  - **优先级**:P2 - Phase 3前解决

- [ ] **问题8:依赖供应链安全**
  - **影响**:项目依赖的第三方库可能存在漏洞:
    - SQLite驱动、LLM SDK、可视化库等
    - 传递依赖中的已知CVE漏洞
    - 恶意包投毒(typosquatting)
  - **建议**:
    - 集成Dependabot/Snyk自动扫描依赖漏洞
    - 锁定依赖版本(requirements.txt/package-lock.json)
    - 定期审计依赖树,移除不必要的依赖

  - **影响**:

## 2. 跨领域关注点

### 架构层面的安全观察

1. **三层架构的信任边界模糊**
   - 当前设计中,关联层(PromiseLink)直接读取记忆层(CarryMem)的数据,但未定义信任边界
   - 如果CarryMem被攻破,PromiseLink的关联图谱也会被污染
   - **建议**:定义明确的数据验证层,PromiseLink消费CarryMem数据时需验证数据完整性(HMAC签名)

2. **合规性的前瞻性考虑**
   - GDPR要求:
     - 数据可携带权(导出所有个人数据)
     - 被遗忘权(删除所有相关数据,包括关联)
     - 数据处理透明度(用户需知道哪些关联是如何生成的)
   - 中国《个人信息保护法》:
     - 敏感个人信息(人脉关系)需单独同意
     - 自动化决策(关联推理)需提供人工干预途径
   - **建议**:
     - 设计"数据删除"功能时,需级联删除所有关联
     - 提供"关联解释"功能,满足透明度要求
     - 用户协议中明确数据处理范围

### 产品层面的安全观察

3. **用户onboarding的安全教育缺失**
   - 商务人士可能不理解人脉图谱的敏感性
   - 可能在不安全的设备上使用(公共电脑、未加密硬盘)
   - **建议**:
     - 首次使用时强制安全教育(2分钟视频或交互式教程)
     - 检测不安全环境并警告(未加密磁盘、公共WiFi)
     - 提供"安全评分"功能,实时显示当前安全状态

4. **与CRM集成的数据边界问题**
   - CRM系统通常有自己的权限模型,PromiseLink如何对接?
   - 双向同步可能导致权限提升(PromiseLink中的私有数据同步到CRM后被团队看到)
   - **建议**:
     - 集成时明确数据流向:只读 vs 双向同步
     - 提供"同步白名单":用户选择哪些实体可同步
     - 同步前风险提示:"此操作将使XX数据对团队可见"

## 3. Top 5 关键安全问题

### 🥇 1. 本地数据库无加密 (P0)
**为什么排第一**:这是最基础的安全防护,缺失将导致所有其他安全措施失效。商务人脉图谱的敏感性决定了这是不可妥协的底线。

**行动项**:
- Week 1: 集成SQLCipher,实现基础加密
- Week 2: 实现密钥管理(系统密钥环集成)
- Week 3: 测试密钥轮换和数据库迁移

### 🥈 2. 团队共享权限模型缺失 (P0)
**为什么排第二**:这是商业化的前置条件,也是最容易出现IDOR漏洞的地方。权限模型的设计错误会导致大规模数据泄露。

**行动项**:
- 设计权限矩阵和访问控制策略
- 实现行级权限过滤(数据库层+API层)
- 编写权限测试用例(覆盖所有IDOR场景)

### 🥉 3. API Key管理不安全 (P0)
**为什么排第三**:LLM是核心功能,API Key泄露会导致财务损失和服务中断。这是用户最容易犯错的地方。

**行动项**:
- 实现API Key加密存储
- 添加使用量监控和异常检测
- 提供API Key轮换工具

### 4. 实体归一的隐私泄露 (P1)
**为什么排第四**:这是产品独特性的核心,但也是隐私风险最高的环节。错误的归一可能导致身份关联泄露。

**行动项**:
- 设计归一确认流程(用户审核)
- 实现"隔离身份"功能
- 本地LLM选项(避免数据出境)

### 5. 多跳查询DoS风险 (P1)
**为什么排第五**:云端版本的可用性保障,防止资源滥用。这是从免费版到付费版的关键安全边界。

**行动项**:
- 实现查询深度和超时限制
- 添加速率限制和成本预估
- 监控查询性能指标

```python
def evaluate_association_discovery(test_cases):
    results = {
        "precision": [],  # 发现的关联中有多少是正确的
        "recall": [],     # 应该发现的关联中有多少被发现了
        "latency": [],    # 发现关联的延迟(事件数)
        "false_positive_rate": [],  # 误报率
    }
    
    for case in test_cases:
        discovered = run_association_pipeline(case.events)
        
        # 计算precision和recall
        tp = len(set(discovered) & set(case.expected_associations))
        fp = len(set(discovered) - set(case.expected_associations))
        fn = len(set(case.expected_associations) - set(discovered))
        
        precision = tp / (tp + fp) if (tp + fp) > 0 else 0
        recall = tp / (tp + fn) if (tp + fn) > 0 else 0
        
        results["precision"].append(precision)
        results["recall"].append(recall)
    
    return {
        "avg_precision": mean(results["precision"]),
        "avg_recall": mean(results["recall"]),
        "f1_score": 2 * avg_precision * avg_recall / (avg_precision + avg_recall),
    }
```

```python
                "MEDIUM": 0,
                "LOW": 0,
            }
        }

        """
        user_action: "clicked" | "dismissed" | "ignored" | "snoozed"

            self.metrics["relevance_score"].append(1.0)
        elif user_action == "dismissed":
            self.metrics["relevance_score"].append(0.0)
            self.metrics["noise_ratio"] += 1
```

# PromiseLink 产品设计全面评审 - 系统架构师视角

## 1. 角色专业视角评审(系统架构师)

### 架构设计问题

- [ ] **问题1:三层架构的依赖方向存在隐患**
  - **影响**:当前设计"关联层消费记忆层数据但不修改记忆层代码",通过 relationship metadata 扩展存储。这种设计在 CarryMem 数据模型变更时会导致 PromiseLink 被动适配,且 metadata 作为弱类型存储会导致数据一致性问题。如果 CarryMem 删除或重构某个 memory,PromiseLink 的关联图谱会出现悬空引用。
  - **建议**:
    1. 在 CarryMem 层引入 Event Bus 或 Change Data Capture (CDC) 机制,发布数据变更事件
    2. PromiseLink 订阅这些事件并维护自己的物化视图(materialized view)
    3. 定义明确的 Integration Contract,包含版本化的数据模型和变更通知协议
    4. 实现双向健康检查:PromiseLink 定期验证引用的 CarryMem 数据仍然有效
  - **优先级**:P0 - 这是架构稳定性的基础

- [ ] **问题2:SQLite 邻接表方案的扩展性瓶颈明确但缺乏迁移路径**
  - **影响**:
    - 多跳查询(如"找出所有与张三有3度关联的人")需要递归 CTE,在 SQLite 中性能会随图规模指数级下降
    - 当实体数超过 10 万、关联数超过 50 万时,查询延迟会超过 5 秒
    - 无法利用图数据库的原生遍历算法(如 Dijkstra、PageRank)
    - 并发写入会导致数据库锁竞争,SQLite 的 WAL 模式只能缓解但无法根本解决
  - **建议**:
    1. **Phase 1 (MVP)**:保持 SQLite,但限制查询深度≤2跳,添加查询超时保护(500ms)
    2. **Phase 2**:引入混合存储架构
       - SQLite 存储热数据(最近 3 个月的事件和关联)
       - 冷数据归档到 DuckDB 或 Parquet 文件
       - 添加查询路由层,根据时间范围选择存储引擎
    3. **Phase 3**:当用户数据规模达到阈值时,提供迁移工具导出到图数据库
       - 本地版:Neo4j Community Edition (嵌入式模式)
       - 云端版:Amazon Neptune 或 Neo4j Aura
       - 提供 SQLite → 图数据库的无缝迁移脚本
    4. 在代码中抽象 Graph Repository 接口,隔离存储实现细节
  - **优先级**:P1 - MVP 可接受,但必须在 Phase 2 前解决

- [ ] **问题3:数据一致性保障机制缺失**
  - **影响**:
    - 实体归一过程中,如果合并两个实体 A 和 B 为 C,但关联更新失败,会导致图谱中出现孤岛
    - 事件处理管道的四个步骤(抽取→归一→发现→生成)如果中途失败,会产生不一致状态
    - 并发场景:两个事件同时引用同一实体,可能产生重复实体或关联
  - **建议**:
    1. 引入事务边界管理
       - 每个事件处理作为一个事务单元
       - 使用 SQLite 的 SAVEPOINT 实现子事务
       - 失败时回滚到事件处理前的状态
    2. 实现幂等性保障
       - 为每个事件生成唯一 ID(基于内容哈希)
       - 处理前检查是否已处理过
       - 关联创建使用 UNIQUE 约束防止重复
    3. 添加数据修复工具
       - 定期扫描孤立实体和悬空关联
       - 提供手动触发的一致性检查命令
       - 记录修复日志供审计
    4. 实现乐观锁机制
       - 为实体和关联添加 version 字段
       - 更新时检查版本号,冲突时重试或合并
  - **优先级**:P0 - 数据一致性是核心可靠性保障

- [ ] **问题4:与 CarryMem 的集成不够优雅,存在紧耦合风险**
  - **影响**:
    - 直接读取 CarryMem 的 SQLite 数据库文件,违反了封装原则
    - 如果 CarryMem 改变存储格式或加密方式,PromiseLink 会直接崩溃
    - 无法支持 CarryMem 的远程部署场景(如 CarryMem 在服务器,PromiseLink 在本地)
  - **建议**:
    1. 定义 CarryMem Integration API
       - 提供 RESTful 或 gRPC 接口暴露必要的数据
       - 包含:查询 memories、订阅变更通知、批量导出
       - 使用 API 版本控制(如 /v1/memories)
    2. 实现适配器模式

       - 模拟 CarryMem 的各种状态(正常、数据库锁定、版本不兼容)
       - 验证 PromiseLink 的降级行为
  - **优先级**:P1 - 影响长期可维护性

  - **影响**:

    - MCP 工具扩展在 AI 助手并发调用时会产生竞态条件
    - 云端版本的多用户并发写入会导致性能瓶颈
  - **建议**:
    1. **本地版并发控制**
       - 使用文件锁(flock)保证同一时刻只有一个进程写入
       - 读操作使用 SQLite 的 WAL 模式,允许并发读
       - CLI 命令添加 --wait 参数,等待锁释放而不是直接失败
    2. **云端版并发控制**
       - 使用消息队列(如 Redis Streams)序列化写操作
       - 实现 CQRS 模式:写操作异步处理,读操作查询物化视图
       - 添加分布式锁(如 Redis Redlock)保护关键操作
    3. **性能优化**
       - 批量操作使用事务合并多个写入
       - 实现写缓冲区,积累一定数量后批量提交
       - 为高频查询添加缓存层(如 Redis)
  - **优先级**:P2 - MVP 可以接受单进程限制,但 Phase 3 前必须解决

### 技术选型问题

- [ ] **问题6:LLM 调用的架构设计不够健壮**
  - **影响**:
    - 直接调用 LLM API 会导致响应时间不可控(可能 10 秒以上)
    - API 限流或故障时会阻塞整个事件处理流程
    - 成本无法预测和控制
  - **建议**:
    1. 实现异步处理架构
       - 事件提交后立即返回,后台异步调用 LLM
       - 使用任务队列(如 Celery + Redis)管理 LLM 任务
       - 提供任务状态查询接口
    2. 添加降级策略
       - 实现基于规则的 Fallback 抽取器(正则表达式 + 词典匹配)
       - LLM 失败时自动降级到规则引擎
       - 记录降级事件供后续人工审核
    3. 成本控制机制
       - 为每个用户设置月度 Token 配额
       - 实现智能缓存:相似事件复用抽取结果
       - 提供"快速模式"(仅规则引擎)和"精确模式"(LLM)供用户选择
    4. 可观测性
       - 记录每次 LLM 调用的延迟、Token 消耗、成功率
       - 提供成本仪表盘
  - **优先级**:P1 - 影响用户体验和成本可控性

- [ ] **问题7:实体归一算法的架构设计不清晰**
  - **影响**:
    - 文档中只提到"实体归一"但未说明算法选择
    - 不同算法(精确匹配、模糊匹配、语义相似度)的准确率和性能差异巨大
    - 错误的归一会导致图谱污染,且难以修复
  - **建议**:
    1. 实现分层归一策略
       - **L1:精确匹配**(名称+类型完全相同) - 自动合并
       - **L2:模糊匹配**(编辑距离 < 2 或拼音相同) - 标记为候选,需人工确认
       - **L3:语义匹配**(Embedding 相似度 > 0.85) - 仅提示,不自动合并
    2. 引入置信度机制
       - 每个实体归一决策记录置信度分数
       - 低置信度(<0.7)的合并需要人工审核
       - 提供"撤销合并"功能
    3. 实现增量学习
       - 记录用户的确认/拒绝决策
       - 训练个性化的归一模型(如 Siamese Network)
       - 定期更新模型权重
    4. 性能优化
       - 使用 Locality-Sensitive Hashing (LSH) 加速候选查找
       - 为实体名称建立倒排索引
       - 限制每次归一的候选数量(Top 10)
  - **优先级**:P0 - 这是核心功能的准确性保障

### 扩展性问题

- [ ] **问题8:缺乏微服务化的演进路径**
  - **影响**:
    - 当前单体架构在云端版本会遇到扩展瓶颈
    - 不同功能模块(实体抽取、关联发现、提醒生成)无法独立扩展
    - 无法支持多租户隔离和资源配额管理
  - **建议**:
    1. **Phase 1-2**:保持单体架构,但做好模块化准备
       - 使用清晰的模块边界和接口定义
       - 每个模块独立的配置和日志
       - 避免跨模块的直接数据库访问
    2. **Phase 3**:引入服务化架构(不是完全微服务)
       - 拆分为 3 个服务:
         - **EventProcessor Service**:处理事件输入和实体抽取
         - **GraphEngine Service**:管理图谱存储和关联发现
         - **AlertService**:生成和推送提醒
       - 使用消息队列(Kafka/RabbitMQ)解耦服务
       - 每个服务独立部署和扩展
    3. **Phase 4**:完全微服务化
       - 添加 API Gateway 统一入口
       - 实现服务注册与发现(Consul/Eureka)
       - 引入分布式追踪(Jaeger/Zipkin)
    4. 提供迁移工具
       - 单体 → 服务化的数据迁移脚本
       - 配置转换工具
       - 回滚方案
  - **优先级**:P2 - 不影响 MVP,但需要提前规划

### 数据架构问题

- [ ] **问题9:缺乏数据版本管理和迁移策略**
  - **影响**:
    - 数据模型变更时(如添加新的实体类型或关联类型)无法平滑升级
    - 用户的历史数据可能与新版本不兼容
    - 无法支持 A/B 测试或灰度发布
  - **建议**:
    1. 引入 Schema 版本管理
       - 在数据库中添加 schema_version 表
       - 每次模型变更递增版本号
       - 应用启动时检查版本兼容性
    2. 实现数据迁移框架
       - 类似 Alembic/Flyway 的迁移工具
       - 每个版本提供 upgrade 和 downgrade 脚本
       - 支持增量迁移和全量迁移
    3. 向后兼容策略
       - 新增字段使用默认值
       - 废弃字段保留但标记为 deprecated
       - 提供数据转换适配器
    4. 备份与恢复
       - 迁移前自动备份数据库
       - 提供一键回滚功能
       - 定期导出数据为 JSON 格式(便于跨版本恢复)
  - **优先级**:P1 - 影响产品的长期演进能力

- [ ] **问题10:多租户数据隔离架构未明确**
  - **影响**:
    - 云端版本的团队共享功能需要多租户支持
    - 不同租户的数据必须严格隔离,防止数据泄露
    - 性能隔离:一个租户的大量查询不应影响其他租户
  - **建议**:
    1. 选择合适的多租户模式
       - **方案 A:共享数据库 + 行级过滤**(推荐 MVP)
         - 所有表添加 tenant_id 字段
         - 使用 Row-Level Security (RLS) 或 ORM 全局过滤器
         - 优点:成本低,易于管理
         - 缺点:性能隔离较弱
       - **方案 B:独立数据库**(推荐生产环境)
         - 每个租户独立的 SQLite 文件或数据库实例
         - 优点:完全隔离,性能可控
         - 缺点:运维复杂度高
    2. 实现租户上下文管理
       - 在请求入口注入 tenant_id
       - 使用 Context Variable 或 Thread Local 传递
       - 所有数据库查询自动添加租户过滤条件
    3. 配额与限流
       - 为每个租户设置资源配额(存储、API 调用次数)
       - 实现租户级别的限流(Token Bucket 算法)
       - 超出配额时返回明确的错误信息
    4. 审计与监控
       - 记录跨租户访问尝试(安全审计)
       - 监控每个租户的资源使用情况
       - 提供租户级别的性能仪表盘
  - **优先级**:P1 - Phase 3 前必须解决

## 4. 整体评估

### 可行性评分:7.5/10

**评分理由**:
- ✅ **优势**:
  - 技术选型务实(SQLite + Python),降低了 MVP 的实现难度
  - 核心价值清晰("帮你发现你错过了什么"),差异化明显
  - 分阶段路线图合理,避免了过度工程化
  - 本地优先的架构符合隐私敏感用户的需求

- ⚠️ **风险**:
  - 数据一致性和实体归一的准确性是两大技术难点,需要大量测试和迭代
  - 与 CarryMem 的集成方式存在紧耦合风险,影响长期可维护性
  - SQLite 的扩展性瓶颈会在用户数据增长后显现,需要提前规划迁移路径
  - LLM 调用的成本和延迟可能影响用户体验

### 最大风险点

**实体归一的准确率问题**

这是整个系统的"阿喀琉斯之踵":
- 如果归一过于激进,会错误合并不同实体(如"张伟"和"张伟"),污染图谱
- 如果归一过于保守,会产生大量重复实体,降低关联发现的价值
- 错误的归一决策难以修复,因为关联已经建立
- 用户对错误的容忍度很低,一次错误可能导致信任崩塌

**缓解措施**:
1. 在 MVP 阶段使用保守策略(仅精确匹配),牺牲召回率保证准确率
2. 提供明确的"撤销合并"和"手动合并"功能,让用户参与决策
3. 实现置信度机制,低置信度的合并需要人工确认
4. 建立测试数据集,持续评估归一算法的 Precision 和 Recall

### 核心建议

1. **架构层面**:
   - 优先解决数据一致性问题,实现事务边界管理和幂等性保障
   - 定义 CarryMem Integration Contract,解耦两个系统
   - 抽象 Graph Repository 接口,为未来迁移到图数据库做准备

2. **技术实现层面**:
   - 实现分层实体归一策略,平衡准确率和召回率
   - 为 LLM 调用添加异步处理和降级策略
   - 引入 Schema 版本管理,支持数据模型的平滑演进

3. **产品策略层面**:
   - 在 MVP 阶段聚焦核心场景(手动输入 + 简单关联),验证价值假设
   - 提供"导入示例数据"功能,解决冷启动问题
   - 实现智能提醒策略,避免过度打扰用户

4. **风险控制层面**:
   - 建立实体归一的测试数据集,持续评估算法性能
   - 为关键操作添加审计日志,便于问题排查
   - 提供数据导出功能,降低用户的迁移成本(避免锁定)

**总结**:PromiseLink 的产品定位和技术方向是正确的,但在数据一致性、实体归一准确性、系统扩展性三个方面存在明显的架构风险。建议在 MVP 阶段优先解决 P0 问题,并为 Phase 2/3 的演进提前规划技术路径。如果能够在实体归一算法上取得突破,这个产品有很大的成功潜力。

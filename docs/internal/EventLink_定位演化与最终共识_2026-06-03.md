**日期**：2026-06-03
**类型**：产品定位决策文档
**参与方**：林总（产品负责人）、李总（战略顾问）、CarryMem团队（技术/产品舵手）

## 📌 TL;DR

- EventLink定位经过三次演化，最终确定为**私密关系经营助手**
- 三次调整方向一致，都在降低冷启动门槛，只是切入角度不同：数据主权→录入意愿→心理阻力
- Slogan终稿：主"让重要的人，不止停留在微信里" + 副"记住需要，兑现承诺，让合作自然发生"
- 核心逻辑转变：第一问从"他能为我做什么"→"他正在关心什么/我答应过他什么"
- PoC验证指标从技术指标改为产品指标：承诺完成率≥50%、4周留存≥60%
- 许总场景保留TTS/语音（不随李总"轻量演示"降级）
- 渐进路线已确认：PoC先利他闭环，Phase2再开资源匹配（李总已认同）

## 一、三次定位演化的完整时间线

| 阶段 | 时间 | 定位 | 核心逻辑 | 解决的问题 | 遗留问题 |
|------|------|------|---------|-----------|---------|
| **原版PRD** | v3.2及之前 | AI合作关系经营助手 | 事件→实体→关联→提醒 | 产品从0到1的定义 | "提醒你联系谁"不是"帮你做成什么" |
| **第1次调整** | 2026-06-02 | AI资源关系经营与匹配平台 | 人脉+资源供给+需求匹配 | 回答"联系他干嘛"的问题 | 撮合平台的双边密度+隐私焦虑 |
| **第2次调整** | 2026-06-02 | 私密资源经营助手 | 你的需求匹配你人脉的供给 | 隐私焦虑（数据在自己手里）+冷启动（你本来就认识这些人） | 仍然是"调用"逻辑，录入时有功利感 |
| **第3次调整** | 2026-06-03 | **私密关系经营助手** | 先成就关系，再促成合作 | 功利感（"给朋友标价"）+录入意愿（更愿意记录"对方关心什么"） | 待验证：利他闭环能否自然沉淀出合作 |

### 三次调整的本质

三次调整方向一致，都在回答同一个问题：**用户凭什么第一次就录入真实信息？**

| 版本 | 用户心理 | 录入动力 |
|------|---------|---------|
| 资源撮合 | "我为什么要公开人脉？" | ❌ 阻力极大 |
| 私密助手 | "数据在自己手里，可以记" | ⚠️ 理性认可但感性犹豫 |
| 关系价值 | "我想成为更靠谱的人" | ✅ 内在驱动 |

## 二、定位调整的核心逻辑

### 2.1 从"调用"到"成就"

```
旧逻辑（自利视角）：
  我认识谁 → 他有什么资源 → 我有什么需求 → 系统建议我去调用他

新逻辑（利他视角）：
  我认识了谁 → 他正在关心什么 → 我答应过他什么 → 我能先为他做什么 → 双方是否自然出现合作
```

这不是改功能，是改**产品与用户交互的第一个动作**：

| 旧版第一个问题 | 新版第一个问题 | 用户感受差异 |
|--------------|--------------|------------|
| "他有什么资源？" | "他正在关心什么？" | 从审讯变关心 |
| "你需要什么？" | "你答应过什么？" | 从索取变承诺 |
| "帮你匹配资源" | "帮你兑现承诺" | 从利用变靠谱 |

### 2.2 关系管理是入口，资源智能是差异化

两个定位不矛盾，是同一个产品的两个层次：

> - **你在解决什么问题？** 商务关系经营（认识那么多人，没有下文）
> - **你靠什么解决？** 自有资源智能（AI帮你把人脉转化为可调用的能力）
> - **用户凭什么开始？** 利他闭环（帮助我成为更靠谱的人）

用比喻说：关系管理是**地图**，资源管理是**导航**。没有地图导航无从谈起，但用户真正需要的是导航。

### 2.3 利他不是"不做资源"，是"先积累信任再开放能力"

```
PoC（阶段0）：承诺兑现闭环
  → 用户价值："我不会忘记重要的人，也不会忘记答应过他的事"
  → 功能：交流记录、对方关注点、我的承诺、行动提醒、反馈记录

Phase1（阶段1）：双向价值关系卡
  → 用户价值："我知道对方需要什么，也知道我为他做过什么"
  → 功能：帮助记录、关系时间线、合作方向候选、值得关心的人提醒

Phase2（阶段2）：善意连接建议
  → 用户价值："当两个我关心的人可能互相帮助时，我能在尊重双方意愿的前提下促成连接"
  → 功能：经确认的需要与能力、双方意愿确认、引荐建议

Phase2+（阶段3）：私密资源经营能力
  → 用户价值："系统更准确地帮我判断哪些合作连接值得推进"
  → 功能：Resource/Demand/MatchResult、合作结果复用、关系冷却和透支保护
```

## 三、Slogan终稿

### 3.1 演化过程

| 版本 | 主slogan | 副slogan | 问题 |
|------|---------|---------|------|
| 原版 | 让重要的人，不止停留在微信里 | 每次见面，都不白费 | 副slogan"不白费"是负面表述 |
| CarryMem建议 | 见面只是开始 | 下次联系的理由，我们给你找好了 | 行动感强但产品指向模糊 |
| 第2次调整 | 让重要的人，不止停留在微信里 | 连接资源，匹配需求，推进合作 | 副slogan是产品语言不是用户语言 |
| **终稿** | **让重要的人，不止停留在微信里** | **记住需要，兑现承诺，让合作自然发生** | 主slogan提问题，副slogan给答案 |

### 3.2 终稿选择理由

1. **主slogan保留"让重要的人，不止停留在微信里"**——李总三次建议都保留，因为它击中真实痛点，且在利他视角下反而更对（不只是"资源停留在微信里"，更是"关心和承诺停留在微信里"）
2. **副slogan"记住需要，兑现承诺，让合作自然发生"**——
   - "记住需要"= 利他动作（记住对方的）
   - "兑现承诺"= 自律动作（完成自己的）
   - "让合作自然发生"= 结果（不强求、水到渠成）
   - 比"连接资源，匹配需求"更温和，比"不白费"更正面

### 3.3 备选文案（按场景选用）

| 文案 | 适合位置 |
|------|---------|
| 先记住对方的需要，再发现彼此的合作 | 产品理念页 |
| 不辜负每一次认识，不错过每一次合作 | 路演/品牌传播 |
| 先帮助，后连接；先信任，后合作 | 价值观表达 |
| 记住重要的人，也记住能为他做什么 | 新用户引导 |
| 把人看作值得被成就的关系 | 品牌宣言 |

## 五、数据模型映射（PoC→Phase1→Phase2）

### 5.1 PoC：映射到现有表

| 李总第3版业务对象 | 映射到PRD v3.6 | 说明 |
|----------------|--------------|------|
| Person | Entity(entity_type=person) + Person.properties JSONB | 加concern/promise字段 |
| Interaction | Event(event_type=meeting/call/manual) | raw_text存交流内容 |
| Concern | Todo(todo_type=care) + Entity.properties.concern JSONB | 关注点提取后标待确认 |
| Promise | Todo(todo_type=promise) | 含due_at、completion_note |
| Contribution | Todo(todo_type=help) + Entity.properties.contribution JSONB | 帮助记录 |
| Feedback | Todo(todo_type=followup) + 关联的metadata | 对方回应 |
| Todo | Todo表（类型扩展） | 新增promise/help/care/followup/cooperation_signal |

### 5.2 Phase1：新增业务视图

- Person/Interaction/Concern/Promise/Contribution/Feedback 作为**业务层视图**（VIEW或Python dataclass）
- 底层仍然映射到Event/Entity/Todo，但业务代码只操作业务对象
- 逐步验证哪些字段需要独立索引/查询

### 5.3 Phase2：独立实体表

- Resource/Demand/MatchOpportunity 拆为独立表
- 需要独立查询的场景（"谁能提供X类资源？"）出现后才拆

### 6.3 为什么产品指标比技术指标更重要

技术指标回答"能不能做"，产品指标回答"该不该做"。

PoC最大的风险不是"AI抽不准"，而是"用户不愿录入真实信息"。如果技术指标全过但4周留存<30%，说明产品方向需要调整。如果技术指标勉强过但承诺完成率>60%且4周留存>60%，说明方向对了，技术可以迭代。

## 七、许总场景的特殊处理

李总第3版建议TTS"轻量演示即可"、语音录入"许总试点可保留"。但对许总来说，这不是增强，是他愿意用的前提条件。

| 李总建议 | 我方判断 | 理由 |
|---------|---------|------|
| TTS轻量演示 | ⚠️ 许总保留完整TTS | "眼神不好喜欢听"是许总的刚需，不是锦上添花 |
| 语音录入许总试点 | ✅ 同意 | 和许总场景完全匹配 |
| 名片仅为身份补充 | ✅ 同意 | 但保留扫码能力（许总线下见人扫名片是高频动作） |
| 首次体验从"记录一次交流"开始 | ✅ 采纳 | 对许总更友好——语音一句话即可 |

### 许总的杀手场景

> "我知道老王能搞定装修，但三个月没联系了，手头刚好有个客户要装修。"

这个场景同时覆盖了：
- **利他闭环**：我答应过给老王介绍客户（promise）→ 我该兑现了
- **资源智能**：我认识能搞装修的人 → 客户有装修需求 → 自然匹配
- **人脉提醒**：三个月没联系了 → 值得关心

**利他和资源不是对立的，是同一个场景的两面。** 先从利他面切入（"我该兑现承诺了"），资源面自然浮现（"他能做什么"）。

## 八、AI能力边界（李总第3版的核心贡献）

### 8.1 AI应该做的

| 能力 | 要求 |
|------|------|
| 从一句话识别联系人 | 支持用户修正 |
| 提取对方关注点 | 必须由用户确认 |
| 提取用户承诺 | 必须由用户确认 |
| 提议提醒时间 | 用户可修改 |
| 生成温和回应建议 | 用户编辑后使用 |
| 基于历史互动提示"值得关心" | 提供依据 |
| 识别潜在合作信号 | 只作为提示，不作为事实 |

### 8.2 AI不应该做的

| 行为 | 原因 |
|------|------|
| 自动判定对方掌握何种资源 | 信息不够且易冒犯 |
| 自动建议用户索取资源 | 违背首期利他逻辑 |
| 自动将两个人撮合 | 缺少双方同意 |
| 自动发送微信或邮件 | 用户必须掌控 |
| 自动推断人格、禁忌、利益关系 | 错误会伤害真实关系 |
| 把推测写成确定事实 | 必须区分AI推测与用户确认 |

### 8.3 输出语言规则

**错误示例**：
> 王主任拥有社区资源，建议调用。

**正确示例**：
> 王主任此前提到社区长者活动的运营需求。你已答应发送方案，建议先兑现承诺；如果她反馈积极，再进一步了解是否适合开展试点合作。

## 九、首次体验重设计

### 9.1 旧流程（删除）

```
首次打开 → 扫第一张名片 → 展示关联发现 → 生成破冰建议
```

问题：名片信息不足、一个人无法形成关联、展示出来的机会显得牵强。

### 9.2 新流程（采纳）

**第1屏：信任承诺**
> 这是你的私人关系助手。你记录的人和交流内容，只用于帮助你更好地回应重要关系。不公开、不匹配给其他用户、不自动联系任何人。
> 
> 按钮：**记录一次重要交流**

**第2屏：一句话输入**
> 最近，你和谁有过一次值得记住的交流？
> 
> 输入示例：今天认识王主任，她在做社区长者活动，提到缺少年轻化运营和线上宣传。我答应明天发一份活动方案给她。
> 
> 按钮：**帮我整理下一步**

**第3屏：用户确认重点**
> 我整理出了这些重点，请你确认

| 项目 | 提取内容 |
|------|---------|
| 联系人 | 王主任 |
| 她正在关心 | 社区长者活动运营与线上宣传 |
| 我答应的事 | 发送一份活动方案 |
| 建议完成时间 | 明天下午前 |

**第4屏：立即获得价值**
> 你可以先为王主任做一件小事
> 
> 建议行动：明天下午前发送一份简版活动案例
> 
> 可复制话术：（生成温和微信表达）
> 
> 按钮：添加提醒 / 复制话术 / 稍后处理

## ⚠️ 待确认 / 假设 / Non-goals

### 待确认
- 许总是否认同杀手场景"别把我已经知道的资源忘了"？
- PoC首批5-10位种子用户，每人每周产生多少会议/名片记录？

### 已确认
- ✅ 定位确定为"私密关系经营助手"（林总审定，"价值"太功利，"经营"更有动作感）
- ✅ 副slogan确定为"记住需要，兑现承诺，让合作自然发生"
- ✅ 渐进路线：PoC先利他闭环，Phase2再开资源匹配（李总已认同）

### 假设
- EventLink定位为"私密关系经营助手"，核心是利他闭环，资源能力是后续演进
- PoC不改核心数据模型（不加新表），只扩展JSONB字段和Todo类型
- 首批用户更可能从会议/交流记录中确认关注点和承诺，而非主动手写录入
- 名片保留为信息采集方式之一，但不作为MVP首次价值演示路径
- 许总TTS/语音能力是刚需，不随"轻量演示"降级

### Non-goals（首期不做）
- 不做资源撮合平台（双边匹配、公开资源、自动引荐）
- 不做社交网络（关系图谱公开、好友发现、动态分享）
- 不做CRM系统（线索管理、销售漏斗、团队协作）
- 不做自动触达（系统不自动发微信/邮件/短信给第三方）
- 不做AI替用户定义关系（不自动推断人格、禁忌、合作意愿）

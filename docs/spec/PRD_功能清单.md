## 3. 交付阶段划分

### 3.0 PoC / Phase 1 / Phase 2 范围定义

> **原则**：PoC验证核心假设，Phase 1交付可用产品，Phase 2扩展能力边界。每阶段有明确的验收标准和退出条件。

| 维度 | PoC（概念验证） | Phase 1（最小可用产品） | Phase 2（能力扩展） |
|------|---------------|---------------------|-------------------|
| **目标** | 验证AI能否从沟通记录提取商机+生成行动建议+识别资源线索 | 许总团队可日常使用的完整产品 | 多场景/商业化 |
| **定位** | 个人商务关系经营助手 | 个人商务关系经营助手 | 个人商务关系经营助手 |
| **用户** | 内部测试(≤5人) | 许总团队(≤100人) | 付费客户(1000+人) |
| **时长** | 3周 | PoC后6周 | Phase 1后持续迭代 |
| **数据** | 预置样本数据+手动录入 | CSV导入+名片扫描+语音录入 | 全量日历同步+邮件解析 |
| **存储** | SQLite（零新依赖，和CarryMem一致） | PostgreSQL 15 + Redis 7 | PG + Redis + (按需ES) |
| **前端** | API+Swagger UI | 名片小程序(原生+H5) | 独立小程序+Web管理后台 |
| **部署** | Docker单机 | Docker Compose(单节点) | K8s(多节点+监控) |
| **认证** | API Key | JWT RS256+临时授权码 | OAuth2（无RBAC） |
| **CarryMem** | NullMemoryProvider | 可选接入 | 完整集成 |
| **资源匹配** | 六维匹配(JSONB) | 六维匹配+需求录入 | 向量语义搜索+资源聚合 |
| **敏感度** | 2级(可匹配/不匹配) | 2级 | 2级(不变) |

**PoC退出条件**（全部满足才进入Phase 1）：
1. ✅ LLM实体抽取准确率≥90%（100条样本）
2. ✅ 实体归一误合并率<5%
3. ✅ 关联发现F1>0.65
4. ✅ 端到端延迟：名片→Todo<5秒，会议→Todo<60秒
5. ✅ 许总团队确认"这个方向对"
6. ✅ 承诺兑现闭环验证率≥50%（原"资源线索确认率"，调整为聚焦承诺兑现核心闭环）
7. ✅ 4周持续使用率≥60%（种子用户4周内持续活跃比例）
8. ✅ 输入分类准确率≥95%（F-44 input_scope分类器指标）
9. ✅ 承诺责任人识别准确率≥90%（F-45 Promise双向动作模型指标）

**PoC退出条件测试方法学（C1 [PM意见]）**：

| 维度 | 说明 |
| ---- | ------------------------------------------------------------ |
| 测试责任人 | 产品经理主导，开发配合 |
| 测试数据 | 100条真实会议记录（脱敏后） |
| 通过标准 | PM+Arch双签确认 |
| 时间窗口 | Sprint 2结束前完成 |

**Phase 1退出条件**（全部满足才进入Phase 2）：
1. ✅ 许总团队DAU≥30人，持续2周
2. ✅ TTS播报使用率≥40%（播报次数/画像查询次数）
3. ✅ Todo完成率≥50%
4. ✅ 语音录入成功率≥85%
5. ✅ 零P0安全漏洞
6. ✅ "你的需求匹配你人脉"场景验证通过（≥3个真实匹配案例）

**明确排除**（私密助手不需要，各阶段均不规划）：
- ❌ RBAC权限模型 / 资源授权共享 / 团队协作
- ❌ 他人可提供资源匹配（只能匹配自己人脉的供给）
- ❌ 多租户隔离 / 企业管理看板
- ❌ 跨项目/跨团队资源撮合
- ❌ 语音交互支持多用户场景[v4.4新增]（语音交互仅支持单用户，不做多人语音识别/方言适配）

## 3.1 功能需求清单

### 3.1 MVP-Core（PoC必须，3周交付）

> **核心假设**：验证AI能否从非结构化沟通记录中自动提取商机实体、发现关联关系，并生成可执行的行动建议，形成从数据到行动的完整闭环。

### 3.1 核心引擎

#### F-01: 事件上报接口

| 属性     | 说明                                                                                                                                                                                                    |
| ------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述   | 接收多来源事件数据（名片JSON、音频转写文本、会议纪要），标准化为统一Event Schema，并根据event\_type路由至对应处理管线                                                                                                                              |
| API    | `POST /api/v1/events`                                                                                                                                                                                 |
| 输入     | event\_type, source, title, timestamp, raw\_text, metadata                                                                                                                                            |
| 输出     | event\_id, status(processing), pipeline(路由管线名), estimated\_completion                                                                                                                                 |
| 事件语义路由 | card\_scan→轻量管线(秒级响应); meeting→深度管线(分钟级); call→要点管线(分钟级); manual→补全管线(异步)                                                                                                                             |
| 验收标准   | ① API响应时间P95<200ms ② 名片JSON解析准确率>95%（基于20张样本名片） ③ 支持card\_scan/meeting/call/manual四种event\_type ④ 重复事件（相同source+timestamp+title）自动去重，去重率100% ⑤ 路由准确率100%（event\_type与管线一一对应） ⑥ 返回结果包含pipeline字段标识路由目标 |

#### F-02: 实体抽取服务（管线化）

| 属性   | 说明                                                                                                                                                                                                                               |
| ---- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述 | 根据事件语义路由结果，走不同深度的抽取管线                                                                                                                                                                                                            |
| 主路径  | LLM抽取（Moka AI Claude Sonnet / OpenAI）                                                                                                                                                                                            |
| 降级路径 | spaCy + 自定义词典规则引擎                                                                                                                                                                                                                |
| 管线定义 | **轻量管线**(card\_scan): 仅抽取person/organization/technology三类实体，延迟P95<1s; **深度管线**(meeting): 抽取全部5类实体+话题标签(见§5.5)+决策项+行动项，延迟P95<5s; **要点管线**(call): 抽取person/organization+承诺项+待确认项，延迟P95<3s; **补全管线**(manual): 仅抽取缺失实体+属性丰富，延迟P95<2s |
| 验收标准 | ① LLM抽取准确率≥90%（基于100条人工标注样本） ② 降级路径抽取准确率≥75% ③ 轻量管线延迟P95<1s ④ 深度管线延迟P95<5s ⑤ 抽取结果包含实体类型、名称、属性字典 ⑥ 深度管线额外输出话题标签、决策项、行动项                                                                                                           |

#### F-03: 实体归一引擎

| 属性    | 说明                                                                                                                                     |
| ----- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述  | 5步级联归一算法，判断新实体是否与已有实体为同一对象                                                                                                             |
| 算法步骤  | Step 1: 精确匹配（confidence=1.0）→ Step 2: 别名扩展查找 → Step 3: 上下文相似度计算（公司+0.3, 技能+0.2, 共现+0.3）→ Step 4: 置信度分级处理 → Step 5: Human-in-the-Loop确认 |
| 置信度分级 | ≥0.85: 自动合并（记录日志，支持撤回）；\[0.70, 0.85): 进入待确认队列；<0.70: 创建新实体                                                                             |
| 撤回机制  | 所有自动合并支持一键撤回，撤回后实体恢复独立状态，关联关系还原，操作日志完整记录                                                                                               |
| 人工确认  | 展示完整属性对比 + AI归一推理 + 置信度分数 + 历史相似案例；操作项：确认归一/拒绝归一/稍后处理；确认结果反馈至模型持续改进                                                                    |
| 验收标准  | ① 精确匹配准确率100% ② 别名扩展召回率≥80%（基于50组别名对测试） ③ 上下文相似度计算延迟<500ms ④ 自动合并误合并率<5%（人工抽检100条） ⑤ 撤回操作成功率100%，数据一致性无损 ⑥ 待确认队列展示完整归一依据，操作响应<200ms    |

#### F-04: 关联发现引擎

| 属性   | 说明                                                                                                                                                 |
| ---- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述 | 基于共现分析和类型推断发现实体间8种关联关系                                                                                                                             |
| **用户视角语言** | **对外沟通时使用"主题互通"表达**（而非技术术语"关联发现"）。术语映射：话题标签=主题，关联发现=互通，实体图谱=主题网络。Phase 1 Plus计划：话题节点可视化（D3.js图谱组件） |
| 关联类型 | alumni, ex\_colleague, same\_city, competitor, tech\_overlap, deal\_link, risk\_link, supply\_chain                                                |
| 算法步骤 | A. 共现频率分析 → B. 类型推断（同公司→ex\_colleague, 同城市→same\_city, 技能重叠→tech\_overlap, 竞对DB匹配→competitor）→ C. 时间衰减+过滤（strength × exp(-λ × Δt), λ=0.01, 保留>0.3） |
| 验收标准 | ① Precision\@5>70% ② Recall\@10>60% ③ F1>0.65 ④ 单次关联发现延迟<2s ⑤ 8种关联类型全部可生成 ⑥ 关联强度计算误差<0.05（与人工标注对比）                                                 |

#### F-05: 商机匹配度计算

| 属性   | 说明                                                                                                                                                                                                                                 |
| ---- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述 | 五维打分法计算供需双方商机匹配度（含话题语义相似度维度），匹配维度不限于技术，也包括业务互补、资源对接等                                                                                                                                                                               |
| 维度   | 需求-能力关键词重叠(Jaccard相似度): +0.30; 行业领域一致性（见§5.4领域分类体系）: +0.25; 话题语义相似度（见§5.5话题标签体系）: +0.20; LLM语义理解: +0.15; 历史合作信号: +0.05                                                                                                             |
| 阈值   | ≥0.80: 强匹配（推送⚪机会Todo）; \[0.60, 0.80): 潜在机会（记录不推送）; <0.60: 不推送                                                                                                                                                                      |
| 验收标准 | ① 匹配度计算延迟<1s ② 强匹配推送准确率≥80%（人工评估50条） ③ Jaccard相似度计算与手工计算偏差<0.02 ④ 总分示例：0.30+0.25+0.20+0.15+0.05=0.95(95%) ⑤ 行业领域一致性评分规则：L1相同=0.25，L1相邻=0.15，L1不相关=0 ⑥ 话题语义相似度计算：余弦相似度≥0.7计满分0.20，\[0.4,0.7)按比例折算，<0.4计0 ⑦ 匹配维度覆盖技术供需/业务互补/资源对接三类场景 |

#### F-06: Todo生成与追踪引擎（原Action路由引擎/提醒生成引擎）

| 属性       | 说明                                                                                                                                                                                                                                                  |
| -------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 功能描述     | 基于关联发现结果和事件类型，生成差异化Todo项（信息型+行动型），支持状态追踪和完成度统计                                                                                                                                                                                                      |
| Todo类型   | **MVP范围（打法B）**: 🤝 Promise（承诺追踪）; 🔄 Followup（跟进待办）。**Phase 2+**: 新增🫶 Help（帮助建议）; 💙 Care（关怀提醒）; ✨ Cooperation_Signal（合作信号）; 🔴 Risk（风险预警）                                                                |
| 状态流转     | pending → in\_progress → done / dismissed / snoozed; snoozed到期后自动恢复pending                                                                                                                               |
| Action路由 | **MVP范围（打法B）**: card_scan→🤝 Promise; meeting→🤝 Promise+🔄 Followup(按会议类型A/B/C/D); call→🤝 Promise+🔄 Followup; manual→🤝 Promise增量。**Phase 2+**: 新增💙 Care/🫶 Help/✨ Cooperation_Signal/🔴 Risk生成能力                                                                |
| API      | `GET /api/v1/todos?status=pending&limit=20`; `PATCH /api/v1/todos/{id}` (更新状态); `GET /api/v1/todos/stats` (完成度统计); `POST /api/v1/todos/{id}/feedback`                                                                                               |
| 输出Schema | card\_save: `{todo, actions[🔄]}`; meeting: `{todos[], actions[🔄{task,who,when,priority}], decisions[], pending_decisions[]}`; call: `{todos[], commitments[], followups[🔄]}`; manual: `{todos_delta[], merge_suggestions[]}` |
| 验收标准     | ① Todo生成延迟<500ms ② **MVP：2种Todo类型（Promise/Followup）** ③ 状态流转API响应<200ms ④ 完成度统计API响应<500ms ⑤ meeting Todo包含🔄行动项列表 ⑥ 不同event\_type的输出Schema符合各自定义 ⑦ Todo去重率100%（同一关联24小时内不重复生成） ⑧ **Phase 2+ 完成6种Todo类型全部可生成**                                                 |

#### F-07: Todo反馈与追踪闭环

| 属性    | 说明                                                                                              |
| ----- | ----------------------------------------------------------------------------------------------- |
| 功能描述  | 用户对Todo进行状态更新和反馈，反馈数据用于优化推荐算法，追踪数据用于完成度统计                                                       |
| 状态操作  | 信息型: done / dismissed; 行动型: in\_progress / done / dismissed                                     |
| 反馈类型  | useful / not\_useful                                                                            |
| 完成度统计 | 按日/周/月统计Todo完成率，区分信息型和行动型                                                                       |
| 验收标准  | ① 状态更新API响应<200ms ② 反馈数据持久化存储，保留期≥365天 ③ 完成度统计API响应<500ms ④ 同一Todo不可重复提交反馈 ⑤ 行动型Todo完成率可按时间维度查询 |

#### F-44: input_scope输入分类器

| 属性 | 说明 |
| ---- | ------------------------------------------------------------ |
| 功能描述 | 在事件处理管线入口对输入材料进行分类，区分身份资料/真实互动/会议纪要/产品反馈/内部评审等类型。分类结果影响后续管线路由（partner_feedback和internal_review不进入关系推进主流程） |
| scope枚举 | identity / relationship_interaction / meeting_minutes / partner_feedback / internal_review / intent_document / execution_record / result_record / **voice_query[v4.4新增]**（共9种） |
| PoC范围 | 启用前6种（identity/relationship_interaction/meeting_minutes/partner_feedback/internal_review/intent_document），后3种保留枚举值不在UI展示 |

## 二、逐项判定

### Q1: Pipeline 性能 — 是否随实体数增长？超时必要性？

**用户原话**: Pipeline 所花的时间会不会随着实体的增多而线性增长甚至指数级增长？有没有做过性能测试？Pipeline 的时间限制 max45 或 90 秒的作用和必要性到底是什么？

**技术分析**:
- 当前 Pipeline 是**事件级别串行处理**：每个事件独立走 13 步，不随实体总数增长
- 但单事件内 Step02(实体提取) + Step10(关联发现) 会查询已有实体做合并/匹配，**理论上 O(n)**
- 实测：5 个实体时 ~40-50s/事件；100+ 实体时可能明显变慢
- **当前无全局 PIPELINE_TIMEOUT**，仅 LLM 单次调用有 60s 超时
- E2E 测试的 90s 超时是测试脚本自己设的，非系统配置

| 角色 | 判定 |
|------|------|
| PM | 中优先级 — 用户感知到"慢"但不知道为什么，需给预期管理 |
| Architect | 需加全局超时 + 性能基线 + 实体数阈值告警 |
| Tester | 缺少压力测试（10/50/100 实体场景） |
| Coder | 工作量 0.5d |

**判定: P1** — 加 `PIPELINE_TIMEOUT=120s` 全局超时 + 前端超时提示 + 压力测试用例

### Q3: LLM 不可用时通知用户

**用户原话**: LLM 不可用或不稳定时，能不能第一时间通知用户？而不是用降级的办法来处理导致处理质量很差？

**现状分析**:
- LLMClient 有三级降级策略：Moka AI → 本地 sentence-transformers → hash 伪 embedding
- **降级后质量确实差**：hash embedding 导致语义搜索失效，entity extraction 可能返回空
- 前端轮询只看 event.status，**不区分"成功但质量差"和"完全失败"**
- 无 `llm_degraded` 或 `quality_warning` 状态码

| 角色 | 判定 |
|------|------|
| PM | **P0** — 用户宁可选择"稍后再试"也不愿接受低质量结果 |
| Architect | 需新增 pipeline 状态: `degraded` + quality_score 字段 |
| Tester | 需增加 LLM 不可用场景测试 |
| Coder | 工作量 1d |

**判定: P0** — 新增 LLM 降级通知机制（状态码 + 前端警告横幅 + 重试按钮）

### Q4: 承诺"待确认"画面 API 422 错误

**用户原话**: "承诺"的"待确认"画面有 api error 422 错误。

**根因定位**:
- 前端调用 `GET /todos/pending-confirmations` 时参数格式可能与后端 Pydantic 模型不匹配
- 422 = Validation Error，通常是字段类型/必填项不匹配
- **需要实际复现并查看请求/响应详情**

| 角色 | 判定 |
|------|------|
| PM | **P0** — 功能完全不可用 |
| Tester | 必须先复现，抓包确认请求差异 |
| Coder | 工作量 0.5d（如果是简单参数问题）|

**判定: P0** — 立即排查修复 422 错误

### Q5: 待办"进行中"Tab 数据

**用户原话**: "待办"的"进行中" tag 下，什么情况下能看到数据？

**现状分析**:
- Todo 的 `status` 字段值域: `pending` / `in_progress` / `done` / `dismissed`
- **当前没有任何代码路径将 todo 设为 `in_progress`**
- Pipeline 生成的 todo 默认都是 `pending`
- 用户手动"开始做"的操作入口**不存在**

| 角色 | 判定 |
|------|------|
| PM | 功能半完成 — 有 Tab 但永远为空，用户困惑 |
| UI Designer | 需加"开始处理"按钮或自动转换逻辑 |
| Coder | 工作量 0.5d |

**判定: P1** — 补全 `in_progress` 状态流转（加操作入口或自动规则）

### Q6: 关怀提醒位置

**用户原话**: 关怀提醒是放在首页还是应该放在人脉？

**多角色共识**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| A. 留在首页 | 发现效率高，每日必看 | 首页信息过载 |
| B. 移到人脉页 | 与实体强关联，上下文完整 | 发现率降低 |

**判定: 保持首页 + 人脉页双展示**
- 首页显示 Top 3 条摘要（已有）
- 人脉页 Entity Detail 内显示该实体的关怀提醒（增强）
- 不需要大改，仅补充人脉侧数据

**判定: P2** — 小改动，人脉 Detail 补充关怀提醒

### Q7: 今日事件"未命名"

**用户原话**: 今日事件中"12:21 未命名"是什么情况？

**根因**:
- Event 创建时 `title` 默认值为 `"未命名"`（model 定义）
- Pipeline Step01 或 Step12 应生成标题，但如果 **LLM 返回异常/空**，标题保持默认
- 属于静默失败 — 不报错但不生成标题

| 角色 | 判定 |
|------|------|
| PM | 用户困惑："我明明输入了内容，为什么显示未命名？" |
| Coder | 需在 pipeline 结束时兜底：若 title 仍为"未命名"，从 raw_text 截取前 20 字 |

**判定: P1** — 标题兜底逻辑（0.2d）

### Q8: 信用分全 D/50

**用户原话**: 人脉的关系信用分都是一样的 D，50，没有通过新的事件来改变这个分数吗？我们的 e2e 测试真的都覆盖了所有场景吗？

**根因**:
- 信用分算法中 `my_fulfillment_rate` 和 `their_fulfillment_rate` 在**无 fulfilled promise 数据时默认 0.5**
- `consistency` 在 interaction < 2 时默认 50
- **E2E 测试从未执行过"完成承诺"操作**，所以所有信用分计算结果相同
- 公式本身没问题，是**测试场景缺失**导致无法验证动态变化

| 角色 | 判定 |
|------|------|
| PM | **P0** — 核心功能看起来"坏了"，虽然公式对但数据不够 |
| Tester | **严重测试缺口**：缺少"承诺→兑现"完整生命周期测试 |
| Coder | 信用分算法无需改动，需补测试场景 |

**判定: P0** — 补充"承诺兑现"E2E 场景 + 验证信用分动态变化

### Q10: 页面间关联跳转

**用户原话**: 事件、待办、人脉、承诺，能根据关联相互跳转吗？

**现状（已实现 vs 缺失）**:

| 跳转链路 | 状态 |
|----------|------|
| 录入 → 待办 | ✅ 已有 |
| 首页 → 人脉（健康度卡片） | ✅ 已有（但无参数传递） |
| 待办 → 人脉（点击关联实体） | ❌ 缺失 |
| 承诺 → 人脉（点击关联实体） | ❌ 缺失 |
| 事件 → 待办展开 | ❌ 缺失 |
| 人脉 → 该实体待办/事件 | ❌ 缺失 |

**技术限制**: Taro `switchTab` 不支持 URL 参数传参，需用 globalStore 或 EventBus

| 角色 | 判定 |
|------|------|
| PM | **P1** — 核心交互缺陷，严重影响使用流畅度 |
| Coder | 工作量 1d（引入全局状态管理） |

**判定: P1** — 实现核心跳转链路（至少 Todo↔Entity、Promise↔Entity）

### Q11: 事件 ID 对用户无意义

**用户原话**: 事件 ID 过长，而且对用户没有意义。

**分析**:
- 事件 UUID（36字符）**仅在调试日志中出现**
- 前端用户界面**不显示事件 ID**
- 如果用户在某处看到了 ID，那是开发遗留

**判定: 不需要做** — 用户界面上本就不应显示内部 ID。如有泄漏需移除。

### Q12: 测试场景不全面

**用户原话**: 事件可以模拟去年的输入、两个月前、一个月前、上周和今日。不用都在同一天输入。也可以模拟用户在上周完成了待办后点击某项已完成等真实世界发生的场景。

**测试缺口清单**:

| 缺失场景 | 重要性 | 影响 |
|----------|--------|------|
| 历史事件（去年/上月/上周） | 高 | 健康度"最近交互"计算不准 |
| 待办完成操作 (`status → done`) | **极高** | 信用分不变的根本原因 |
| 待办逾期操作 (`fulfillment_status → overdue`) | 高 | 健康度 promise 维度无效 |
| 承诺确认/拒绝操作 | 高 | 承诺页面功能未验证 |
| 多日累积数据场景 | 中 | Dashboard 日视图边界情况 |
| LLM 不可用降级场景 | 高 | Q3 相关 |

| 角色 | 判定 |
|------|------|
| Tester | **P0** — 当前 E2E 只覆盖了"创建"未覆盖"完整生命周期" |
| Coder | 工作量 2d（编写新测试脚本 + 执行验证） |

**判定: P0** — 补充完整生命周期 E2E 测试套件

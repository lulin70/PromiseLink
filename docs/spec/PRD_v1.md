## 1. 产品概述

### 1.1 产品愿景

**EventLink — 见面只是开始。**

> 主Slogan：让每一次连接，都有回应。
> 副Slogan：记住需要，兑现承诺，让合作自然发生。
> 品牌短句：见面只是开始。
> （路演/传播场景可继续使用"让重要的人，不止停留在微信里"）

EventLink 是 **AI驱动的个人商务关系经营助手——先成就关系，再促成合作**。通过一句话记录、会议纪要和合作档案导入，帮助用户沉淀关键联系人、记住对方关注点、兑现自己的承诺、主动提供帮助，把每一次认识逐步推进为真实合作。

> **Slogan使用场景说明**：主Slogan"让每一次连接，都有回应"用于产品核心价值传达和用户心智建立；路演/传播场景可继续使用"让重要的人，不止停留在微信里"以强化差异化定位。

**产品侧重"人"而非"知识"** — 客户买单的驱动力来自"帮我经营关系"，知识是经营关系过程中的自然沉淀。

**产品是"私密助手"而非"撮合平台"** — 所有数据归用户个人所有，匹配逻辑是"你的需求匹配你人脉的供给"，不存在资源公开、授权共享或团队协作场景。资源永远附着在具体人身上（"张总能搞定工厂"），不脱离具体人独立存在。

**首期核心闭环：互动 → 关注 → 承诺 → 帮助 → 反馈**

> 从利他视角出发：我认识了谁 → 他正在关心什么 → 我答应过他什么 → 我能先为他做什么 → 双方是否自然出现合作

产品不只是记录联系人、提取待办、推荐机会，更帮助用户理解：
- 这个人和我是什么关系阶段，该怎么维护
- 他正在关心什么，我答应过他什么
- 我能先为他做什么，让合作自然发生
- 下次该用什么方式、什么理由去联系

长期愿景：成为用户的"随身关系经营助手"，软硬件一体（录音卡/摄像头/会议系统→文本→AI处理→推送提醒），让每一次接触都不白费。

**长期愿景（v4.3新增）**：成为用户的**终身关系经营智能体**——越用越懂你，CarryMem记忆层提供持续学习能力。CarryMem的7种记忆类型（事实记忆/情景记忆/程序记忆/语义记忆/工作记忆/短期记忆/长期记忆）支撑"终身"属性：系统随使用时长积累的关系洞察越来越精准，用户数据始终可携带（数据导出功能提前到Phase 1，见§3.3 F-21）。

**明确排除的平台级功能**（私密助手不需要）：
- ❌ RBAC权限模型 / 资源授权共享 / 团队协作
- ❌ 他人可提供资源匹配（只能匹配自己人脉的供给）
- ❌ 多租户隔离 / 企业管理看板
- ❌ 跨项目/跨团队资源撮合
- ❌ 商机匹配引擎（首期暂停，Phase2条件性重启）
- ❌ 关系图谱作为主展示（推进卡为主，图谱为辅）
- ❌ 竞对风险主动推送（Phase2+）
- ❌ 批量名片商机发现（与定位冲突）

### 1.5 部署模型与数据主权

> **核心决策**：EventLink采用"阶段递进式部署"——PoC本地优先、Phase 1云端托管、Phase 2弹性扩展。前端始终以微信小程序为主入口，不开发原生APP。

#### 1.5.0 为什么不做原生APP？

| 维度 | 原生APP | 微信小程序 | 结论 |
|------|---------|-----------|------|
| LLM调用 | 必须走云端 | 必须走云端 | 无差异 |
| 语音录入 | 原生能力 | 微信同声传译 | 小程序够用 |
| TTS播报 | 原生能力 | 微信插件 | 小程序够用 |
| 名片扫描 | 原生相机 | wx.scanCode | 小程序够用 |
| 推送通知 | APNs/FCM | 服务号模板消息 | 小程序够用 |
| 获客成本 | 高（下载APP） | 低（扫码即用） | 小程序胜 |
| 开发成本 | 双端开发 | 单端开发 | 小程序胜 |
| 生态优势 | 无 | 微信分享/名片/服务号 | 小程序胜 |

**结论**：LLM推理必须走云端，手机只是展示层。微信小程序已覆盖EventLink全部核心交互，原生APP无增量价值。

#### 1.5.1 阶段递进式部署模型

```
PoC（你的电脑）              Phase 1（云服务器）           Phase 2（云集群）
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Docker单机       │         │ Docker Compose   │         │ K8s集群          │
│ ├─ FastAPI       │  迁移   │ ├─ FastAPI       │  扩展   │ ├─ FastAPI×N     │
│ ├─ SQLite        │ ──────→ │ ├─ PostgreSQL    │ ──────→ │ ├─ PG+Redis+ES  │
│ └─ NullMemory    │         │ ├─ Redis         │         │ └─ pgvector     │
│                  │         │ └─ CarryMem(opt) │         │                  │
└─────────────────┘         └─────────────────┘         └─────────────────┘
     ↕ API                       ↕ API                       ↕ API
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Swagger UI      │         │ 名片小程序        │         │ 独立小程序+Web   │
│ (开发测试用)     │         │ ├─ 原生(语音/TTS) │         │ ├─ 原生+Web     │
│                 │         │ └─ H5(查询/管理)  │         │ └─ 全功能       │
└─────────────────┘         └─────────────────┘         └─────────────────┘
```

| 维度 | PoC（概念验证） | Phase 1（最小可用产品） | Phase 2（能力扩展） |
|------|---------------|---------------------|-------------------|
| **后台位置** | 用户个人电脑 | 云服务器（单节点） | 云集群（多节点） |
| **部署方式** | Docker单机 | Docker Compose | K8s |
| **数据库** | SQLite（本地文件） | PostgreSQL 15 + Redis 7 | PG + Redis + (按需ES) |
| **前端** | API + Swagger UI | 名片小程序(原生+H5) | 独立小程序+Web管理后台 |
| **认证** | API Key | JWT RS256+临时授权码 | OAuth2（无RBAC） |
| **运维** | 用户自己 | 运维团队集中管理 | 运维团队+监控告警 |
| **升级方式** | git pull + 重启 | CI/CD自动部署 | 滚动更新 |
| **备份** | 用户自己 | 自动每日备份 | 自动+异地容灾 |
| **可用性** | PC开机时可用 | 99.5% | 99.9% |
| **成本** | 零（本地） | ~¥500/月（单节点云） | 按用量弹性 |

#### 1.5.2 数据主权原则

> **核心原则**：数据归用户所有，EventLink是数据的加工者而非所有者。

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **用户数据所有权** | 用户创建的所有数据（事件/实体/关联/Todo）归用户个人所有 | user_id数据隔离，无跨用户访问 |
| **数据可携带** | 用户可随时导出全部数据 | F-21数据导出（**Phase 1**，v4.3提前），标准格式 |
| **数据可删除** | 用户可随时删除全部数据 | API支持批量删除+账号注销 |
| **数据最小化** | 只收集必要数据，不收集媒体文件 | 仅文本输入，无音频/视频/图片 |
| **处理透明** | AI处理过程可追溯 | 匹配推荐可解释，Todo来源可追溯 |
| **私密助手定位** | 不存在资源公开/授权共享场景 | 无RBAC、无跨用户匹配、无团队协作 |

**PoC阶段数据主权**：
- 数据存储在用户个人电脑的SQLite文件中
- LLM调用走HTTPS，仅发送脱敏后的文本片段
- 用户完全控制数据的启动/停止/备份/删除

**Phase 1+数据主权**：
- 数据存储在云端PostgreSQL，user_id隔离
- 客户端加密可选（Phase 2+）
- 数据导出/删除API始终可用
- 云端备份加密存储

#### 1.5.3 PoC→Phase 1 迁移路径

**迁移成本极低**：SQLite→PG只需改连接字符串（SQLAlchemy抽象层），API层零改动。

| 迁移项 | PoC | Phase 1 | 迁移方式 |
|--------|-----|---------|---------|
| 数据库 | SQLite | PostgreSQL | SQLAlchemy连接字符串切换 |
| 缓存 | 内存dict | Redis | 配置切换 |
| 认证 | API Key | JWT | 中间件替换 |
| CarryMem | NullMemory | 可选接入 | 配置切换 |
| 前端 | Swagger UI | 小程序 | 新增，不影响API |
| 数据 | 手动迁移脚本 | pg_dump/sqlite3→pg | 一次性脚本 |

#### 1.5.4 自建小程序前端备选方案（v4.2新增）

> **背景**：许总团队前端进展可能缓慢，或数字名片数据对接不能打通。EventLink需要有自建小程序前端的备选方案，确保不因前端依赖阻塞产品验证。

**方案概述**：

| 维度 | 说明 |
| ---- | ------------------------------------------------------------ |
| 框架选型 | Taro 3.x + Vue3 + NutUI（兼容微信小程序，支持跨端扩展） |
| 核心页面 | ① 首页（今天需要我回应的连接）② 录入页（语音+文字输入）③ 人物详情页（F-47关系推进卡）④ Todo列表页⑤ 设置页 |
| API兼容性 | 与现有后端API完全兼容，无需改动任何后端代码 |
| 开发策略 | Phase 1可先用H5+Swagger UI验证API正确性，小程序同步开发；PoC阶段H5先行，Phase 1小程序接替 |
| 开发周期估计 | 2-3周完成MVP页面（5个核心页面+基础导航） |

**技术栈详情**：
- 前端框架：Taro 3.x（跨端编译：微信/H5/RN）
- UI组件库：NutUI 4.x（京东出品，微信小程序原生体验）
- 状态管理：Pinia
- HTTP请求：Taro.request封装（与现有Swagger API对接）
- 语音录入：微信wx.getRecorderManager API

**决策触发条件**（满足任一即启动自建前端开发）：

| 触发条件 | 判断标准 | 负责人 |
| -------- | ------- | ------ |
| 许总团队无法按时启动 | 确认2周内无法启动前端开发 | PM |
| 数字名片API对接阻塞 | 对接评估超过3周无结论 | Architect |
| 种子用户测试需要独立环境 | 需要独立前端进行用户测试 | PM |

> **备注**：此方案为备选路径，优先级低于许总团队原生小程序方案。仅在触发条件满足时启动。

### 1.6 用户旅程定义

> **设计原则**：3条核心旅程覆盖产品最关键的使用场景，每条旅程定义完整的操作步骤、系统响应和体验目标。
> **架构约束**：EventLink不处理媒体文件（音频/视频/图片），所有媒体→文本转换由外部应用完成，EventLink仅接收文本输入。

#### 1.5.1 旅程一：新用户首次体验（冷启动）

> **【v4.2更新】首次体验固定为"记录一次重要交流"4屏流，不以扫码为主入口。名片扫描降级为补充身份输入方式。**

```
用户目标：快速理解产品价值，完成第一个行动
体验目标：从打开到完成第一个行动 < 2分钟

步骤1: 信任承诺（第1屏）
  系统 → 显示信任承诺页
         "这是你的私人关系助手。你记录的人和交流内容，只用于帮助你更好地回应重要关系。
          不公开、不匹配给其他用户、不自动联系任何人。"
  用户 → 点击"记录一次重要交流"

步骤2: 一句话输入（第2屏）【核心入口：语音+文字】
  系统 → 显示输入界面
         "最近，你和谁有过一次值得记住的交流？"
         输入示例：今天认识王主任，她在做社区长者活动，提到缺少年轻化运营和线上宣传。我答应明天发一份活动方案给她。
  用户 → 输入/语音录入交流内容（语音为推荐方式）
  用户 → 点击"帮我整理下一步"
  备注：名片扫描按钮在此屏作为补充入口（小图标，非主CTA）

步骤3: 用户确认重点（第3屏）
  系统 → 展示AI提取的重点，请用户确认
         | 项目 | 提取内容 |
         |------|---------|
         | 联系人 | 王主任 |
         | 她正在关心 | 社区长者活动运营与线上宣传 |
         | 我答应的事 | 发送一份活动方案 |
         | 建议完成时间 | 明天下午前 |
  用户 → 确认/修正提取内容

步骤4: 立即获得价值（第4屏）
  系统 → 展示行动建议
         "你可以先为王主任做一件小事"
         建议行动：明天下午前发送一份简版活动案例
         可复制话术：（生成温和微信表达）
  用户 → 点击"添加提醒" / "复制话术" / "稍后处理"
  系统 → 标记为in_progress，设置默认截止时间(可修改)
  系统 → 显示"已添加到待办"确认提示
```

**关键体验指标**:

- 引导到首次录入转化率>80%
- AI提取内容用户确认率>70%
- 从安装到完成第一个行动完成率>60%

#### 1.5.2 旅程二：日常使用（会前准备→会中记录→会后跟进）

```
用户目标：围绕会议全流程，不遗漏商机和行动
体验目标：每个环节操作<30秒

环节A: 会前准备(每日早晨自动+按需手动)
  自动流程(每日早晨):
    系统 → 检查当天日历(本地日程+已同步的外部日历)
    系统 → 对每个今日会议，自动生成准备清单:
           - 参会人基础信息+关联关系
           - 破冰话题建议(校友/前同事/同城市等)
           - 竞对/风险提示(如有)
    系统 → 推送今日会议准备摘要(与F-19每日简报合并推送)
  
  手动流程(按需):
    用户 → 点击某会议的"会前准备"按钮
```

**验收标准**: 每日8:00自动推送简报（可配置时间）；简报包含：今日待办Top 5+新发现关联数+今日会议清单；推送为系统通知+应用内消息；支持关闭/调整推送时间。

**US-27: 日程表双向对接**

**验收标准**: 支持Apple Calendar和Google Calendar(Android)两种日历源；读取未来7天会议并自动关联已有实体；会前1小时推送准备清单；会议结束后30分钟内推送"记录会议"提醒；Todo可一键添加到日历；日历事件变更（取消/改期）自动更新关联Todo；iOS和Android双平台均可用。

**验收标准**: 每日21:00自动推送事件汇总（可配置推送时间）；汇总包含：今日新增事件数+新增关联数+待跟进Todo数+未记录会议提醒；未记录的会议（日历中有但未录入纪要）高亮提醒；汇总支持一键补录遗漏的会议纪要；推送为系统通知+应用内消息。

### 4.2 安全需求

| 维度        | 方案                                            | 验收标准                  |
| --------- | --------------------------------------------- | --------------------- |
| 传输加密      | TLS 1.3 + HSTS                                | 安全扫描无TLS漏洞            |
| 身份认证      | JWT（access\_token: 15min, refresh\_token: 7d） | 未认证请求100%返回401        |
| 权限控制      | JWT认证 + 单用户数据隔离（无RBAC）               | 跨用户数据访问100%被阻止         |

### 5.5 话题标签体系设计（Topic Tag Taxonomy）

> **设计背景**：许总分享的分类截图更偏向文本话题维度（非纯行业归类）。EventLink需要在行业分类（§5.4）之外，建立话题标签体系，用于提升匹配精度、丰富实体属性、以及会议类型分类。话题标签通过不同管线融入处理流程——轻量管线(card\_scan)不提取话题，深度管线(meeting)和要点管线(call)提取话题标签。

#### 5.5.1 话题标签与行业分类的分工

| 维度         | 行业分类(§5.4 L1/L2)   | 话题标签(§5.5)                     |
| ---------- | ------------------ | ------------------------------ |
| **分类对象**   | 组织实体（公司/机构）        | 文本内容（需求描述/技能表述/会议议题）           |
| **解决什么问题** | 这家公司跟谁竞争？属于哪个赛道？   | 这个需求在说什么？能跟谁的能力配？这场会议属于什么类型？   |
| **数据来源**   | 公司属性、公开信息          | 事件raw\_text中的LLM抽取             |
| **使用管线**   | 所有管线               | 仅meeting管线和call管线              |
| **参与匹配度**  | F-05行业一致性维度(+0.25) | F-05话题语义相似度维度(+0.20)           |
| **会议类型分类** | 不参与                | meeting管线通过话题标签判定会议类型(A/B/C/D) |

**核心洞察**：同一行业的两家公司可能关注完全不同的话题；不同行业的公司可能关注同一话题。行业分类解决组织层面问题，话题标签解决内容层面问题。话题标签还承担会议类型分类的职责——将许总分享的各类文本分类归纳到会议的四大类型及子类。

#### 5.5.2 话题标签生成规则

```
话题标签来源（仅meeting和call管线触发）：
├── LLM从raw_text中提取（标记为provisional）
│   ├── meeting管线：提取需求话题、技能话题、行业热词
│   └── call管线：提取关键话题、痛点话题
├── 用户手动标注（优先级最高，直接confirmed）
└── 反馈闭环收敛（用户确认后provisional→confirmed）

话题标签与实体的关联：
├── person实体 → skills话题标签（如"边缘计算""React"）
├── organization实体 → 业务话题标签（如"工厂智能化改造""私域运营"）
├── technology实体 → 技术话题标签（如"大模型推理""IoT网关"）
└── project实体 → 需求话题标签（如"产线数字化""供应链优化"）
```

#### 5.5.3 话题标签在管线中的使用

| 管线                 | 话题标签使用方式                            | 输出              |
| ------------------ | ----------------------------------- | --------------- |
| **card\_scan(轻量)** | 不提取话题标签，仅使用已有confirmed标签参与匹配        | 匹配结果沿用已有话题维度    |
| **meeting(深度)**    | 从会议纪要中提取话题标签→丰富实体属性→参与F-05话题语义相似度计算 | 新增话题标签+匹配结果+行动项 |
| **call(要点)**       | 从通话记录中提取关键话题→丰富实体属性                 | 新增话题标签+跟进计划     |
| **manual(补全)**     | 不提取话题标签，仅确认/拒绝待定标签                  | 标签状态更新          |

#### 5.5.4 话题标签与F-05商机匹配度的关系

F-05商机匹配度五维打分法：

| 维度                  | 权重        | 说明                         | 变化     |
| ------------------- | --------- | -------------------------- | ------ |
| 技能-需求关键词重叠(Jaccard) | +0.30     | 原有，权重从0.35微调               | ↓0.05  |
| 行业领域一致性(§5.4)       | +0.25     | 原有，不变                      | —      |
| **话题语义相似度(§5.5)**   | **+0.20** | **新增：需求话题与技能话题在话题空间的语义距离** | **新增** |
| LLM语义理解             | +0.15     | 原有，权重从0.20微调               | ↓0.05  |
| 历史合作信号              | +0.05     | 原有，不变                      | —      |
| **合计**              | **0.95**  | 预留0.05给未来维度扩展              | —      |

**话题语义相似度计算**：

- 输入：实体A的话题标签集合 + 实体B的话题标签集合
- 计算：LLM embedding向量的余弦相似度，取均值
- 阈值：相似度≥0.7计满分0.20，\[0.4, 0.7)按比例折算，<0.4计0

#### 5.5.5 话题标签生命周期

```
LLM提取 → provisional标签
              │
              ├─ 用户确认 → confirmed标签 → 参与F-05匹配度计算
              ├─ 用户拒绝 → discarded → 不再显示
              └─ 30天无操作 → auto_expired → 清理
```

#### 5.5.6 话题标签与领域分类的关联

| 关联方式    | 说明                         | 示例                                 |
| ------- | -------------------------- | ---------------------------------- |
| 话题→L1归属 | 每个话题标签可映射到1-2个L1领域         | "工厂智能化改造"→D02智能制造                  |
| 交叉验证    | 话题标签推断的L1与实体已有L1不一致时，标记待确认 | 公司L1=D06零售电商，但话题="供应链优化"→可能L1应为D08 |
| L2标签来源  | 高频confirmed话题标签可升级为L2领域标签  | "边缘计算"出现≥20次→升级为D02的L2标签           |

#### 5.5.7 话题标签与会议类型分类的映射

meeting管线在提取话题标签后，通过话题组合判定会议类型（对应F-14的4类输出Schema）：

| 会议类型       | 话题标签特征           | 典型话题组合                   | 对应F-14输出                                        |
| ---------- | ---------------- | ------------------------ | ----------------------------------------------- |
| **A.内部协同** | 权责、分账、团队、合伙人、招商  | \["团队分工","分账比例","合伙人机制"] | topics+consensus+actions+faq                    |
| **B.对外商务** | 客户、方案、报价、竞标、谈判   | \["客户需求","方案汇报","商务条款"]  | clientBg+painPoints+solution+commercial+actions |
| **C.项目复盘** | 复盘、回顾、总结、改进、GRAI | \["项目回顾","经验总结","流程优化"]  | goal+keep+improve+prevent+insights+actions      |
| **D.知识沉淀** | SOP、话术、培训、标准、规范  | \["流程标准化","话术整理","培训大纲"] | knowledgeTags+assets+sop+talkScript             |

**分类逻辑**：

1. LLM从会议raw\_text中提取话题标签
2. 话题标签与4类会议的关键词特征匹配
3. 取匹配度最高的类型作为会议类型
4. 匹配度均<0.4时，默认为A类（内部协同）
5. 用户可在前端修正会议类型，修正结果反馈至模型

**与许总文本分类的关系**：许总分享的各类文本分类（如行业标签、话题标签、内容类型等），可通过本节映射机制归纳到4大会议类型及其子类。例如：许总分类中的"商务谈判"→B类子类，"团队管理"→A类子类，"案例复盘"→C类子类，"流程规范"→D类子类。

### 5.7.1 原始文件存储策略

> **核心架构决策**：EventLink**不存储、不处理**音频/视频/图片等媒体文件。所有媒体→文本的转换由外部应用（IAMHERE数字名片、恒智易R1录音卡、腾讯会议等）完成，EventLink仅接收文本输入。这是"不重复造轮子"原则——信息采集由专业应用负责，EventLink专注于文本→洞察→行动的智能引擎。

#### 输入边界定义

| 输入类型 | 谁负责转文本 | EventLink接收什么 | 说明 |
|---------|------------|-----------------|------|
| 名片照片 | IAMHERE数字名片小程序 | 结构化JSON（姓名/公司/职位/联系方式） | 许总负责的前端层 |
| 录音 | 恒智易R1录音卡/腾讯会议 | 转写文本 | 外部应用完成ASR |
| 视频会议 | 腾讯会议/飞书/Zoom | 会议纪要文本 | 平台自带转写功能 |
| 通话录音 | 手机/录音App | 转写文本 | 外部ASR服务 |
| 手动输入 | 用户直接输入 | 文本 | 无需转换 |
| CSV/Excel | 用户上传 | 解析后的文本数据 | 唯一的"文件"输入 |

#### 存储策略（仅文本）

| 数据类型 | 存储方式 | 保留策略 | 大小限制 |
|---------|---------|---------|---------|
| 事件原始文本 | Event.raw_text（PostgreSQL TEXT） | 永久保留 | ≤500KB（约15万字） |
| 名片结构化数据 | Event.raw_text（JSON字符串） | 永久保留 | ≤10KB |
| CSV/Excel导入 | 临时解析，不保留原始文件 | 导入后即删除 | ≤50MB |
| AI提取结果 | Entity/Association/Todo表 | 永久保留 | — |

> **注意**：EventLink不需要对象存储(S3/OSS)，不需要CDN，不需要媒体处理管线。所有存储均为PostgreSQL数据库内的文本字段。

### 5.7.2 LLM上下文窗口处理策略

> **核心问题**：当前主流LLM上下文窗口为128K-200K tokens（约5-8万字中文），但长会议转写可能超过此限制。需要分段处理策略。

#### 分段处理流程

```
原始文本输入
│
├─ 文本 ≤ 3万字（约8K tokens）
│  └─ 直接整段送入LLM处理 ✅
│
├─ 3万字 < 文本 ≤ 15万字
│  └─ 滑动窗口分段处理：
│     1. 按段落/发言者切分为N段（每段≤3万字）
│     2. 每段独立提取实体+要点
│     3. 合并阶段：将各段提取结果汇总，去重实体，合并关联
│     4. 全局推理：基于合并结果生成Todo和关联发现
│
└─ 文本 > 15万字（极端情况）
   └─ 先摘要后处理：
      1. 第一轮LLM：按段生成摘要（每段压缩至3000字）
      2. 第二轮LLM：基于全文摘要提取实体+关联+Todo
      3. 原始文本保留，用户可查看但不再二次处理
```

#### 各管线处理策略

| 管线 | 典型文本长度 | 处理策略 | LLM调用次数 |
|------|------------|---------|------------|
| card_save | <500字 | 整段处理 | 1次 |
| call | 3000-3万字 | 整段或2段处理 | 1-2次 |
| meeting | 1万-15万字 | 滑动窗口分段 | 2-5次 |
| manual | <3万字 | 整段处理 | 1次 |

#### Token消耗预估与成本控制

| 场景 | 输入tokens | 输出tokens | 单次成本(估算) | 日均上限 |
|------|----------|----------|-------------|---------|
| 名片扫描 | ~500 | ~300 | ¥0.01 | 100次/天 |
| 通话记录 | ~5K | ~2K | ¥0.05 | 50次/天 |
| 会议纪要(短) | ~20K | ~5K | ¥0.15 | 20次/天 |
| 会议纪要(长) | ~80K | ~15K | ¥0.60 | 10次/天 |

**成本控制措施**：
- 单用户每日LLM调用Token上限：500K tokens
- 超限后降级为规则引擎处理（spaCy+NLP规则）
- 相同文本不重复调用LLM（基于raw_text哈希去重）
- 缓存LLM提取结果，实体属性增量更新而非全量重提取

#### 5.10.2 离线存储策略

```
前端存储层次：
├─ Memory Cache(当前会话数据，5分钟TTL)
├─ IndexedDB(持久化离线数据)
│  ├─ cards(名片基础信息，最近500条)
│  ├─ todos(待办事项，最近100条)
│  ├─ entities(实体摘要，最近200条)
│  └─ tags(标签缓存)
└─ 操作队列(待同步操作)
   ├─ pending_uploads(待上传的名片/会议纪要)
   ├─ pending_updates(待更新的Todo状态)
   └─ pending_deletes(待删除的记录)
```

#### 5.10.3 数据同步策略

| 同步场景    | 策略                         | 冲突解决                      |
| ------- | -------------------------- | ------------------------- |
| 网络恢复    | 自动上传操作队列(按时间戳顺序)→拉取服务端增量更新 | Last-Write-Wins(服务端时间戳为准) |
| 多设备同时编辑 | 服务端检测冲突→推送冲突提示             | 用户选择保留版本                  |
| 删除冲突    | 删除优先                       | 删除操作始终生效                  |
| 同步状态展示  | 顶部指示器(在线/同步中/离线)           | 同步失败显示重试入口                |

#### 5.10.4 弱网优化

- 请求超时：正常5秒→弱网自动延长至15秒
- 自动重试：指数退避(1s, 2s, 4s, 8s)，最多4次
- 降级策略：关联发现失败→仅显示基础信息；图谱加载失败→显示列表视图
- 预加载：WiFi环境下预加载最近7天数据

### 5.11 Todo生命周期增强

> **设计背景**：客户视角评审发现F-06仅定义了"生成与追踪"，缺少批量操作、智能提醒、延期等关键交互。本节补充Todo管理的完整交互设计。

#### 5.11.1 Todo状态机增强

```
                    ┌──────────────┐
                    │   pending    │ ← 新建/延期恢复
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
      ┌───────────┐ ┌──────────┐ ┌──────────┐
      │in_progress│ │ dismissed│ │ snoozed  │
      └─────┬─────┘ └──────────┘ └────┬─────┘
            │                           │
            ▼                           │(到期后)
      ┌──────────┐                      │
      │   done   │ ←────────────────────┘
      └──────────┘

新增状态:
- snoozed: 延期状态，到期后自动恢复为pending
- dismissed: 不感兴趣/不相关，不进入完成度统计
```

#### 5.11.2 批量操作

| 操作   | 触发方式                   | 说明                 |
| ---- | ---------------------- | ------------------ |
| 批量完成 | 多选模式→标记完成              | 适用于一次性处理多个信息型Todo  |
| 批量延期 | 多选模式→选择延期时间(明天/下周/自定义) | 适用于会议后批量推迟低优先级Todo |
| 批量归档 | 多选模式→归档                | 将已完成的Todo移出活跃列表    |
| 批量删除 | 多选模式→删除(二次确认)          | 适用于清理不相关的Todo      |

#### 5.11.3 智能提醒

| 提醒类型 | 触发条件                  | 提醒方式           |
| ---- | --------------------- | -------------- |
| 到期提醒 | 行动型Todo到达due\_date前1天 | Push通知+应用内红点   |
| 逾期提醒 | 行动型Todo超过due\_date未完成 | 每日8:00推送逾期清单   |
| 会前提醒 | 关联日历事件的Todo，会前1小时     | Push通知+会议准备清单  |
| 沉默提醒 | 同一关联24小时内不重复生成Todo    | 自动去重，仅更新已有Todo |

#### 5.11.4 延期(Snooze)操作

| 延期选项 | 说明            | 效果                     |
| ---- | ------------- | ---------------------- |
| 明天   | 延期至次日9:00     | due\_date更新，状态→snoozed |
| 下周   | 延期至下周一9:00    | due\_date更新，状态→snoozed |
| 自定义  | 用户选择日期和时间     | due\_date更新，状态→snoozed |
| 会前   | 延期至关联日历事件前1小时 | F-33日程表对接支持            |

### 5.12 关联暴增下的搜索与性能策略

> **设计背景**：随着数据量增长（500联系人+200公司+5000条沟通记录），关联可能从几十条暴增到几千甚至上万条。需要在架构层面提前规划搜索和性能方案。

#### 5.12.1 数据规模预估

| 数据规模 | 实体数 | 关联数(预估) | 搜索响应要求 |
|---------|--------|------------|------------|
| PoC期 | <100 | <500 | <200ms |
| 早期用户 | 500-1000 | 2000-5000 | <500ms |
| 成熟用户 | 3000+ | 10000+ | <1s |

#### 5.12.2 分层查询策略

| 层级 | 数据范围 | 存储介质 | 响应时间 | 适用场景 |
|------|---------|---------|---------|---------|
| L1 热数据 | 最近30天+置信度>0.7 | Redis缓存 | <50ms | 首页待办、高频关联 |
| L2 温数据 | 最近1年全部 | PostgreSQL索引查询 | <200ms | 实体详情、搜索 |
| L3 冷数据 | 历史归档 | PostgreSQL分区表 | <1s | 历史回溯、报表 |

#### 5.12.3 索引策略

```
关联表核心索引：
- (source_type, source_id, confidence DESC) -- 查"某实体的所有关联"
- (target_type, target_id, confidence DESC) -- 反向查询
- (relation_type, confidence DESC) -- 按关联类型筛选
- (source_type, target_type, confidence DESC) -- 类型过滤

实体表核心索引：
- (entity_type, name) -- 按类型+名称搜索
- GIN索引(name, company) -- 全文搜索
```

#### 5.12.4 搜索方案演进

| 阶段 | 方案 | 适用规模 | 说明 |
|------|------|---------|------|
| MVP-Core | PostgreSQL LIKE + 关联表JOIN | <1000条 | 简单够用 |
| MVP-Plus | PostgreSQL全文搜索(tsvector+tsquery) | <10万条 | 支持中文分词 |
| V1.0+ | Elasticsearch | >10万条 | 复杂查询、聚合、推荐 |

#### 5.12.5 前端性能保障

| 策略 | 说明 |
|------|------|
| 分页加载 | 关联列表默认Top 20（按置信度排序），滚动加载更多 |
| 懒加载 | 详情页关联图谱按需展开，不一次性渲染 |
| 聚合视图 | 不展示所有关联，显示"与10家公司、25个人、8个商机相关" |
| 智能排序 | 优先显示"最近活跃"和"高置信度"的关联 |
| 后台计算 | 关联发现引擎异步计算，不阻塞用户操作 |
| 缓存策略 | 高频查询结果缓存5分钟，写入时失效 |



### 5.13 Person实体扩展画像

> **设计原则**：Person是EventLink的核心实体，因为人是一切商务关系的交汇点。画像字段由AI从事件文本中自动提取，用户无需手动填写表单。详细字段定义见附录A。

#### 5.13.1 画像维度

Person实体通过`properties` JSONB字段支持扩展画像，按需丰富，不强制填充：

| 维度 | 包含信息 | Phase 1 | 数据来源 |
|------|----------|---------|----------|
| 基础信息 | 姓名/公司/职位/城市/行业 | ✅ | 名片JSON+LLM提取 |
| 沟通偏好 | 偏好渠道/响应速度 | ✅ | LLM从交互记录推断 |
| 决策画像 | 决策权力/决策风格 | ✅ | LLM推理+用户确认 |
| 资源能力 | 核心资源/能力标签 | ✅ | LLM提取 |
| 关系状态 | 关系强度/维护频率/下次抓手 | ✅ | 系统计算+LLM推理 |
| 禁忌敏感 | 禁忌话题/敏感信息 | Phase 2 | LLM从事件文本提取 |
| 事业轨迹 | 职业高峰/背景经历 | Phase 2 | LLM提取+用户补充 |

#### 5.13.2 渐进式填充

> 画像不是用户填出来的，是AI从每次互动中逐步积累的。

```
创建联系人 → 自动填充基础信息（来自名片/事件文本）
每次互动后 → AI提取沟通偏好/决策画像/资源能力等增量信息
查看详情时 → AI弹出1-2个选择题确认（如"决策权力？A.拍板 B.建议 C.传话"）
积累5+次互动 → 关系洞察质量显著提升
```

### 5.16 触达通道与前端整合设计

> **核心问题**：用户在外拜访/开会时，如何收到AI的行动建议？这是产品闭环的"最后一公里"。

#### 5.16.1 场景-通道矩阵

| 用户状态 | 需要什么 | 时间窗口 | Phase 1通道 |
|----------|----------|----------|------------|
| 会前（即将见某人） | 关系画像+沟通建议 | 会前15-60分钟 | 微信服务号推送 / 小程序主动查询 |
| 会中（正在开会） | 不打扰，静默采集 | 全程 | 无推送 |
| 会后（刚结束） | 快速录入+行动项 | 会后5-30分钟 | 小程序语音录入 |
| 日常（碎片时间） | 维护提醒+早晚报 | 早晨/晚间 | 微信服务号 |

#### 5.16.2 名片小程序+EventLink整合方案

EventLink与许总数字名片小程序合并为**一个小程序**，名片是入口，关系助手是增值功能。

**整合后的用户流程**：

```
许总名片小程序
├── 首页：我的名片 + 扫一扫
├── 名片夹：扫过的名片列表
│   └── 点击某张名片 → 名片详情
│       ├── 基础信息（名片原有）
│       └── 🆕 关系助手（EventLink）
│           ├── 关系画像（阶段/偏好/决策权）
│           ├── 交流要点（上次聊了什么）
│           ├── 🔊 语音播报（开车时听）
│           └── 📝 快速记录（语音/文字录入）
├── 🆕 今日日程：今天要见的人
│   └── 会前速览 + TTS播报
└── 🆕 待办提醒：行动项+维护提醒
```

**分阶段实施**：

| 阶段 | 方案 | 许总改动量 | 用户体验 |
|------|------|-----------|---------|
| Phase 1 | 名片小程序内嵌EventLink H5页面 | 1个按钮+1个WebView | 基本无缝，名片数据通过URL参数传递 |
| Phase 2 | EventLink核心页面做成小程序原生页面 | 需开发原生页面 | 完全原生体验，TTS/录音用微信API |

**为什么合并而非分开**：
- 用户视角：扫名片→看关系画像是同一流程，不应跳转另一个小程序
- 数据视角：名片JSON直接作为EventLink的card_save事件输入，无需跨小程序传数据
- 商业视角：名片小程序是许总的入口产品，EventLink是增值功能，合并后ARPU更高

#### 5.16.3 小程序核心能力

**查询能力**：
- 搜索今天要见的人 → 展示简介、关系阶段、沟通偏好
- 查看与某人的历史交流要点 → 上次聊了什么、待跟进事项
- 浏览关系图谱 → 一度/二度人脉

**录入能力**：
- 语音录入 → "今天和张总聊了新项目预算" → AI自动建档
- 文字录入 → 快速记录要点
- 名片扫描 → 许总名片小程序已有能力，扫描结果直接触发EventLink事件处理

**播报能力**：
- 点击语音按钮 → TTS读出人物简介和交流要点
- 会前场景：开车/走路时听一遍对方信息，不用看屏幕

#### 5.16.4 Phase 1 三通道

1. **名片小程序+EventLink H5**（主交互入口）— 查询/录入/播报
2. **微信服务号模板消息**（推送通道）— 会前提醒/行动建议/维护提醒/早晚报
3. **手机推送通知**（APNs/FCM）— 紧急提醒（风险预警/到期Todo）

#### 5.16.5 Phase 2 扩展

- EventLink小程序原生页面（替代H5）
- 录音卡屏幕/语音播报（许总硬件）
- Apple Watch/智能手环
- 邮件日报

#### 5.16.6 防打扰策略

**推送优先级分级**：

| 优先级 | 类型 | 通道 | 频次限制 | 说明 |
|--------|------|------|----------|------|
| P1-紧急 | 会前提醒(1小时内) | 服务号模板消息 | 不受限(仅推1次) | 最重要，必须触达 |
| P1-紧急 | 逾期Todo | 服务号模板消息 | 不受限 | 行动项过期 |
| P2-重要 | 新行动建议 | 小程序卡片 | 每小时≤3条 | 会后AI生成 |
| P3-普通 | 关系维护提醒 | 应用内消息 | 每日≤2次(早晚) | 合并推送 |
| P4-低优 | 新发现关联 | 早报合并 | 每日1次 | 不单独推送 |

**防打扰规则**：
- 同一用户每日总推送≤10条（P1不受限）
- 静默时段（默认22:00-7:00）：仅P1可穿透
- 用户可在设置中自定义：各类型开关+静默时段+提前提醒时间
- APNs授权失败 → 自动切换到服务号推送
- 服务号推送失败 → 记录到应用内消息中心

## 8. 术语表

| 术语                | 英文                      | 定义                                                                      |

| 事件                | Event                   | 系统处理的基本单元，如名片扫描、会议记录等                                                   |

| 商机匹配度             | Opportunity Match Score | 供需双方商机匹配程度（含技术供需/业务互补/资源对接），五维打分法计算                                     |

| 别名                | Alias                   | 同一实体的不同称呼，如"李总"和"李明"                                                    |
| Human-in-the-Loop | HITL                    | 人工参与确认的关键决策环节                                                           |

## 1. 产品概述

### 1.1 产品愿景

EventLink 是 IAMHERE 数字名片与 CarryMem AI 关联发现引擎的融合产品，自动发现商务社交中隐藏的人脉关联和商业线索。产品定位不是"帮助记忆"，而是"帮助发现被忽略的机会"。

### 1.2 目标用户

| 用户画像 | 核心需求 | 使用频率 | 价值感知 |
|---------|---------|---------|---------|
| 商务BD/销售 | 拜访客户前了解关联背景 | 高频（每日） | 避免信息盲区，提高拜访效率 |
| 投资人/FA | 发现项目间隐藏关联 | 中频（每周3-5次） | 发现投资组合交叉机会 |
| 创业者 | 了解合作伙伴/竞对关系 | 中频（每周2-3次） | 规避商业风险，发现合作可能 |

### 1.3 核心价值主张

| 维度 | 传统方案 | EventLink |
|------|---------|-----------|
| 信息获取 | 被动查询CRM/通讯录 | 主动推送隐藏关联 |
| 关联发现 | 依赖人工记忆和经验 | AI自动发现跨事件关联 |
| 风险预警 | 事后才知道竞对关系 | 实时竞对关系预警 |
| 数据输入 | 手动录入CRM | 名片/录音/会议自动采集 |

### 1.4 三层架构

```
┌─────────────────────────────────────────────────────────┐
│  L1 应用层（信息触点）— 合作方许总团队负责               │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │IAMHERE   │  │AI录音卡   │  │视频会议   │              │
│  │数字名片   │  │(恒智易R1) │  │(腾讯/飞书)│              │
│  └──────────┘  └──────────┘  └──────────┘              │
└────────────────────────┬────────────────────────────────┘
                         │ HTTP API
┌────────────────────────▼────────────────────────────────┐
│  L2 引擎层（事件标准化）— CarryMem 团队负责              │
│  多源数据接入 → 格式标准化 → AI实体抽取 → 元数据丰富     │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────▼────────────────────────────────┐
│  L3 引擎层（关联发现引擎）— CarryMem 团队负责            │
│  实体归一引擎 → 关联发现引擎 → 提醒生成引擎              │
│  底层: 实体图谱(NetworkX) + 规则引擎                     │
└─────────────────────────────────────────────────────────┘
```

**分工原则**:
- 应用层负责前端展示和用户操作，无需服务器部署、无需AI模型维护、无需后端团队
- 引擎层负责后端服务和核心算法，提供标准HTTP API（含SDK和示例代码）
- 文档采用第三人称客观语言，不使用"你做/我做"等指派口吻

## 2. 用户故事

### 2.1 机会发现场景（⚪ Opportunity）

**US-01: 技术匹配商机发现**

> 作为商务BD，当扫描一张数字名片时，系统自动发现该名片持有人的技术需求与系统中另一联系人的技术能力高度匹配（匹配度≥0.80），并在5秒内推送商机提醒，以便及时牵线搭桥。

**验收标准**: 技术匹配度≥0.80时，系统在事件上报后10秒内生成⚪机会提醒，提醒包含双方姓名、匹配维度和匹配分数。

**US-02: 跨事件合作线索发现**

> 作为投资人，当两次不同会议中出现同一技术领域的关键人物时，系统自动发现两人存在技术重叠关联，推送潜在合作机会提醒，以便评估联合投资可能。

**验收标准**: 当两个person实体在同一technology实体下共现且tech_overlap关联强度>0.5时，系统生成⚪机会提醒。

**US-03: 供应链关联发现**

> 作为创业者，当系统发现新接触的供应商与现有客户存在供应链上下游关系时，推送供应链关联提醒，以便优化采购或销售策略。

**验收标准**: 当两个organization实体间建立supply_chain关联且strength>0.4时，系统生成⚪机会提醒。

### 2.2 风险预警场景（🔴 Risk）

**US-04: 竞对关系预警**

> 作为商务人士，当扫描一张名片时，系统发现该名片所属公司与系统中已有联系人所在公司存在直接竞争关系，立即推送🔴风险提醒，以便在沟通中注意信息边界。

**验收标准**: 当两个organization实体间建立competitor关联且strength>0.5时，系统在5秒内生成🔴高风险提醒，提醒包含竞对依据和沟通建议。

**US-05: 风险关联链预警**

> 作为商务人士，当系统发现新接触的人与已知风险实体存在风险关联链（如竞对公司的前员工），推送🔴风险提醒，以便评估合作风险。

**验收标准**: 当person实体与已有risk_link关联的实体建立新关联时，系统生成🔴风险提醒。

### 2.3 背景补充场景（🔵 Context）

**US-06: 校友关系背景补充**

> 作为商务人士，当查看今日提醒时，系统推送🔵背景提醒，告知即将会面的联系人与自己是校友关系，以便在沟通中建立亲近感。

**验收标准**: 当两个person实体间建立alumni关联时，系统生成🔵背景提醒，提醒包含学校名称和毕业年份（如有）。

**US-07: 投资偏好背景补充**

> 作为创业者，当系统发现潜在投资人的投资方向标签与自身项目领域匹配时，推送🔵背景提醒，以便在路演中针对性强调相关亮点。

**验收标准**: 当person实体的properties中investment_preferences字段与project实体的industry字段匹配度>0.6时，系统生成🔵背景提醒。

### 2.4 实体归一场景

**US-08: 实体归一人工确认**

> 作为系统管理员，当实体归一引擎产生置信度在[0.70, 0.85)区间的归一建议时，系统在管理界面展示待确认队列，包含完整属性对比、AI归一推理、置信度分数和历史相似案例，管理员可执行"确认归一"/"拒绝归一"/"稍后处理"三种操作，确认结果反馈至模型用于持续改进。

**验收标准**: 待确认队列展示所有[0.70, 0.85)区间的归一建议，每条建议包含属性对比表、AI推理说明、置信度数值；操作后系统在200ms内响应。

**US-09: 实体归一撤回**

> 作为系统管理员，当发现已自动合并的实体（置信度≥0.85）存在误合并时，可在管理界面一键撤回，系统自动拆分实体并恢复原始状态，同时记录操作日志。

**验收标准**: 撤回操作后实体恢复为合并前的独立状态，关联关系正确还原，操作日志完整记录撤回时间、操作人和原始合并信息。

### 2.5 数据录入场景

**US-10: 名片扫描事件上报**

> 作为IAMHERE小程序用户，当扫描或查看一张数字名片时，小程序自动将名片数据上报至EventLink API，系统在200ms内返回确认，后台异步完成实体抽取和关联发现。

**验收标准**: API在200ms内返回event_id和processing状态；后台处理完成后，相关提醒在10秒内可查询。

## 3. 功能需求清单

### 3.1 P0 功能（Phase 1 必须）

#### F-01: 事件上报接口

| 属性 | 说明 |
|------|------|
| 功能描述 | 接收多来源事件数据（名片JSON、音频转写文本、会议纪要），标准化为统一Event Schema |
| API | `POST /api/v1/events` |
| 输入 | event_type, source, title, timestamp, raw_text, metadata |
| 输出 | event_id, status(processing), estimated_completion |
| 验收标准 | ① API响应时间P95<200ms ② 名片JSON解析准确率>95%（基于20张样本名片） ③ 支持card_scan/meeting/call/manual四种event_type ④ 重复事件（相同source+timestamp+title）自动去重，去重率100% |

#### F-02: 实体抽取服务

| 属性 | 说明 |
|------|------|
| 功能描述 | 从事件raw_text中抽取person/organization/technology/project/attribute五类实体 |
| 主路径 | LLM抽取（Moka AI Claude Sonnet / OpenAI） |
| 降级路径 | spaCy + 自定义词典规则引擎 |
| 验收标准 | ① LLM抽取准确率≥90%（基于100条人工标注样本） ② 降级路径抽取准确率≥75% ③ 单次抽取延迟P95<2s（LLM）/ <200ms（规则） ④ 抽取结果包含实体类型、名称、属性字典 |

#### F-03: 实体归一引擎

| 属性 | 说明 |
|------|------|
| 功能描述 | 5步级联归一算法，判断新实体是否与已有实体为同一对象 |
| 算法步骤 | Step 1: 精确匹配（confidence=1.0）→ Step 2: 别名扩展查找 → Step 3: 上下文相似度计算（公司+0.3, 技能+0.2, 共现+0.3）→ Step 4: 置信度分级处理 → Step 5: Human-in-the-Loop确认 |
| 置信度分级 | ≥0.85: 自动合并（记录日志，支持撤回）；[0.70, 0.85): 进入待确认队列；<0.70: 创建新实体 |
| 撤回机制 | 所有自动合并支持一键撤回，撤回后实体恢复独立状态，关联关系还原，操作日志完整记录 |
| 人工确认 | 展示完整属性对比 + AI归一推理 + 置信度分数 + 历史相似案例；操作项：确认归一/拒绝归一/稍后处理；确认结果反馈至模型持续改进 |
| 验收标准 | ① 精确匹配准确率100% ② 别名扩展召回率≥80%（基于50组别名对测试） ③ 上下文相似度计算延迟<500ms ④ 自动合并误合并率<5%（人工抽检100条） ⑤ 撤回操作成功率100%，数据一致性无损 ⑥ 待确认队列展示完整归一依据，操作响应<200ms |

#### F-04: 关联发现引擎

| 属性 | 说明 |
|------|------|
| 功能描述 | 基于共现分析和类型推断发现实体间7种关联关系 |
| 关联类型 | alumni, ex_colleague, competitor, tech_overlap, deal_link, risk_link, supply_chain |
| 算法步骤 | A. 共现频率分析 → B. 类型推断（同公司→ex_colleague, 技能重叠→tech_overlap, 竞对DB匹配→competitor）→ C. 时间衰减+过滤（strength × exp(-λ × Δt), λ=0.01, 保留>0.3） |
| 验收标准 | ① Precision@5>70% ② Recall@10>60% ③ F1>0.65 ④ 单次关联发现延迟<2s ⑤ 7种关联类型全部可生成 ⑥ 关联强度计算误差<0.05（与人工标注对比） |

#### F-05: 技术匹配度计算

| 属性 | 说明 |
|------|------|
| 功能描述 | 四维打分法计算技术供需匹配度 |
| 维度 | 技能-需求关键词重叠(Jaccard相似度): +0.35; 行业领域一致性: +0.25; LLM语义理解: +0.20; 历史合作信号: +0.05 |
| 阈值 | ≥0.80: 强匹配（推送⚪机会提醒）; [0.60, 0.80): 潜在机会（记录不推送）; <0.60: 不推送 |
| 验收标准 | ① 匹配度计算延迟<1s ② 强匹配推送准确率≥80%（人工评估50条） ③ Jaccard相似度计算与手工计算偏差<0.02 ④ 总分示例：0.35+0.25+0.20+0.05=0.85(85%) |

#### F-06: 提醒生成引擎

| 属性 | 说明 |
|------|------|
| 功能描述 | 基于关联发现结果生成3类提醒 |
| 提醒类型 | ⚪ Opportunity（商机发现，tech_match≥0.80=强匹配）; 🔴 Risk（竞对关系预警）; 🔵 Context（背景信息补充，校友/投资偏好等） |
| API | `GET /api/v1/alerts/today?limit=10`; `POST /api/v1/alerts/{id}/feedback` |
| 验收标准 | ① 提醒生成延迟<500ms（从关联发现完成到提醒入库） ② 3类提醒类型全部可生成 ③ 提醒内容包含title/detail/suggestion三要素 ④ 反馈API响应<200ms ⑤ 每日提醒去重率100%（同一关联24小时内不重复推送） |

#### F-07: 用户反馈闭环

| 属性 | 说明 |
|------|------|
| 功能描述 | 用户对提醒进行有用/无用/忽略反馈，反馈数据用于优化推荐算法 |
| 反馈类型 | useful / not_useful / dismissed |
| 验收标准 | ① 反馈API响应<200ms ② 反馈数据持久化存储，保留期≥365天 ③ 反馈数据可导出用于离线分析 ④ 同一提醒不可重复提交反馈 |

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 事件上报API响应时间 | P95<200ms | 压测工具（locust/k6），1000次请求取P95 |
| 实体抽取延迟（LLM路径） | P95<2s | 计时器测量，100次调用取P95 |
| 实体抽取延迟（规则路径） | P95<200ms | 计时器测量，1000次调用取P95 |
| 关联发现延迟 | P95<2s | 计时器测量，100次调用取P95 |
| 端到端延迟（事件上报→提醒可查） | <10s | 完整链路计时，20次取最大值 |
| 技术匹配度计算延迟 | P95<1s | 计时器测量，100次调用取P95 |
| 提醒查询API响应时间 | P95<500ms | 压测工具，1000次请求取P95 |
| 并发用户支持 | ≥50并发 | 压测工具，50并发下P95延迟不退化 |
| 数据库查询（单实体） | <50ms | EXPLAIN ANALYZE验证 |
| 数据库查询（关联遍历2跳） | <200ms | EXPLAIN ANALYZE验证 |

### 4.2 安全需求

| 维度 | 方案 | 验收标准 |
|------|------|---------|
| 传输加密 | TLS 1.3 + HSTS | 安全扫描无TLS漏洞 |
| 身份认证 | JWT（access_token: 15min, refresh_token: 7d） | 未认证请求100%返回401 |
| 权限控制 | RBAC（admin/member/viewer）+ 行级隔离 | 越权访问100%返回403 |
| 数据加密 | AES-256-GCM 字段级加密（敏感属性） | 敏感字段数据库存储为密文 |
| API Key管理 | 操作系统钥匙串存储（Keychain/Credential Manager） | API Key不明文出现在配置文件和日志中 |
| 审计日志 | 所有写操作记录（操作人/IP/时间/内容） | 审计日志完整性100%，保留期≥90天 |
| 合规性 | GDPR支持（数据导出Art.20/删除Art.17/撤回同意） | 数据删除请求24h内完成，包含级联关联清理 |
| 输入验证 | SQL注入/XSS/CSRF防护 | OWASP Top 10扫描无高危漏洞 |

### 4.3 可用性需求

| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 系统可用性 | ≥99.5%（月度） | 监控系统统计，月度停机<3.6h |
| 故障恢复时间（RTO） | <1h | 故障注入测试 |

## 5.5 领域分类体系设计（Domain Taxonomy）

> **设计原则（来自许总反馈）**：大厂有成熟的行业分类体系，EventLink不需要照搬，但需要了解后选择自己的方向。本节定义EventLink自有的轻量级领域分类策略。

### 5.5.1 分类策略：方案D — 一级预定义 + 二级动态扩展

| 层级 | 定义方式 | 数量 | 用途 | 维护方式 |
|------|---------|------|------|---------|
| **L1 一级领域** | 硬编码预定义 | 18±3个 | F-05技术匹配度"行业一致性"维度、F-08竞对判断的行业归属 | 配置文件，版本管理 |
| **L2 二级标签** | LLM动态生成 + 用户反馈收敛 | 不设上限 | 实体属性丰富化、精准匹配 | Provisional标记，人工确认后固化 |

**选择理由**：
- L1硬编码确保核心功能的可验证性和可复现性
- L2动态扩展避免过早锁定细分方向，允许从用户数据中学习
- 完美平衡"知道大厂分类→选自己方向→不完全照搬"的战略定位

### 5.5.3 L2 二级标签生成规则

```
L2标签来源：
├── LLM抽取（从实体raw_text中提取，标记为provisional）
├── 用户手动标注（优先级最高）
├── 公开网页信息检索（标记为provisional）
└── 反馈闭环收敛（用户确认后provisional→confirmed）

Provisional标签规则：
├── 自动生成时标记为provisional
├── provisional标签不参与F-05匹配度计算（仅L1参与）
├── 用户或管理员确认后升级为confirmed
└── confirmed标签计入F-05匹配度计算
```

### 5.5.4 分类体系与非功能需求的关联

| 维度 | 要求 | 验证方式 |
|------|------|---------|
| 可维护性 | L1领域列表通过YAML配置文件管理，支持热加载 | 修改配置后无需重启即可生效 |
| 可扩展性 | L1支持动态增删（≤30个），L2无上限 | 压测验证1000个L2标签下性能不退化 |
| 一致性 | 同一组织实体的L1领域归属唯一 | 数据库约束 + 归一引擎校验 |
| 可追溯 | 每个L2标签记录来源（LLM/用户/网页）+ 创建时间 | metadata字段记录 |

## 8. 术语表

| 术语 | 英文 | 定义 |
|------|------|------|
| 事件 | Event | 系统处理的基本单元，如名片扫描、会议记录等 |
| 实体 | Entity | 从事件中抽取的命名对象，含5种类型 |
| 实体归一 | Entity Resolution | 判断不同表述是否指向同一真实对象的过程 |
| 关联 | Association | 两个实体间的关系，含7种类型 |
| 关联强度 | Association Strength | 关联的置信程度，取值0.0~1.0 |
| 提醒 | Alert | 系统主动推送给用户的有价值信息，含3种类型 |
| 技术匹配度 | Tech Match Score | 技术供需双方的匹配程度，四维打分法计算 |
| 置信度 | Confidence | 实体归一算法对归一结果的确定性评估 |
| 共现 | Co-occurrence | 两个实体在同一事件中同时出现 |
| 时间衰减 | Time Decay | 关联强度随时间推移而降低的机制，λ=0.01 |
| 别名 | Alias | 同一实体的不同称呼，如"李总"和"李明" |

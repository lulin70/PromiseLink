## 1. 产品愿景

### 1.1 一句话定位

**EventLink — AI驱动的个人商务关系经营助手。先成就关系，再促成合作。**

> 主Slogan：让每一次连接，都有回应。
> 副Slogan：记住需要，兑现承诺，让合作自然发生。
> 品牌短句：见面只是开始。

### 1.2 我们解决什么问题

多数商务人士面临三大困境：

| 痛点 | 具体表现 |
|------|----------|
| **记不住** | 见过的人太多，重要细节遗忘——上次聊了什么、对方关心什么、我答应过什么 |
| **想不清** | 不知道对方能调动什么资源、该用什么方式维护、关系走到哪一步了 |
| **顾不上** | 日常事务繁忙，错过最佳维护时机，承诺拖延，关系自然冷却 |

### 1.3 核心闭环

**互动 → 关注 → 承诺 → 帮助 → 反馈**

从利他视角出发：
1. **互动**：我认识了谁
2. **关注**：他正在关心什么
3. **承诺**：我答应过他什么
4. **帮助**：我能先为他做什么
5. **反馈**：双方是否自然出现合作

> 产品不只是记录联系人和提取待办，更帮助用户理解：这个人和我是什么关系阶段、该怎么维护、下次该用什么方式去联系。

### 1.4 产品定位边界

**EventLink 是什么：**
- 个人私密助手——数据归用户个人所有
- 关系经营工具——帮你经营人脉关系资产
- 利他驱动——先为对方做点什么，让合作自然发生

**EventLink 不是什么：**
- 不是撮合平台——不公开资源、不授权共享、不团队协作
- 不是销售CRM——不管理销售漏斗、不团队协作
- 不是通用AI助手——语音助手仅限结构化数据查询，不做开放域闲聊
- 不做关系图谱为主展示——推进卡为主，图谱为辅
- 不做竞对风险主动推送（Phase 2+ 才考虑）
- 不做批量名片商机发现

## 4. 用户旅程

### 4.1 旅程一：新用户首次体验（冷启动）

> **核心设计**：首次体验固定为"记录一次重要交流"4屏流程，不以扫名片为主入口。名片扫描降级为补充身份输入方式。

**体验目标**：从打开到完成第一个行动 < 2分钟

```
第1屏：信任承诺
  "这是你的私人关系助手。你记录的人和交流内容，只用于帮助你更好地回应重要关系。
   不公开、不匹配给其他用户、不自动联系任何人。"
  → 点击"记录一次重要交流"

第2屏：一句话输入（核心入口：语音 + 文字）
  "最近，你和谁有过一次值得记住的交流？"
  示例：今天认识王主任，她在做社区长者活动，提到缺少年轻化运营和线上宣传。
       我答应明天发一份活动方案给她。
  → 语音录入或文字输入 → 点击"帮我整理下一步"
  备注：名片扫描按钮在此屏作为补充入口（小图标，非主操作）

第3屏：用户确认重点
  AI提取重点，请用户确认：
  | 项目       | 提取内容               |
  |-----------|----------------------|
  | 联系人     | 王主任                |
  | 她正在关心 | 社区长者活动运营与线上宣传 |
  | 我答应的事 | 发送一份活动方案        |
  | 建议完成时间 | 明天下午前             |
  → 确认/修正提取内容

第4屏：立即获得价值
  "你可以先为王主任做一件小事"
  建议行动：明天下午前发送一份简版活动案例
  可复制话术：（生成温和微信表达）
  → 点击"添加提醒" / "复制话术" / "稍后处理"
  → 显示"已添加到待办"确认提示
```

### 4.2 旅程二：日常使用

**会前准备**
- 每日早晨自动推送今日会议准备清单：参会人信息 + 关联关系 + 破冰话题建议
- 外出拜访场景：微信服务号推送会前提醒 → 小程序查看今日日程 → 语音播报人物简介（5秒极简/30秒完整）

**会中记录**
- 主方案：录音卡自动录音 → 后台转写文本
- 替代方案：小程序内"快速记录"模板 / 会后语音补录
- EventLink不录音、不转写，仅等待外部应用提交文本

**会后跟进**
- AI自动处理会议纪要 → 生成行动项和待办
- 微信小程序卡片推送AI行动建议
- 语音录入会议要点 → AI处理 → 推送结果

**持续追踪**
- 每日早晨自动推送：今日到期待办 + 逾期提醒 + 会议准备信息
- 日常碎片时间：微信服务号推送维护提醒 → 一键发微信/标记已联系

### 4.3 旅程三：数据管理与维护

- 修正AI提取的错误信息
- 确认AI推断的标签
- 批量清理过期待办
- 导出数据备份
- 导入存量数据（CSV/Excel）

## 5. 功能亮点

### 5.1 六种待办类型

EventLink将所有待办分为6种类型，每种有独特的颜色标识和触发逻辑：

| 类型 | 颜色 | 含义 | 典型场景 |
|------|------|------|----------|
| 🤝 承诺 | 雾绿 | 我答应对方要做的事 | "明天发一份方案给王主任" |
| 🫶 帮助 | 雾紫 | 我可以主动为对方做的事 | 资源有效性复核、透支提醒 |
| 💙 关注 | 雾蓝 | 对方正在关心的事 | 人脉关系背景补充 |
| 🔄 跟进 | 雾金 | 需要后续跟进的事项 | AI提取的线索需用户确认 |
| ✨ 合作信号 | 雾白 | 商机匹配发现 | 商机匹配度≥0.80 |
| 🔴 风险 | 烟粉 | 竞对关系预警（Phase 2+） | 竞对公司关系提醒 |

### 5.2 关系推进卡（F-47）

为每个重点联系人生成一张全貌视图卡片，回答"这个人的关系走到哪一步了"：

- **当前阶段**：这段关系处于哪个阶段
- **最近交流**：上次聊了什么
- **对方关注**：他正在关心什么
- **待回应承诺**：我答应过他什么（≤3条）
- **需求洞察**：他可能需要什么
- **下一自然动作**：建议下一步做什么
- **建议触达方式**：用什么方式联系最合适

### 5.3 关系阶段标记（F-48）

7阶段关系推进模型：

```
新连接 → 了解需求 → 价值回应 → 合作探索 → 意图确认 → 执行中 → 回顾复盘
```

- PoC阶段仅启用前3阶段
- **阶段不可由AI自动升级，必须用户确认**——AI可建议，但最终决策权在用户

### 5.4 智能语音助手（F-50）

专为视障用户和车载场景设计：

| 子功能 | 阶段 | 能力 |
|--------|------|------|
| 语音问答引擎 | Phase 1.1 | 日程查询 / 承诺追踪 / 关系推进问询 |
| 多轮对话管理 | Phase 1.2 | 追问上下文保持、会话超时、话题切换 |
| 主动语音提醒 | Phase 2 | 基于时间的语音推送 |

**唤醒方式**：小程序内长按语音按钮（推荐）

**能力边界**：仅支持结构化数据查询，不做开放域闲聊

### 5.5 动态优先级排序

待办不是静态列表，而是动态行动网络：

**PoC阶段（二维简化版）**：

```
优先级 = 0.4 × 紧急性 + 0.6 × 重要性
```

- 紧急性：截止时间越近越紧急
- 重要性：关系权重越高的联系人越重要

**Phase 1扩展（四维完整版）**：

```
优先级 = 0.3 × 紧急性 + 0.35 × 重要性 + 0.2 × 依赖性 + 0.15 × 场景匹配
```

- 依赖性：其他待办是否依赖此项完成
- 场景匹配：未来24小时是否有相关会议/通话

### 5.6 语义搜索

- 支持自然语言查询："上次见的那个做AI的张总"
- 支持模糊搜索和关联搜索
- 向量化语义能力：理解查询意图，而非仅匹配关键词

## 6. 产品架构概览

### 6.1 三层架构

```
┌──────────────────────────────────────────────┐
│  应用层（用户接触面）                          │
│  微信小程序（主入口）                          │
│  ├─ 名片夹 / 关系助手 / 今日日程              │
│  └─ EventLink H5（查询画像/语音录入/播报）     │
│  外部设备：AI录音卡 / 视频会议                 │
└──────────────────────────────────────────────┘
                    ↕
┌──────────────────────────────────────────────┐
│  引擎层（AI处理核心）                          │
│  多源接入 → 格式标准化 → 事件语义路由          │
│  ├─ 名片管线：轻量实体匹配（秒级）             │
│  ├─ 会议管线：深度结构化（实体+话题+决策+待办） │
│  ├─ 通话管线：要点提炼（承诺+待确认+跟进）     │
│  └─ 手动管线：补全丰富（合并+属性扩展）        │
└──────────────────────────────────────────────┘
                    ↕
┌──────────────────────────────────────────────┐
│  引擎层（关联发现+待办生成）                    │
│  实体归一 → 关联发现 → 待办生成与追踪          │
│  底层：实体图谱 + 话题标签体系 + 规则引擎       │
└──────────────────────────────────────────────┘
```

### 6.2 三大沟通通道

| 通道 | 用途 | 说明 |
|------|------|------|
| 微信服务号 | 推送通知 | 模板消息推送早晚报、待办提醒、会前准备 |
| 手机推送 | 重要提醒 | 待办到期、逾期提醒 |
| 微信小程序卡片 | 行动建议 | AI处理结果推送，用户可直接操作 |

### 6.3 微信小程序为主入口

**为什么不做原生APP？**

| 维度 | 原生APP | 微信小程序 | 结论 |
|------|---------|-----------|------|
| AI推理 | 必须走云端 | 必须走云端 | 无差异 |
| 语音能力 | 原生 | 微信同声传译 | 小程序够用 |
| 获客成本 | 高（需下载） | 低（扫码即用） | 小程序胜 |
| 开发成本 | 双端开发 | 单端开发 | 小程序胜 |
| 生态优势 | 无 | 微信分享/名片/服务号 | 小程序胜 |

**结论**：AI推理必须走云端，手机只是展示层。微信小程序已覆盖全部核心交互，原生APP无增量价值。

## 7. 部署与数据主权

### 7.1 阶段递进式部署

| 维度 | PoC（概念验证） | Phase 1（最小可用产品） | Phase 2（能力扩展） |
|------|----------------|---------------------|-------------------|
| 目标 | 验证AI能否从沟通记录提取商机+生成行动建议 | 许总团队可日常使用的完整产品 | 多场景/商业化 |
| 用户 | 内部测试（≤5人） | 许总团队（≤100人） | 付费客户（1000+人） |
| 后台位置 | 用户个人电脑 | 云服务器（单节点） | 云集群（多节点） |
| 前端 | API测试界面 | 微信小程序（原生+H5） | 独立小程序+Web管理后台 |
| 运维 | 用户自己 | 运维团队集中管理 | 运维团队+监控告警 |
| 可用性 | PC开机时可用 | 99.5% | 99.9% |
| 成本 | 零（本地） | ~500元/月 | 按用量弹性 |

### 7.2 数据主权原则

> **核心原则**：数据归用户所有，EventLink是数据的加工者而非所有者。

| 原则 | 说明 |
|------|------|
| **用户数据所有权** | 所有数据（事件/实体/关联/待办）归用户个人所有 |
| **数据可携带** | 用户可随时导出全部数据（Phase 1即支持） |
| **数据可删除** | 用户可随时删除全部数据 |
| **数据最小化** | 只收集必要数据，不收集媒体文件 |
| **处理透明** | AI处理过程可追溯，推荐可解释 |
| **私密助手定位** | 不存在资源公开/授权共享场景 |

### 7.3 长期愿景

成为用户的**终身关系经营智能体**——越用越懂你。系统随使用时长积累的关系洞察越来越精准，用户数据始终可携带。

软硬件一体愿景：录音卡/摄像头/会议系统 → 文本 → AI处理 → 推送提醒，让每一次接触都不白费。

## 8. 交付节奏

### 8.1 PoC（概念验证，3周）

**验证核心假设**：AI能否从沟通记录中提取商机、生成行动建议、识别资源线索

核心功能：
- 事件上报与AI处理（名片/会议/通话/手动4条管线）
- 实体抽取与归一
- 关联发现
- 待办生成与追踪
- 关系推进卡与关系阶段
- 输入分类与承诺识别
- 动态优先级排序（二维）

**PoC退出条件**：
- AI实体抽取准确率≥90%
- 关联发现F1>0.65
- 承诺兑现闭环验证率≥50%
- 4周持续使用率≥60%

### 8.2 Phase 1（最小可用产品，PoC后6周）

**目标**：许总团队可日常使用的完整产品

新增功能：
- 微信小程序前端（原生+H5）
- 语音录入与TTS播报
- 智能语音助手（F-50）
- 推送通知（三通道）
- 早晚报
- 名片扫描
- CSV导入
- 数据导出
- 日程管理（本地）
- 语义搜索
- 动态优先级排序（四维）

**Phase 1退出条件**：
- 许总团队DAU≥30人，持续2周
- 待办完成率≥50%
- 语音录入成功率≥85%
- 零P0安全漏洞

### 8.3 Phase 2（能力扩展，持续迭代）

**目标**：多场景/商业化

扩展方向：
- 独立小程序 + Web管理后台
- 日历双向同步
- 竞对风险预警
- 向量语义搜索增强
- 邮件解析
- 云集群弹性扩展

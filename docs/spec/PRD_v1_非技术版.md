## 1. 产品概述

### 1.1 产品愿景

**EventLink — 见面只是开始。**

> 品牌slogan：让重要的人，不止停留在微信里。
> 副slogan：记住需要，兑现承诺，让合作自然发生。
> 品牌短句：见面只是开始。

EventLink 是 **AI驱动的个人商务关系经营助手——先成就关系，再促成合作**。通过一句话记录、会议纪要和合作档案导入，帮助用户沉淀关键联系人、记住对方关注点、兑现自己的承诺、主动提供帮助，把每一次认识逐步推进为真实合作。

**产品侧重"人"而非"知识"** — 客户买单的驱动力来自"帮我经营关系"，知识是经营关系过程中的自然沉淀。

**产品是"私密助手"而非"撮合平台"** — 所有数据归用户个人所有，匹配逻辑是"你的需求匹配你人脉的供给"，不存在资源公开、授权共享或团队协作场景。资源永远附着在具体人身上（"张总能搞定工厂"），不脱离具体人独立存在。

**首期核心闭环：互动 → 关注 → 承诺 → 帮助 → 反馈**

> 从利他视角出发：我认识了谁 → 他正在关心什么 → 我答应过他什么 → 我能先为他做什么 → 双方是否自然出现合作

产品不只是记录联系人、提取待办、推荐机会，更帮助用户理解：

- 这个人和我是什么关系阶段，该怎么维护
- 他正在关心什么，我答应过他什么
- 我能先为他做什么，让合作自然发生
- 下次该用什么方式、什么理由去联系

长期愿景：成为用户的"随身关系经营助手"，软硬件一体（录音卡/摄像头/会议系统→文本→AI处理→推送提醒），让每一次接触都不白费。

**明确排除的平台级功能**（私密助手不需要）：

- ❌ RBAC权限模型 / 资源授权共享 / 团队协作
- ❌ 他人可提供资源匹配（只能匹配自己人脉的供给）
- ❌ 多租户隔离 / 企业管理看板
- ❌ 跨项目/跨团队资源撮合

### 1.2 目标用户

| 用户画像    | 核心需求        | 使用频率       | 价值感知          |
| ------- | ----------- | ---------- | ------------- |
| 商务BD/销售 | 拜访客户前了解关联背景 | 高频（每日）     | 避免信息盲区，提高拜访效率 |
| 投资人/FA  | 发现项目间隐藏关联   | 中频（每周3-5次） | 发现投资组合交叉机会    |
| 创业者     | 了解合作伙伴/竞对关系 | 中频（每周2-3次） | 规避商业风险，发现合作可能 |

### 1.3 核心价值主张

**见面只是开始。** 多数商务人士面临三大困境：**记不住**（见过的人太多，重要细节遗忘）、**想不清**（不知道对方能调动什么资源、该用什么方式维护）、**顾不上**（日常事务繁忙，错过最佳维护时机）。

**分工原则**:

- 应用层负责前端展示和用户操作，无需后端团队。但需要服务器部署、AI模型维护。
- 引擎层负责后端服务和核心算法，提供标准HTTP API（含SDK和示例代码）

### 1.5 部署模型与数据主权

> **核心决策**：EventLink采用"阶段递进式部署"——PoC本地优先、Phase 1云端托管、Phase 2弹性扩展。前端始终以微信小程序为主入口，不开发原生APP。

#### 1.5.0 为什么不做原生APP？

| 维度    | 原生APP    | 微信小程序       | 结论    |
| ----- | -------- | ----------- | ----- |
| LLM调用 | 必须走云端    | 必须走云端       | 无差异   |
| 语音录入  | 原生能力     | 微信同声传译      | 小程序够用 |
| TTS播报 | 原生能力     | 微信插件        | 小程序够用 |
| 名片扫描  | 原生相机     | wx.scanCode | 小程序够用 |
| 推送通知  | APNs/FCM | 服务号模板消息     | 小程序够用 |
| 获客成本  | 高（下载APP） | 低（扫码即用）     | 小程序胜  |
| 开发成本  | 双端开发     | 单端开发        | 小程序胜  |
| 生态优势  | 无        | 微信分享/名片/服务号 | 小程序胜  |

**结论**：LLM推理必须走云端，手机只是展示层。微信小程序已覆盖EventLink全部核心交互，原生APP无增量价值。

#### 1.5.1 阶段递进式部署模型

```
PoC（你的电脑）              Phase 1（云服务器）           Phase 2（云集群）
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Docker单机       │         │ Docker Compose   │         │ K8s集群          │
│ ├─ FastAPI       │  迁移   │ ├─ FastAPI       │  扩展   │ ├─ FastAPI×N     │
│ ├─ SQLite        │ ──────→ │ ├─ PostgreSQL    │ ──────→ │ ├─ PG+Redis+ES  │
│ └─ NullMemory    │         │ ├─ Redis         │         │ └─ pgvector     │
│                  │         │ └─ CarryMem(opt) │         │                  │
└─────────────────┘         └─────────────────┘         └─────────────────┘
     ↕ API                       ↕ API                       ↕ API
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Swagger UI      │         │ 名片小程序        │         │ 独立小程序+Web   │
│ (开发测试用)     │         │ ├─ 原生(语音/TTS) │         │ ├─ 原生+Web     │
│                 │         │ └─ H5(查询/管理)  │         │ └─ 全功能       │
└─────────────────┘         └─────────────────┘         └─────────────────┘
```

| 维度       | PoC（概念验证）        | Phase 1（最小可用产品）         | Phase 2（能力扩展）       |
| -------- | ---------------- | ----------------------- | ------------------- |
| **后台位置** | 用户个人电脑           | 云服务器（单节点）               | 云集群（多节点）            |
| **部署方式** | Docker单机         | Docker Compose          | K8s                 |
| **数据库**  | SQLite（本地文件）     | PostgreSQL 15 + Redis 7 | PG + Redis + (按需ES) |
| **前端**   | API + Swagger UI | 名片小程序(原生+H5)            | 独立小程序+Web管理后台       |
| **认证**   | API Key          | JWT RS256+临时授权码         | OAuth2（无RBAC）       |
| **运维**   | 用户自己             | 运维团队集中管理                | 运维团队+监控告警           |
| **升级方式** | git pull + 重启    | CI/CD自动部署               | 滚动更新                |
| **备份**   | 用户自己             | 自动每日备份                  | 自动+异地容灾             |
| **可用性**  | PC开机时可用          | 99.5%                   | 99.9%               |
| **成本**   | 零（本地）            | \~¥500/月（单节点云）          | 按用量弹性               |

#### 1.5.2 数据主权原则

> **核心原则**：数据归用户所有，EventLink是数据的加工者而非所有者。

| 原则          | 说明                              | 实现方式                   |
| ----------- | ------------------------------- | ---------------------- |
| **用户数据所有权** | 用户创建的所有数据（事件/实体/关联/Todo）归用户个人所有 | user\_id数据隔离，无跨用户访问    |
| **数据可携带**   | 用户可随时导出全部数据                     | F-21数据导出（Phase 2），标准格式 |
| **数据可删除**   | 用户可随时删除全部数据                     | API支持批量删除+账号注销         |
| **数据最小化**   | 只收集必要数据，不收集媒体文件                 | 仅文本输入，无音频/视频/图片        |
| **处理透明**    | AI处理过程可追溯                       | 匹配推荐可解释，Todo来源可追溯      |
| **私密助手定位**  | 不存在资源公开/授权共享场景                  | 无RBAC、无跨用户匹配、无团队协作     |

**PoC阶段数据主权**：

- 数据存储在用户个人电脑的SQLite文件中
- LLM调用走HTTPS，仅发送脱敏后的文本片段
- 用户完全控制数据的启动/停止/备份/删除

**Phase 1+数据主权**：

- 数据存储在云端PostgreSQL，user\_id隔离
- 客户端加密可选（Phase 2+）
- 数据导出/删除API始终可用
- 云端备份加密存储

#### 1.5.3 PoC→Phase 1 迁移路径

**迁移成本极低**：SQLite→PG只需改连接字符串（SQLAlchemy抽象层），API层零改动。

| 迁移项      | PoC        | Phase 1             | 迁移方式              |
| -------- | ---------- | ------------------- | ----------------- |
| 数据库      | SQLite     | PostgreSQL          | SQLAlchemy连接字符串切换 |
| 缓存       | 内存dict     | Redis               | 配置切换              |
| 认证       | API Key    | JWT                 | 中间件替换             |
| CarryMem | NullMemory | 可选接入                | 配置切换              |
| 前端       | Swagger UI | 小程序                 | 新增，不影响API         |
| 数据       | 手动迁移脚本     | pg\_dump/sqlite3→pg | 一次性脚本             |

### 1.6 用户旅程定义

> **设计原则**：3条核心旅程覆盖产品最关键的使用场景，每条旅程定义完整的操作步骤、系统响应和体验目标。
> **架构约束**：EventLink不处理媒体文件（音频/视频/图片），所有媒体→文本转换由外部应用完成，EventLink仅接收文本输入。

#### 1.5.1 旅程一：新用户首次体验（冷启动）

```
用户目标：快速理解产品价值，完成第一个行动
体验目标：从打开到完成第一个行动 < 2分钟

步骤1: 信任承诺（第1屏）
  系统 → 显示信任承诺页
         "这是你的私人关系助手。你记录的人和交流内容，只用于帮助你更好地回应重要关系。
          不公开、不匹配给其他用户、不自动联系任何人。"
  用户 → 点击"记录一次重要交流"

步骤2: 一句话输入（第2屏）
  系统 → 显示输入界面
         "最近，你和谁有过一次值得记住的交流？"
         输入示例：今天认识王主任，她在做社区长者活动，提到缺少年轻化运营和线上宣传。我答应明天发一份活动方案给她。
  用户 → 输入/语音录入交流内容
  用户 → 点击"帮我整理下一步"

步骤3: 用户确认重点（第3屏）
  系统 → 展示AI提取的重点，请用户确认
         | 项目 | 提取内容 |
         |------|---------|
         | 联系人 | 王主任 |
         | 她正在关心 | 社区长者活动运营与线上宣传 |
         | 我答应的事 | 发送一份活动方案 |
         | 建议完成时间 | 明天下午前 |
  用户 → 确认/修正提取内容

步骤4: 立即获得价值（第4屏）
  系统 → 展示行动建议
         "你可以先为王主任做一件小事"
         建议行动：明天下午前发送一份简版活动案例
         可复制话术：（生成温和微信表达）
  用户 → 点击"添加提醒" / "复制话术" / "稍后处理"
  系统 → 标记为in_progress，设置默认截止时间(可修改)
  系统 → 显示"已添加到待办"确认提示
```

**关键体验指标**:

- 引导到首次录入转化率>80%
- AI提取内容用户确认率>70%
- 从安装到完成第一个行动完成率>60%

#### 1.5.2 旅程二：日常使用（会前准备→会中记录→会后跟进）

```
用户目标：围绕会议全流程，不遗漏商机和行动
体验目标：每个环节操作<30秒

环节A: 会前准备(每日早晨自动+按需手动)
  自动流程(每日早晨):
    系统 → 检查当天日历(本地日程+已同步的外部日历)
    系统 → 对每个今日会议，自动生成准备清单:
           - 参会人基础信息+关联关系
           - 破冰话题建议(校友/前同事/同城市等)
           - 竞对/风险提示(如有)
    系统 → 推送今日会议准备摘要(与F-19每日简报合并推送)
  
  手动流程(按需):
    用户 → 点击某会议的"会前准备"按钮
    系统 → 按最新数据重新生成准备清单(实时查询)
    系统 → 如该会议不在外部日历中:
           → 询问"是否添加到外部日历？"
           → 用户确认后同步到外部日历(F-30)

环节B: 会中(外部应用记录，EventLink不参与)
  用户 → 使用腾讯会议/飞书/录音卡等外部应用记录会议
  外部应用 → 完成录音转写，生成会议纪要文本
  (EventLink不录音、不转写、不记录会议，仅等待外部应用提交文本)

环节B+: 外出拜访场景（移动端，许总核心场景）
  用户目标：在外拜访时，不漏信息、不费眼睛，靠听和说完成交互
  体验目标：会前速览<10秒，语音播报<30秒，语音录入<5秒

  会前（开车/走路去拜访）:
    系统 → 微信服务号推送会前提醒（提前15-60分钟）
         → 仅包含：时间+姓名+公司（不含敏感信息，详见§5.16.4推送分级）
    用户 → 打开名片小程序 → 点击"今日日程"
    系统 → 展示今天要见的人列表
    用户 → 点击某人 → 选择播报级别：
      🔊极简播报(5秒): "张总，XX科技供应链总监。上次聊了新项目。"
      🔊完整播报(30秒): "张总，XX科技供应链总监。关系阶段：单独见过面。
           偏好电话沟通。上次聊了新项目预算。建议：可提华南区代工厂对接。"
    用户 → 边开车边听完，不用看屏幕（许总反馈：眼神不好，喜欢听语音）
    注：播报级别默认standard，可在设置中调整；basic级适合公共场合

  会中（录音卡静默采集 + 替代方案）:
    主方案：手机背后录音卡自动录音（无需操作）→ 后台转写文本
    替代方案（对方不同意录音/录音卡不可用时）:
      - 快捷笔记：小程序内"快速记录"模板 "见了[谁] 聊了[话题] 下次[行动]"
      - 会后补录：会议结束后语音录入或手动输入
    (EventLink不录音、不转写、不记录会议，仅等待外部应用提交文本)

  会后（刚结束，还在外面）:
    系统 → 微信小程序卡片推送AI行动建议
    用户 → 打开名片小程序 → 查看行动建议
    用户 → 或直接语音录入："今天和张总聊了新项目预算，他提到华南区3个代工厂"
    小程序 → 语音转文本 → 展示识别结果供确认（✓提交/✗重录/✎编辑）
    用户 → 确认提交
    小程序 → 调用EventLink API
    EventLink → meeting管线处理 → 生成行动项
    系统 → 推送处理结果（微信服务号模板消息）

  日常碎片（等电梯/坐车）:
    系统 → 微信服务号推送维护提醒："李总2周未联系，建议发微信问候"
    用户 → 点击提醒 → 打开名片小程序 → 一键发微信/标记已联系

环节C: 会后跟进(接收外部文本→AI处理→行动项)
```

## 2. 用户故事

### 2.1 机会发现场景（⚪ Opportunity）

**US-01: 商机匹配发现**

> 作为商务BD，当扫描一张数字名片时，系统自动发现该名片持有人与系统中另一联系人存在商机匹配（技术供需/业务互补/资源对接，匹配度≥0.80），并在5秒内推送⚪商机Todo+🟢破冰行动，以便及时建立连接或牵线搭桥。

**验收标准**: 商机匹配度≥0.80时，系统在事件上报后10秒内生成⚪机会Todo，包含双方姓名、匹配维度、匹配分数和1-3条🟢破冰行动（如共同话题、引荐话术、切入角度）。

**US-02: 跨事件合作线索发现**

> 作为投资人，当两次不同会议中出现同一技术领域的关键人物时，系统自动发现两人存在技术重叠关联，推送潜在合作机会Todo，以便评估联合投资可能。

**验收标准**: 当两个person实体在同一technology实体下共现且tech\_overlap关联强度>0.5时，系统生成⚪机会Todo。

**US-03: 供应链关联发现**

> 作为创业者，当系统发现新接触的供应商与现有客户存在供应链上下游关系时，推送供应链关联Todo，以便优化采购或销售策略。

**验收标准**: 当两个organization实体间建立supply\_chain关联且strength>0.4时，系统生成⚪机会Todo。

### 2.2 风险预警场景（🔴 Risk）— Phase 2+ 实现

> **优先级说明**：Phase 1数据源有限（无爬取、无付费API），竞对关系数据极度稀疏，硬做风险预警易产生大量误报。Phase 1仅建立competitor/risk\_link关联类型的数据结构，不主动推送🔴风险Todo。Phase 2引入竞对数据管理(F-08)后启用。

**US-04: 竞对关系预警**（Phase 2）

> 作为商务人士，当扫描一张名片时，系统发现该名片所属公司与系统中已有联系人所在公司存在直接竞争关系，立即推送🔴风险Todo，以便在沟通中注意信息边界。

**验收标准**: 当两个organization实体间建立competitor关联且strength>0.5时，系统在5秒内生成🔴高风险Todo，包含竞对依据和沟通建议。

**US-05: 风险关联链预警**（Phase 2）

> 作为商务人士，当系统发现新接触的人与已知风险实体存在风险关联链（如竞对公司的前员工），推送🔴风险Todo，以便评估合作风险。

**验收标准**: 当person实体与已有risk\_link关联的实体建立新关联时，系统生成🔴风险Todo。

### 2.3 背景补充场景（🔵 Context）

**US-06: 人脉关系背景补充**

> 作为商务人士，当查看今日待办时，系统推送🔵背景Todo，告知即将会面的联系人与自己存在人脉关联（校友/前同事/同城/同行业圈/投资偏好匹配等），以便在沟通中建立亲近感或针对性准备。

**验收标准**: 当两个person实体间建立alumni/ex\_colleague/同城等关联，或投资偏好与项目领域匹配度>0.6时，系统生成🔵背景Todo，包含关系类型、关系依据和破冰话题建议。

### 2.4 实体归一场景

**US-08: 实体归一人工确认**

> 作为系统管理员，当实体归一引擎产生置信度在\[0.70, 0.85)区间的归一建议时，系统在管理界面展示待确认队列，包含完整属性对比、AI归一推理、置信度分数和历史相似案例，管理员可执行"确认归一"/"拒绝归一"/"稍后处理"三种操作，确认结果反馈至模型用于持续改进。

**验收标准**: 待确认队列展示所有\[0.70, 0.85)区间的归一建议，每条建议包含属性对比表、AI推理说明、置信度数值；操作后系统在200ms内响应。

**US-09: 实体归一撤回**

> 作为系统管理员，当发现已自动合并的实体（置信度≥0.85）存在误合并时，可在管理界面一键撤回，系统自动拆分实体并恢复原始状态，同时记录操作日志。

**验收标准**: 撤回操作后实体恢复为合并前的独立状态，关联关系正确还原，操作日志完整记录撤回时间、操作人和原始合并信息。

### 2.5 数据录入场景

**US-10: 名片扫描事件上报**

**验收标准**: API在200ms内返回event\_id和processing状态；后台处理完成后，相关Todo在10秒内可查询。

### 2.6 行动建议场景

**US-11: 会议行动项自动分解**

### 2.7 数据管理场景

**US-13: 实体信息浏览与修正**

**验收标准**: 实体详情页展示全部属性+来源标记(AI提取/手动录入/用户确认)；provisional属性可一键确认或修正；属性编辑后200ms内持久化；支持删除实体（级联清理关联）。

**US-14: 事件与关联的浏览管理**

**验收标准**: 会议结束后推送"记录会议"提醒；语音/文字输入1分钟内完成；自动生成分类Todo并关联到相关实体；默认跟进时间3天后提醒。

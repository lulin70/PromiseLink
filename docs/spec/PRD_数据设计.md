## 4. 非功能需求

### 4.1 性能需求

| 指标                 | 目标值       | 测量方法                        |
| ------------------ | --------- | --------------------------- |
| 事件上报API响应时间        | P95<200ms | 压测工具（locust/k6），1000次请求取P95 |
| 实体抽取延迟（LLM路径）      | P95<2s    | 计时器测量，100次调用取P95            |
| 实体抽取延迟（规则路径）       | P95<200ms | 计时器测量，1000次调用取P95           |
| 关联发现延迟             | P95<2s    | 计时器测量，100次调用取P95            |
| 端到端延迟（事件上报→Todo可查） | <10s      | 完整链路计时，20次取最大值              |
| 商机匹配度计算延迟          | P95<1s    | 计时器测量，100次调用取P95            |
| Todo查询API响应时间      | P95<500ms | 压测工具，1000次请求取P95            |
| 并发用户支持             | ≥50并发     | 压测工具，50并发下P95延迟不退化          |
| 数据库查询（单实体）         | <50ms     | EXPLAIN ANALYZE验证           |
| 数据库查询（关联遍历2跳）      | <200ms    | EXPLAIN ANALYZE验证           |

### 4.2 安全需求

| 维度        | 方案                                            | 验收标准                  |
| --------- | --------------------------------------------- | --------------------- |
| 传输加密      | TLS 1.3 + HSTS                                | 安全扫描无TLS漏洞            |
| 身份认证      | JWT（access\_token: 15min, refresh\_token: 7d） | 未认证请求100%返回401        |
| 权限控制      | JWT认证 + 单用户数据隔离（无RBAC）               | 跨用户数据访问100%被阻止         |
| 数据加密      | AES-256-GCM 字段级加密（敏感属性）                       | 敏感字段数据库存储为密文，无媒体文件存储 |
| API Key管理 | 操作系统钥匙串存储（Keychain/Credential Manager）        | API Key不明文出现在配置文件和日志中 |
| 审计日志      | 所有写操作记录（操作人/IP/时间/内容）                         | 审计日志完整性100%，保留期≥90天   |
| 合规性       | GDPR支持（数据导出Art.20/删除Art.17/撤回同意）              | 数据删除请求24h内完成，包含级联关联清理 |
| 输入验证      | SQL注入/XSS/CSRF防护                              | OWASP Top 10扫描无高危漏洞   |

## 5. 数据需求

### 5.1 数据来源

| 数据类别     | Phase 1来源               | Phase 2+来源               | 排除项                          |
| -------- | ----------------------- | ------------------------ | ---------------------------- |
| 名片/联系人信息 | IAMHERE数字名片JSON         | 录音卡App、视频会议纪要            | —                            |
| 竞对关系数据   | 公开网页信息检索（公司官网/新闻稿/行业报告） | 专业数据服务（验证后引入）            | ❌公开网页爬取（法律风险）、❌采购数据爬取（仅口头沟通） |
| 人脉关系数据   | 用户手动输入                  | 第三方付费服务（投资人画像API等）       | —                            |
| 设备音视频    | AI眼镜/大疆设备录制（Phase 2+）   | 声纹识别+多声轨分离               | 当前设备仅支持3人声轨                  |
| 线下活动数据   | 活动参与者自我介绍批量采集（Phase 2+） | 标准化信息收集流程                | 许总每周45场活动作为数据源               |
| 会议/对话内容  | 手动输入                    | AI录音卡（恒智易R1）转写、腾讯/飞书会议纪要 | —                            |

**关键决策**:

- Phase 1完全基于自有数据和用户主动输入运行
- 每期核心功能不依赖后续期数据源
- 第三方付费数据服务延期至付费版

### 5.2 数据流

```
数据源                    标准化+路由                引擎处理                  输出
───────                  ──────────                ────────                ────

IAMHERE名片JSON ──→  Event标准化管道 ──→  实体归一引擎 ──→  Todo生成与追踪引擎
                      │                  │                  │
录音卡转写文本 ──→    │ 格式转换         │ 别名扩展         │ ┌─ card_save:
                      │ 事件语义路由     │ 上下文相似度     │ │  ⚪🔵信息型+🟢破冰行动
会议纪要 ──→        │ ┌─card_save轻量  │ 置信度分级       │ │
手动输入 ──→          │ │ meeting深度    │ 人工确认         │ ├─ meeting:
                      │ │ call要点       │                  │ │  ⚪🔵信息型+🟢行动分解
                      │ └─manual补全    │                  │ │  (按A/B/C/D类型)
                      │                  │                  │
                      │ 实体抽取(管线化) │                  │ ├─ call:
                      │ ┌─轻量:3类实体   │                  │ │  ⚪🔵信息型+🟢跟进行动
                      │ │ 深度:5类+话题  │                  │ │
                      │ │   +决策+行动   │ 关联发现引擎     │ └─ manual:
                      │ │ 要点:2类+承诺  │ │                │    ⚪🔵增量+合并建议
                      │ └─补全:缺失实体  │ │ 共现分析        │
                      │                  │ │ 类型推断        │
                      │ 话题标签提取     │ │ 话题相似度      │
                      │ (见§5.5)         │ │ 时间衰减        │
                      │ 人脉关系提取     │ ↓                │
                      │ (meeting/call)   │ 商机匹配度计算   │
                      │                  │ (含话题维度)     │
                      │                  └──────────────────┘
```

### 5.3 核心数据模型

#### Event（事件）

| 字段          | 类型        | 必填 | 说明                                                  |
| ----------- | --------- | -- | --------------------------------------------------- |
| id          | UUID v4   | 是  | 唯一标识                                                |
| event\_type | enum      | 是  | card\_scan / meeting / call / manual                |
| source      | string    | 是  | iamhere / recording\_r1 / tencent\_meeting / manual |
| title       | string    | 是  | 人类可读标题，≤200字符                                       |
| timestamp   | ISO 8601  | 是  | 事件发生时间                                              |
| raw\_text   | text      | 是  | 原始内容（JSON字符串/转写文本），≤500KB                          |
| entities    | Entity\[] | 否  | 抽取出的实体列表（引擎填充）                                      |
| metadata    | JSONB     | 否  | 扩展字段（来源应用信息、时长、参会人数等，不含媒体文件URL）        |

#### Entity（实体）

| 字段             | 类型        | 必填 | 说明                                                       |
| -------------- | --------- | -- | -------------------------------------------------------- |
| id             | UUID v4   | 是  | 唯一标识                                                     |
| entity\_type   | enum      | 是  | person / organization / technology / project / attribute |
| name           | string    | 是  | 显示名称，≤100字符                                              |
| aliases        | string\[] | 否  | 别名列表，如\["李总","老李"]                                       |
| properties     | JSONB     | 否  | 属性字典（company/title/skills/education等）+ **resource**（可调用资源列表）+ **demand**（需求列表） |
| source\_events | UUID\[]   | 是  | 来源事件ID列表                                                 |
| created\_at    | ISO 8601  | 是  | 创建时间                                                     |
| updated\_at    | ISO 8601  | 是  | 更新时间                                                     |

#### Association（关联）

| 字段             | 类型       | 必填 | 说明                                                                                                         |
| -------------- | -------- | -- | ---------------------------------------------------------------------------------------------------------- |
| id             | UUID v4  | 是  | 唯一标识                                                                                                       |
| source\_entity | UUID     | 是  | 源实体ID                                                                                                      |
| target\_entity | UUID     | 是  | 目标实体ID                                                                                                     |
| relation\_type | enum     | 是  | alumni / ex\_colleague / same\_city / competitor / tech\_overlap / deal\_link / risk\_link / supply\_chain |
| strength       | float    | 是  | 关联强度 0.0\~1.0                                                                                              |
| evidence       | UUID\[]  | 否  | 支撑证据（事件ID列表）                                                                                               |
| discovered\_at | ISO 8601 | 是  | 发现时间                                                                                                       |
| status         | enum     | 是  | active / stale / dismissed                                                                                 |

#### Todo（待办项，原Alert）

> **语义**：Todo = 因为某个事件(原因)，对某个实体(对象)，维护某段关系(目标)

| 字段                   | 类型       | 必填 | 说明                                                  |
| -------------------- | -------- | -- | --------------------------------------------------- |
| id                   | UUID v4  | 是  | 唯一标识                                                |
| todo\_type           | enum     | 是  | promise🤝 / help🫶 / care💙 / followup🔄 / cooperation\_signal✨ / risk🔴 |
| priority             | enum     | 是  | high / medium / low                                 |
| title                | string   | 是  | Todo标题，≤100字符                                       |
| detail               | string   | 是  | Todo详情                                              |
| suggestion           | string   | 否  | 行动建议                                                |
| pipeline             | enum     | 是  | card\_save / meeting / call / manual — 标识来源管线       |
| actions              | JSONB    | 否  | 行动项列表（仅🟢action类型和meeting管线），结构因管线而异                |
| source\_event        | UUID     | 否  | **为什么**产生此Todo（来源事件ID，系统提醒无来源事件时为null） |
| target\_entities     | UUID[]   | 是  | **对谁**做（目标实体ID列表，可多人）                           |
| target\_association  | UUID     | 否  | **维护哪段关系**（关联ID，关系维护型Todo必填）              |
| status               | enum     | 是  | pending / in\_progress / done / dismissed / snoozed |
| due\_date            | ISO 8601 | 否  | 截止时间（行动型必有，信息型可选）                                   |
| user\_feedback       | enum     | 否  | useful / not\_useful / null                         |
| created\_at          | ISO 8601 | 是  | 创建时间                                                |
| updated\_at          | ISO 8601 | 是  | 更新时间（状态变更时更新）                                       |
| completed\_at        | ISO 8601 | 否  | 完成时间                                                |

### 5.4 领域分类体系设计（Domain Taxonomy）

> **设计原则（来自许总反馈）**：大厂有成熟的行业分类体系，EventLink不需要照搬，但需要了解后选择自己的方向。本节定义EventLink自有的轻量级领域分类策略。

#### 5.4.1 分类策略：方案D — 一级预定义 + 二级动态扩展

| 层级          | 定义方式             | 数量    | 用途                               | 维护方式                  |
| ----------- | ---------------- | ----- | -------------------------------- | --------------------- |
| **L1 一级领域** | 硬编码预定义           | 18±3个 | F-05商机匹配度"行业一致性"维度、F-08竞对判断的行业归属 | 配置文件，版本管理             |
| **L2 二级标签** | LLM动态生成 + 用户反馈收敛 | 不设上限  | 实体属性丰富化、精准匹配                     | Provisional标记，人工确认后固化 |

**选择理由**：

- L1硬编码确保核心功能的可验证性和可复现性
- L2动态扩展避免过早锁定细分方向，允许从用户数据中学习
- 完美平衡"知道大厂分类→选自己方向→不完全照搬"的战略定位

#### 5.4.3 L2 二级标签生成规则

```
L2标签来源：
├── LLM抽取（从实体raw_text中提取，标记为provisional）
├── 用户手动标注（优先级最高）
├── 公开网页信息检索（标记为provisional）
└── 反馈闭环收敛（用户确认后provisional→confirmed）

Provisional标签规则：
├── 自动生成时标记为provisional
├── provisional标签不参与F-05匹配度计算（仅L1参与）
├── 用户或管理员确认后升级为confirmed
└── confirmed标签计入F-05匹配度计算
```

| 维度   | 要求                          | 验证方式                |

### 5.5 话题标签体系设计（Topic Tag Taxonomy）

> **设计背景**：许总分享的分类截图更偏向文本话题维度（非纯行业归类）。EventLink需要在行业分类（§5.4）之外，建立话题标签体系，用于提升匹配精度、丰富实体属性、以及会议类型分类。话题标签通过不同管线融入处理流程——轻量管线(card\_scan)不提取话题，深度管线(meeting)和要点管线(call)提取话题标签。

#### 5.5.1 话题标签与行业分类的分工

| 维度         | 行业分类(§5.4 L1/L2)   | 话题标签(§5.5)                     |
| ---------- | ------------------ | ------------------------------ |
| **分类对象**   | 组织实体（公司/机构）        | 文本内容（需求描述/技能表述/会议议题）           |
| **解决什么问题** | 这家公司跟谁竞争？属于哪个赛道？   | 这个需求在说什么？能跟谁的能力配？这场会议属于什么类型？   |
| **数据来源**   | 公司属性、公开信息          | 事件raw\_text中的LLM抽取             |
| **使用管线**   | 所有管线               | 仅meeting管线和call管线              |
| **参与匹配度**  | F-05行业一致性维度(+0.25) | F-05话题语义相似度维度(+0.20)           |
| **会议类型分类** | 不参与                | meeting管线通过话题标签判定会议类型(A/B/C/D) |

**核心洞察**：同一行业的两家公司可能关注完全不同的话题；不同行业的公司可能关注同一话题。行业分类解决组织层面问题，话题标签解决内容层面问题。话题标签还承担会议类型分类的职责——将许总分享的各类文本分类归纳到会议的四大类型及子类。

#### 5.5.2 话题标签生成规则

```
话题标签来源（仅meeting和call管线触发）：
├── LLM从raw_text中提取（标记为provisional）
│   ├── meeting管线：提取需求话题、技能话题、行业热词
│   └── call管线：提取关键话题、痛点话题
├── 用户手动标注（优先级最高，直接confirmed）
└── 反馈闭环收敛（用户确认后provisional→confirmed）

话题标签与实体的关联：
├── person实体 → skills话题标签（如"边缘计算""React"）
├── organization实体 → 业务话题标签（如"工厂智能化改造""私域运营"）
├── technology实体 → 技术话题标签（如"大模型推理""IoT网关"）
└── project实体 → 需求话题标签（如"产线数字化""供应链优化"）
```

#### 5.5.3 话题标签在管线中的使用

| 管线                 | 话题标签使用方式                            | 输出              |
| ------------------ | ----------------------------------- | --------------- |
| **card\_scan(轻量)** | 不提取话题标签，仅使用已有confirmed标签参与匹配        | 匹配结果沿用已有话题维度    |
| **meeting(深度)**    | 从会议纪要中提取话题标签→丰富实体属性→参与F-05话题语义相似度计算 | 新增话题标签+匹配结果+行动项 |
| **call(要点)**       | 从通话记录中提取关键话题→丰富实体属性                 | 新增话题标签+跟进计划     |
| **manual(补全)**     | 不提取话题标签，仅确认/拒绝待定标签                  | 标签状态更新          |

#### 5.5.4 话题标签与F-05商机匹配度的关系

F-05商机匹配度五维打分法：

| 维度                  | 权重        | 说明                         | 变化     |
| ------------------- | --------- | -------------------------- | ------ |
| 技能-需求关键词重叠(Jaccard) | +0.30     | 原有，权重从0.35微调               | ↓0.05  |
| 行业领域一致性(§5.4)       | +0.25     | 原有，不变                      | —      |
| **话题语义相似度(§5.5)**   | **+0.20** | **新增：需求话题与技能话题在话题空间的语义距离** | **新增** |
| LLM语义理解             | +0.15     | 原有，权重从0.20微调               | ↓0.05  |
| 历史合作信号              | +0.05     | 原有，不变                      | —      |
| **合计**              | **0.95**  | 预留0.05给未来维度扩展              | —      |

**话题语义相似度计算**：

- 输入：实体A的话题标签集合 + 实体B的话题标签集合
- 计算：LLM embedding向量的余弦相似度，取均值
- 阈值：相似度≥0.7计满分0.20，\[0.4, 0.7)按比例折算，<0.4计0

#### 5.5.5 话题标签生命周期

```
LLM提取 → provisional标签
              │
              ├─ 用户确认 → confirmed标签 → 参与F-05匹配度计算
              ├─ 用户拒绝 → discarded → 不再显示
              └─ 30天无操作 → auto_expired → 清理
```

#### 5.5.6 话题标签与领域分类的关联

| 关联方式    | 说明                         | 示例                                 |
| ------- | -------------------------- | ---------------------------------- |
| 话题→L1归属 | 每个话题标签可映射到1-2个L1领域         | "工厂智能化改造"→D02智能制造                  |
| 交叉验证    | 话题标签推断的L1与实体已有L1不一致时，标记待确认 | 公司L1=D06零售电商，但话题="供应链优化"→可能L1应为D08 |
| L2标签来源  | 高频confirmed话题标签可升级为L2领域标签  | "边缘计算"出现≥20次→升级为D02的L2标签           |

#### 5.5.7 话题标签与会议类型分类的映射

meeting管线在提取话题标签后，通过话题组合判定会议类型（对应F-14的4类输出Schema）：

| 会议类型       | 话题标签特征           | 典型话题组合                   | 对应F-14输出                                        |
| ---------- | ---------------- | ------------------------ | ----------------------------------------------- |
| **A.内部协同** | 权责、分账、团队、合伙人、招商  | \["团队分工","分账比例","合伙人机制"] | topics+consensus+actions+faq                    |
| **B.对外商务** | 客户、方案、报价、竞标、谈判   | \["客户需求","方案汇报","商务条款"]  | clientBg+painPoints+solution+commercial+actions |
| **C.项目复盘** | 复盘、回顾、总结、改进、GRAI | \["项目回顾","经验总结","流程优化"]  | goal+keep+improve+prevent+insights+actions      |
| **D.知识沉淀** | SOP、话术、培训、标准、规范  | \["流程标准化","话术整理","培训大纲"] | knowledgeTags+assets+sop+talkScript             |

**分类逻辑**：

1. LLM从会议raw\_text中提取话题标签
2. 话题标签与4类会议的关键词特征匹配
3. 取匹配度最高的类型作为会议类型
4. 匹配度均<0.4时，默认为A类（内部协同）
5. 用户可在前端修正会议类型，修正结果反馈至模型

**与许总文本分类的关系**：许总分享的各类文本分类（如行业标签、话题标签、内容类型等），可通过本节映射机制归纳到4大会议类型及其子类。例如：许总分类中的"商务谈判"→B类子类，"团队管理"→A类子类，"案例复盘"→C类子类，"流程规范"→D类子类。

### 5.7.1 原始文件存储策略

> **核心架构决策**：EventLink**提供但解耦**媒体处理能力（ASR/TTS/OCR）。媒体文件仅在内存中处理，不持久化存储；仅提取的文本/结构化数据入库。媒体服务通过独立API前缀(/media/)提供，可通过配置禁用。外部应用仍可直接提交文本输入，媒体处理是可选增强而非强制依赖。这是"提供便利但不耦合"原则——EventLink既支持文本直入，也提供一站式媒体→文本→洞察→行动的完整链路。

#### 输入边界定义

| 输入类型 | 谁负责转文本 | EventLink接收什么 | 说明 |
|---------|------------|-----------------|------|
| 名片照片 | IAMHERE数字名片小程序 | 结构化JSON（姓名/公司/职位/联系方式） | 许总负责的前端层 |
| 录音 | 恒智易R1录音卡/腾讯会议 | 转写文本 | 外部应用完成ASR |
| 视频会议 | 腾讯会议/飞书/Zoom | 会议纪要文本 | 平台自带转写功能 |
| 通话录音 | 手机/录音App | 转写文本 | 外部ASR服务 |
| 手动输入 | 用户直接输入 | 文本 | 无需转换 |
| CSV/Excel | 用户上传 | 解析后的文本数据 | 唯一的"文件"输入 |

#### 存储策略（仅文本）

| 数据类型 | 存储方式 | 保留策略 | 大小限制 |
|---------|---------|---------|---------|
| 事件原始文本 | Event.raw_text（PostgreSQL TEXT） | 永久保留 | ≤500KB（约15万字） |
| 名片结构化数据 | Event.raw_text（JSON字符串） | 永久保留 | ≤10KB |
| CSV/Excel导入 | 临时解析，不保留原始文件 | 导入后即删除 | ≤50MB |
| AI提取结果 | Entity/Association/Todo表 | 永久保留 | — |

> **注意**：EventLink不需要对象存储(S3/OSS)，不需要CDN。媒体文件在内存中处理完毕后立即释放，不写入持久化存储。所有持久化存储均为PostgreSQL数据库内的文本字段。TTS音频通过流式响应返回，不落盘。

#### 外部应用对接规范

外部应用调用EventLink API时，必须提供已转写的文本：

```
POST /api/v1/events
{

  "source": "tencent_meeting",   // iamhere / recording_r1 / tencent_meeting / manual
  "title": "与华为技术合作讨论",
  "timestamp": "2026-06-01T14:00:00+08:00",
  "raw_text": "张总：我们对AI方向很感兴趣...（已转写文本）",
  "metadata": {

```

**对接要求**：
- 外部应用必须在调用API前完成OCR/ASR转文本
- raw_text字段为必填，不可为空
- metadata中可携带来源应用信息，但不包含媒体文件URL
- EventLink不提供媒体上传接口

### 5.7.2 LLM上下文窗口处理策略

> **核心问题**：当前主流LLM上下文窗口为128K-200K tokens（约5-8万字中文），但长会议转写可能超过此限制。需要分段处理策略。

#### 分段处理流程

```
原始文本输入
│
├─ 文本 ≤ 3万字（约8K tokens）
│  └─ 直接整段送入LLM处理 ✅
│
├─ 3万字 < 文本 ≤ 15万字
│  └─ 滑动窗口分段处理：
│     1. 按段落/发言者切分为N段（每段≤3万字）
│     2. 每段独立提取实体+要点
│     3. 合并阶段：将各段提取结果汇总，去重实体，合并关联
│     4. 全局推理：基于合并结果生成Todo和关联发现
│
└─ 文本 > 15万字（极端情况）
   └─ 先摘要后处理：
      1. 第一轮LLM：按段生成摘要（每段压缩至3000字）
      2. 第二轮LLM：基于全文摘要提取实体+关联+Todo
      3. 原始文本保留，用户可查看但不再二次处理
```

#### 各管线处理策略

| 管线 | 典型文本长度 | 处理策略 | LLM调用次数 |
|------|------------|---------|------------|
| card_save | <500字 | 整段处理 | 1次 |
| call | 3000-3万字 | 整段或2段处理 | 1-2次 |
| meeting | 1万-15万字 | 滑动窗口分段 | 2-5次 |
| manual | <3万字 | 整段处理 | 1次 |

#### Token消耗预估与成本控制

| 场景 | 输入tokens | 输出tokens | 单次成本(估算) | 日均上限 |
|------|----------|----------|-------------|---------|
| 名片扫描 | ~500 | ~300 | ¥0.01 | 100次/天 |
| 通话记录 | ~5K | ~2K | ¥0.05 | 50次/天 |
| 会议纪要(短) | ~20K | ~5K | ¥0.15 | 20次/天 |
| 会议纪要(长) | ~80K | ~15K | ¥0.60 | 10次/天 |

**成本控制措施**：
- 单用户每日LLM调用Token上限：500K tokens
- 超限后降级为规则引擎处理（spaCy+NLP规则）
- 相同文本不重复调用LLM（基于raw_text哈希去重）
- 缓存LLM提取结果，实体属性增量更新而非全量重提取

### 5.10 离线与弱网策略

> **设计原则**：商务人士在地铁/电梯/地下会议室等弱网环境高频使用，核心功能必须离线可用。

#### 5.10.1 离线能力矩阵

| 功能          | 在线  | 弱网(<100KB/s) | 离线            |
| ----------- | --- | ------------ | ------------- |
| 名片扫描(本地OCR) | ✅完整 | ✅完整          | ✅完整(本地OCR)    |
| 名片扫描(关联发现)  | ✅完整 | ⚠️延迟增加       | ❌仅保存本地，联网后同步  |
| Todo查看      | ✅完整 | ✅完整          | ✅完整(缓存最近100条) |
| Todo状态操作    | ✅完整 | ⚠️延迟同步       | ✅本地操作，联网后同步   |
| 实体搜索        | ✅完整 | ⚠️仅本地        | ⚠️仅本地缓存数据     |
| 会议纪要上传      | ✅完整 | ❌超时风险        | ❌本地暂存，联网后上传   |
| 关联图谱可视化     | ✅完整 | ❌禁用          | ❌禁用           |
| 数据管理中心      | ✅完整 | ⚠️只读         | ⚠️只读(缓存数据)    |

#### 5.10.2 离线存储策略

```
前端存储层次：
├─ Memory Cache(当前会话数据，5分钟TTL)
├─ IndexedDB(持久化离线数据)
│  ├─ cards(名片基础信息，最近500条)
│  ├─ todos(待办事项，最近100条)
│  ├─ entities(实体摘要，最近200条)
│  └─ tags(标签缓存)
└─ 操作队列(待同步操作)
   ├─ pending_uploads(待上传的名片/会议纪要)
   ├─ pending_updates(待更新的Todo状态)
   └─ pending_deletes(待删除的记录)
```

#### 5.10.3 数据同步策略

| 同步场景    | 策略                         | 冲突解决                      |
| ------- | -------------------------- | ------------------------- |
| 网络恢复    | 自动上传操作队列(按时间戳顺序)→拉取服务端增量更新 | Last-Write-Wins(服务端时间戳为准) |
| 多设备同时编辑 | 服务端检测冲突→推送冲突提示             | 用户选择保留版本                  |
| 删除冲突    | 删除优先                       | 删除操作始终生效                  |
| 同步状态展示  | 顶部指示器(在线/同步中/离线)           | 同步失败显示重试入口                |

#### 5.10.4 弱网优化

- 请求超时：正常5秒→弱网自动延长至15秒
- 自动重试：指数退避(1s, 2s, 4s, 8s)，最多4次
- 降级策略：关联发现失败→仅显示基础信息；图谱加载失败→显示列表视图
- 预加载：WiFi环境下预加载最近7天数据

### 5.11 Todo生命周期增强

> **设计背景**：客户视角评审发现F-06仅定义了"生成与追踪"，缺少批量操作、智能提醒、延期等关键交互。本节补充Todo管理的完整交互设计。

#### 5.11.1 Todo状态机增强

```
                    ┌──────────────┐
                    │   pending    │ ← 新建/延期恢复
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
              ▼            ▼            ▼
      ┌───────────┐ ┌──────────┐ ┌──────────┐
      │in_progress│ │ dismissed│ │ snoozed  │
      └─────┬─────┘ └──────────┘ └────┬─────┘
            │                           │
            ▼                           │(到期后)
      ┌──────────┐                      │
      │   done   │ ←────────────────────┘
      └──────────┘

新增状态:
- snoozed: 延期状态，到期后自动恢复为pending
- dismissed: 不感兴趣/不相关，不进入完成度统计
```

#### 5.11.2 批量操作

| 操作   | 触发方式                   | 说明                 |
| ---- | ---------------------- | ------------------ |
| 批量完成 | 多选模式→标记完成              | 适用于一次性处理多个信息型Todo  |
| 批量延期 | 多选模式→选择延期时间(明天/下周/自定义) | 适用于会议后批量推迟低优先级Todo |
| 批量归档 | 多选模式→归档                | 将已完成的Todo移出活跃列表    |
| 批量删除 | 多选模式→删除(二次确认)          | 适用于清理不相关的Todo      |

#### 5.11.3 智能提醒

| 提醒类型 | 触发条件                  | 提醒方式           |
| ---- | --------------------- | -------------- |
| 到期提醒 | 行动型Todo到达due\_date前1天 | Push通知+应用内红点   |
| 逾期提醒 | 行动型Todo超过due\_date未完成 | 每日8:00推送逾期清单   |
| 会前提醒 | 关联日历事件的Todo，会前1小时     | Push通知+会议准备清单  |
| 沉默提醒 | 同一关联24小时内不重复生成Todo    | 自动去重，仅更新已有Todo |

#### 5.11.4 延期(Snooze)操作

| 延期选项 | 说明            | 效果                     |
| ---- | ------------- | ---------------------- |
| 明天   | 延期至次日9:00     | due\_date更新，状态→snoozed |
| 下周   | 延期至下周一9:00    | due\_date更新，状态→snoozed |
| 自定义  | 用户选择日期和时间     | due\_date更新，状态→snoozed |
| 会前   | 延期至关联日历事件前1小时 | F-33日程表对接支持            |

### 5.12 关联暴增下的搜索与性能策略

> **设计背景**：随着数据量增长（500联系人+200公司+5000条沟通记录），关联可能从几十条暴增到几千甚至上万条。需要在架构层面提前规划搜索和性能方案。

#### 5.12.1 数据规模预估

| 数据规模 | 实体数 | 关联数(预估) | 搜索响应要求 |
|---------|--------|------------|------------|
| PoC期 | <100 | <500 | <200ms |
| 早期用户 | 500-1000 | 2000-5000 | <500ms |
| 成熟用户 | 3000+ | 10000+ | <1s |

#### 5.12.2 分层查询策略

| 层级 | 数据范围 | 存储介质 | 响应时间 | 适用场景 |
|------|---------|---------|---------|---------|
| L1 热数据 | 最近30天+置信度>0.7 | Redis缓存 | <50ms | 首页待办、高频关联 |
| L2 温数据 | 最近1年全部 | PostgreSQL索引查询 | <200ms | 实体详情、搜索 |
| L3 冷数据 | 历史归档 | PostgreSQL分区表 | <1s | 历史回溯、报表 |

#### 5.12.3 索引策略

```
关联表核心索引：
- (source_type, source_id, confidence DESC) -- 查"某实体的所有关联"
- (target_type, target_id, confidence DESC) -- 反向查询
- (relation_type, confidence DESC) -- 按关联类型筛选
- (source_type, target_type, confidence DESC) -- 类型过滤

实体表核心索引：
- (entity_type, name) -- 按类型+名称搜索
- GIN索引(name, company) -- 全文搜索
```

#### 5.12.4 搜索方案演进

| 阶段 | 方案 | 适用规模 | 说明 |
|------|------|---------|------|
| MVP-Core | PostgreSQL LIKE + 关联表JOIN | <1000条 | 简单够用 |
| MVP-Plus | PostgreSQL全文搜索(tsvector+tsquery) | <10万条 | 支持中文分词 |
| V1.0+ | Elasticsearch | >10万条 | 复杂查询、聚合、推荐 |

#### 5.12.5 前端性能保障

| 策略 | 说明 |
|------|------|
| 分页加载 | 关联列表默认Top 20（按置信度排序），滚动加载更多 |
| 懒加载 | 详情页关联图谱按需展开，不一次性渲染 |
| 聚合视图 | 不展示所有关联，显示"与10家公司、25个人、8个商机相关" |
| 智能排序 | 优先显示"最近活跃"和"高置信度"的关联 |
| 后台计算 | 关联发现引擎异步计算，不阻塞用户操作 |
| 缓存策略 | 高频查询结果缓存5分钟，写入时失效 |

### 5.16 触达通道与前端整合设计

> **核心问题**：用户在外拜访/开会时，如何收到AI的行动建议？这是产品闭环的"最后一公里"。

#### 5.16.1 场景-通道矩阵

| 用户状态 | 需要什么 | 时间窗口 | Phase 1通道 |
|----------|----------|----------|------------|
| 会前（即将见某人） | 关系画像+沟通建议 | 会前15-60分钟 | 微信服务号推送 / 小程序主动查询 |
| 会中（正在开会） | 不打扰，静默采集 | 全程 | 无推送 |
| 会后（刚结束） | 快速录入+行动项 | 会后5-30分钟 | 小程序语音录入 |
| 日常（碎片时间） | 维护提醒+早晚报 | 早晨/晚间 | 微信服务号 |

#### 5.16.2 名片小程序+EventLink整合方案

EventLink与许总数字名片小程序合并为**一个小程序**，名片是入口，关系助手是增值功能。

**整合后的用户流程**：

```
许总名片小程序
├── 首页：我的名片 + 扫一扫
├── 名片夹：扫过的名片列表
│   └── 点击某张名片 → 名片详情
│       ├── 基础信息（名片原有）
│       └── 🆕 关系助手（EventLink）
│           ├── 关系画像（阶段/偏好/决策权）
│           ├── 交流要点（上次聊了什么）
│           ├── 🔊 语音播报（开车时听）
│           └── 📝 快速记录（语音/文字录入）
├── 🆕 今日日程：今天要见的人
│   └── 会前速览 + TTS播报
└── 🆕 待办提醒：行动项+维护提醒
```

**分阶段实施**：

| 阶段 | 方案 | 许总改动量 | 用户体验 |
|------|------|-----------|---------|
| Phase 1 | 名片小程序内嵌EventLink H5页面 | 1个按钮+1个WebView | 基本无缝，名片数据通过URL参数传递 |
| Phase 2 | EventLink核心页面做成小程序原生页面 | 需开发原生页面 | 完全原生体验，TTS/录音用微信API |

**为什么合并而非分开**：
- 用户视角：扫名片→看关系画像是同一流程，不应跳转另一个小程序
- 数据视角：名片JSON直接作为EventLink的card_save事件输入，无需跨小程序传数据
- 商业视角：名片小程序是许总的入口产品，EventLink是增值功能，合并后ARPU更高

#### 5.16.3 小程序核心能力

**查询能力**：
- 搜索今天要见的人 → 展示简介、关系阶段、沟通偏好
- 查看与某人的历史交流要点 → 上次聊了什么、待跟进事项
- 浏览关系图谱 → 一度/二度人脉

**录入能力**：
- 语音录入 → "今天和张总聊了新项目预算" → AI自动建档
- 文字录入 → 快速记录要点
- 名片扫描 → 许总名片小程序已有能力，扫描结果直接触发EventLink事件处理

**播报能力**：
- 点击语音按钮 → TTS读出人物简介和交流要点
- 会前场景：开车/走路时听一遍对方信息，不用看屏幕

#### 5.16.4 Phase 1 三通道

1. **名片小程序+EventLink H5**（主交互入口）— 查询/录入/播报
2. **微信服务号模板消息**（推送通道）— 会前提醒/行动建议/维护提醒/早晚报
3. **手机推送通知**（APNs/FCM）— 紧急提醒（风险预警/到期Todo）

#### 5.16.5 Phase 2 扩展

- EventLink小程序原生页面（替代H5）
- 录音卡屏幕/语音播报（许总硬件）
- Apple Watch/智能手环
- 邮件日报

#### 5.16.6 防打扰策略

**推送优先级分级**：

| 优先级 | 类型 | 通道 | 频次限制 | 说明 |
|--------|------|------|----------|------|
| P1-紧急 | 会前提醒(1小时内) | 服务号模板消息 | 不受限(仅推1次) | 最重要，必须触达 |
| P1-紧急 | 逾期Todo | 服务号模板消息 | 不受限 | 行动项过期 |
| P2-重要 | 新行动建议 | 小程序卡片 | 每小时≤3条 | 会后AI生成 |
| P3-普通 | 关系维护提醒 | 应用内消息 | 每日≤2次(早晚) | 合并推送 |
| P4-低优 | 新发现关联 | 早报合并 | 每日1次 | 不单独推送 |

**防打扰规则**：
- 同一用户每日总推送≤10条（P1不受限）
- 静默时段（默认22:00-7:00）：仅P1可穿透
- 用户可在设置中自定义：各类型开关+静默时段+提前提醒时间
- APNs授权失败 → 自动切换到服务号推送
- 服务号推送失败 → 记录到应用内消息中心

### 5.17 数据接入层架构（v4.5新增）

> **设计背景**：Pipeline是source-agnostic的，不关心数据从哪来，只关心event_type+raw_text。数据接入层负责将不同数据源转换为统一的Event格式。

#### 5.17.1 DataSourceAdapter接口

```
DataSourceAdapter (interface)
  ├── ManualInputAdapter    (已有 - API直接创建)
  ├── VoiceAdapter          (已有 - F-50 NLU语音交互)
  ├── WeChatForwardAdapter  (Phase 1 - 小程序转发)
  ├── EmailAdapter          (Phase 1 - IMAP/Exchange)
  └── CalendarAdapter       (Phase 2 - CalDAV/Exchange)
```

所有adapter的输出都是统一的`Event(event_type, raw_text, source)`，Pipeline不需要改。

#### 5.17.2 邮件场景设计（Phase 1）

**核心设计**：原子事件+溯源边

- 邮件原文作为完整Event存储（raw_text归档）
- 邮件中的承诺/待办拆解为原子化Todo（source_event_id链回原始邮件）
- 一封邮件一个Event，通过Todo的action_type字段区分方向（my_promise/their_promise）

**约束**：
- 需要OAuth授权，用户信任门槛高
- 邮件内容敏感，隐私合规要求严
- 适合Phase 1而非PoC

#### 5.17.3 微信消息接入约束

微信不开放个人消息API，可行路径：
- 小程序内交互（已有）
- 企业微信API（需企业微信环境）
- 用户手动粘贴/转发到EventLink小程序（Phase 1）

PoC阶段聚焦**降低输入摩擦**而非自动抓取，语音输入（F-50已实现）是最低摩擦的输入方式。

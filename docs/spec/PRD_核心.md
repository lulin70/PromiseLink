## 1. 产品概述

### 1.1 产品愿景

**PromiseLink — 见面只是开始。**

> 主Slogan：让每一次连接，都更有价值。
> 副Slogan：记住需要，兑现承诺，让合作自然发生。
> 品牌短句：见面只是开始。
> （路演/传播场景可继续使用"让重要的人，不止停留在微信里"）

PromiseLink 是 **AI驱动的个人商务关系经营助手——先成就关系，再促成合作**。通过一句话记录、会议纪要和合作档案导入，帮助用户沉淀关键联系人、记住对方关注点、兑现自己的承诺、主动提供帮助，把每一次认识逐步推进为真实合作。

> **Slogan使用场景说明**：主Slogan"让每一次连接，都更有价值"用于产品核心价值传达和用户心智建立；路演/传播场景可继续使用"让重要的人，不止停留在微信里"以强化差异化定位。

**产品侧重"人"而非"知识"** — 客户买单的驱动力来自"帮我经营关系"，知识是经营关系过程中的自然沉淀。

**产品是"私密助手"而非"撮合平台"** — 所有数据归用户个人所有，匹配逻辑是"你的需求匹配你人脉的供给"，不存在资源公开、授权共享或团队协作场景。资源永远附着在具体人身上（"张总能搞定工厂"），不脱离具体人独立存在。

**首期核心闭环：互动 → 关注 → 承诺 → 帮助 → 反馈**

> 从利他视角出发：我认识了谁 → 他正在关心什么 → 我答应过他什么 → 我能先为他做什么 → 双方是否自然出现合作

产品不只是记录联系人、提取待办、推荐机会，更帮助用户理解：
- 这个人和我是什么关系阶段，该怎么维护
- 他正在关心什么，我答应过他什么
- 我能先为他做什么，让合作自然发生
- 下次该用什么方式、什么理由去联系

长期愿景：成为用户的"随身关系经营助手"，软硬件一体（录音卡/摄像头/会议系统→文本→AI处理→推送提醒），让每一次接触都不白费。

**长期愿景（v4.3新增）**：成为用户的**终身关系经营智能体**——越用越懂你，CarryMem记忆层提供持续学习能力。CarryMem的7种记忆类型（事实记忆/情景记忆/程序记忆/语义记忆/工作记忆/短期记忆/长期记忆）支撑"终身"属性：系统随使用时长积累的关系洞察越来越精准，用户数据始终可携带（数据导出功能提前到Phase 1，见§3.3 F-21）。

**明确排除的平台级功能**（私密助手不需要）：
- ❌ RBAC权限模型 / 资源授权共享 / 团队协作
- ❌ 他人可提供资源匹配（只能匹配自己人脉的供给）
- ❌ 多租户隔离 / 企业管理看板
- ❌ 跨项目/跨团队资源撮合
- ❌ 商机匹配引擎（首期暂停，Phase2条件性重启）
- ❌ 关系图谱作为主展示（推进卡为主，图谱为辅）
- ❌ 竞对风险主动推送（Phase2+）
- ❌ 批量名片商机发现（与定位冲突）
- ❌ 通用AI助手（不是ChatGPT/Siri竞品）[v4.4新增]: 语音助手仅限结构化数据查询(日程/待办/关系)，不做开放域闲聊

### 1.5 部署模型与数据主权

> **核心决策**：PromiseLink采用"阶段递进式部署"——PoC本地优先、Phase 1云端托管、Phase 2弹性扩展。前端始终以微信小程序为主入口，不开发原生APP。

#### 1.5.0 为什么不做原生APP？

| 维度 | 原生APP | 微信小程序 | 结论 |
|------|---------|-----------|------|
| LLM调用 | 必须走云端 | 必须走云端 | 无差异 |
| 语音录入 | 原生能力 | 微信同声传译 | 小程序够用 |
| TTS播报 | 原生能力 | 微信插件 | 小程序够用 |
| 名片扫描 | 原生相机 | wx.scanCode | 小程序够用 |
| 推送通知 | APNs/FCM | 服务号模板消息 | 小程序够用 |
| 获客成本 | 高（下载APP） | 低（扫码即用） | 小程序胜 |
| 开发成本 | 双端开发 | 单端开发 | 小程序胜 |
| 生态优势 | 无 | 微信分享/名片/服务号 | 小程序胜 |

**结论**：LLM推理必须走云端，手机只是展示层。微信小程序已覆盖PromiseLink全部核心交互，原生APP无增量价值。

#### 1.5.1 阶段递进式部署模型

```
PoC（你的电脑）              Phase 1（云服务器）           Phase 2（云集群）
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Docker单机       │         │ Docker Compose   │         │ K8s集群          │
│ ├─ FastAPI       │  迁移   │ ├─ FastAPI       │  扩展   │ ├─ FastAPI×N     │
│ ├─ SQLite        │ ──────→ │ ├─ PostgreSQL    │ ──────→ │ ├─ PG+Redis+ES  │
│ └─ NullMemory    │         │ ├─ Redis         │         │ └─ pgvector     │
│                  │         │ └─ CarryMem(opt) │         │                  │
└─────────────────┘         └─────────────────┘         └─────────────────┘
     ↕ API                       ↕ API                       ↕ API
┌─────────────────┐         ┌─────────────────┐         ┌─────────────────┐
│ Swagger UI      │         │ 名片小程序        │         │ 独立小程序+Web   │
│ (开发测试用)     │         │ ├─ 原生(语音/TTS) │         │ ├─ 原生+Web     │
│                 │         │ └─ H5(查询/管理)  │         │ └─ 全功能       │
└─────────────────┘         └─────────────────┘         └─────────────────┘
```

| 维度 | PoC（概念验证） | Phase 1（最小可用产品） | Phase 2（能力扩展） |
|------|---------------|---------------------|-------------------|
| **后台位置** | 用户个人电脑 | 云服务器（单节点） | 云集群（多节点） |
| **部署方式** | Docker单机 | Docker Compose | K8s |
| **数据库** | SQLite（本地文件） | PostgreSQL 15 + Redis 7 | PG + Redis + (按需ES) |
| **前端** | API + Swagger UI | 名片小程序(原生+H5) | 独立小程序+Web管理后台 |
| **认证** | API Key | JWT RS256+临时授权码 | OAuth2（无RBAC） |
| **运维** | 用户自己 | 运维团队集中管理 | 运维团队+监控告警 |
| **升级方式** | git pull + 重启 | CI/CD自动部署 | 滚动更新 |
| **备份** | 用户自己 | 自动每日备份 | 自动+异地容灾 |
| **可用性** | PC开机时可用 | 99.5% | 99.9% |
| **成本** | 零（本地） | ~¥500/月（单节点云） | 按用量弹性 |

#### 1.5.2 数据主权原则

> **核心原则**：数据归用户所有，PromiseLink是数据的加工者而非所有者。

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **用户数据所有权** | 用户创建的所有数据（事件/实体/关联/Todo）归用户个人所有 | user_id数据隔离，无跨用户访问 |
| **数据可携带** | 用户可随时导出全部数据 | F-21数据导出（**Phase 1**，v4.3提前），标准格式 |
| **数据可删除** | 用户可随时删除全部数据 | API支持批量删除+账号注销 |
| **数据最小化** | 只收集必要数据，不收集媒体文件 | 仅文本输入，无音频/视频/图片 |
| **处理透明** | AI处理过程可追溯 | 匹配推荐可解释，Todo来源可追溯 |
| **私密助手定位** | 不存在资源公开/授权共享场景 | 无RBAC、无跨用户匹配、无团队协作 |

**PoC阶段数据主权**：
- 数据存储在用户个人电脑的SQLite文件中
- LLM调用走HTTPS，仅发送脱敏后的文本片段
- 用户完全控制数据的启动/停止/备份/删除

**Phase 1+数据主权**：
- 数据存储在云端PostgreSQL，user_id隔离
- 客户端加密可选（Phase 2+）
- 数据导出/删除API始终可用
- 云端备份加密存储

#### 1.5.3 托管PoC部署模式（面向非技术用户）

针对不具备服务器运维能力的用户（如许总方），提供托管PoC部署选项：

| 项目 | 自助PoC | 托管PoC |
|------|---------|---------|
| 部署位置 | 用户PC | 我方云服务器 |
| 运维责任 | 用户 | 我方 |
| 数据存储 | 本地SQLite | 云端SQLite(Phase1→PG) |
| 访问方式 | localhost | 域名+HTTPS |
| 成本 | 0 | ≈50元/月(轻量云) |
| 用户体验 | Docker一键启动 | 扫码即用 |

托管PoC模式下，用户仅通过微信小程序访问，无需接触任何技术组件。我方负责服务器运维、LLM Key管理、数据备份。

#### 1.5.4 PoC→Phase 1 迁移路径

**迁移成本极低**：SQLite→PG只需改连接字符串（SQLAlchemy抽象层），API层零改动。

| 迁移项 | PoC | Phase 1 | 迁移方式 |
|--------|-----|---------|---------|
| 数据库 | SQLite | PostgreSQL | SQLAlchemy连接字符串切换 |
| 缓存 | 内存dict | Redis | 配置切换 |
| 认证 | API Key | JWT | 中间件替换 |
| CarryMem | NullMemoryProvider | 可选接入 | 配置切换 |
| 前端 | Swagger UI | 小程序 | 新增，不影响API |
| 数据 | 手动迁移脚本 | pg_dump/sqlite3→pg | 一次性脚本 |

托管PoC用户迁移路径：我方执行数据导出→新环境导入→切换域名指向，用户无感知。

#### 1.5.5 自建小程序前端备选方案（v4.2新增）

> **背景**：许总团队前端进展可能缓慢，或数字名片数据对接不能打通。PromiseLink需要有自建小程序前端的备选方案，确保不因前端依赖阻塞产品验证。

**方案概述**：

| 维度 | 说明 |
| ---- | ------------------------------------------------------------ |
| 框架选型 | Taro 3.x + Vue3 + NutUI（兼容微信小程序，支持跨端扩展） |
| 核心页面 | ① 首页（今天需要我回应的连接）② 录入页（语音+文字输入）③ 人物详情页（F-47关系推进卡）④ Todo列表页⑤ 设置页 |
| API兼容性 | 与现有后端API完全兼容，无需改动任何后端代码 |
| 开发策略 | Phase 1可先用H5+Swagger UI验证API正确性，小程序同步开发；PoC阶段H5先行，Phase 1小程序接替 |
| 开发周期估计 | 2-3周完成MVP页面（5个核心页面+基础导航） |

**技术栈详情**：
- 前端框架：Taro 3.x（跨端编译：微信/H5/RN）
- UI组件库：NutUI 4.x（京东出品，微信小程序原生体验）
- 状态管理：Pinia
- HTTP请求：Taro.request封装（与现有Swagger API对接）
- 语音录入：微信wx.getRecorderManager API

**决策触发条件**（满足任一即启动自建前端开发）：

| 触发条件 | 判断标准 | 负责人 |
| -------- | ------- | ------ |
| 许总团队无法按时启动 | 确认2周内无法启动前端开发 | PM |
| 数字名片API对接阻塞 | 对接评估超过3周无结论 | Architect |
| 目标用户测试需要独立环境 | 需要独立前端进行用户测试 | PM |

> **备注**：此方案为备选路径，优先级低于许总团队原生小程序方案。仅在触发条件满足时启动。

### 1.6 用户旅程定义

> **设计原则**：3条核心旅程覆盖产品最关键的使用场景，每条旅程定义完整的操作步骤、系统响应和体验目标。
> **架构约束**：PromiseLink提供媒体处理服务（ASR/TTS/OCR），但严格解耦：媒体文件仅在内存中处理，不持久化存储；仅提取的文本/结构化数据入库；媒体服务通过独立API前缀(/media/)提供，可按需禁用。

#### 1.5.1 旅程一：新用户首次体验（冷启动）

> **【v4.2更新】首次体验固定为"记录一次重要交流"4屏流，不以扫码为主入口。名片扫描降级为补充身份输入方式。**

```
用户目标：快速理解产品价值，完成第一个行动
体验目标：从打开到完成第一个行动 < 2分钟

步骤1: 信任承诺（第1屏）
  系统 → 显示信任承诺页
         "这是你的私人关系助手。你记录的人和交流内容，只用于帮助你更好地回应重要关系。
          不公开、不匹配给其他用户、不自动联系任何人。"
  用户 → 点击"记录一次重要交流"

步骤2: 一句话输入（第2屏）【核心入口：语音+文字】
  系统 → 显示输入界面
         "最近，你和谁有过一次值得记住的交流？"
```

### 1.7 智能定义与边界（v4.5新增）

> **设计背景**：基于与DeepSeek的架构讨论，明确PromiseLink"智能"的具体含义和各阶段验证边界。

#### 1.7.1 智能定义

PromiseLink的智能，定义为**以个人知识图谱为基础，对商务关系的理解、记忆与预见能力**。具体体现为三个层次：

| 层次 | 能力 | 说明 | 验证阶段 |
|------|------|------|---------|
| **理解** | 从非结构化沟通中提取实体、关注点、承诺 | AI替代手动录入 | PoC |
| **记忆** | 跨事件关联发现、关系推进卡积累 | AI替代人脑记忆 | PoC |
| **预见** | 动态优先级排序、行动建议 | AI主动推送而非被动查询 | PoC(二维)→Phase1(四维) |

#### 1.7.2 核心机制：四层架构

```
第一层：事件感知与结构化层
  多源输入 → 实体/关注点/承诺/能力提取 → 结构化存储

第二层：索引与知识图谱层
  实体索引 + 主题索引 + 关系索引 → 个人知识图谱

第三层：关联匹配引擎层
  需求-资源匹配器 + 承诺闭环追踪 + 模式匹配规则库

第四层：行动建议与交互层
  动态优先级排序 + 隐式反馈学习 → 智能Todo清单
```

#### 1.7.3 PoC验证边界

PoC阶段验证的核心闭环：**原子化事件 → 图谱关联 → 动态优先级 → 行动建议**

| 维度 | PoC范围 | Phase 1扩展 | Phase 2扩展 |
|------|---------|------------|------------|
| 实体提取粒度 | Person的concern/capability字段强化 | 邮件/日历等被动数据源 | 向量语义搜索 |
| 关联发现 | 6种结构化关联+3种冷类型 | 依赖性分析 | 图数据库 |
| 优先级排序 | 二维模型(紧急性+重要性) | 四维模型(+依赖性+场景) | 上下文感知推送 |
| 反馈学习 | 完成顺序隐式反馈 | 长按降权主动反馈 | 自适应权重 |

#### 1.7.4 Person实体concern/capability强化

**核心决策**：PoC阶段不扩展实体类型（不新增"概念/痛点/需求"实体），而是通过强化Person的concern和capability字段来覆盖深层关联需求。

设计原则：
- 关联的不是两个抽象概念，而是两个人的"痛点"和"方案"
- concern和capability直接绑定到具体的"人"，使需求-资源匹配有了坚实的数据基础
- 商务关系本质上是人和人之间concern和capability的交换网络

字段设计（受控词表+自由文本混合模式）：

```json
{
  "concerns": [
    {"tag": "会议效率", "detail": "记不住见客户时答应的方案和事", "source_event_id": "evt_xxx"}
  ],
  "capabilities": [
    {"tag": "智能体方案", "detail": "可提供AI会议纪要+承诺追踪方案", "source_event_id": "evt_yyy"}
  ]
}
```

- tag：受控词表（保证召回率）
- detail：自由文本（保证覆盖长尾）
- source_event_id：溯源边（链回原始事件）

#### 1.7.5 动态优先级排序模型

**核心决策**：Todo排序从静态列表升级为动态行动网络，"二维起步、四维演进"。

PoC阶段（二维简化版）：

```
Score = 0.4 × 紧急性(截止时间衰减) + 0.6 × 重要性(关系权重)
```

- **紧急性**：截止时间指数衰减函数；无截止时间的Todo赋予极缓慢衰减值，避免被"雪藏"
- **重要性**：复用Brief.score(0-100)作为关系权重基础

Phase 1扩展（四维完整版）：

```
Score = w1×紧急性 + w2×重要性 + w3×依赖性 + w4×场景匹配
```

权重配置：
```
PoC:      w1=0.4, w2=0.6, w3=0.0, w4=0.0
Phase 1:  w1=0.3, w2=0.35, w3=0.2, w4=0.15
```

**维度3：依赖性（dependency）— 全图谱路径分析**

核心问题：Todo A被Todo B阻塞，B没完成A就无法推进。PromiseLink的核心价值是承诺闭环，依赖性维度让系统理解"谁在等谁"。

算法：全图谱路径分析
1. 从Association图谱中提取所有"承诺"类型的边（promise类型Todo关联的Entity对）
2. 构建有向依赖图：如果"我答应A"是"A答应我"的前置条件，则存在依赖边
3. 检测阻塞链：如果Todo X的完成是Todo Y的前置条件，且X未完成，则Y的依赖性得分提升
4. 间接依赖：支持多跳路径分析（A→B→C），但限制最大深度为3跳避免过度膨胀

依赖性得分计算：
```
dependency_score = Σ(1/depth) × blocked_weight
```
- depth: 依赖链深度（1=直接依赖, 2=间接依赖, 3=三跳依赖）
- blocked_weight: 被阻塞的Todo数量权重

示例：
- 我答应李总发AI项目资料（promise, pending）
- 李总答应对接LP资源（promise, pending）
- 依赖分析：我的promise是李总promise的前置条件
- 结果：我的"发资料"Todo依赖性得分提升（有人在等我）

**维度4：场景匹配（context_match）— Event表驱动**

核心问题：即将见某人时，与该人相关的Todo应该临时提升优先级。

算法：Event表驱动场景匹配
1. 扫描Event表中未来24h的meeting/call类型事件
2. 提取这些事件关联的Entity（通过source_event_id关联）
3. 匹配当前pending Todo中related_entity_id与即将见面Entity相同的Todo
4. 根据距离见面时间计算场景匹配得分

场景匹配得分计算：
```
context_score = max(0, 1 - hours_until_meeting / 24)
```
- hours_until_meeting < 1h: score ≈ 1.0（即将见面，最高提升）
- hours_until_meeting = 6h: score ≈ 0.75
- hours_until_meeting = 24h: score ≈ 0.0（不提升）

示例：
- 下午3点要见李总（Event: meeting, 14:00）
- 上午10点查看Todo列表
- "发AI项目资料给李总"（promise, related_entity=李总）
- context_score = 1 - 4/24 ≈ 0.83 → 优先级显著提升

Phase 2扩展：上下文感知推送，从"工具"向"助理"进化

#### 1.7.6 隐式反馈学习机制

**核心决策**：PoC阶段不增加任何额外交互，通过观察用户完成Todo的顺序来学习真实优先级。

原理：
```
系统排序: [A(许总, score=85), B(同事, score=72), C(李总, score=68)]
用户实际完成顺序: C → A → B
→ 推断: C的实际优先级 > A > B
→ 调整关系权重: 李总权重↑, 同事权重↓
```

实现：
- 记录每个Todo的completed_at时间戳（已有）
- 新增completed_rank字段（完成序号）
- 每日计算：如果某人的Todo总是被提前完成，提升该人的关系权重

Phase 1扩展：长按Todo → "以后少提醒"按钮（小程序原生手势支持）
Phase 2扩展：滑动手势语义分化（需原生APP）

#### 1.7.7 向量化语义能力（v4.7新增）

> **设计背景**：结构化匹配（关键词/标签/行业）无法发现语义相似但字面不同的关联。例如"投资决策"和"融资"在结构化匹配中无交集，但语义高度相关。向量化能力让系统理解"意思相近"而非仅"字面相同"。

**核心问题**：结构化匹配无法发现语义相似但字面不同的关联。

**两个功能**：

| 功能编号 | 功能名称 | 说明 |
|----------|----------|------|
| **F-57** | 语义搜索 | embedding向量化Entity/Event，支持自然语言查询 |
| **F-58** | 关联发现增强 | 用embedding余弦相似度补充结构化匹配，发现语义层面的关联 |

**F-57 语义搜索**：
- 将Entity和Event的文本信息向量化为embedding，支持自然语言查询
- 用户可以用"上次和张总聊代工厂的事"这类自然语言进行搜索，系统通过语义匹配定位到3月15日会议纪要
- 用户可以用"谁懂AI"进行搜索，系统找到capability含"技术架构"的人
- 搜索范围：Entity + Event

**F-58 关联发现增强**：
- 在现有结构化匹配基础上，用embedding余弦相似度补充语义层面的关联发现
- 当结构化匹配得分为0时，语义相似度仍可发现关联
- 混合得分：final_score = 0.7 × structured_score + 0.3 × semantic_score

**技术选型**：

| 组件 | 选型 | 说明 |
|------|------|------|
| Embedding模型 | text-embedding-3-small | 通过Moka AI API，API模式768维/本地降级384维，兼容OpenAI SDK |
| 向量存储 | sqlite-vec扩展 | 与PoC部署模型一致，零额外依赖 |
| Phase 2迁移路径 | sqlite-vec → pgvector | PostgreSQL迁移时 |

**向量化对象**：
- Entity：concern + capability + basic信息组合文本
- Event：raw_text摘要

**查询示例**：
- "上次和张总聊代工厂的事" → 语义匹配 → 定位3月15日会议纪要
- "谁懂AI" → 语义搜索 → 找到capability含"技术架构"的人

**性能目标**：
- embedding计算 < 500ms/条
- 语义搜索 < 200ms（100条以内）

> 作为商务BD，当扫描一张数字名片时，系统自动发现该名片持有人与系统中另一联系人存在商机匹配（技术供需/业务互补/资源对接，匹配度≥0.80），并在5秒内推送⚪商机Todo+🟢破冰行动，以便及时建立连接或牵线搭桥。

**验收标准**: 商机匹配度≥0.80时，系统在事件上报后10秒内生成⚪机会Todo，包含双方姓名、匹配维度、匹配分数和1-3条🟢破冰行动（如共同话题、引荐话术、切入角度）。

**US-02: 跨事件合作线索发现**

> 作为投资人，当两次不同会议中出现同一技术领域的关键人物时，系统自动发现两人存在技术重叠关联，推送潜在合作机会Todo，以便评估联合投资可能。

**验收标准**: 当两个person实体在同一technology实体下共现且tech\_overlap关联强度>0.5时，系统生成⚪机会Todo。

**US-03: 供应链关联发现**

> 作为创业者，当系统发现新接触的供应商与现有客户存在供应链上下游关系时，推送供应链关联Todo，以便优化采购或销售策略。

**验收标准**: 当两个organization实体间建立supply\_chain关联且strength>0.4时，系统生成⚪机会Todo。

### 2.2 风险预警场景（🔴 Risk）— Phase 2+ 实现

> **优先级说明**：Phase 1数据源有限（无爬取、无商业API），竞对关系数据极度稀疏，硬做风险预警易产生大量误报。Phase 1仅建立competitor/risk\_link关联类型的数据结构，不主动推送🔴风险Todo。Phase 2引入竞对数据管理(F-08)后启用。

**US-04: 竞对关系预警**（Phase 2）

> 作为商务人士，当扫描一张名片时，系统发现该名片所属公司与系统中已有联系人所在公司存在直接竞争关系，立即推送🔴风险Todo，以便在沟通中注意信息边界。

**验收标准**: 当两个organization实体间建立competitor关联且strength>0.5时，系统在5秒内生成🔴高风险Todo，包含竞对依据和沟通建议。

**US-05: 风险关联链预警**（Phase 2）

> 作为商务人士，当系统发现新接触的人与已知风险实体存在风险关联链（如竞对公司的前员工），推送🔴风险Todo，以便评估合作风险。

**验收标准**: 当person实体与已有risk\_link关联的实体建立新关联时，系统生成🔴风险Todo。

### 2.3 背景补充场景（🔵 Context）

**US-06: 人脉关系背景补充**

> 作为商务人士，当查看今日待办时，系统推送🔵背景Todo，告知即将会面的联系人与自己存在人脉关联（校友/前同事/同城/同行业圈/投资偏好匹配等），以便在沟通中建立亲近感或针对性准备。

**验收标准**: 当两个person实体间建立alumni/ex\_colleague/同城等关联，或投资偏好与项目领域匹配度>0.6时，系统生成🔵背景Todo，包含关系类型、关系依据和破冰话题建议。

### 2.4 实体归一场景

**US-08: 实体归一人工确认**

> 作为系统管理员，当实体归一引擎产生置信度在\[0.70, 0.85)区间的归一建议时，系统在管理界面展示待确认队列，包含完整属性对比、AI归一推理、置信度分数和历史相似案例，管理员可执行"确认归一"/"拒绝归一"/"稍后处理"三种操作，确认结果反馈至模型用于持续改进。

**验收标准**: 待确认队列展示所有\[0.70, 0.85)区间的归一建议，每条建议包含属性对比表、AI推理说明、置信度数值；操作后系统在200ms内响应。

**US-09: 实体归一撤回**

> 作为系统管理员，当发现已自动合并的实体（置信度≥0.85）存在误合并时，可在管理界面一键撤回，系统自动拆分实体并恢复原始状态，同时记录操作日志。

**验收标准**: 撤回操作后实体恢复为合并前的独立状态，关联关系正确还原，操作日志完整记录撤回时间、操作人和原始合并信息。

### 2.5 数据录入场景

**US-10: 名片扫描事件上报**

> 作为IAMHERE小程序用户，当扫描或查看一张数字名片时，小程序自动将名片数据上报至PromiseLink API，系统在200ms内返回确认，后台异步完成实体抽取和关联发现。

**验收标准**: API在200ms内返回event\_id和processing状态；后台处理完成后，相关Todo在10秒内可查询。

### 2.6 行动建议场景

**US-11: 会议行动项自动分解**

> 作为商务BD，当上传一场客户会议纪要后，系统不仅推送关联Todo，还自动从会议内容中提取行动项（任务、负责人、截止时间），生成结构化的待办清单，以便会后立即执行跟进。

**验收标准**: meeting事件处理后，输出包含actions列表，每条action包含task/who/when/priority四要素；行动项提取准确率≥75%（基于20场会议人工评估）。

### 2.7 数据管理场景

**US-13: 实体信息浏览与修正**

> 作为商务人士，当需要了解某位联系人的完整信息时，可通过数据管理中心查看该实体的所有属性（基本信息、人脉关系、话题标签、关联实体），并可对AI自动提取的provisional信息进行确认/修正/删除，或手动补充缺失属性。

**验收标准**: 实体详情页展示全部属性+来源标记(AI提取/手动录入/用户确认)；provisional属性可一键确认或修正；属性编辑后200ms内持久化；支持删除实体（级联清理关联）。

**US-14: 事件与关联的浏览管理**

> 作为商务人士，当需要回顾历史事件或检查关联关系时，可通过数据管理中心浏览已存取的事件列表和关联图谱，可修正错误的关联类型、删除不准确的关联、或手动添加遗漏的关联。

**验收标准**: 事件列表支持按时间/类型/关键词筛选；关联列表支持按类型/实体筛选；关联可编辑类型/强度/删除；手动添加关联需指定两实体+关联类型；操作响应<500ms。

### 2.8 新用户与日常使用场景

**US-15: 新用户首次体验引导**

> 作为新用户，首次打开产品时，系统展示简洁引导（3屏以内），引导完成首次名片扫描，并在扫描后立即展示关联发现结果和第一个行动建议，以便快速理解产品价值并决定继续使用。

**验收标准**: 首次打开显示引导页（≤3屏，含产品定位/核心功能/隐私承诺）；引导用户扫描第一张名片；扫描后即使仅1条关联也立即展示；生成第一个🟢行动建议；从安装到完成第一个行动的完成率>60%。

**US-16: 会前背景快速准备**

> 作为商务BD，在见客户前，通过输入客户姓名/公司，3秒内获取基础信息和关联关系（校友/前同事/竞对等），并生成会议准备清单（背景调研/话题建议），以便提高会议效率。

**验收标准**: 输入客户信息后3秒内返回基础信息；展示关联关系和破冰话题建议；生成会议准备清单；支持添加到日历（F-33），会前1小时推送提醒。

**验收标准**: 会议结束后推送"记录会议"提醒；语音/文字输入1分钟内完成；自动生成分类Todo并关联到相关实体；默认跟进时间3天后提醒。

**验收标准**: 支持模糊搜索和关联搜索；搜索结果按相关度排序；支持按类型筛选（人/公司/Todo/会议）；搜索响应<500ms；支持拼音首字母搜索。

**验收标准**: 支持连续扫描模式；自动识别已存在联系人并去重；批量生成关联报告；批量添加标签；单次批量处理支持≥20张名片。

**验收标准**: 支持JSON和CSV两种导出格式；导出内容包含实体/关联/Todo三类数据；导出前显示数据量和预估时间；1万实体规模导出操作<30s。

### 6.5 语音交互范式设计 [v4.4新增]

> **定位说明**：语音是PromiseLink的输入/输出方式增强，不改变产品定位（AI驱动的个人商务关系经营助手）。核心循环不变：互动→关注→承诺→帮助→反馈。Slogan不变："让每一次连接，都更有价值"。

#### 6.5.1 交互流程

```
用户唤醒 → ASR语音转文字 → NLU意图识别 → API查询 → NLG回答生成 → TTS语音播报
         → 中间态反馈("请稍等,我去看一下")
```

**各环节说明**：

| 环节 | 技术方案 | 复用现有功能 |
| ---- | -------- | ----------- |
| ASR语音转文字 | 小程序原生录音+第三方ASR服务(如微信同声传译/讯飞) | F-10语音录入扩展 |
| NLU意图识别 | 基于F-44 input_scope分类器扩展，新增voice_query意图分类 | F-44 input_scope分类器 |
| API查询 | 复用现有API（日程/Todo/关系推进卡） | F-30/F-06/F-47 |
| NLG回答生成 | 结构化数据→自然语言转换层(基于模板+LLM) | F-41 TTS播报NLG层 |
| TTS语音播报 | 复用现有TTS服务 | F-41 语音输出 |

#### 6.5.2 唤醒方式

| 方式 | 描述 | Phase | 适用场景 |
| ---- | ---- | ----- | -------- |
| **方式A: 长按语音按钮** | 小程序内长按麦克风图标开始录音，松开结束并发送 | Phase 1.1 | 推荐，最自然的交互方式（尤其适合视障用户） |
| **方式B: 唤醒词** | 说出"PromiseLink"唤醒语音助手 | Phase 1.2 | 解放双手场景（驾驶/运动） |
| **降级: 文字输入** | 公共场合自动检测环境音，推荐切换为文字模式 | Phase 1.1 | 噪音环境/公共场合隐私保护 |

**公共场合智能降级规则**：
- 检测到环境噪音 > 60dB 时，提示"当前环境较嘈杂，建议使用文字模式"
- 检测到连续2次ASR识别失败时，自动推荐文字输入
- 用户可手动切换模式，选择记忆偏好

#### 6.5.3 中间态反馈规范

语音交互的异步特性要求在处理过程中提供即时反馈，避免用户以为系统无响应：

| 场景 | 反馈话术 | 触发条件 |
| ---- | -------- | -------- |
| 处理中 | "正在查询..." / "让我看看..." | API调用发起后立即播放 |
| 网络异常 | "网络好像有问题,稍后再试" | 网络请求超时/失败 |
| 超时 | "查询时间有点长,要不您先看看文字版?" | 端到端响应 > 8秒 |
| 成功 | （直接返回查询结果语音） | 正常完成 |

**技术实现**：中间态反馈采用预录制音频片段拼接，不依赖TTS实时生成，确保<200ms内播放。

#### 6.5.4 错误处理

| 错误类型 | 反馈话术 | 处理策略 |
| -------- | -------- | -------- |
| **ASR失败** | "没听清,能再说一次吗?" | 自动重试1次，仍失败则提示切换文字模式 |
| **NLU不确定** | "我不太确定您的意思,是想问今天的日程吗?" | Top-2意图置信度均 < 0.7时，主动确认（提供候选选项） |
| **无结果** | "没有找到相关信息" | API返回空结果时的友好提示 |
| **能力边界** | "这个问题我还学不会,试试问我今天的日程?" | 识别到超出Phase 1能力范围时，引导至已知能力 |

**NLU不确定处理详细策略**：
1. 取Top-2意图及其置信度
2. 若最高置信度 ≥ 0.7：直接执行
3. 若0.5 ≤ 最高置信度 < 0.7：生成确认问题（"是想问X吗?"）
4. 若最高置信度 < 0.5：返回能力边界引导语

#### 6.5.5 隐私保护

语音交互涉及敏感的音频数据和个人信息，需严格遵守隐私保护原则：

| 保护措施 | 说明 | 实现方式 |
| -------- | ---- | -------- |
| **不存储原始媒体** | 音频/图片文件仅在ASR/OCR处理后即丢弃，不留存 | 内存中处理完毕后释放，不写入持久化存储；TTS音频流式返回不落盘 |
| **识别文字可选删除** | ASR转写后的文字记录，用户可选择删除或保留 | 设置页提供"清除语音历史"选项 |
| **公共场合检测** | 自动检测环境音量，推荐文字模式以保护隐私 | 麦克风音量采样 + 阈值判断 + UI提示 |
| **敏感信息确认** | 手机号、金额等敏感字段播报前需用户确认 | NLG层标记敏感字段，TTS前插入确认交互 |
| **会话隔离** | 每次语音对话独立，不跨会话关联上下文（Phase 1.1） | 会话ID隔离，上下文仅当次有效 |

#### 6.5.6 MVP能力边界

明确Phase 1不做的事情，避免期望管理偏差：

**✅ Phase 1.1 支持的能力（单轮问答）**：
- 日程查询："今天有什么安排?" / "明天几点的会议?"
- 承诺追踪："我还有什么待办?" / "答应张总的事办了吗?"
- 关系推进："李总的关系走到哪一步了?" / "最近联系过谁?"

**✅ Phase 1.2 新增能力（多轮对话）**：
- 多轮追问："那后天呢?" / "具体什么内容?"
- 日期范围查询："这周有哪些会议?"
- 简单创建待办："提醒我下周给王总打电话"

**❌ Phase 1 不做（明确排除）**：
- 开放域闲聊（不是ChatGPT/Siri竞品）
- 复杂推理（如"帮我分析一下这个人的合作可能性"需要看屏幕详情）
- 多人对话识别（仅支持单人语音输入）
- 方言支持（Phase 1仅支持标准普通话）
- 语音编辑Event（语音仅用于查询和简单创建，编辑操作需走文字界面）

## 二、逐项判定（v2 修订版）

### Q1: Pipeline 性能 — O(n) 问题与超时策略

**用户原话**: Pipeline 所花的时间会不会随实体数增长？全局超时对用户的意义在哪里？用户情愿慢一点但是事件要处理完，只要不是系统卡死。

**v1 判定回顾**: P1 — 加 PIPELINE_TIMEOUT=120s 全局超时

**用户反馈（关键决策变更）**:
> ❌ "不要用超时来杀掉 Pipeline"
> ✅ "慢但完整 > 快但残缺"
> ⚠️ "一个商务人士认识 1000+ 人很正常" — O(n) 必须解决

**技术分析（深化）**:
- Step02 `entity_extractor`：查询已有实体做名称相似度匹配 → **O(n) 全表扫描**
- Step10 `association_discovery`：增量发现 + SQL pushdown → **已部分优化，但冷类型仍 O(n)**
- 1000 实体时 Step02 匹配可能需 5-10s（纯 Python 字符串比较）
- **SQLite 无问题**（有索引），瓶颈在 Python 内存中的实体匹配

**v2 策略调整**:

| 维度 | v1 方案 | v2 方案（用户确认）|
|------|---------|-------------------|
| 超时策略 | 全局 120s 超时终止 | **不终止**：超时仅用于前端"预计等待"提示 + 后端日志告警 |
| O(n) 优化 | 未涉及 | **加索引层**：实体名 → 倒排索引 / SQLite FTS5 / 内存字典树 |
| 用户体验 | "处理超时"错误 | "正在处理中，预计还需 XX 秒" + 可关闭页面后台运行 |

| 角色 | v2 判定 |
|------|---------|
| PM | **P1** — 核心体验问题，1000+ 实体必须可用 |
| Architect | Step02 加实体名倒排索引（dict prefix tree），将 O(n)→O(1) 查找；Pipeline 加步骤进度回调 |
| Tester | 补充 50/200/500/1000 实体压力测试基线 |
| Coder | 工作量 1d（索引优化 0.7d + 进度回调 0.3d）|

**判定: P1（策略调整）** — 实体匹配索引优化 + Pipeline 进度回调（非超时终止）

### Q3: LLM 不可用时策略

**用户原话**: LLM 不可用时能不能第一时间通知？而不是降级导致质量很差？

**v1 判定回顾**: P0 — 新增降级通知机制（状态码 + 警告横幅 + 重试按钮）

**用户反馈（关键决策变更）**:
> ❌ "不要静默降级导致质量差"
> ✅ "用户宁愿等久一点也要结果正确"
> 💡 核心诉求：**重试 > 降级 > 放弃**

**v2 策略调整**:

| 维度 | v1 方案 | v2 方案（用户确认）|
|------|---------|-------------------|
| LLM 失败时 | 降级到本地模型/hash | **自动重试 3 次**（指数退避 2s/4s/8s），每次重试前端显示 "第 N 次重试中..." |
| 重试全失败 | 降级并标记 `degraded` | **暂停处理 + 明确告知用户**："AI 暂不可用，已保存原始内容，稍后将自动重试" |
| 降级路径 | 三级降级仍执行 | **仅在用户手动确认后才降级**（弹窗选择："等待重试" vs "先用基础处理"） |
| 前端展示 | 警告横幅 | 分状态展示：processing → retrying(N) → waiting → degraded(用户确认后) → completed |

| 角色 | v2 判定 |
|------|---------|
| PM | **P0** — 质量是生命线，不能偷工减料 |
| Architect | LLMClient 改为 "重试优先 + 用户确认降级" 模式；新增 pipeline 状态 `retrying`, `waiting_retry` |
| Coder | 工作量 1.5d（LLMClient 改造 1d + 前端状态展示 0.5d）|

**判定: P0（策略升级）** — 重试优先 + 用户确认降级（而非静默降级）

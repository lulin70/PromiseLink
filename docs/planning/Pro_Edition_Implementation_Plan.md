> **版本**: v1.0

## 0. 文档定位

本文档是 PromiseLink 专业版的**分阶段实现计划**，基于架构设计文档 `Pro_Edition_Architecture.md`，提供：
- 6个Phase的任务分解、依赖关系、验收标准
- 优先级排序（基于用户价值和技术难度）
- 风险评估与缓解措施
- E2E测试计划（含模拟真实用户使用的测试）

**重要前提**: 专业版大部分功能模块的**业务代码已实现**（ASR/TTS/OCR/邮件/微信/隐私/CSV），本计划的重点是：
1. **云端AI网关**（核心缺失，从零实现）
2. **relay_client**（本地侧中继客户端，从零实现）
3. **集成联调**（已实现模块与网关对接）
4. **E2E测试**（模拟真实用户全链路）

### 4.2 E2E测试场景

#### 场景1: 许总的一天（核心场景E2E）

```
[早晨 8:00] 许总打开小程序
  → 语音查询"今天有什么安排"
  → TTS播报今日日程
  → 验证: 3秒内播报，内容正确

[上午 10:00] 许总见客户，交换名片
  → 名片扫描 → 自动建人脉
  → 验证: 姓名/公司/职位/电话/邮箱正确提取

[中午 12:00] 许总吃完饭，语音录入
  → 语音"今天和张总吃了午饭，聊了新项目合作，答应下周给他发方案"
  → 验证: 事件创建，实体"张总"提取，Todo"发方案"生成，承诺识别

[下午 15:00] 许总开车，语音查询
  → 语音"李总最近怎么样"
  → TTS播报李总关系简报
  → 验证: 3秒内播报，内容正确

[晚上 20:00] 许总回家，打开电脑
  → 浏览器访问本地Docker H5
  → 查看今日录入的事件和待办
  → 验证: 数据同步，AI调用经网关代理（场景2路径）
```

#### 场景2: 邮件同步E2E

```
[配置] 许总配置QQ邮箱IMAP
  → 输入邮箱 + 应用专用密码
  → 验证: IMAP连接成功

[同步] 手动触发邮件同步
  → 拉取未读邮件
  → 验证: 邮件解析为事件，发件人提取为Entity

[定时] 等待5分钟
  → 自动同步新邮件
  → 验证: 增量同步，不重复处理
```

#### 场景3: 微信转发E2E

```
[转发] 许总长按微信消息 → 转发到PromiseLink小程序
  → 粘贴群聊记录
  → 验证: 正确识别说话人+时间+内容

[解析] 自动解析为事件
  → 验证: 事件创建，实体提取，Todo生成
```

#### 场景4: CSV批量导入E2E

```
[上传] 许总上传CRM导出的CSV
  → 预览前10行
  → 验证: 列正确识别

[映射] 配置列映射
  → 姓名→name, 电话→phone, 公司→company
  → 验证: 映射正确

[导入] 确认导入
  → 批量写入
  → 验证: 成功N条/跳过M条/失败K条，去重正确
```

#### 场景5: 隐私数据E2E

```
[加密] 查看数据库
  → 验证: phone/email字段AES加密存储

[脱敏] H5查看人脉列表
  → 验证: 电话显示为138****1234

[审计] 查看审计日志
  → 验证: view/export/delete操作有记录

[GDPR] 数据导出
  → 验证: 导出完整数据，含所有事件/实体/Todo
```

#### 场景6: 降级与容错E2E

```
[网关宕机] 停止网关服务
  → 验证: 本地基础功能（查询/展示）仍可用
  → 验证: AI功能返回降级提示

[网关恢复] 启动网关服务
  → 验证: relay_client自动重连
  → 验证: AI功能恢复

[额度用尽] 模拟用户Token用量达100%
  → 验证: AI调用被拒绝，返回"额度已用完"
  → 验证: 基础功能仍可用

[Key限流] 模拟LLM API返回429
  → 验证: Key池自动切换到备用Key
  → 验证: 用户无感知
```

### 4.4 安全测试

| 测试项 | 方法 | 预期 |
|--------|------|------|
| JWT伪造 | 使用错误密钥签发JWT | 401拒绝 |

## 5. 风险评估

### 5.1 技术风险

| 风险 | 级别 | 概率 | 影响 | 缓解措施 |
|------|------|------|------|----------|
| **网关WebSocket稳定性** | 高 | 中 | 高 | 指数退避重连 + 心跳检测 + 降级模式 |
| **AI代理延迟过高** | 高 | 中 | 高 | Key池负载均衡 + 缓存 + 超时降级 |
| **ASR中文准确率不足** | 中 | 低 | 中 | Moka AI Whisper + 本地Whisper双路 |
| **小程序WebView兼容性** | 中 | 中 | 中 | 核心交互走原生，H5仅只读查询 |
| **IMAP邮件解析兼容性** | 中 | 中 | 低 | 支持主流邮箱(QQ/163/Gmail)，异常邮件跳过 |
| **PII加密性能影响** | 低 | 低 | 低 | AES-256-GCM性能足够，仅加密phone/email |

### 5.2 商业风险

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| **AI成本超预期** | 中 | 用量计费+红黄绿灯+月度配额上限 |
| **用户增长慢** | 中 | 早鸟价¥29/月 + 许总团队种子用户 |
| **网关被攻击** | 中 | 网关不存业务数据 + WAF + 限流 |
| **AGPL合规争议** | 低 | Open Core模型，网关独立服务不触发传染 |

### 5.3 运营风险

| 风险 | 级别 | 缓解措施 |
|------|------|----------|
| **网关单点故障** | 高 | 监控告警 + 快速重启 + 降级模式 |
| **LLM API变更** | 中 | API Key池多Provider支持(DeepSeek+Moka) |
| **证书过期** | 低 | Let's Encrypt自动续期 + 监控 |
| **数据库备份** | 中 | PG定时备份 + 用量数据可重建 |

  510 ÷ 48.55 ≈ 11个付费用户 (常规价)

## 8. 附录

### 8.1 Phase优先级排序理由

| Phase | 优先级 | 排序理由 |
|-------|--------|----------|
| Phase 0 网关 | P0 | 专业版核心价值"零配置AI"的基础，所有AI功能依赖网关 |
| Phase 1 relay_client | P0 | 网关的本地侧配套，没有它网关无法工作 |
| Phase 2 语音助手 | P1 | 许总核心场景"开车听简介"，最高用户价值 |
| Phase 3 媒体处理 | P1 | 名片扫描是商务刚需，高频使用 |
| Phase 4 邮件同步 | P2 | 中频功能，不阻塞核心发布 |
| Phase 5 微信/CSV | P3 | 低频功能，代码已实现仅需联调 |
| Phase 6 隐私增强 | P2 | 隐私是信任基础，但加密已实现，增强可后置 |

### 8.2 关键决策记录

| 决策 | 选项 | 选择 | 理由 |
|------|------|------|------|
| 网关架构 | 单体 vs 分布式 | 单体FastAPI | 专业版v1快速验证，定制版再考虑分布式 |
| AI调用策略 | 用户自带Key vs 网关代理 | 网关代理 | 专业版核心价值"零配置AI" |
| 数据存储 | 本地SQLite vs 云端PG | 本地SQLite | 隐私优先，网关不存业务数据 |
| TTS方案 | 后端TTS vs 微信插件 | 微信插件(前端)+Moka(后端) | 前端免费原生，后端统一认证 |
| relay_client | 独立容器 vs 嵌入式 | 嵌入式Task | 零额外部署开销，共享容器资源 |
| 小程序语音 | H5 WebView vs 原生 | 原生页面 | 微信同声传译仅支持原生，<3秒响应 |

## 1. 架构师 (Architect) Review

### 1.1 架构合理性评估

**优点**:
- ✅ 三层架构 (小程序/网关/本地Docker) 职责划分清晰，符合"隐私优先"核心价值
- ✅ "网关无状态中继 + 用户PC主动出站"设计巧妙，规避了公网IP和端口映射的部署门槛
- ✅ "同一套代码库 + APP_EDITION配置区分"的Open Core模型合理，降低维护成本
- ✅ relay_client作为FastAPI嵌入式Task而非独立容器，零额外部署开销，设计精炼
- ✅ API Key池管理设计完备 (轮询+健康率+熔断+冷却恢复)

**问题**:
- ⚠️ **场景3链路延迟风险 (阻断项 A-1)**: 小程序→网关→本地Docker→网关→LLM→原路返回，共4跳网络往返。架构文档§4.6场景3的路径为 `小程序→网关→本地Docker(业务逻辑)→网关(AI代理)→LLM`，这意味着AI调用需要从本地Docker经网关再到LLM，加上小程序到本地Docker的中继，总延迟可能无法满足<3秒目标。文档未对此链路做延迟分解分析。
- ⚠️ **网关水平扩展缺失**: 架构文档明确"网关无状态中继"，但部署架构(§7)仅单VPS单实例，未说明未来如何水平扩展。WebSocket连接映射存在Redis中理论上支持多实例，但文档未明确说明。
- ⚠️ **配置项未落地 (阻断项 A-2)**: 架构文档§8.1描述的专业版配置项 (`relay_gateway_url`, `relay_user_token`, `ai_mode`, `pro_license_key`等)在 `src/promiselink/config.py` 中**尚未实现**。文档声称"已实现"但代码验证显示这些字段不存在，存在文档与代码不一致问题。

### 1.2 技术选型评估

- LLM建议增加Moka AI作为DeepSeek的备选Provider (架构文档§9提到但未在Key池设计中明确双Provider切换逻辑)

## 3. 安全专家 (Security) Review

### 3.1 安全模型完备性

**优点**:
- ✅ 四层防护体系 (路由/认证/网络/数据) 框架完整
- ✅ PII加密密钥独立于JWT签名密钥 (即使JWT密钥泄露，PII仍安全)
- ✅ AES-256-GCM + PBKDF2-SHA256 (100K迭代) 符合工业标准
- ✅ 网关不存储业务payload，仅存用户账户+用量

**问题**:
- 🔴 **AI内容经网关中转的隐私边界模糊 (阻断项 S-1)**: 架构文档§6.4声明"网关仅转发LLM请求/响应，不持久化对话内容"，但AI调用路径中，用户的业务数据 (如"今天和张总吃了午饭") 会作为LLM prompt经网关转发。虽然"不持久化"，但网关进程内存中会经过这些数据。文档未说明:
  - 网关是否记录请求日志 (即使不存payload，URL/method/timestamp日志是否记录？)
  - 网关进程内存中的数据生命周期 (是否有内存dump风险？)
  - LLM Provider (DeepSeek/Moka AI) 侧的数据留存政策 (第三方AI厂商是否会存储/训练用户数据？)
  
  **建议**: 补充§6.4，明确 (1) 网关日志策略 (仅记metadata不记body)，(2) LLM Provider数据政策声明，(3) 用户隐私协议中的第三方AI披露条款。

- ⚠️ **许可证防破解能力不足**: PRO_LICENSE_KEY验证流程 (§4.3) 依赖JWT签名，但:
  - JWT使用HS256对称签名，网关密钥泄露即可伪造任意用户token
  - 未提及设备绑定/硬件指纹，单Key可多设备共用
  - 未提及许可证吊销机制 (用户退款后如何立即失效？)
  
  **建议**: (1) 考虑非对称签名(RS256)，(2) 增加设备指纹绑定，(3) 增加许可证吊销列表(CRL)。

### 3.2 API Key池安全性

- ✅ Key仅存网关侧，用户侧零Key配置 (核心安全优势)
- ✅ Key加密存储在环境变量
- ⚠️ **Key泄露检测缺失**: 未提及Key异常使用检测 (如突然用量激增可能表示Key泄露)
- ⚠️ **Key轮换策略缺失**: 未提及定期轮换Key的计划

### 3.3 用量计费防刷

- ✅ 网关侧计量，用户侧无法篡改
- ✅ 红黄绿灯机制 + 月度配额上限
- ⚠️ **缺少用户级速率限制**: 架构文档仅提到Key池级别的RPM/TPM限流，未提到单用户级别的速率限制。恶意用户可在配额内高频调用消耗AI成本。
- ⚠️ **缺少异常用量告警**: 未定义"异常用量"阈值 (如单用户1小时内消耗>10万Token触发告警)

### 3.4 DDoS防护

- 🔴 **DDoS防护缺失**: 网关暴露在公网 (api.promiselink.com)，仅靠Nginx + Let's Encrypt，未提及:
  - Cloudflare/CDN前置防护
  - 连接数限制 (单个IP的WebSocket连接数上限)
  - SYN flood防护
  
  **建议**: 至少配置Cloudflare免费版前置，或云厂商的DDoS基础防护。

### 3.5 安全专家投票

**有条件通过** — 需修复阻断项 S-1 (AI内容隐私边界明确化) 和补充DDoS防护方案。防破解和防刷建议作为P1改进。

**问题**:

  - 100个WebSocket连接同时活跃的网关资源消耗
  - AI Key池在高并发下的切换行为
  
  **建议**: 补充并发测试场景，至少覆盖10用户并发 (发布检查清单§7.2提到但未在E2E计划中细化)。

- ⚠️ **场景3性能基线存疑**: 性能目标"语音查询<3秒"，但场景3链路为 小程序→网关→本地Docker→网关→LLM→返回，4跳网络+ASR+NLU+查询+NLG+TTS。未做延迟分解:
  - ASR: ~500ms (Moka AI Whisper)
  - NLU (LLM意图分类): ~800ms
  - 查询: ~100ms (本地SQLite)
  - NLG (LLM生成): ~800ms
  - TTS: ~300ms (微信同声传译)
  - 网络中继: ~200ms×4跳 = 800ms
  - **总计估算: ~3.3秒** — 可能超标
  
  **建议**: 在Phase 2增加延迟分解基准测试，若超标需并行化 (ASR与NLU无法并行，但NLG与TTS可流水线化)。

### 6.2 部署架构评估

**优点**:
- ✅ 用户侧零额外容器 (relay_client嵌入式)，部署门槛低
- ✅ 网关侧Docker Compose编排清晰 (gateway + postgres + redis + nginx)

**问题**:
- 🔴 **网关单点故障 (SPOF) 缓解不足 (阻断项 D-1)**: 
  - 网关部署在单台4核8G VPS上，无高可用
  - 虽然架构文档§7.4提到"降级模式"，但网关宕机意味着所有专业版用户的AI功能不可用
  - 未提及数据库备份策略 (PG中存用户账户+用量+许可证，丢失影响大)
  - 未提及Redis持久化策略 (WebSocket连接映射丢失需所有客户端重连)
  
  **建议**: (1) PG定时备份 (至少每日pg_dump)，(2) Redis AOF持久化，(3) 网关镜像快照备份，(4) 文档化灾难恢复RTO/RPO。

### 6.3 监控告警评估

**现状**:
- ✅ 基础版已有Prometheus + alerts.yml (监控API健康/错误率/P95延迟)
- ⚠️ 网关监控完全缺失

**建议新增网关监控指标**:

| 指标 | 类型 | 告警阈值 |
|------|------|----------|
| 网关健康检查 | 可用性 | 1m无响应→Critical |
| WebSocket活跃连接数 | 容量 | >500→Warning |
| AI调用成功率 | 业务 | <99%→Warning, <95%→Critical |
| Key池可用Key数 | 容量 | <2→Critical |
| LLM API延迟P95 | 性能 | >2s→Warning |
| 用户用量异常 | 安全 | 单用户1h>10万Token→Warning |
| PG连接池使用率 | 容量 | >80%→Warning |
| Redis内存使用 | 容量 | >80%→Warning |
| TLS证书剩余天数 | 运维 | <14天→Warning, <7天→Critical |

### 6.4 容灾备份方案

| 数据 | 当前方案 | 建议 |
|------|----------|------|
| 用户业务数据 (SQLite) | 用户本地 | 用户自行备份 (文档提示) |
| 网关用户账户 (PG) | ❌ 无备份 | 每日pg_dump + 7天保留 |
| 网关用量记录 (PG) | ❌ 无备份 | 每日pg_dump + 30天保留 |
| WebSocket映射 (Redis) | ❌ 无持久化 | AOF持久化 (可接受重启后重连) |
| 网关配置 (环境变量) | 手动管理 | 纳入Secret管理 (如GitHub Secrets) |

### 6.5 DevOps投票

**有条件通过** — 需修复阻断项 D-1 (网关SPOF缓解 + 备份策略) 后方可进入P3。

## 7. UI设计师 (UI) Review

### 7.1 语音助手交互设计

**优点**:
- ✅ 小程序原生页面录音 (wx.startRecord) 而非H5 WebView，保证录音质量和响应速度
- ✅ TTS播报走微信同声传译插件 (免费、原生体验)
- ✅ 语音查询支持多轮对话 (VoiceSession管理)

**建议**:
- ⚠️ **语音交互反馈态设计缺失**: 文档未描述录音中的视觉反馈 (波形动画/计时器)、ASR处理中的过渡态 ("识别中...")、AI查询中的加载态。这些对<3秒感知等待至关重要。
- ⚠️ **语音查询错误态设计缺失**: ASR识别失败/NLU意图未识别/查询无结果时的UI反馈未定义。

### 7.2 媒体处理UI

**优点**:
- ✅ 名片扫描走wx.scanCode原生能力，交互自然
- ✅ CSV导入有预览+列映射确认流程，符合用户心智

**建议**:
- ⚠️ **名片扫描结果确认页设计缺失**: OCR提取后应允许用户编辑修正再保存，文档未描述此交互。
- ⚠️ **媒体上传进度反馈缺失**: 25MB音频上传需进度条，文档未描述。

### 7.3 邮件同步配置UI

- ⚠️ **邮件配置引导缺失**: 非技术用户不知道如何获取"应用专用密码"，需提供QQ/163/Gmail的图文引导。
- ⚠️ **同步状态可视化缺失**: 未设计邮件同步状态面板 (上次同步时间/成功数/失败数/重试按钮)。

### 7.4 专业版与基础版UI一致性

**优点**:
- ✅ 专业版小程序 + 基础版H5共用Taro前端框架，组件可复用
- ✅ H5查询管理页面在专业版中通过WebView嵌入，保证一致性

**建议**:
- ⚠️ **用量计费UI设计缺失**: 红黄绿灯状态、剩余额度、历史用量图表未在UI设计中体现。这是付费用户最关心的功能之一。
- ⚠️ **降级提示UI设计缺失**: 网关宕机/额度用尽时的降级提示组件未设计。

### 7.5 UI设计师投票

**通过** — 交互方案整体合理。建议补充加载态/错误态/用量面板/降级提示的UI设计作为P1改进项。

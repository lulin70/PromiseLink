> **版本**: v1.0

## 1. 概述

**产品**: EventLink — AI驱动的个人商务关系经营助手

**数据主权**: 用户数据所有权归用户，EventLink仅是数据处理者，非数据所有者。用户可随时导出、删除、查看自己的全部数据。

**部署模型**:

| 阶段 | 部署方式 | 数据库 | 缓存 | 安全等级 |
|------|---------|--------|------|---------|
| PoC | 本地 Docker | SQLite | 无 | 本地可信网络 |
| Phase 1 | 云端 Docker Compose | PostgreSQL | Redis | 生产级 |
| Phase 2 | K8s 集群 | PostgreSQL HA | Redis Cluster | 企业级 |

## 2. 信任边界

```
┌─────────────────────────────────────────────────────────────┐
│                        信任边界图                            │
│                                                             │
│  [用户/小程序] ──①──▶ [EventLink API] ──②──▶ [数据库]       │
│                           │                                 │
│                           ├──③──▶ [LLM Provider (Moka AI)]  │
│                           │                                 │
│                           ├──④──▶ [微信 OAuth 服务]          │
│                           │                                 │
│                           └──⑤──▶ [邮件服务商 (IMAP)]       │
└─────────────────────────────────────────────────────────────┘
```

| 边界 | 描述 | 跨边界数据 | 当前防护 | PoC状态 |
|------|------|-----------|---------|---------|
| ① 用户 ↔ API | 用户身份认证与请求传输 | JWT token、请求体 | JWT HS256 + IP白名单 | HTTP（无TLS） |
| ② API ↔ 数据库 | 数据持久化读写 | 全部业务数据 | SQLite本地文件 | 本地访问，无加密 |
| ③ API ↔ LLM | AI推理请求 | 脱敏后的文本 | sanitize_llm_input + redact_pii_from_text | HTTPS（Moka AI侧） |
| ④ API ↔ 微信 | 小程序OAuth认证 | OAuth code → openid | 服务端code换session | 已实现 |
| ⑤ API ↔ 邮件 | IMAP邮件同步 | 邮件凭据+邮件内容 | IMAP主机白名单防SSRF | 凭据仅在内存中 |

## 3. 威胁分析（STRIDE）

基于 Security_Design_v1.md 的STRIDE分析，以下为诚实评估：

### S — Spoofing（欺骗）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| 伪造JWT身份冒充登录 | ✅ HS256签名验证 + iss/aud声明 | — | ✅ 已实现 |
| 微信小程序身份伪造 | ✅ 服务端code换openid | — | ✅ 已实现 |
| PoC匿名访问被滥用 | ✅ IP白名单(127.0.0.1/::1) | 非白名单IP无防护 | ✅ 已实现 |
| poc_secret暴力破解 | ✅ hmac.compare_digest防时序攻击 | 无失败次数限制/锁定 | ⚠️ 部分实现 |

### T — Tampering（篡改）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| API请求参数中间人篡改 | — | PoC无TLS，HTTP明文传输 | ❌ PoC不防护 |
| JWT token被篡改 | ✅ HS256签名校验 | — | ✅ 已实现 |
| 数据库数据被篡改 | — | SQLite文件无完整性校验 | ❌ PoC不防护 |

### R — Repudiation（抵赖）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| 用户否认执行了某操作 | — | 无操作审计日志 | ❌ 未实现 |
| PoC匿名访问无法追溯 | ⚠️ 日志记录IP和时间戳 | 无用户级审计链 | ⚠️ 部分实现 |

### I — Information Disclosure（信息泄露）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| PII数据（手机号/邮箱/微信）泄露 | ✅ redact_pii_from_text脱敏 | 数据库存储未加密 | ⚠️ 传输层脱敏，存储层未加密 |
| LLM响应中泄露PII | ✅ sanitize_llm_input输入消毒 | LLM输出无法完全控制 | ⚠️ 降低风险，未消除 |
| 数据库文件被直接读取 | — | SQLite文件无加密 | ❌ PoC不防护 |
| 邮件凭据泄露 | ✅ 不持久化存储 | 请求体明文传输（HTTP） | ⚠️ 内存安全，传输不安全 |

### D — Denial of Service（拒绝服务）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| LLM端点被恶意大量调用 | ✅ Rate Limit 20次/分钟 | — | ✅ 已实现 |
| 通用API端点被刷 | ✅ 认证用户60次/分钟 | 非关键端点无独立限流 | ⚠️ 部分实现 |
| 数据库耗尽 | — | SQLite无连接池限制 | ❌ PoC不防护 |

### E — Elevation of Privilege（权限提升）

| 场景 | 已防护 | 未防护 | 状态 |
|------|--------|--------|------|
| 越权访问其他用户数据 | ✅ user_id应用层强制过滤 | — | ✅ 已实现 |
| input_scope伪造 | ✅ 服务端校验 | — | ✅ 已实现 |
| SQL注入获取数据库权限 | ✅ SQLAlchemy参数化查询 | — | ✅ 已实现 |

## 4. PoC模式安全边界

PoC模式是EventLink当前阶段的核心安全上下文，以下为完整说明：

### 4.1 poc_anonymous_access（匿名访问）

- **配置项**: `poc_anonymous_access: bool = False`
- **机制**: 开启后，无JWT token的请求自动使用默认用户ID `00000000-0000-0000-0000-000000000001`
- **IP白名单**: 仅允许 `127.0.0.1` 和 `::1`（可通过 `POC_ALLOWED_IPS` 环境变量自定义）
- **审计**: 每次匿名访问均记录⚠️级别日志（含客户端IP、时间戳）
- **非白名单IP**: 返回 403 Forbidden

### 4.2 poc_secret（共享密钥登录）

- **配置项**: `EVENTLINK_POC_SECRET` 环境变量
- **机制**: 使用 `hmac.compare_digest` 进行常量时间比较，防止时序攻击
- **未设置时**: 端点返回 403，强制使用微信OAuth登录
- **已知限制**: 无失败次数限制、无账户锁定、无暴力破解防护

### 4.3 单用户模型

- 无多租户隔离
- 无RBAC权限模型
- 所有数据属于单一用户
- user_id过滤是应用层逻辑，非数据库层隔离

### 4.4 本地部署

- 无TLS要求（localhost流量）
- SQLite文件存储，无加密
- Docker网络隔离，无外部暴露

### 4.5 PoC已知风险清单

| 风险 | 影响 | 可接受性 | 理由 |
|------|------|---------|------|
| 无审计追踪 | 操作不可追溯 | ⚠️ PoC可接受 | 单用户，开发者自用 |
| 无会话管理 | JWT无法主动撤销 | ⚠️ PoC可接受 | 本地环境，token有效期短(15分钟) |
| 无Refresh Token轮换 | Token刷新无安全增强 | ⚠️ PoC可接受 | PoC未实现refresh token |
| HTTP明文传输 | 中间人可嗅探 | ⚠️ PoC可接受 | 仅localhost流量 |
| 无暴力破解防护 | poc_secret可被枚举 | ⚠️ PoC可接受 | 本地网络，攻击面极小 |

## 5. 已知安全缺口

以下为当前实现中**明确未防护**的安全项：

### 5.1 传输层

- **PoC无TLS**: 所有HTTP通信为明文，包括JWT token和请求体
- **影响**: 同网络攻击者可嗅探全部流量
- **缓解**: 仅限localhost访问，不暴露到外部网络

### 5.2 会话管理

- **无Token撤销**: JWT签发后无法主动失效，只能等待过期
- **无Refresh Token轮换**: 当前仅access_token，无refresh机制
- **影响**: Token泄露后无法紧急止损

### 5.3 凭据处理

- **邮件凭据**: 通过请求体传入IMAP host/user/password，不持久化但存在于内存中
- **影响**: 内存转储可能泄露凭据；HTTP传输为明文
- **缓解**: IMAP主机白名单防SSRF，凭据不写入数据库

### 5.4 LLM安全

- **Prompt注入**: `sanitize_llm_input` 降低风险但无法完全消除
- **影响**: 恶意构造的输入可能操纵LLM输出
- **缓解**: 输入消毒 + PII脱敏 + 输出后处理

### 5.5 限流

- **非全端点限流**: 仅LLM相关端点有独立限流（20次/分钟），通用限流较宽松
- **影响**: 非LLM端点可能被滥用
- **缓解**: 认证用户60次/分钟基础限流

### 5.6 安全测试

- **无自动化安全测试**: CI中无OWASP ZAP/DAST扫描
- **影响**: 安全漏洞无法在CI中自动发现
- **缓解**: 代码审查 + 手动安全评估

## 7. 假设与前提

| 编号 | 假设 | 影响范围 | 违反后果 |
|------|------|---------|---------|
| A1 | PoC运行在可信本地网络 | 全部PoC安全决策 | 中间人攻击、数据泄露 |
| A2 | 每个部署实例为单用户 | 无多租户隔离 | 数据越权访问 |
| A3 | 用户具备技术能力（开发者/PM级别） | 安全配置与操作 | 误配置导致安全风险 |
| A4 | PoC阶段无合规要求 | 审计、加密、日志 | 无法满足合规审计 |
| A5 | LLM提供商(Moka AI)可信处理文本数据 | LLM交互安全 | 数据被滥用或泄露 |
| A6 | 本地Docker环境网络隔离 | 无TLS、无WAF | 外部攻击者直接访问 |
| A7 | 用户不会主动恶意操作 | 单用户模型 | 自我数据破坏（可接受） |

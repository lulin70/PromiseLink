## 1. 设计原则

### 1.1 核心原则 [0.2.0更新]
- **私密优先**：所有数据仅属用户本人，无分享、无协作、无他人可见
- **移动优先**：以微信小程序为第一适配目标（375px-428px），单手可操作；基础版使用 Taro H5 编译（同一代码库，浏览器目标）
- **回应驱动**：首页从"信息展示"转向"行动驱动"，三段式布局（问候+日程+紧急待办）引导用户快速行动
- **信息密度适中**：莫兰迪色系降低视觉疲劳，长时间使用不刺眼
- **快速操作≤3点击**：核心操作不超过3次点击到达目标
- **微信生态适配**：遵循小程序设计规范

### 1.2 明确排除 [0.2.0更新]
- ❌ RBAC / 多租户 / 团队协作
- ❌ 他人资源匹配（仅匹配自己的人脉资源）
- ❌ 分享功能（私密助手，不分享任何数据）
- ❌ 原生APP（微信小程序为主入口 + H5 WebView备选）
- ~~❌ 人脉推荐~~ → **[0.2.0] 已正式移除**：与"私密助手"定位冲突，不做主动推荐（D6-9）
- ❌ 外部数字名片平台对接(基础版/专业版阶段)

### 1.3 关键术语 [0.2.0更新]
| 术语 | 说明 | 字段名 |
|------|------|--------|
| Todo类型 | 6种：cooperation_signal/risk/care/promise/followup/help | `todo_type` |
| 动作类型 | 6种：my_promise/their_promise/my_followup/mutual_action/system_reminder/unclear | `action_type` |
| 可联系度 | 人的可联系程度（0.0-1.0） | `callability` |
| 敏感度 | 2级：matchable / no_match | `sensitivity` |
| 关系阶段 | 5阶段枚举（专业版简化模型） | `RelationshipStage` |
| 关系推进卡 | F-47 RelationshipBrief，12模块全貌视图 | `relationship_briefs` |

## 2. 视觉规范

### 2.1 色彩系统

#### 2.1.1 莫兰迪主色系（Todo类型色）

产品核心视觉语言采用莫兰迪色系——低饱和度、灰调柔和，传达私密助手的温和专业感。6种Todo类型各对应一种莫兰迪色，贯穿整个产品视觉体系。

| 色名 | 色值 | Todo类型 | 语义 | 色板 |
|------|------|----------|------|------|
| 雾白 | `#B8C4C0` | cooperation_signal（合作信号） | 合作机会、可能性 | 🟩 |
| 烟粉 | `#C4A7A0` | risk（风险） | 警示、需关注 | 🟫 |
| 雾蓝 | `#A0B0C4` | care（关心） | 关心、关注 | 🟦 |
| 雾绿 | `#A0C4A8` | promise（承诺） | 承诺、兑现 | 🟩 |
| 雾金 | `#C4C0A0` | followup（跟进） | 跟进、待定 | 🟨 |
| 雾紫 | `#B0A0C4` | help（帮助） | 帮助、支持 | 🟪 |

[0.2.0新增] **关系推进卡模块色标**：

| 模块类别 | 推荐色值 | 使用场景 |
|---------|---------|---------|
| 阶段标识 | 雾绿 `#A0C4A8` | 当前关系阶段进度条、阶段标签 |
| 对方关注 | 雾蓝 `#A0B0C4` | concerns模块标签 |
| 承诺事项 | 雾绿 `#A0C4A8` | active_promises模块 |
| 需求洞察 | 雾白 `#B8C4C0` | need_insights模块 |
| 帮助贡献 | 雾紫 `#B0A0C4` | contribution/help模块 |
| 下一动作 | 雾金 `#C4C0A0` | next_natural_action模块 |

**使用规范**：
- 主色用于Todo类型标识（左侧色条、标签背景、图标着色）
- 主色作为背景时，文字使用深色（`#3A3F42`）确保可读性
- 主色作为标签时，使用10%透明度背景 + 100%边框

#### 2.1.2 功能色

功能色复用莫兰迪色系，保持视觉统一性。

| 功能 | 色值 | 来源 | 使用场景 |
|------|------|------|---------|
| 成功 | `#A0C4A8` | 雾绿 | 操作成功提示、完成状态 |
| 警告 | `#C4C0A0` | 雾金 | 警告提示、待处理状态 |
| 错误 | `#C4A7A0` | 烟粉 | 错误提示、风险状态 |
| 信息 | `#A0B0C4` | 雾蓝 | 信息提示、背景说明 |

#### 2.1.3 中性色

| 用途 | 色值 | 说明 |
|------|------|------|
| 主文字 | `#3A3F42` | 标题、重要信息 |
| 次文字 | `#6B7280` | 正文、描述 |
| 辅助文字 | `#9CA3AF` | 时间戳、标签 |
| 占位文字 | `#C4C8CC` | 输入框placeholder |
| 分割线 | `#DDE0E3` | 列表分割、区域分隔 |
| 页面背景 | `#F5F6F7` | 全局背景色 |
| 卡片背景 | `#FFFFFF` | 卡片/弹窗背景 |
| 禁用背景 | `#EDF0F2` | 禁用状态/PoC不可用阶段 |
| 禁用文字 | `#C4C8CC` | 禁用状态文字/PoC不可用阶段文字 |

#### 2.1.4 色彩使用规范

**对比度要求**：
- 主文字（`#3A3F42`）在白色背景上：对比度 ≥ 12:1 ✅
- 次文字（`#6B7280`）在白色背景上：对比度 ≥ 5.5:1 ✅
- 莫兰迪主色在白色背景上：对比度约 2.5-3.0:1，**仅用于装饰性色条/图标**

**色盲友好**：
- 不单独依赖颜色传达信息，必须配合图标/文字标签
- Todo类型标识 = 颜色色条 + 图标 + 文字标签，三重编码

## 3. 组件库

> 以下组件沿用v1.2定义，新增组件标记🆕

### 3.1 PersonCard（人物卡片）

展示人物基本信息 + 资源标签 + 敏感度标识 + 快速操作。

**属性**：
| 属性 | 类型 | 说明 |
|------|------|------|
| name | string | 人物姓名 |
| company | string | 公司 |
| title | string | 职位 |
| avatar | string | 头像URL |
| resource_tags | string[] | 资源标签 |
| sensitivity | enum | matchable / no_match |
| callability | float | 可联系度 0.0-1.0 |

**视觉规范**：
- 卡片背景：`#FFFFFF`，圆角12px，阴影E1
- 敏感度标识：`matchable`不显示；`no_match`显示🔒 + "不可匹配"
- 点击整卡跳转推进卡详情页

### 3.2 TodoItem（待办事项）

展示Todo类型标识 + 状态 + 描述 + 操作按钮。6种类型对应6种莫兰迪色。

**属性**：
| 属性 | 类型 | 说明 |

| todo_type | enum | cooperation_signal/risk/care/promise/followup/help |
| action_type | enum | my_promise/their_promise/my_followup/mutual_action/system_reminder/unclear [0.2.0新增] |
| status | enum | pending/in_progress/done/dismissed/snoozed |
| priority | enum | high/medium/low |
| description | string | Todo描述 |
| due_date | datetime | 截止时间 |
| related_entity | object | 关联人物摘要 |

**morandi_color映射**：
```
cooperation_signal → #B8C4C0（雾白）
risk              → #C4A7A0（烟粉）
care              → #A0B0C4（雾蓝）
promise           → #A0C4A8（雾绿）
followup          → #C4C0A0（雾金）

```

### 3.3 SearchBar（搜索栏）

输入框 + 搜索按钮。背景`#EDF0F2`，圆角8px。

### 3.4 VoiceRecorder（语音录入）

按住说话 + 波形动画。录音按钮56px圆形，雾蓝背景。

### 3.6 ResourceTag（资源标签）

展示资源/需求/技能标签。resource=雾蓝，demand=雾白，skill=雾绿。

### 3.7 SensitivityBadge（敏感度标识）

`matchable`不显示；`no_match`显示🔒 + "不可匹配"。

### 3.8 🆕 RelationshipProgressBar（关系阶段进度条）[0.2.0新增, v2.4更新]

5阶段水平进度条组件，可视化RelationshipStage。专业版采用简化5阶段模型，替代原7阶段模型。

```
┌─────────────────────────────────────────────────────┐
│  关系阶段                                            │
│  ○───●───○───○───○                                 │
│  初识 建立信任 深化关系 稳定合作 战略伙伴              │
│       ↑ 当前阶段                                    │
│            [确认升级到：深化关系]                      │
└─────────────────────────────────────────────────────┘
```

**属性**：
| 属性 | 类型 | 说明 |
|------|------|------|
| current_stage | enum | 当前阶段（RelationshipStage枚举） |
| stages | array | 5个阶段的完整定义 |
| can_advance | bool | 是否可升级到下一阶段 |
| next_stage | enum | 建议的下一阶段 |

**5阶段定义**：

| # | 阶段key | 阶段中文名 | 语义 |
|---|---------|-----------|------|
| 1 | initial_contact | 初识 | 刚建立联系，正在互相了解 |
| 2 | building_trust | 建立信任 | 知道对方关心什么，理解其业务痛点 |
| 3 | deepening_relationship | 深化关系 | 已为对方提供帮助或价值，获得正面反馈 |
| 4 | stable_cooperation | 稳定合作 | 多次互惠往来，建立了深度信任关系 |
| 5 | strategic_partnership | 战略伙伴 | 稳定的长期合作伙伴关系 |

**视觉规范**：
- 进度条水平排列，每个阶段 = 圆形节点(20px) + 连线 + 阶段名称(10px)
- 已完成阶段：节点填充雾绿`#A0C4A8`，连线实线雾绿，显示✓
- 当前阶段：节点填充雾绿 + 外圈发光效果(2px雾绿)，阶段名称加粗
- 未来阶段：空心圆(16px) + 雾绿边框
- RS-01强制确认：阶段升级按钮仅在AI建议升级时出现，需用户主动点击确认

### 3.9 🆕 RelationshipBriefCard（关系推进卡容器）[0.2.0新增]

F-47关系推进卡的UI容器，包含12个模块按优先级分层展示。

**属性**：
| 属性 | 类型 | 说明 |
|------|------|------|
| person_id | string | 关联人物ID |
| stage | enum | 当前关系阶段 |
| brief_data | object | 推进卡完整数据（12模块） |

#### 问候区

**信息架构**：
- 问候语：根据时段自动切换（夜深了/早上好/中午好/下午好/晚上好）
- 日期：`{月}月{日}日`
- 完整日期行：`{年}年 · {星期}`

#### 今日日程

**信息架构**：
- 数据来源：当天类型为meeting/call的Event，按时间升序排列
- 每条日程 = 时间 + 主题 + 参会人 + 类型标识（📅会议/📞电话）
- 横向可滚动（ScrollView scrollX），适合多日程场景
- 右侧显示日程总数（如"3项"）
- 空状态："今天暂无日程安排"

#### 紧急待办

**信息架构**：
- 数据来源：status不为completed/cancelled的Todo，按dynamic_score降序排列
- 最多显示5条
- 每条待办 = Todo类型色条 + 标题 + 截止日期 + 类型标签 + 动态评分
- 右侧显示"按优先级"标识
- 空状态："暂无紧急待办"
- 点击待办卡片 → 查看待办详情
- 支持一键完成操作

#### 浮动快捷操作区 [v2.4更新]

3个浮动QuickAction按钮，固定在首页底部内容区：

| 入口 | 图标 | 颜色 | 功能 | 触发行为 |
|------|------|------|------|---------|
| 🎤 语音 | 麦克风 | 雾蓝 `#A0B0C4` | 打开VoiceAssistant对话面板 | 打开Chat Panel（语音模式） |
| 📷 拍照 | 相机 | 雾绿 `#A0C4A8` | 拍照/选图→OCR识别→创建Event | 调用Taro.chooseImage→api.ocrCreateEvent |
| 💼 名片 | 名片 | 雾紫 `#B0A0C4` | 拍照→OCR识别名片信息 | 调用Taro.chooseImage→api.ocrRecognize |

**交互细节**：
- 语音按钮：打开VoiceAssistant组件（Chat Panel模式），默认语音输入
- 拍照按钮：调用相机/相册→上传图片→后端OCR→自动创建互动记录Event
- 名片按钮：调用相机/相册→上传图片→后端OCR→识别名片结构化数据（姓名/公司/职位等）
- H5预览环境降级：拍照/名片按钮弹出提示"请在微信小程序中使用拍照功能"

> 与"私密助手"定位冲突，不做主动推荐。原首页中的"人脉推荐"区域已移除。

## 5. RelationshipStage可视化规范 [0.2.0新增, v2.4更新]

### 5.1 5阶段定义（专业版简化模型）

> **v2.4变更**：从原7阶段模型简化为5阶段模型。5阶段更简洁实用，适合专业版快速迭代。后端仍保留7阶段枚举（含deep_trust/active_cooperation/long_term_partner/dormant），前端StageFlow组件使用5阶段展示。

| # | 阶段key | 阶段中文名 | 语义 | 后端映射 |
|---|---------|-----------|------|---------|
| 1 | initial_contact | 初识 | 刚建立联系，正在互相了解 | new_connection |
| 2 | building_trust | 建立信任 | 知道对方关心什么，理解其业务痛点 | understanding_needs |
| 3 | deepening_relationship | 深化关系 | 已为对方提供帮助或价值，获得正面反馈 | value_response |
| 4 | stable_cooperation | 稳定合作 | 多次互惠往来，建立了深度信任关系 | deep_trust + active_cooperation |
| 5 | strategic_partnership | 战略伙伴 | 稳定的长期合作伙伴关系 | long_term_partner |

**后端7阶段→前端5阶段映射规则**：
- `new_connection` → 初识
- `understanding_needs` → 建立信任
- `value_response` → 深化关系
- `deep_trust` / `active_cooperation` → 稳定合作
- `long_term_partner` → 战略伙伴
- `dormant` → 不在进度条中展示，单独显示"沉寂"标签

### 5.2 进度条样式规范

**布局**：水平方向，5个等宽节点

**节点状态**：
| 状态 | 视觉表现 | 色值 |
|------|---------|------|
| 已完成 | 实心圆(16px) + 雾绿填充 + ✓ | `#A0C4A8` |
| 当前阶段 | 实心圆(20px) + 雾绿填充 + 2px外圈发光 | `#A0C4A8` |
| 未来阶段 | 空心圆(16px) + 雾绿边框 | `#A0C4A8` |

**连线状态**：
| 状态 | 视觉表现 |
|------|---------|
| 已走过 | 实线 2px 雾绿 |
| 未走到 | 虚线 1px `#DDE0E3` |

**阶段名称**：
- 位置：节点下方
- 已完成/当前：10px `#3A3F42` 正常字重
- 未来阶段：10px `#9CA3AF`
- 当前阶段名称加粗(600weight)

### 5.3 RS-01 强制确认按钮

当AI检测到满足升级条件时，在当前阶段节点旁显示升级提示：

```
┌──────────────────────────────┐
│  💡 建议升级到：深化关系      │
│  理由：已连续3次价值回应...   │
│                              │
│  [确认升级]    [暂时忽略]     │
└──────────────────────────────┘
```

- 出现条件：AI判断≥80%概率应该升级时
- 位置：进度条下方，浮动卡片
- "确认升级"按钮：雾绿填充，触发PATCH /stage API（需带version乐观锁）
- "暂时忽略"按钮：灰色边框，7天内不再提示此升级建议
- **硬约束**：用户不点击"确认升级"，阶段永远不会自动改变

### 11.3 后端API集成与本地NLU降级 [v2.4新增]

#### 优先后端API

用户输入文字后，优先调用后端Voice Query API：

```
用户输入文字/语音转文字
    ↓
POST /api/v1/voice/query { user_id, text }
    ↓ (成功)
显示AI回答气泡（result.response）
    ↓ (语音模式)
TTS播报回答（api.ttsSynthesize → InnerAudioContext播放）
```

**后端Pipeline**：voice→ASR→NLU→DB query→NLG→TTS

#### 本地NLU降级

后端API不可用时，降级到本地NLU意图匹配：

```
用户输入文字
    ↓
后端API调用失败
    ↓
本地matchIntent()匹配5种意图：
  - query_todos: 待办查询
  - query_todo_detail: 待办详情
  - query_schedule: 日程查询
  - query_contact: 联系人查询
  - record_note: 记录备忘
    ↓
本地generateResponse()生成回答
    ↓
显示AI回答气泡
```

**本地NLU能力**：
- 基于正则匹配5种核心意图
- 支持提取实体（序号、人名）
- 使用Mock数据生成回答
- 语音模式下仍尝试TTS播报

### 11.6 错误与异常状态的UI处理

| 场景 | UI表现 | 用户可执行操作 |
|------|--------|--------------|
| ASR失败 | 用户气泡显示"语音识别失败，请重试或用文字输入" | 重新录音 / 切换文字输入 |

## 19. 三级产品模型功能可用性 [v3.1新增]

> **v3.1变更**：产品从 PoC→Phase1→Phase2 演进模型，更新为**基础版/专业版/定制版**三级产品模型。基础版使用 Taro H5 编译（同一代码库，浏览器目标），专业版为微信小程序，定制版支持深度定制。

### 19.1 基础版 Taro H5 UI 适配考量

基础版使用 Taro H5 编译，与专业版（微信小程序）共享同一代码库，但目标运行环境为浏览器。以下为 H5 与小程序的关键差异及适配策略：

| 差异项 | 微信小程序（专业版） | 浏览器 H5（基础版） | 适配策略 |
|--------|---------------------|---------------------|----------|
| **语音录入** | Taro.getRecorderManager + 微信同声传译插件 | ❌ 不支持 | 基础版为纯文本交互，语音录入为专业版专属功能 |
| **TTS播报** | 小程序原生 InnerAudioContext + 后端TTS API | ❌ 不支持 | 基础版为纯文本交互，TTS为专业版专属功能 |
| **拍照/选图** | Taro.chooseImage（调用微信相机） | ❌ 不支持 | 基础版不处理图片，名片扫描为专业版专属功能 |
| **名片扫描** | 小程序相机+OCR | ❌ 不支持 | 基础版为纯文本交互，名片扫描为专业版专属功能 |
| **微信登录** | wx.login() → code → openid | 用户名密码 / 邮箱登录 | 基础版使用独立认证方案（无微信OAuth） |
| **推送通知** | 微信服务号模板消息 | 浏览器 Notification API | 基础版使用浏览器通知（需用户授权） |
| **本地存储** | Taro.setStorageSync | localStorage | 统一使用 Taro 存储API，H5自动降级到localStorage |
| **页面路由** | Taro.navigateTo / switchTab | 浏览器 History API | Taro路由自动适配 |
| **安全区域** | env(safe-area-inset-*) | CSS env()（部分浏览器支持） | 基础版使用CSS fallback |
| **分享** | onShareAppMessage | Web Share API（兼容性有限） | 基础版不提供分享功能（与私密助手定位一致） |

**基础版 H5 特有设计注意**：
- 首次访问引导用户"添加到主屏幕"（PWA体验）
- 浏览器地址栏和底部工具栏占用空间，页面内容区需额外预留 60px
- 桌面浏览器响应式：≥768px 宽度时采用双栏布局（列表+详情）
- 不支持微信支付等微信生态能力，基础版无需相关UI

# PromiseLink UI 改进方案

**版本**: v1.0
**日期**: 2026-07-03
**范围**: 基础版（宽屏 H5）+ 专业版（微信小程序）
**参考文档**: UI_UX_Design_v1.md v3.1, UI_Design_Gap_Analysis_2026-06-19.md
**设计原则**: 两端功能一致、场景适配差异

---

## 核心设计原则

### 场景分工

| 维度 | 基础版（宽屏 H5） | 专业版（微信小程序） |
|------|-------------------|---------------------|
| 核心场景 | 整理归纳、深度分析、批量操作 | 快速录入、即时查询、移动提醒 |
| 信息密度 | 高（宽屏可展示更多模块） | 精炼（单手操作，核心信息优先） |
| 交互方式 | 键盘/鼠标为主 | 触屏/语音/拍照为主 |
| TabBar | 可保留更多功能入口（屏幕空间充足） | 精简为 5 个高频入口 |
| 详情展示 | 偏好右侧面板或独立详情页 | 偏好独立全屏详情页 |

### 功能一致性要求

- 两端调用同一套后端 API
- 核心实体（人脉/待办/承诺/事件/提醒）两端均可访问和操作
- 状态变更一端操作后，另一端刷新即可同步
- 专业版独有功能：语音录入、拍照识别、名片扫描、微信消息推送

## 一、P0 级问题（功能缺失）

### 1.1 基础版设置页功能不足

**现状**:
- 小程序版 `pages/mine/index.tsx`：已实现完整设置页，包含专业版激活、数据导出、CSV 导入、隐私与数据管理、邮件同步配置、语音查询、提醒设置、需求录入、退出登录等 11 个菜单项
- 基础版 `pages/mine/index.tsx`：仅有 4 个菜单项（升级专业版、导出数据、关于、退出），且无隐私与数据管理、无邮件同步配置、无 CSV 导入

**问题**:
- 用户在宽屏端无法管理隐私设置（导出/删除数据）
- 无法配置邮件同步（专业版功能，但基础版应展示入口并提示升级）
- 无法设置提醒偏好（每日上限、免打扰时段）

**建议方案**:

补充以下功能模块，与小程序版对齐：

| 菜单项 | 基础版行为 | 说明 |
|--------|-----------|------|
| 升级专业版 | 跳转外链 | 已有，保留 |
| 数据导出 | 导出 JSON 文件 | 已有，保留 |
| CSV 导入 | 选择文件 + 上传 | 新增，与小程序对齐 |
| 隐私与数据 | 导出 / 删除数据 | 新增，删除需二次确认 |
| 邮件同步配置 | 提示"专业版功能" | 新增，展示但标记需升级 |
| 提醒设置 | 偏好设置面板 | 新增，调用 `getReminderPreferences` / `updateReminderPreferences` API |
| 关于 PromiseLink | 弹窗展示版本信息 | 已有，保留 |
| 退出登录 | 清除凭证并跳转 | 已有，保留 |

宽屏版优势：设置页可以做成左侧导航 + 右侧表单/详情的布局，而非小程序的纵向菜单列表。

### 1.2 全局搜索页缺失

**现状**:
- 小程序版：无独立全局搜索页，各列表页有局部关键词搜索（人脉页搜索 `contacts/index.tsx`，Todo 页无搜索，承诺页有搜索）
- 基础版：无独立全局搜索页，人脉页和承诺页有局部搜索，其余页面无搜索

**问题**:
- 用户无法跨实体类型搜索（比如搜一个人名，同时看到相关事件、待办、承诺）
- 搜索仅限于关键词匹配，缺少语义搜索能力
- 无搜索历史 / 热门搜索

**建议方案**:

新增全局搜索页 `/search`，功能包括：

1. **统一搜索入口**
   - 基础版：顶部导航栏搜索框（常驻）
   - 小程序版：首页顶部搜索框 + TabBar 可加"搜索"入口，或在首页下拉露出搜索框

2. **混合搜索结果**
   - 搜索结果按类型分组展示：人物 / 事件 / 承诺 / 待办
   - 每个结果项显示：类型图标 + 标题 + 摘要 + 相关度分数
   - 支持关键词匹配（前端即时过滤）+ 语义搜索（后端 `/api/v1/search` 接口）

3. **搜索辅助**
   - 搜索历史（最近 10 条，本地存储）
   - 热门/推荐搜索词（基于用户最近互动频率）
   - AI 搜索建议（"试试这样搜"）

4. **快捷筛选**
   - 搜索结果上方提供类型筛选 Tab：全部 / 人物 / 事件 / 承诺 / 待办
   - 时间范围筛选：本周 / 本月 / 近三个月

5. **后端 API**
   - 新增 `/api/v1/search` 统一搜索接口，接受 `query`、`types[]`、`date_range` 参数
   - 利用已有的 `semantic_search` 服务做语义匹配

### 1.3 每日提醒页 — 基础版完全缺失

**现状**:
- 小程序版 `pages/reminders/index.tsx`：已完整实现，包括 4 级优先级提醒卡（高优/中优/低优/参考）、操作按钮（已完成/推迟 N 天/忽略）、偏好设置（每日上限/免打扰时段/推送开关）
- 基础版：无每日提醒页面。API `getDailyReminders` / `reminderAction` / `getReminderPreferences` 已在后端就绪，但基础版前端无页面消费

**问题**:
- 基础版用户无法查看和管理每日提醒
- 宽屏版更适合批量处理提醒（列表展示 + 批量操作），但该功能完全空白

**建议方案**:

基础版新增 `/reminders` 页面，与小程序版功能对齐但适配宽屏交互：

1. **布局**: 左侧提醒列表 + 右侧选中提醒的详情/操作面板（三栏布局的一部分，或独立两栏页面）
2. **提醒列表**: 按优先级分组（高优/中优/低优/参考），每组显示色标和计数
3. **提醒卡片**: 展示联系人姓名、提醒原因、上下文（到期日期/关系阶段/互动间隔天数）、操作按钮（已完成/推迟/忽略）
4. **批量操作**: 宽屏版独有优势 — 支持多选提醒后批量完成/推迟
5. **偏好设置**: 嵌入页面底部或右侧面板，包含每日推送上限、免打扰时段、推送开关
6. **入口**:
   - 基础版 TabBar 不建议占用一个位置，建议在首页"今日日程"区域上方显示提醒摘要条（"今日有 3 条提醒"），点击进入完整页面
   - 也可在设置页"提醒设置"菜单项中提供入口

## 二、P1 级问题（交互设计）

### 2.1 详情页展示策略不统一

**现状**:

| 页面 | 基础版展示方式 | 小程序版展示方式 |
|------|---------------|----------------|
| 人脉（Entity） | Modal 弹窗 (`detailVisible` in `entities/index.tsx`) | 独立详情页 (`relationship/index.tsx`，10 个模块） |
| 待办（Todo） | 独立详情页 (`todos/detail.tsx`) | 独立详情页 (`todos/detail/index.tsx`) |
| 承诺（Promise） | 独立详情页 (`promises/detail.tsx`) | 无独立详情页（承诺页列表内操作） |
| 事件（Event） | 独立详情页 (`events/detail.tsx`) | 独立详情页 (`events/detail/index.tsx`) |
| 推进卡（Brief） | 无 | 独立详情页 (`relationship/index.tsx`，即人脉详情） |

**问题**:
- 基础版人脉详情用 Modal 弹窗，信息承载量极有限，只能展示基础属性。设计稿定义的推进卡详情有 12 个模块（基础信息/最近交流/对方关注/我方承诺/对方待确认/需求洞察/下一节点/推荐回应/反馈记录等），弹窗根本无法承载
- 基础版原本有三栏布局（sidebar + content + 右侧详情栏），但右栏被移除（代码注释显示"Q1: showing placeholder text on all other pages"），导致宽屏空间浪费
- 小程序版关系推进卡详情内容丰富（7 阶段进度条 + 10 个模块），但基础版完全缺失对应能力

**建议方案**:

统一详情展示策略：

1. **核心实体（人脉/推进卡）— 独立详情页**
   - 基础版：新增 `/entities/:id` 独立详情页，复用小程序版 `relationship/index.tsx` 的模块结构，适配宽屏两栏/三栏布局
   - 人脉列表页的点击行为从"打开 Modal"改为"跳转详情页"
   - Modal 降级为"快速预览"用途（列表页长按或特定按钮触发），仅展示姓名/公司/阶段/待办数 3-4 个关键指标

2. **简单实体（待办/承诺/事件）— 保持独立详情页**
   - 两端均已实现，保持现状
   - 小程序版承诺页补充独立详情页

3. **基础版宽屏右栏复活（可选）**
   - 不恢复全局右栏，而是在特定列表页（人脉列表）采用"左列表 + 右详情"的主从布局
   - 用户在左栏选中一个联系人，右栏即时展示该联系人的关键模块（阶段/待办/最近互动），无需跳转
   - 点击右栏的"查看完整详情"按钮再跳转到独立详情页
   - 这种模式在 CRM/通讯录类产品中是经典宽屏交互模式

4. **小程序版保持全屏详情页**
   - 移动端不存在"右栏"概念，全屏详情页是最优解

### 2.2 Todo 页缺少 action_type 维度视图

**现状**:
- 小程序版 `todos/index.tsx`：Tab 按 `todo_type` 分为"全部/承诺/关注/跟进/风险"
- 基础版 `todos/index.tsx`：Tab 按 `status` + `todo_type` 组合筛选
- 设计稿要求：按 `action_type` 维度分为"我的承诺/等待回应/全部/已完成"

**问题**:
- 用户最高频的需求是"我答应了别人什么"和"别人答应了我什么"，而非按 todo_type（承诺/关注/跟进）分类
- `action_type` 维度的"等待回应"视图在两端都完全缺失
- 现有 todo_type 维度有价值但不是唯一的分类方式

**建议方案**:

两端均新增 action_type 维度 Tab：

| Tab | action_type 筛选条件 | 说明 |
|-----|---------------------|------|
| 我的承诺 | `my_promise` | 我对别人的承诺，需要我主动履行 |
| 等待回应 | `their_promise` | 别人对我的承诺，需要我追踪确认 |
| 跟进事项 | `my_followup` + `mutual_action` | 需要主动跟进的互动 |
| 全部 | 不过滤 | 按优先级排序的全部待办 |
| 已完成 | `status = done/completed` | 已完成的待办归档 |

- 基础版宽屏优势：可以在"等待回应"Tab 中增加"提醒对方"批量操作，或展示等待天数统计
- 小程序版：每个 Tab 下方显示该分类的待办数量 Badge

### 2.3 新手引导内容与形态改进

**现状**:
- 小程序版 `components/Guide/index.tsx`：4 步弹窗，内容为功能介绍（核心功能/开始记录/专业版功能）
- 基础版 `components/Guide/Guide.tsx`：4 步弹窗，内容为桌面布局说明（三栏布局/记录交流/AI 解析）

**问题**:
- 基础版引导内容讲"三栏布局"等 UI 结构，用户不关心布局，关心的是"我能做什么"
- 两端引导都是文字说明弹窗，无实际交互操作
- 设计稿定义的是"记录一次重要交流"的 4 屏交互流（实际录入 → 确认重点 → 获得价值），与当前实现完全不同

**建议方案**:

1. **内容重写** — 从"讲布局"改为"讲场景"

   小程序版引导步骤：
   - 第 1 步："外出时，随时用语音或文字记录一次交流"（配录入入口动效）
   - 第 2 步："AI 自动提取人物、承诺和关注点"（配 AI 解析动画）
   - 第 3 步："回到首页，查看今日待办和日程"（配首页预览）
   - 第 4 步："专业版还支持拍照识别和名片扫描"（配功能亮点）

   基础版引导步骤：
   - 第 1 步："在电脑上，整理你的人脉关系全景"（配人脉列表预览）
   - 第 2 步："查看每个人的关系阶段和待跟进事项"（配推进卡预览）
   - 第 3 步："外出时用小程序快速录入，回到电脑上深度整理"（配双端协作示意）
   - 第 4 步："设置每日提醒，不错过任何重要跟进"（配提醒预览）

2. **交互改进**（可选，优先级 P2）
   - 引导完成后自动跳转到录入页（小程序）或人脉页（基础版），而非仅关闭弹窗
   - 引导过程支持"跳过"和"下次不再显示"

## 三、P2 级问题（视觉与体验打磨）

### 3.1 ContactCard 信息密度不足

**现状**:
小程序版 `ContactCard/index.tsx` 展示：首字母头像 + 姓名 + 公司职位 + 关注标签（最多 2 个）+ 最近互动日期

**设计稿要求**:
关系阶段标签 + 待办数（"2 项待办"）+ 上次联系时间 + 敏感度标识（🔒不可匹配）+ 资源标签

**建议方案**:

在小程序版 ContactCard 中补充以下信息行：

```
┌────────────────────────────────┐
│ [头像] 张三          合作中    │  ← 补充关系阶段标签
│        某公司 · 产品总监        │
│        [关注A] [关注B]         │
│        2项待办 · 3天前联系过    │  ← 补充待办数 + 联系间隔
└────────────────────────────────┘
```

基础版人脉列表同步补充。敏感度标识（🔒）在联系人标记为 `no_match` 时显示在姓名右侧。

### 3.2 动态优先级分数可视化

**现状**:
- 后端已计算 `dynamic_score`（0-100），前端在排序时使用但完全不展示
- 用户无法感知"为什么这个待办排在前面"

**建议方案**:

在 TodoCard 上展示优先级分数：

| 分数区间 | 颜色 | 标签 | 说明 |
|---------|------|------|------|
| ≥ 85 | 烟粉 `#C4A7A0` | 紧急 | 需立即处理 |
| 60-84 | 雾金 `#C4C0A0` | 重要 | 今日内关注 |
| 40-59 | 雾蓝 `#A0B0C4` | 一般 | 本周内跟进 |
| < 40 | 雾绿 `#A0C4A8` | 低优 | 有空再看 |

展示形式：
- 小程序版：TodoCard 右上角显示分数 + 色点（如"72"配黄色圆点）
- 基础版：Todo 列表可显示分数列，支持按分数排序

### 3.3 空状态视觉改进

**现状**:
各页空状态使用单字占位（"搜"/"醒"/"诺"/"待"），视觉粗糙

**建议方案**:
- 统一使用 SVG 线性插图（4-5 个通用插图：空搜索/空列表/空提醒/空承诺/加载中）
- 配合引导性文案（如"暂无待办，录入一次交流后这里会出现待办事项"）
- 可考虑使用莫兰迪色系的简约线条风格

### 3.4 语音助手视觉打磨（小程序版独有）

**现状**:
- 录音状态使用文字 "●" 作为图标
- 对话气泡无 user/AI 色差区分
- 快捷问题标签仅 3 个

**建议方案**:
- 录音脉冲动画：使用 CSS 动画的同心圆脉冲效果替代文字
- 对话气泡着色：用户消息气泡使用雾蓝 `#A0B0C4`，AI 回复气泡使用白色 `#FFFFFF`（设计稿§11.2 定义）
- 快捷问题标签补充至 4 个，增加"帮我记一下"类录入引导标签

## 四、两端共通问题

### 4.1 色彩变量系统不统一

**现状**:
- 小程序版：完善的 `theme.scss`（莫兰迪色系定义）+ `variables.scss`（间距/圆角/阴影/字体 Mixin）
- 基础版：色彩值散落在各页面 SCSS 和 TSX 内联样式中，无统一变量文件

**建议方案**:
- 基础版新建 `frontend/src/styles/theme.scss`，从小程序版 `theme.scss` 复制并适配（rpx → px）
- 新建 `frontend/src/styles/variables.scss`，包含间距/圆角/阴影/字体等设计 Token
- 所有页面 SCSS 统一引用变量，禁止硬编码色值

### 4.2 缺少统一的加载状态组件

**现状**:
两端均使用 `Text>加载中...</Text>` 作为加载状态

**建议方案**:
- 新建 Skeleton 骨架屏组件，用于数据密集页面（人脉列表/推进卡详情/承诺列表）
- 小程序版可用 View + 背景色模拟骨架形状
- 基础版可用 CSS animation 实现闪烁效果
- 空内容加载时用 Skeleton 替代 loading 文字，数据返回后平滑过渡到真实内容

### 4.3 基础版承诺状态枚举不一致

**现状**:
- 基础版 `promises/index.tsx` STATUS_FILTERS 包含 `broken`（已违背）
- 小程序版 `promises/index.tsx` 使用 `expired`（已失效）
- 设计稿定义的是 `expired`

**建议方案**:
统一为设计稿定义的枚举：`pending / overdue / fulfilled / expired`，基础版移除 `broken`

| 优先级 | 编号 | 改进项 | 影响端 | 预估工作量 |

| P0 | 1.1 | 基础版设置页补齐 | 基础版 | 0.5 天 |
| P0 | 1.2 | 全局搜索页 | 两端 | 1.5 天 |
| P0 | 1.3 | 基础版每日提醒页 | 基础版 | 0.5 天 |
| P1 | 2.1 | 详情页展示策略统一 | 基础版为主 | 2 天 |
| P1 | 2.2 | Todo 增加 action_type 视图 | 两端 | 0.5 天 |

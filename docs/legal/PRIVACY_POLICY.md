## 1. 数据存储位置

### 1.1 基础版（本地部署）

PromiseLink 基础版采用**本地优先**架构，您的所有业务数据存储在您自己的电脑上：

| 数据类型 | 存储位置 | 说明 |
|---------|---------|------|
| 事件记录 | 本地 SQLite (`/app/data/promiselink.db`) | 会议、通话、交往记录 |
| 人脉信息 | 本地 SQLite | 姓名、职位、联系方式 |
| 待办事项 | 本地 SQLite | 跟进任务、提醒 |
| 承诺追踪 | 本地 SQLite | 承诺记录、状态 |
| 关联关系 | 本地 SQLite | 人脉-事件-待办关联 |

**核心承诺**：您的业务数据**从不出家门**——不上传到任何云服务器，不经过任何第三方存储。

### 1.2 专业版（云端网关）

专业版的**网关服务**部署在云端（阿里云 ECS，域名 `gateway.promiselink.cn`），但网关**不存储您的业务数据**：

- 网关只做**加密转发**：接收 AI 请求 → 转发给 LLM 提供方 → 返回结果
- 网关日志只记录**计费元数据**（调用时间、token 数量、license_key 前 8 位），不记录请求内容
- 网关数据库只存储**许可证信息**（license_key、激活状态、到期时间）

## 2. 数据处理方式

### 2.1 AI 增强调用（重要）

当您使用 AI 功能（如会议纪要生成、人脉分析、智能提醒）时，**部分业务数据会经网关发给 LLM 提供方进行处理**：

| AI 功能 | 发送的数据 | 数据流向 |
|--------|-----------|---------|
| 会议纪要生成 | 事件文本内容 | 本地 → 网关 → Moka AI → 网关 → 本地 |
| 人脉关系分析 | 人脉姓名、职位、关系描述 | 本地 → 网关 → Moka AI → 网关 → 本地 |
| 智能提醒 | 待办事项内容 | 本地 → 网关 → Moka AI → 网关 → 本地 |
| 语音转文字（专业版） | 音频文件 | 小程序 → 网关 → Moka AI → 网关 → 小程序 |
| 图片识别（专业版） | 图片文件 | 小程序 → 网关 → Moka AI → 网关 → 小程序 |

**重要说明**：
- AI 调用是**用户主动触发**的，不会在后台自动发送数据
- 发送给 LLM 的数据**不被网关存储**，只做实时转发
- LLM 提供方（Moka AI）有自己的数据处理政策，请参阅其隐私政策

### 2.2 WSS 中继（专业版）

专业版使用 WSS 长连接实现小程序访问本地数据：

- **连接方向**：用户电脑主动出站连接网关（无需公网IP）
- **数据流向**：小程序 → 网关（WSS 中继）→ 用户电脑 → 网关 → 小程序
- **加密方式**：WSS（WebSocket over TLS）+ JWT 认证
- **网关角色**：纯转发，不存储、不解析业务数据

**API 端点**：`GET /api/v1/privacy/data-summary`

## 5. 数据安全措施

### 5.1 传输安全

- 所有网络通信使用 HTTPS/WSS（TLS 1.2+）
- JWT 认证（RS256 非对称签名，专业版）
- 网关不存储业务数据，只做加密转发

### 5.2 存储安全

- 基础版数据存储在本地 SQLite，受用户电脑自身安全保护
- 网关数据库（PostgreSQL）存储在阿里云 ECS，受云安全组保护
- 敏感环境变量（API Key、密钥）通过 Docker secrets 管理，不硬编码

### 5.3 访问控制

- 基础版：本地访问，无需认证（localhost）
- 专业版网关：所有端点需要 JWT 认证（license_key 激活后获取）
- 网关 admin API：需要 ADMIN_API_KEY（仅运营人员持有）

## 6. License Key 安全

### 6.1 当前状态

- License key 用于专业版激活和用户身份识别
- License key 格式：`PL-PRO-xxxx-xxxx-xxxx`
- 激活后生成 JWT（含 user_id + license_key claims）

### 6.2 安全措施（已实现）

License key 安全机制已在网关 v0.7.0+ 实现：

- **单设备绑定**：`device_fingerprint`（SHA-256）绑定，`max_devices=1`，激活时检查设备指纹匹配，防止跨设备滥用
- **吊销机制**：`revoke_license` 方法支持按 `user_id` 或 `license_key` 吊销，所有活跃 JWT 通过 Redis CRL 黑名单（`jwt_blacklist:{jti}`）实时失效
- **JWT 追踪**：每个 JWT 的 `jti` 在 Redis 中追踪（`jwt_active:{license_key}`），支持吊销时批量失效
- **使用追踪**：`quota_used_tokens/media` 字段记录使用量，审计日志（`AuditLog`）记录所有激活/刷新/吊销操作
- **防抢绑**：license_key 首次激活后绑定 `user_id`，其他用户尝试激活会被拒绝（409 LicenseAlreadyActivated）
- **测试覆盖**：39 个测试覆盖单设备绑定、吊销、JWT 追踪等场景（`gateway/tests/test_license_service.py`）

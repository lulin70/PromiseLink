# PromiseLink AGPL v3 传染边界法律确认

> **文档版本**: v1.0
> **日期**: 2026-06-18
> **状态**: 技术团队初步评估，待法律顾问正式确认
> **关联文档**: `docs/architecture/Repo_Split_Decision.md` §7、`docs/architecture/edition_architecture.md`

---

## 1. 文档目的

本文档明确 PromiseLink 项目中 AGPL v3 许可证的传染边界（copyleft boundary），定义基础版与专业版之间的法律隔离机制，确保：

1. 基础版（PromiseLink）的 AGPL v3 开源义务得到完整履行
2. 专业版（PromiseLink-Pro）的商业闭源模式不受 AGPL v3 传染
3. 用户、开发者、商业合作伙伴清晰理解各自的权利与义务

> ⚠️ **免责声明**：本文档为技术团队基于对 AGPL v3 的理解所做的初步评估，**不构成法律意见**。AGPL v3 的传染性边界在法律实践中存在争议，正式发布前必须由具备开源协议经验的执业律师出具书面法律意见。

## 2. 许可证架构

### 2.1 双版本许可体系

| 版本 | 仓库 | 许可证 | 传染性 | 商业模式 |
|------|------|--------|--------|----------|
| **基础版**（PromiseLink） | `lulin70/PromiseLink`（公开） | GNU AGPL v3 | 强 copyleft，修改后必须开源 | 开源免费 |
| **专业版**（PromiseLink-Pro） | `lulin70/PromiseLink-Pro`（私有） | 私有商业许可 | 不受 AGPL 约束 | 订阅付费 |
| **协议层**（promiselink-contracts） | 公开 | MIT License | 不传染 | 开源免费 |
| **工具层**（promiselink-utils） | 公开 | MIT License | 不传染 | 开源免费 |

### 2.2 物理隔离原则

基础版与专业版采用**物理隔离**（分仓库），而非单仓库多分支：

- 基础版仓库：包含完整的 13 步管线、实体提取、记忆、Todo、Promise、关联发现、Docker、H5、适配器
- 专业版仓库：包含网关、小程序、语音云端、图谱融合、定制版功能
- 两仓库之间通过 HTTP API 通信，无源码级依赖（非 submodule）

## 3. AGPL v3 传染边界定义

### 3.1 AGPL v3 核心传染规则

根据 GNU AGPL v3 协议（特别是第 13 条"远程网络交互"和第 5 条"修改后的源码"），传染性触发条件如下：

| 触发条件 | 是否传染 | 说明 |
|----------|----------|------|
| 修改 AGPL v3 源码并分发 | ✅ 传染 | 修改后的代码必须以 AGPL v3 开源 |
| 修改 AGPL v3 源码并通过网络提供服务 | ✅ 传染 | AGPL v3 第 13 条独有的"网络交互"条款，即使不分发也触发 |
| 静态链接 AGPL v3 代码 | ✅ 传染 | 静态链接构成"衍生作品"（derivative work） |
| 动态链接 AGPL v3 代码 | ⚠️ 争议 | FSF 认为传染，部分法律观点认为不传染，保守起见视为传染 |
| 独立进程通过管道/命令行调用 AGPL v3 程序 | ❌ 不传染 | 独立程序间通信不构成"衍生作品" |
| 通过网络 API 调用 AGPL v3 服务 | ❌ 不传染 | 网络通信不构成"链接" |
| 仅运行未修改的 AGPL v3 程序 | ❌ 不传染 | 未修改 + 未分发，不触发 copyleft |

### 3.2 FSF 对 AGPL v3 第 13 条的解释

根据自由软件基金会（FSF）对 AGPL v3 第 13 条"远程网络交互"（Remote Network Interaction）的官方解释：

> **第 13 条核心内容**：如果你修改了 AGPL v3 程序，并通过网络让用户与之交互，你必须向所有用户提供修改后完整源代码的获取途径。

**关键边界**：
- 第 13 条仅适用于**修改后的** AGPL v3 程序。未修改的程序通过网络提供服务**不触发**开源义务。
- 第 13 条的"网络交互"指用户与**修改后的程序本身**交互，而非通过 API 调用该程序。
- 独立程序通过 HTTP/HTTPS API 调用 AGPL v3 程序，两程序各自独立运行，**不构成衍生作品关系**。

**参考来源**：
- FSF 官方 FAQ: https://www.gnu.org/licenses/gpl-faq.html
- AGPL v3 全文: https://www.gnu.org/licenses/agpl-3.0.html
- FSF 关于 AGPL 与网络服务的解释: https://www.gnu.org/philosophy/network-services-freedom.html

### 3.3 PromiseLink 的传染边界判定

基于上述规则，PromiseLink 各集成场景的传染性判定：

| 集成场景 | 技术方式 | 是否触发传染 | 分析依据 |
|----------|----------|------------|----------|
| **专业版通过 HTTP API 调用基础版** | 网络通信，两进程独立 | ❌ 不传染 | 独立进程间通过 HTTP 通信，不构成"链接"，类似 Apache/nginx 与 GPL 模块的边界 |
| **专业版网关通过 WSS 调用 relay_client** | 网络通信 | ❌ 不传染 | relay_client（基础版公开子包）与网关（专业版私有）通过 WebSocket 通信，独立进程，不传染 |
| **用户修改基础版源码并自用** | 源码修改 | ✅ 传染 | 修改 AGPL v3 源码后通过网络提供服务，触发第 13 条，必须开源修改后的代码 |
| **用户修改基础版源码仅本地运行** | 源码修改 | ⚠️ 视情况 | 若不通过网络对外提供服务，不触发第 13 条；但若分发则触发第 5 条 |

## 4. 桥接接口不构成"衍生作品"的法律分析

### 4.1 API 桥接的隔离原理

PromiseLink 专业版通过 HTTP API 调用基础版，这种"桥接"（bridging）关系不构成"衍生作品"，理由如下：

1. **进程隔离**：基础版（FastAPI 进程）与专业版（网关进程）是两个独立的操作系统进程，各自有独立的内存空间、进程 ID、生命周期。

2. **通信方式**：两进程通过标准 HTTP/HTTPS 协议通信，与浏览器访问 Web 服务器的技术本质相同。FSF 明确表示，通过标准网络协议通信的程序之间不构成"链接"关系。

3. **类比先例**：此模式类似于：
   - Apache HTTP Server（Apache License）调用 GPL 许可的模块
   - PostgreSQL（PostgreSQL License）被 GPL 程序通过 libpq 调用
   - Linux 内核（GPL）运行专有用户态程序
   这些场景中，FSF 和开源社区均认可进程隔离 + 网络通信不触发 copyleft。

4. **契约式接口**：基础版对外暴露的 API 是稳定的契约式接口（见 `docs/design/API_Design_v1.md`），专业版依赖的是接口契约而非实现源码。接口契约本身（`promiselink-contracts`）采用 MIT License，不传染。

### 4.2 专业版通过 API 调用基础版不触发 AGPL 传染

```
┌─────────────────────────────────────────────────────────┐
│  专业版（PromiseLink-Pro，私有商业许可）                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │  网关进程     │  │  小程序后端   │  │  语音云端     │  │
│  │  (gateway/)  │  │  (miniapp/)  │  │  (asr/tts)   │  │
│  └──────┬───────┘  └──────┬───────┘  └──────────────┘  │
│         │ HTTP API         │ HTTP API                   │
└─────────┼──────────────────┼───────────────────────────┘
          │                  │
══════════╪══════════════════╪═══════════════════════════
          │ AGPL v3 传染边界 │  (网络通信，不传染)
══════════╪══════════════════╪═══════════════════════════
          │                  │
┌─────────▼──────────────────▼───────────────────────────┐
│  基础版（PromiseLink，AGPL v3）                          │
│  ┌──────────────────────────────────────────────────┐  │
│  │  FastAPI 进程（13步管线/实体/Todo/Promise/关联）   │  │
│  │  独立运行，对外暴露 HTTP API                       │  │
│  └──────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

**结论**：专业版通过 HTTP API 调用基础版，两进程独立运行，不构成"衍生作品"，不触发 AGPL v3 传染。专业版可保持闭源商业许可。

## 5. 用户使用基础版的开源义务

### 5.1 用户修改基础版后的义务

任何用户修改 PromiseLink 基础版源码后，必须遵守 AGPL v3 的以下义务：

| 使用方式 | 开源义务 | 适用条款 |
|----------|----------|----------|
| 修改后仅本地自用，不联网 | 无需开源 | AGPL v3 第 13 条不适用（无网络交互） |
| 修改后通过网络对外提供服务 | **必须开源**修改后的完整源代码 | AGPL v3 第 13 条（远程网络交互） |
| 修改后分发给他人 | **必须开源**修改后的完整源代码 | AGPL v3 第 5 条（分发修改版） |
| 未修改，仅运行 | 无需开源 | AGPL v3 不限制运行未修改的程序 |
| 未修改，通过网络对外提供服务 | 无需开源 | 第 13 条仅适用于"修改后的"程序 |

### 5.2 开源义务的具体要求

若用户修改基础版并通过网络提供服务，必须：

1. **提供完整源代码**：包括所有修改后的代码，以及与之链接的代码（适用范围内）。
2. **保留 AGPL v3 许可证**：修改后的代码必须继续以 AGPL v3 许可，不得更改许可证。
3. **提供获取途径**：向所有通过网络交互的用户提供源代码获取途径（如下载链接）。
4. **保留版权声明**：保留原始版权声明和作者信息。
5. **声明修改内容**：在源代码中注明修改日期和修改内容。

### 5.3 商业用户建议

- **如需闭源使用**：建议购买专业版（PromiseLink-Pro）商业许可，不受 AGPL v3 约束。
- **如需定制功能**：可通过专业版的插件机制/配置项实现定制，避免修改基础版源码。
- **如需修改基础版**：修改后的版本必须按 AGPL v3 开源，建议将修改贡献回上游（Pull Request），惠及社区。

## 6. 风险缓解措施

### 6.1 技术隔离措施

1. **物理分仓库**：基础版与专业版分属不同 Git 仓库，无 submodule 依赖。
2. **进程隔离**：基础版与专业版作为独立进程运行，通过 HTTP API 通信。
3. **接口契约隔离**：共享的协议层（`promiselink-contracts`）和工具层（`promiselink-utils`）采用 MIT License，不传染。
4. **不修改原则**：专业版通过 pip install 引入基础版，不修改任何基础版源码。

### 6.2 法律审查清单

正式发布前，必须完成以下法律审查（由执业律师出具书面意见）：

- [ ] HTTP API 调用 AGPL v3 服务是否完全不触发传染（确认进程隔离 + 网络通信的边界）
- [ ] AGPL v3 第 13 条"网络交互"条款对专业版云端部署的适用性
- [ ] WebView 嵌入基础版 H5 是否触发传染（针对小程序 WebView 方案）
- [ ] 商业 License 与 AGPL v3 的兼容性声明
- [ ] 用户修改基础版后通过 SaaS 提供服务的开源义务告知文案

### 6.3 备选方案

若法律审查认为当前方案存在传染风险，备选方案为：

1. **基础版 H5 改用 MPL 2.0**：弱 copyleft，文件级隔离，允许闭源文件与开源文件混合。
2. **专业版自建查询页面**：不嵌入基础版 H5，专业版小程序自建 UI，仅通过 API 获取数据。
3. **双许可模式**：基础版同时以 AGPL v3 + 商业双许可发布，付费用户可选择商业许可免除 AGPL 义务。

## 7. 结论

1. **基础版（PromiseLink）**：采用 AGPL v3，修改后通过网络提供服务必须开源。
2. **专业版（PromiseLink-Pro）**：采用私有商业许可，通过 HTTP API 调用基础版，不构成"衍生作品"，不受 AGPL v3 约束。
3. **桥接接口**：API 桥接（HTTP 通信）不触发 AGPL v3 传染，FSF 明确认可进程隔离 + 网络通信的隔离边界。
4. **用户义务**：修改基础版并通过网络提供服务的用户，必须按 AGPL v3 开源修改后的完整源代码。

> ⚠️ **再次声明**：本结论为技术团队的初步评估，正式发布前必须由具备开源协议经验的执业律师出具书面法律意见。本团队不对本文档的准确性承担法律责任。

```python
def recording_to_event(recording_data: dict) -> Event:
    """
    录音文件/转写文本 → Event 转换
    """
    # Step 1: 如果是原始录音，先调用转写API
    if recording_data.get("audio_url"):
        transcript = call_transcription_api(
            audio_url=recording_data["audio_url"],
            provider="whisper_or_iflytek"  # 可选讯飞/Whisper
        )
    else:
        transcript = recording_data.get("transcript", "")

    # Step 2: 创建 Event
    return Event(
        event_type="meeting",  # 或 "call"/"interview"
        title=f"{recording_data.get('title', '录音记录')}",
        timestamp=recording_data["recorded_at"],
        source="recording_hengzhiyi_r1",  # 标记来源设备
        raw_text=transcript,
        entities=[],  # 后续由 LLM 抽取
        metadata={
            "duration_seconds": recording_data.get("duration"),
            "participants": extract_speakers_from_transcript(transcript),
            "device_info": {
                "model": "R1",
                "serial": recording_data.get("device_serial")
            },
            "location": recording_data.get("location"),
            "audio_url": recording_data.get("audio_url")
        }
    )
```

**关键优化点**：
- **边录边传**：录音卡实时上传到云端，不占用手机存储
- **增量处理**：每5分钟或说话停顿 >3秒 时触发一次 Event 处理
- **离线缓存**：网络不好时本地缓存，恢复后批量上传

#### 第二层：Event 统一层（Event Ingestion Layer）

**核心职责**：将不同来源、不同格式的数据统一转换为标准 Event 对象

```python
class EventIngestionPipeline:
    """
    Event 统一处理管道
    """

    def __init__(self):
        self.sources = {
            "digital_card": DigitalCardProcessor(),
            "recording_r1": RecordingProcessor(),
            "meeting_tencent": MeetingProcessor(),
            "calendar_caldav": CalendarProcessor(),
            "manual_input": ManualInputProcessor()
        }
        self.normalizer = EventNormalizer()
        self.entity_extractor = EntityExtractor()  # LLM-based
        self.deduplicator = EventDeduplicator()

    async def ingest(self, raw_data: dict, source_type: str) -> Event:
        """
        统一入口：任何来源的数据都走这个管道
        """
        # Step 1: 来源特定预处理
        processor = self.sources[source_type]
        preprocessed = await processor.preprocess(raw_data)

        # Step 2: 格式标准化
        normalized = self.normalizer.normalize(preprocessed, source_type)

        # Step 3: 实体抽取（LLM or rule-based）
        if not normalized.entities:
            normalized.entities = await self.entity_extractor.extract(normalized.raw_text)

        # Step 4: 去重（同一事件不要重复处理）
        if self.deduplicator.is_duplicate(normalized):
            logger.info(f"Event duplicated, skipping: {normalized.title}")
            return None

        # Step 5: 存储并返回
        event = await self.persist(normalized)
        return event
```

**统一 Event Schema**：
```python
@dataclass
class StandardizedEvent:
    """所有来源的 Event 都必须符合这个 Schema"""

    # 基础字段
    id: str                           # evt_uuid
    event_type: str                   # meeting/call/chat/card_scan/schedule/manual
    title: str                        # 人类可读标题
    timestamp: datetime               # ISO 8601
    source: str                       # digital_card/recording_r1/meeting_tencent/calendar/manual
    raw_text: str                     # 原始内容（文本/转录结果/JSON字符串）

    # 实体信息（由引擎填充）
    entities: List[Entity]            # 抽取出的实体列表
    entity_count: int                 # 实体数量（用于性能监控）

    # 元数据
    metadata: dict                    # 扩展字段（来源特定）
    duration_minutes: Optional[int]   # 持续时间（如果是会议/通话）
    participants: List[str]          # 参与人列表
    location: Optional[str]           # 地点
    attachments: List[str]            # 附件URL列表

    # 引擎产出（由 EventLink 填充）
    associations: List[Association]   # 发现的关联
    alerts: List[Alert]               # 生成的提醒
    processing_status: str            # pending/processing/completed/failed
    created_at: datetime
    updated_at: datetime
```

#### 第三层：EventLink 核心引擎层（Core Engine Layer）

**这就是 CarryMem 的范围！** 不需要关心数据从哪里来，只负责：

1. **实体归一**："李总"和"李明"是不是同一个人？
2. **关联发现**：新 Event 和历史数据有什么交叉？
3. **提醒生成**：发现了什么有价值的信息要告诉用户？
4. **存储更新**：将新的实体和关联持久化

**关键优势：解耦！**
- 输入层可以无限扩展（新的硬件/软件/数据源）
- 引擎层保持稳定（专注算法和质量）
- 输出层灵活适配（小程序/App/Web/API）

## 🚀 落地路径：MVP 到完整产品的演进计划

### Phase 0.5：概念验证（Proof of Concept）（2-3周）

**目标**：验证"数字名片 + EventLink"的技术可行性

**具体动作**：

1. **许总提供 IAMHERE 小程序的 API 接口**（或测试账号）
2. **CarryMem 开发数字名片数据接入模块**
3. **用 10-20 张真实名片测试 EventLink 引擎**
4. **验证关联发现效果**（能否发现有趣的交叉？）

**交付物**：
- [ ] 数字名片 → Event 转换器（Demo）
- [ ] 关联发现的演示报告（5-10个案例）
- [ ] 技术可行性评估报告

**成功标准**：
- 能从名片数据中抽取出实体（人物/公司/技能）
- 能发现至少 2-3 个有价值的关联（如"这两个人的公司有竞对关系"）
- 处理延迟 <5秒/张名片

---

### Phase 1：MVP - 数字名片版（4-6周）

**目标**：让许总的小程序用户能用上 EventLink 的关联发现功能

**功能范围**：

✅ **必做**：
1. **数字名片扫描/查看事件自动采集**
   - 对接 IAMHERE 小程序 API
   - 每次查看/扫描名片自动创建 Event
2. **基础实体抽取**（规则引擎为主，LLM 为辅）
   - 从名片 JSON 直接解析（姓名/公司/职位/技能）
   - 无需调用外部 API（低成本）
3. **简单关联发现**（3-5种硬编码规则）
   - 同公司同事关系
   - 同行业潜在合作
   - 技能互补提示
4. **提醒展示**（小程序内嵌 H5 页面）
   - 每日摘要（最多 5 条）
   - 点击查看详情

❌ **不做**：
- 录音卡集成（Phase 2）
- 视频会议集成（Phase 2）
- 复杂的多跳推理（Phase 2）
- 团队共享（Phase 3）

**技术架构**：
```
IAMHERE 小程序
    ↓ (Webhook / API)
EventLink Cloud Service (FastAPI)
    ↓
SQLite / PostgreSQL
    ↓
定时任务：每小时跑一次关联发现
    ↓
推送提醒到小程序（模板消息 / 服务通知）
```

**许总需要做的**：
- 开发小程序内嵌页面（展示提醒列表）
- 配置 Webhook 回调（名片事件推送到 EventLink）
- 设计引导流程（让用户开启"智能关联"功能）

**CarryMem 需要做的**：
- 提供 EventLink API 服务（Docker 部署）
- 开发名片数据解析器
- 实现基础关联规则
- 编写 API 文档和集成指南

**里程碑**：
- Week 2: API 对通，能接收名片数据
- Week 4: 关联发现能跑通，有初步结果
- Week 6: 小程序内可以看到提醒，10个种子用户测试

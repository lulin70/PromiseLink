# PromiseLink 基础版开源风险警示与防护建议

> **文档性质**：项目组内部警示文件
> **生成日期**：2026-06-12
> **作者**：产品负责人（林总）授权生成
> **关联决策**：基础版采用 Open Core + AGPL v3 模式开源（2026-06-11 已确认）
> **前置讨论**：定价方案修订（pricing-tier-proposal-2026-06-11.md）、头脑风暴报告（brainstorm-ai-usecases-2026-06-11.md）

## 1. 核心立场

**开源基础版是正确决策**——隐私信任的不可辩驳证据，对占 45-55% 的谨慎型用户是致命一击，Clay/Dex/Monica 全闭源的竞品格局下是结构性差异化。

**但开源不等于无风险。** "相信没人会恶意利用"是产品负责人的失职。防患于未然，比事后追责成本低一个数量级。

---

## 2. 风险清单（按严重程度排序）

### 🔴 P0 — 商业克隆风险

| 风险 | 场景 | 可能性 | 影响 |
|------|------|--------|------|
| 有组织克隆 | 团队拿开源代码 + 自建网关 + 自建小程序 → 以更低价格卖 SaaS 服务 | 低 | 高 |
| 白标转售 | 第三方打包开源代码，改 Logo/品牌名，作为自己的产品出售 | 中 | 中 |
| 海外克隆 | 英语/日语市场有人 Fork 代码，本地化后抢占海外市场 | 中 | 高 |

**判断**：技术上可行（AGPL 代码 + 自建闭源组件可绕过），商业上极难盈利（品牌、信任、获客成本远高于 ¥29-49/月定价空间），但**不应仅靠"商业上不划算"作为防线**。

### 🟡 P1 — 代码滥用风险

| 风险 | 场景 | 可能性 | 影响 |
|------|------|--------|------|
| 去除品牌标识 | 删除 "Powered by PromiseLink"，假装自己原创 | 高 | 低 |
| 修改代码后闭源出售 | 违反 AGPL §5 但偷偷做 | 中 | 中 |
| 恶意植入 | 他人 Fork 后在代码中植入后门/采集工具，损害 PromiseLink 声誉 | 低 | 高 |

### 🟢 P2 — 生态碎片化风险

| 风险 | 场景 | 可能性 | 影响 |
|------|------|--------|------|
| 社区分裂 | 有人 Fork 后做重大架构改动，形成不兼容分支 | 中 | 低 |
| 插件生态分散 | 第三方适配器/插件提交到 Fork 版本而非主仓库 | 中 | 低 |
| 商标稀释 | "PromiseLink" 名字被用于非官方 Fork，用户在不明就里的情况下使用非官方版本 | 中 | 中 |

## 3. 已建立的结构性防护

以下防护已在 2026-06-11 的定价方案和开源策略中确认：

### 3.1 代码级防护

| 防护 | 机制 | 覆盖风险 |
|------|------|---------|
| **微信小程序完全闭源** | 小程序的 .wxml/.wxss 及业务逻辑不在开源仓库 | P0 商业克隆（缺少最关键的用户触点） |
| **中继网关完全闭源** | Token 控制、JWT 签发、推送调度均在闭源网关 | P0 商业克隆（自建网关需要反向工程） |
| **AGPL v3 License** | 网络服务提供者也必须开源修改（§13） | P0/P1（法律约束 + DMCA 武器） |
| **云端 AI Key 服务端代理** | 专业版 DeepSeek Key 从不暴露给客户端 | P0（自建方案需自行签约付费） |

### 3.2 资质级防护

| 防护 | 说明 |
|------|------|
| **微信小程序类目审核** | 企业主体 + 类目资质 + 内容安全审查 → 克隆者需独立过审 |
| **微信支付商户资质** | 企业主体 + 对公账户 → 个人/小团队门槛高 |
| **微信模板消息额度** | 推送通知需要微信平台审批，非简单代码实现 |

### 3.3 品牌级防护

| 防护 | 说明 |
|------|------|
| **22篇公众号文章** | "承诺账本"叙事 + "让每一次连接，都更有价值"品牌资产 → 克隆者无法复制 |
| **Product Hunt / HN 首发** | 开源时的首发声量 → 建立"正宗"认知 |
| **CarryMem 联动** | 姊妹项目交叉背书 → 生态护城河 |

## 4. 建议新增的防护措施

以下措施尚未实施，建议在开源前（M3-M4）落地：

### 4.1 代码级新增

| # | 措施 | 优先级 | 工作量 | 说明 |
|---|------|--------|--------|------|
| 1 | **AI 输出植入 "via PromiseLink" 标识** | 🔴 P0 | 小（1-2天） | 在 Todo 建议/关联发现/Promise 提示末尾自动附加品牌标识。删除标识=改代码=触发 AGPL taint。不改=免费品牌传播 |
| 2 | **CONFIG.md 中 hardcode 官方域名** | 🟡 P1 | 极小 | 默认配置指向 promiselink.im（或最终域名），Fork 者需显式修改才能指向自己的服务 |
| 3 | **API 响应中加入 `X-Powered-By: PromiseLink` 头** | 🟡 P1 | 极小 | HTTP 响应头植入，删除同样触发 AGPL |
| 4 | **启动日志打印 License 声明** | 🟢 P2 | 极小 | Docker 启动时打印 "PromiseLink Standard — AGPL v3. Commercial use requires compliance." |

### 4.2 法律/合规新增

| # | 措施 | 优先级 | 工作量 | 说明 |
|---|------|--------|--------|------|
| 5 | **注册 PromiseLink 商标** | 🔴 P0 | 中（商标流程） | 中国 + 美国商标注册。AGPL 管代码，商标管品牌。未授权使用 "PromiseLink" 商标的 Fork 可以直接商标侵权投诉 |
| 6 | **GitHub 仓库设置 CLA (Contributor License Agreement)** | 🟡 P1 | 小 | 确保社区贡献的代码版权归属于 PromiseLink，防止贡献者后续主张权利 |
| 7 | **准备 DMCA Takedown 模板** | 🟡 P1 | 小（1小时） | 预先写好发给 GitHub/Vercel/阿里云的侵权下架通知模板，遇到克隆时直接发出 |

### 4.3 监控/运营新增

| # | 措施 | 优先级 | 工作量 | 说明 |
|---|------|--------|--------|------|
| 8 | **GitHub Fork 监控** | 🟡 P1 | 中 | 定期扫描 PromiseLink 仓库的 Fork，标记有商业化迹象的 Fork（修改 README、添加定价、隐藏品牌标识）。可用 GitHub API 自动化 |
| 9 | **Google Alerts + 社交媒体监控** | 🟢 P2 | 小 | 设置 "PromiseLink" 关键词告警，监控是否有第三方以 PromiseLink 名义提供收费服务 |
| 10 | **社区行为准则 (CODE_OF_CONDUCT.md)** | 🟢 P2 | 极小 | 明确禁止恶意 Fork 和商标侵权，建立社区举报渠道 |

## 6. 团队共识

### 6.1 我们相信什么

- **开源的信任红利 > 闭源的安全幻觉。** 在隐私敏感场景下，"你可以看我的代码"比"你只能信我"更有力。
- **护城河不在代码里，在微信生态里。** 小程序、推送、支付——这三样比任何代码都难复制。
- **防患于未然，但不过度防御。** 10项防护措施中，P0 四项（品牌标识、商标注册）必须落地，其余按节奏推进。

### 6.2 我们警戒什么

- **有组织的商业化克隆**（团队 + 资金 → 自建全套）
- **海外市场抢跑**（英文/日语 Fork → 本地化 → 抢占海外首发声量）
- **品牌混淆**（第三方以 PromiseLink 名义提供服务，损害用户信任）

### 6.3 我们不担心什么

- **个人开发者自建自用** —— 他们是布道者，不是竞争者
- **小团队拿代码做内部工具** —— 他们是潜在专业版客户
- **合法 Fork 做非商业化改进** —— 这是开源生态的正常运转

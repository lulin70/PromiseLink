# PromiseLink 产品分层与定价方案

**日期**：2026-06-11
**类型**：定价策略方案（已讨论，部分决策已确认）
**状态**：待工程评估 + PoC 验证

---

## 📌 TL;DR

- 原方案按"AI功能额度"分层（基础200人/50次 → 专业全AI → 定制），用户容易感知为"砌墙"
- **新方案按"访问方式"分层**：本地部署（免费）→ 微信小程序中继（付费）→ 定制（按需），升级驱动力是真实场景而非功能阉割
- 定价建议：¥98/月 或 ¥888/年（年付约¥74/月），毛利率93%，AI成本仅占7%
- 核心逻辑：基础版建立信任（数据不出家门），专业版解决便利性（随时随地+推送+语音），天然形成转化漏斗
- **已决策 ✅**：专业版统一走云端AI + 基础版开源（AGPL v3 Open Core）

## 2. 为什么这个分层更好

### 2.1 付费驱动力是"真痛点"，不是"人造墙"

原方案按"200人/50次AI → 解锁更多"分层，本质是在功能上砌墙。用户心理是"你凭什么限制我"。

新方案是"本地能用，出门想用就付费"，用户心理是"确实需要，合理付费"。一个是阻力，一个是引力。

### 2.2 隐私叙事天然分层

| 版本 | 隐私叙事 | 用户感受 |
|------|----------|----------|
| 基础版 | "数据从不出你家门" | 安心，信任建立 |
| 专业版 | "网关只做中继，数据不过夜" | 可接受，已建立信任 |
| 定制版 | "按需部署，数据位置你定" | 大客户放心 |

之前担心"怎么让用户信任把数据放进来"。基础版天然解决——**先让用户在本地用起来，信任是体验出来的，不是承诺出来的**。

### 2.3 升级路径自然

基础版用户用着用着，想在外面用手机查看联系人的承诺 → 自然升级专业版。不是硬造需求，是真实场景驱动。

### 2.4 基础版不是残废版

核心AI能力（实体抽取/关联发现/Todo生成/承诺追踪）三版完全一样，只是访问方式不同。用户不会觉得被阉割。

## 3. 定价分析

### 3.1 成本结构

| 成本项 | 性质 | 月成本 | 说明 |
|--------|------|--------|------|

| 存储+带宽 | 固定 | ¥10-20 | 中继日志、用户配置 |
| **总固定成本** | | **¥80-150** | |

### 3.2 AI成本明细

DeepSeek API 约 ¥1/1M tokens（V3约¥2/1M）。商务用户场景估算：

| 用户状态 | 日交互次数 | 月Token消耗 | 月AI成本 |
|----------|-----------|------------|----------|
| 轻度 | 5次 | 0.75M | ~¥1 |
| 中度 | 10次 | 1.5M | ~¥2-3 |
| 重度 | 20次 | 3M | ~¥3-6 |

**AI成本在¥98定价中占比仅~5%，几乎可以忽略。** 真正吃利润的是服务器固定成本，但分摊到百人级别每人仅¥1。

### 3.3 不同用户量盈利模型

| 用户数 | 月收入 | 固定成本 | AI变量成本 | 毛利 | 毛利率 |
|--------|--------|----------|-----------|------|--------|
| 10人 | ¥980 | ¥100 | ¥30 | ¥850 | 87% |

> **关键发现**：因为AI成本极低（DeepSeek），利润率始终>85%。定价空间非常宽松，可以放开了做年付折扣、促销活动。

## 4. 技术方案：Token控制 + 专业版认证

### 4.1 架构

```
小程序 → 中继网关 → 转发业务请求到本地Docker
                    → 本地Docker标记"AI调用走网关"
                    → 网关代理AI API调用（用我方Key）
                    → 网关计数 user_id → 本月剩余Token
                    → 超限拒绝，降级提示
```

### 4.2 为什么必须走网关

- **本地Docker计数不可信**：用户可以改代码绕过
- **方案A：网关代理AI调用（推荐）**：小程序不直调AI，所有AI请求经网关，精准可控
- **方案C（备选）：控交互次数，不控Token**：按每月N次核心操作计费，简单但粒度粗

### 4.3 基础版 vs 专业版 AI调用路径

```
基础版：
  浏览器 → 本地Docker → 本地Ollama/自带API Key → 返回

专业版：
  小程序 → 中继网关 → 本地Docker（业务逻辑）→ 标注"AI调用"→ 网关代理DeepSeek → 返回
                                                          ↑ 计数、限流在这里
```

### 4.4 专业版用户 Docker 侧的认证方案（已决策 ✅）

**决策（2026-06-11）**：采用方案A — 网关下发JWT，Docker缓存，专业版用户无论浏览器还是小程序均走云端AI。

**认证流程**：

```
1. 用户在小程序完成付费 → 网关记录 user_id = 专业版
2. 用户回家，浏览器打开本地Docker → Docker提示"登录同步"
3. 用户点击 → Docker向中继网关请求 /auth/verify
4. 网关返回 JWT（含专业版标识 + 过期时间，建议24h）
5. Docker缓存JWT → 后续AI调用走网关代理（用我方Key）
6. JWT过期前1h自动刷新，基本无感
```

**体验一致性保证**：

| 场景 | 访问方式 | LLM来源 | 说明 |
|------|----------|----------|------|
| 纯基础版（未付费） | 浏览器局域网 | 本地Ollama / 自带Key | 完全离线 |
| 已购专业版 + 在家用浏览器 | 浏览器局域网 | **云端AI（我方Key）** | 身份已激活，体验一致 |
| 已购专业版 + 出门用小程序 | 微信小程序 | 云端AI（我方Key） | 正常路径 |

> **关键决策**：专业版用户回家用浏览器大屏，AI仍然走云端模型，体验一致。不允许"出门用好的，回家用差的"。

**隐私声明措辞（需在PoC阶段定稿）**：
> "联网仅用于验证您是否为付费用户，不传输任何关系数据。验证令牌缓存在本地，过期前自动刷新。"

### 4.5 专业版一个待解技术问题

小程序用户请求走中继网关到本地Docker后，Docker处理业务逻辑时可能要调AI。此时Docker需要知道"当前是专业版模式，AI走网关不要走本地Ollama"。需要Docker侧做模式切换支持。

## 5. 开源策略（已决策 ✅）

### 5.1 决策结论（2026-06-11）

**采用 Open Core 模式（AGPL v3 License）**：
- ✅ 基础版（PromiseLink-Standard）开源
- 🔒 专业版中继网关、微信小程序、定制版全栈闭源
- 📜 License：AGPL v3（防止他人拿开源代码做 SaaS 竞品，用户在隐私场景下信任度更高）

### 5.2 开源范围界定

```
开源（PromiseLink-Standard）：
  ✅ 13步事件处理管线（完整业务逻辑）
  ✅ 实体提取 + 归一算法
  ✅ 所有记忆类型（7种）
  ✅ 6种Todo生成（含Promise双向）
  ✅ 关联发现12种（含供需匹配supply_demand）
  ✅ 完整的本地Docker部署脚本
  ✅ Taro H5 前端（浏览器访问）
  ✅ Obsidian/飞书/企微 适配器
  ✅ 语义搜索、动态优先级评分
  ✅ 关系阶段识别（7阶段模型）

闭源（PromiseLink Pro / 定制版）：
  🔒 中继网关源码（Token控制、推送调度）
  🔒 微信小程序前端（.wxml/.wxss 混淆）
  🔒 语音 NLU/NLG/TTS 云端模型调用封装
  🔒 关系图谱融合算法（定制版专属）
  🔒 云端管理后台（订阅管理、用量仪表盘）
```

### 5.3 为什么开源是基础版的正确决策

| 理由 | 说明 |
|------|------|
| **隐私信任 = 开源证据** | "数据从不出家门"闭源是承诺，开源是可验证的事实。对占45-55%的谨慎型用户是致命一击 |
| **获客杠杆** | 技术用户自己部署、自己验证，写技术博客推荐，把用户变成传播节点 |
| **竞品无人在做** | Clay/Dex/Monica 全闭源 SaaS。开源基础版在"隐私"最敏感维度有结构性优势 |
| **AGPL v3 防竞争** | 云服务提供者修改代码也必须开源。防止别人拿 PromiseLink-Standard 做 SaaS 跟我们竞争 |
| **基础版本来不赚钱** | 它的使命是获客，开源加速这个循环，不损伤营收 |

### 5.4 开源节奏

```
M1-M2（上线期）：
  基础版Docker先私有分发（TestFlight/内测群）
  收集早期用户反馈，打磨 `docker-compose up` 真的"一键"
  确保离线部署文档完整（README + 3分钟视频）

M3-M4（优化期）：
  正式开源 PromiseLink-Standard（GitHub org: promiselink-oss）
  同步发 Product Hunt / Hacker News
  配合"数据从不出家门"叙事做一波传播
  预留 CONTRIBUTING.md + 插件开发文档

M5-M6（成熟期）：
  社区PR合并，插件生态起步
  定制版按需开放，转介绍机制，行业解决方案
```
